在目标周围构建开发框架。`create_deep_agent` 为您提供一个生产级的基础框架：将其连接到您的数据，调整其行为，并添加您的用例所需的功能。

:::python

:::

```python
from deepagents import create_deep_agent

agent = create_deep_agent(
    model="anthropic:claude-sonnet-4-6",
    system_prompt="You are a helpful assistant.",
    tools=[search, fetch_url],
    memory=["./AGENTS.md"],
    skills=["./skills/"],
)
```

| 参数 | 功能说明 |
|---|---|
| [`model=`](#model) | 指定使用的模型 |
| [`system_prompt=`](#system-prompt) | 代理的自定义指令 |
| [`tools=`](#tools) | 代理可调用的领域工具 |
| [`memory=`](#memory) | 启动时加载的 AGENTS.md 文件 |
| [`skills=`](#skills) | 按需获取知识的技能目录 |
| [`backend=`](#backends) | 文件系统后端（默认为 StateBackend）|
| [`permissions=`](/oss/deepagents/permissions) | 文件系统的路径级访问控制 |
| [`subagents=`](#subagents) | 用于委派任务的自定义子代理 |
| [`middleware=`](#middleware) | 追加到[默认堆栈](#default-stack-main-agent)的额外中间件 |
| [`interrupt_on=`](#human-in-the-loop) | 工具调用前暂停以等待人工批准 |
| [`response_format=`](#structured-output) | 结构化输出模式 |
| [`state_schema=`](/oss/deepagents/context-engineering#custom-state-schema) | 自定义图状态模式 |
| [`context_schema=`](/oss/deepagents/context-engineering#runtime-context) | 每次运行的运行时上下文模式（用户 ID、API 密钥、功能开关） |
| [profiles](#profiles) | 作为可复用包使用的每模型默认值 |

:::python
安装 `langchain-mcp-adapters` 以连接到 MCP 服务器：


:::

```bash
pip install langchain-mcp-adapters
```

```python
import asyncio
from langchain_mcp_adapters.client import MultiServerMCPClient
from deepagents import create_deep_agent

async def main():
    async with MultiServerMCPClient(
        {
            "my_server": {
                "transport": "http",
                "url": "http://localhost:8000/mcp",
            }
        }
    ) as client:
        tools = await client.get_tools()

        agent = create_deep_agent(
            model="openai:gpt-5.5",
            tools=tools,
        )

        result = await agent.ainvoke(
            {"messages": [{"role": "user", "content": "Use the MCP server to help me."}]},
            config={"configurable": {"thread_id": "1"}},
        )

asyncio.run(main())
```

## 系统提示词

Deep Agents 配备内置的系统提示词。Deep Agent 的价值来自 SDK 在模型之上提供的编排层——规划、虚拟文件系统工具和子代理——模型需要知道这些组件的存在以及何时使用它们。内置提示词教会代理如何使用这些脚手架，因此您无需为每个项目重新推导它；通过 [profile](/oss/deepagents/profiles#harness-profiles) 或您自己的 `system_prompt=` 来调整它，而不是逐字复制。

当中间件添加特殊工具（如文件系统工具）时，它会将这些工具追加到系统提示词中。

每个 deep agent 还应包含一个特定于其用例的自定义系统提示词：

### 提示词组装

Deep Agents 从最多四个命名部分构建系统提示词，以便调用者提供的指令、SDK 的内置代理指导以及任何特定于模型的 [profile](/oss/deepagents/profiles) 覆盖都可以与可预测的优先级共存。如果没有这种分层，针对 Claude 调整的 profile 后缀（例如）可能会根据调用顺序覆盖或被您的 `system_prompt=` 参数覆盖；命名槽使排序变得明确和稳定。

在实践中，大多数调用者只遇到两个槽：`USER`（您的 `system_prompt=`）和 `BASE`（SDK 默认值）。选择具有内置 profile 的模型——目前是 Anthropic 或 OpenAI——会添加 `SUFFIX`。完整的四部分组装主要与您编写自定义 `HarnessProfile` 或调试为什么 profile 的文本出现在某处相关。

四个命名部分（每个都可能不存在）：

| 名称     | 来源                                                  | 备注                                                   |
| -------- | ----------------------------------------------------- | ----------------------------------------------------- |
| `USER`   | `create_deep_agent` 的 `system_prompt=` 参数         | `str` 或 `SystemMessage`；未设置时省略。              |
| `BASE`   | SDK 默认值（`BASE_AGENT_PROMPT`）                    | 除非被 profile 的 `CUSTOM` 替换，否则始终存在。       |
| `CUSTOM` | [`HarnessProfile.base_system_prompt`](/oss/deepagents/profiles#harness-profiles) | 当匹配的 profile 设置后，完全替换 `BASE`。           |
| `SUFFIX` | [`HarnessProfile.system_prompt_suffix`](/oss/deepagents/profiles#harness-profiles) | 当匹配的 profile 设置后，追加到最后。                |

顺序始终是 **`USER` -> (`BASE` 或 `CUSTOM`) -> `SUFFIX`**，用空行连接（`\n\n`）。两个不变式如下：

1. **`USER` 始终在最前面。** 调用者的文本优先于任何 SDK 或 profile 内容，因此角色/指令优先，不受所选模型的影响。
2. **`SUFFIX` 始终在最后面。** Profile 后缀最接近对话历史，模型调优指导最能可靠地发挥作用。

组装形状（✓ = 字段已设置，- = 字段未设置）：

| `system_prompt=` | profile `base_system_prompt`（`CUSTOM`） | profile `system_prompt_suffix`（`SUFFIX`） | 最终组装的系统提示词 |
| ---------------- | :--------------------------------------: | :---------------------------------------: | ------------------- |
| `None`           |                    -                     |                     -                     | `BASE`              |
| `None`           |                    -                     |                     ✓                     | `BASE` + `SUFFIX`   |
| `None`           |                    ✓                     |                     -                     | `CUSTOM`            |
| `None`           |                    ✓                     |                     ✓                     | `CUSTOM` + `SUFFIX` |
| `str`            |                    -                     |                     -                     | `USER` + `BASE`     |
| `str`            |                    -                     |                     ✓                     | `USER` + `BASE` + `SUFFIX` |
| `str`            |                    ✓                     |                     -                     | `USER` + `CUSTOM`   |
| `str`            |                    ✓                     |                     ✓                     | `USER` + `CUSTOM` + `SUFFIX` |

工作示例——内置 profile（Anthropic、OpenAI）仅提供 `system_prompt_suffix`，因此典型的调用落在 `str` + `-` + `✓` 行：

```python
agent = create_deep_agent(
    model="anthropic:claude-sonnet-4-6",
    system_prompt="You are a customer-support agent for ACME Corp.",
)
# Final = USER + BASE + SUFFIX
#       = "You are a customer-support agent for ACME Corp."
#         + "\n\n"
#         + BASE_AGENT_PROMPT
#         + "\n\n"
#         + <Claude-specific guidance>
```

<Note>
    传递一个 `SystemMessage`（而非字符串）会触发不同的拼接路径：右侧组件（`BASE` 或 `CUSTOM` 加上任何 `SUFFIX`）作为额外的文本内容块追加到消息现有的 `content_blocks` 上。相同的逻辑顺序适用（调用者块优先），且调用者块上的任何 `cache_control` 标记都会被保留——这对于放置显式的 Anthropic prompt-cache 断点很有用。
</Note>

<AccordionGroup>
    <Accordion title="子代理提示词">
        [提示词组装](#prompt-assembly) 覆盖规则同样适用于声明式 [子代理](/oss/deepagents/subagents)：每个子代理会针对**其自身的模型**重新运行配置解析，然后将已解析配置的 `base_system_prompt` / `system_prompt_suffix` 应用到其编写的 `system_prompt` 上。子代理的 `system_prompt` 扮演 `BASE` 角色；`CUSTOM` 和 `SUFFIX` 来自匹配子代理模型的配置（这可能与主代理的配置不同）。

        | `spec["system_prompt"]` | 配置 `base_system_prompt`（`CUSTOM`） | 配置 `system_prompt_suffix`（`SUFFIX`） | 最终子代理系统提示词 |
        | ----------------------- | :-------------------------------------: | :---------------------------------------: | ---------------------------- |
        | 已编写                  |                    -                    |                     -                     | 已编写                       |
        | 已编写                  |                    -                    |                     ✓                     | 已编写 + `SUFFIX`            |
        | 已编写                  |                    ✓                    |                     -                     | `CUSTOM`                     |
        | 已编写                  |                    ✓                    |                     ✓                     | `CUSTOM` + `SUFFIX`          |

        子代理没有 `USER` 段。规范中编写的 `system_prompt` 是最接近的类比，保留在 `BASE` 槽中。只提供 `system_prompt_suffix` 的配置（内置 Anthropic / OpenAI 配置的常见情况）会追加到子代理作者编写的任何内容之后。而设置 `base_system_prompt` 的配置则会直接替换编写的提示词。
    </Accordion>
    <Accordion title="通用子代理提示词">
        自动添加的[通用子代理](/oss/deepagents/subagents#the-general-purpose-subagent)遵循[提示词组装](#prompt-assembly)覆盖规则，但多了一层处理：GP 基础提示词按以下顺序解析：**`general_purpose_subagent.system_prompt`（如果已设置）-> `HarnessProfile.base_system_prompt`（如果已设置）-> SDK 通用默认值**。配置后缀无论如何都会叠加在上面。

        这两个覆盖字段都可以携带基础提示词替换，但不可互换。`general_purpose_subagent.system_prompt` 是通用代理的特定配置；`base_system_prompt` 是全局覆盖，主要针对主代理。当两者都设置时，**通用代理的特定意图对通用代理获胜**，这样用户在同时调优这两个字段时永远不会看到其 GP 覆盖被静默丢弃：


</Accordion>

        ```python
        register_harness_profile(
            "anthropic",
            HarnessProfile(
                base_system_prompt="You are ACME's support orchestrator.",  # main agent
                general_purpose_subagent=GeneralPurposeSubagentProfile(
                    system_prompt="You are a research subagent. Cite sources.",  # GP subagent
                ),
                system_prompt_suffix="Always think step by step.",
            ),
        )
        ```

<fragment>

        | Stack | Final system prompt |
        | ----- | ------------------- |
        | Main agent | `"You are ACME's support orchestrator." + SUFFIX` |
        | GP subagent | `"You are a research subagent. Cite sources." + SUFFIX` |

        如果 `general_purpose_subagent.system_prompt` 未设置，GP 子代理会回退到 `base_system_prompt`（如果已设置），最终回退到 SDK 的通用默认值。
    </Accordion>
</AccordionGroup>

## 中间件

Deep Agents 支持任何[中间件](/oss/langchain/middleware/overview)，包括下面列出的内置中间件、来自 LangChain 的预构建中间件、提供商特定中间件以及您自己编写的自定义中间件。

:::python
将中间件传递给 `create_deep_agent` 的 `middleware` 参数。自定义中间件在[默认堆栈](#default-stack-main-agent)中的 @[`PatchToolCallsMiddleware`] 之后追加。
:::

:::js
将中间件传递给 `createDeepAgent` 的 `middleware` 参数。自定义中间件在[默认堆栈](#default-stack-main-agent)中的 @[`PatchToolCallsMiddleware`] 之后追加。
:::

默认情况下，Deep Agents 可以访问以下中间件：

### 默认堆栈（主代理）

从第一个到最后一个：

:::python

1. @[`TodoListMiddleware`]：跟踪和管理待办事项列表，用于组织代理任务和工作。
2. @[`SkillsMiddleware`]：仅当您传递 `skills` 时。在待办事项中间件**之后**和**文件系统中间件之前**注入，以便文件工具运行前技能元数据可用。
3. @[`FilesystemMiddleware`]：处理文件系统操作，如读取、写入和导航目录。当您传递 `permissions` 时，文件系统权限强制在此处包含，以便它可以评估代理可能调用的每个工具。
4. @[`SubAgentMiddleware`]：生成并协调子代理，用于将任务委托给专业代理。
5. @[`SummarizationMiddleware`]：压缩消息历史以在对话变长时保持在上下文限制内（通过 @[create_summarization_middleware]）。
6. @[`PatchToolCallsMiddleware`]：修复消息历史中悬空的工具调用，当运行在中断后恢复或收到格式错误的工具调用参数时。在 Anthropic 提示缓存和下面的尾部堆栈**之前**运行。
7. @[`AsyncSubAgentMiddleware`]：仅当您配置异步子代理时。
8. **您的中间件参数**：作为 `middleware` 参数传递的可选中间件在此处追加（在 Patch 之后，尾部堆栈之前）。
9. **Harness profile extras**：来自解析模型 profile 的提供商特定中间件（如果有）。
10. **排除工具过滤**：当 harness profile 列出排除的工具时，中间件从代理中移除这些工具。
11. @[`AnthropicPromptCachingMiddleware`]：始终注册；在非 Anthropic 模型上无操作（`unsupported_model_behavior="ignore"`）。在 Patch 和您的中间件**之后**运行，以便缓存前缀与实际发送给模型的内容匹配。
12. @[`MemoryMiddleware`]：仅当您传递 `memory` 时。

    <Note>

6. @[`PatchToolCallsMiddleware`]：修复消息历史中悬空的工具调用，当运行在中断后恢复或收到格式错误的工具调用参数时。在 Anthropic 提示缓存和下面的尾部堆栈**之前**运行。

9. **Harness profile extras**：来自解析模型 profile 的提供商特定中间件（如果有）。
10. **排除工具过滤**：当 harness profile 列出排除的工具时，中间件从代理中移除这些工具。
11. @[`AnthropicPromptCachingMiddleware`]：当您使用 Anthropic 模型时自动添加。在 Patch 和您的中间件**之后**运行，以便缓存前缀与实际发送给模型的内容匹配。

## Profile

[Harness profile](/oss/deepagents/profiles#harness-profiles) 是可重用的按模型配置包，`create_deep_agent` 在选择匹配模型时自动应用。当您需要随模型（而非调用站点）变化的行为时，Profile 是正确的工具，例如针对 Claude 指令风格调优的系统提示后缀、为 GPT 重写的工具描述，或仅在特定提供商上有意义的额外中间件。

单个 profile 可以包含：自定义基础系统提示（`base_system_prompt`）、追加的后缀（`system_prompt_suffix`）、工具描述覆盖、要排除的工具或中间件、要注入的额外中间件，以及对自动添加的通用子代理的编辑。
