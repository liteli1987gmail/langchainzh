title: 记忆
description: 为使用 Deep Agents 构建的智能体添加持久化记忆，使其能够在对话中学习和改进
---

记忆让你的智能体能够在对话中学习和改进。Deep Agents 将记忆作为一等公民，提供文件系统支持的记忆功能：智能体像读写文件一样读写记忆，而你可以通过[后端](/oss/deepagents/backends)来控制这些文件的存储位置。

<Note>
本页面介绍**长期记忆**：在对话之间持续存在的记忆。关于短期记忆（单次会话中的对话历史和临时文件），请参阅[上下文工程](/oss/deepagents/context-engineering)指南。短期记忆由智能体的[状态](/oss/langgraph/graph-api#state)自动管理。

![短期记忆通过检查点限制在单个线程中；长期记忆通过存储在多个线程间持续存在](/oss/images/deepagents/production/memory.png)
</Note>

## 记忆的工作原理

1. **将智能体指向记忆文件。** 创建智能体时通过 `memory=` 传递文件路径。你还可以通过 `skills=` 传递[技能](/oss/deepagents/skills)以实现过程记忆（可重用的指令，告诉智能体*如何*执行任务）。[后端](/oss/deepagents/backends)控制文件的存储位置和访问权限。
2. **智能体读取记忆。** 智能体可以在启动时将记忆文件加载到系统提示词中，也可以在对话中按需读取。例如，[技能](/oss/deepagents/skills)使用按需加载：智能体在启动时只读取技能描述，然后在匹配任务时才读取完整的技能文件。这使上下文保持精简，直到需要某个能力时才加载。
3. **智能体更新记忆（可选）。** 当智能体学到新信息时，它可以使用内置的 `edit_file` 工具来更新记忆文件。更新可以在对话期间进行（默认），也可以通过[后台整合](#后台整合)在对话之间后台进行。更改会被持久化，并在下一次对话中可用。并非所有记忆都可写：开发者定义的[技能](/oss/deepagents/skills)和[组织策略](#组织级记忆)通常是只读的。详见[只读与可写记忆](#只读与可写记忆)。

两种最常见的模式是[智能体范围记忆](#智能体范围记忆)（在所有用户之间共享）和[用户范围记忆](#用户范围记忆)（按用户隔离）。

## 范围记忆

智能体记忆可以设置范围，使相同的记忆文件可被所有使用该智能体的用户访问，或者记忆文件可以是每个用户独有的。

### 智能体范围记忆

赋予智能体一个随时间演变的持久身份。智能体范围记忆在所有用户之间共享，因此智能体通过每次对话积累自己的角色特征、累积知识和学习偏好。当它与用户互动时，会发展专业知识、改进方法，并记住有效的做法。当拥有写权限时，它还可以学习和更新[技能](/oss/deepagents/skills)。

关键在于后端命名空间：将其设置为 `(assistant_id,)` 意味着此智能体的每次对话都读写同一个记忆文件。

:::python
<Note>
访问 `rt.server_info` 需要 `deepagents>=0.5.0`。在旧版本中，请改为从 `get_config()["metadata"]["assistant_id"]` 读取助手 ID。
</Note>
:::

:::js
<Note>
访问 `rt.serverInfo` 需要 `deepagents>=1.9.0`。在旧版本中，请改为从 `getConfig().metadata.assistantId` 读取助手 ID。
</Note>
:::

### 用户作用域内存

为每个用户提供独立的内存文件。代理会记住每个用户的偏好、上下文和历史记录，而核心代理指令保持不变。如果存储在用户作用域的后端中，用户也可以拥有各自的[技能](/oss/deepagents/skills)。

命名空间使用`(user_id,)`，因此每个用户获得内存文件的独立副本。用户 A 的偏好永远不会泄露到用户 B 的对话中。

:::python
```python
from deepagents import create_deep_agent
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

agent = create_deep_agent(
    model="google_genai:gemini-3.5-flash",
    memory=["/memories/preferences.md"],
    skills=["/skills/"],
    backend=CompositeBackend(
        default=StateBackend(),
        routes={
            "/memories/": StoreBackend(
                namespace=lambda rt: (rt.server_info.user.identity,),
            ),
            "/skills/": StoreBackend(
                namespace=lambda rt: (rt.server_info.user.identity,),
            ),
        },
    ),
)
```
:::

:::js
```typescript
import { createDeepAgent, CompositeBackend, StateBackend, StoreBackend } from "deepagents";
```

## 高级用法

除了内存路径和作用域的基本配置选项外，您还可以为内存配置更高级的参数：

| 维度 | 问题 | 选项 |
|---|---|---|
| **持续时间** | 持续多长时间？ | [短期](/oss/deepagents/context-engineering)（单次对话）或[长期](#scoped-memory)（跨对话） |
| **信息类型** | 它是什么类型的信息？ | [情景记忆](#episodic-memory)（过去的经验）、[程序记忆](/oss/deepagents/skills)（指令和技能）或[语义记忆](/oss/concepts/memory#semantic-memory)（事实） |
| **作用域** | 谁可以查看和修改它？ | [用户](#user-scoped-memory)、[代理](#agent-scoped-memory)或[组织](#organization-level-memory) |
| **更新策略** | 何时写入记忆？ | 对话期间（默认）或[对话之间](#background-consolidation) |
| **检索** | 如何读取记忆？ | 加载到提示词中（默认）或按需（例如[技能](/oss/deepagents/skills)） |
| **代理权限** | 代理可以写入内存吗？ | [读写](#read-only-vs-writable-memory)（默认）或[只读](#read-only-vs-writable-memory)（用于共享策略） |

### 情景记忆

情景记忆存储过去经验的记录：发生了什么、按什么顺序发生、以及结果如何。与语义记忆（存储在 `AGENTS.md` 等文件中的事实和偏好）不同，情景记忆保留了完整的对话上下文，使代理能够回忆问题是如何解决的，而不仅仅是学到了什么。

Deep Agents 已经使用[检查点器](/oss/langgraph/checkpointers#checkpoints)，这是支持情景记忆的机制：每次对话都作为检查点化的线程被持久化。

为了让过去的对话可搜索，请将线程搜索包装在工具中。`user_id` 从运行时上下文中获取，而不是作为参数传递：

:::python
```python
from langgraph_sdk import get_client
from langchain.tools import tool, ToolRuntime

client = get_client(url="<DEPLOYMENT_URL>")


@tool
async def search_past_conversations(query: str, runtime: ToolRuntime) -> str:
    """Search past conversations for relevant context."""
    user_id = runtime.server_info.user.identity  # [!code highlight]
    threads = await client.threads.search(
        metadata={"user_id": user_id},
        limit=5,
    )
    results = []
    for thread in threads:
        history = await client.threads.get_history(thread_id=thread["thread_id"])
        results.append(history)
    return str(results)
```
:::

:::js
```typescript
import { Client } from "@langchain/langgraph-sdk";
import { tool } from "@langchain/core/tools";

const client = new Client({ apiUrl: "<DEPLOYMENT_URL>" });

const searchPastConversations = tool(
  async ({ query }, runtime) => {
    const userId = runtime.serverInfo.user.identity;  // [!code highlight]
    const threads = await client.threads.search({
      metadata: { userId },
      limit: 5,
    });
    const results = [];
    for (const thread of threads) {
      const history = await client.threads.getHistory(thread.threadId);
      results.push(history);
    }
    return JSON.stringify(results);
  },
  {
    name: "search_past_conversations",
    description: "Search past conversations for relevant context.",
  }
);
```
您可以通过调整元数据过滤器来按用户或组织限定线程搜索范围：

:::python

```python
    metadata={"user_id": user_id},
    limit=5,
)

# Search conversations across an organization

    metadata={"org_id": org_id},
    limit=5,
)
```
:::

### 组织级内存

组织级内存遵循与用户级内存相同的模式，但使用的是组织级命名空间而非用户级命名空间。将其用于应在组织内所有用户和智能体中应用的策略或知识。

组织内存通常是 **只读** 的，以防止通过共享状态进行提示注入。详见[只读与可写内存](#read-only-vs-writable-memory)。

### 后台整合

默认情况下，代理在对话期间写入记忆（热路径）。另一种方式是将记忆作为后台任务在**对话之间**处理，有时称为**休眠计算**。一个单独的深度代理会审查最近的对话，提取关键事实，并将其与现有记忆合并。

| 方式 | 优点 | 缺点 |
|---|---|---|
| **热路径**（对话期间）| 记忆立即可用，对用户透明 | 增加延迟，代理需要处理多个任务 |
| **后台**（对话之间）| 无用户面延迟，可以跨多个对话综合 | 记忆直到下次对话才可用，需要第二个代理 |

对于大多数应用，热路径已足够。当您需要减少延迟或提高跨多个对话的记忆质量时，添加后台整合。

推荐模式是部署一个**整合代理**与您的主代理并行——一个读取最近对话历史、提取关键事实并将其合并到记忆存储中的深度代理——并通过 [cron 计划](#cron) 触发它。选择一个反映用户实际与代理交互频率的节奏：每日流量稳定的聊天产品可能每隔几小时整合一次，而每周只使用几次的工具只需要每天或每周运行一次。整合频率比用户对话频率高出太多只会浪费令牌在无操作运行上。

#### 整合代理

整合代理读取最近的对话历史并合并关键事实到记忆存储中。在 `langgraph.json` 中将其与您的主代理一起注册：

:::python
```python consolidation_agent.py
from datetime import datetime, timedelta, timezone

from deepagents import create_deep_agent
from langchain.tools import tool, ToolRuntime
from langgraph_sdk import get_client

sdk_client = get_client(url="<DEPLOYMENT_URL>")


@tool
async def search_recent_conversations(query: str, runtime: ToolRuntime) -> str:
    """Search this user's conversations updated in the last 6 hours."""
    user_id = runtime.server_info.user.identity  # [!code highlight]

    since = datetime.now(timezone.utc) - timedelta(hours=6)
    threads = await sdk_client.threads.search(
        metadata={"user_id": user_id},
        updated_after=since.isoformat(),
        limit=20,
    )
    conversations = []
    for thread in threads:
        history = await sdk_client.threads.get_history(
            thread_id=thread["thread_id"]
        )
        conversations.append(history["values"]["messages"])
    return str(conversations)


agent = create_deep_agent(
    model="google_genai:gemini-3.5-flash",
    system_prompt="""Review recent conversations and update the user's memory file.
Merge new facts, remove outdated information, and keep it concise.""",
    tools=[search_recent_conversations],
)
```
:::

:::js
```typescript src/consolidation-agent.ts
import { createDeepAgent } from "deepagents";
import { Client } from "@langchain/langgraph-sdk";
import { tool } from "@langchain/core/tools";

const sdkClient = new Client({ apiUrl: "<DEPLOYMENT_URL>" });

const searchRecentConversations = tool(
  async ({ query }, runtime) => {
    const userId = runtime.serverInfo.user.identity;  // [!code highlight]

    const since = new Date(Date.now() - 6 * 60 * 60 * 1000).toISOString();
    const threads = await sdkClient.threads.search({
      metadata: { userId },
      updatedAfter: since,
      limit: 20,
    });
    const conversations = [];
    for (const thread of threads) {
      const history = await sdkClient.threads.getHistory(thread.threadId);
      conversations.push(history.values.messages);
    }
    return JSON.stringify(conversations);
  },
  {
    name: "search_recent_conversations",
    description: "Search this user's conversations updated in the last 6 hours.",
  }
);

const agent = createDeepAgent({
  model: "google_genai:gemini-3.5-flash",
  systemPrompt: `Review recent conversations and update the user's memory file.
Merge new facts, remove outdated information, and keep it concise.`,
  tools: [searchRecentConversations],
});

export { agent };
```

#### Cron

[cron job](/langsmith/cron-jobs) 按固定计划运行整合代理。该代理搜索最近的对话并将其综合到记忆中。将计划与您的使用模式相匹配，使整合运行大致跟踪真实活动。

使用 cron 作业调度合并代理：

:::python
```python
from langgraph_sdk import get_client

client = get_client(url="<DEPLOYMENT_URL>")

cron_job = await client.crons.create(
    assistant_id="consolidation_agent",
    schedule="0 */6 * * *",
    input={"messages": [{"role": "user", "content": "Consolidate recent memories."}]},
)
```
:::

:::js
```typescript
import { Client } from "@langchain/langgraph-sdk";

const client = new Client({ apiUrl: "<DEPLOYMENT_URL>" });

const cronJob = await client.crons.create(
  "consolidation_agent",
  {
    schedule: "0 */6 * * *",
    input: { messages: [{ role: "user", content: "Consolidate recent memories." }] },
  },
);
```
:::

<Note>
所有 cron 计划都在 **UTC** 时间内解释。详见 [cron 作业](/langsmith/cron-jobs) 了解管理和删除 cron 作业的详情。
</Note>

<Warning>
cron 间隔必须与整合代理内的回看窗口相匹配。上面的示例每 6 小时运行一次（`0 */6 * * *`），代理的 `search_recent_conversations` 工具回看 `timedelta(hours=6)`——请保持同步。如果 cron 运行频率超过回看频率，您将重复处理相同的对话；如果运行频率低于回看频率，您将丢失落在窗口外的记忆。
</Warning>

有关使用后台流程部署代理的更多信息，请参阅 [投入生产](/oss/deepagents/going-to-production)。

### 只读与可写内存

默认情况下，代理可以读取和写入内存文件。对于组织策略或合规规则等共享状态，您可能希望将内存设为 **只读**，以便代理可以引用但不能修改。这样可以防止通过共享内存进行提示注入，并确保只有您的应用程序代码控制文件内容。

| 权限 | 用例 | 工作原理 |
|---|---|---|
| **读写**（默认） | 用户偏好、代理自我改进、学习的 [技能](/oss/deepagents/skills) | 代理通过 `edit_file` 工具更新文件 |
| **只读** | 组织策略、合规规则、共享知识库、开发者定义的 [技能](/oss/deepagents/skills) | 通过应用程序代码或 [Store API](/langsmith/custom-store) 填充。使用 [权限](/oss/deepagents/permissions) 拒绝写入特定路径，或使用 [策略钩子](/oss/deepagents/backends#add-policy-hooks) 进行自定义验证逻辑。 |

**安全注意事项：** 如果一个用户可以写入另一个用户读取的内存，恶意用户可能会向共享状态注入指令。为缓解此问题：

- **默认为用户范围** `(user_id)`，除非您有特定原因需要共享
- 对共享策略使用 **只读内存**（通过应用程序代码填充，而非代理）
- 在代理写入共享内存之前添加 **人工介入** 验证。使用 [中断](/oss/langgraph/interrupts) 要求对敏感路径的写入进行人工批准。

要强制只读内存，请使用 [权限](/oss/deepagents/permissions) 以声明方式拒绝写入特定路径。对于自定义验证逻辑（速率限制、审计日志、内容检查），请使用 [后端策略钩子](/oss/deepagents/backends#add-policy-hooks)。

### 并发写入

多个线程可以并行写入内存，但对 **同一文件** 的并发写入可能导致后写入者获胜的冲突。对于用户范围的内存，这种情况很少见，因为用户通常一次只有一个活跃对话。对于代理范围或组织范围的内存，请考虑使用 [后台整合](#background-consolidation) 来序列化写入，或将内存结构化为每个主题的单独文件以减少争用。

实际上，如果写入因冲突而失败，LLM 通常足够智能来重试或优雅地恢复，因此丢失一次写入不是灾难性的。

### 同一部署中的多个代理

要在共享部署中为每个代理提供自己的内存，请将 `assistant_id` 添加到命名空间：

:::python
```python
StoreBackend(
    namespace=lambda rt: (
        rt.server_info.assistant_id,  # [!code highlight]
        rt.server_info.user.identity,
    ),
)
```
:::

:::js
```typescript
new StoreBackend({
  namespace: (rt) => [
    rt.serverInfo.assistantId,  // [!code highlight]
    rt.serverInfo.user.identity,
  ],
})
```
:::

如果您只需要按智能体隔离，而不需要按用户进行作用域划分，请单独使用 `assistant_id`。

<Tip>
使用 [LangSmith 追踪](/langsmith/trace-with-langgraph) 来审计您的智能体写入内存的内容。每次文件写入都会在追踪中显示为工具调用。
</Tip>
