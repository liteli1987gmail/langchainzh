## `BG_JOB_ISOLATED_LOOPS`

设置 `BG_JOB_ISOLATED_LOOPS` to `True` 在独立于服务 API 事件循环的隔离事件循环中执行后台运行。

<Warning>
启用此标志不会消除根本问题。它将同步阻塞工作从服务 API 的事件循环中移出，使健康检查不再失败，但阻塞代码继续在后台循环中运行，并且 **将** 继续在生产环境中造成问题，例如吞吐量下降、尾延迟峰值、工作线程饥饿或连接池耗尽（请参阅下面的连接池大小注意事项），以及在负载下扩展性差。

要正确解决这些问题，请在整个代理中使用原生异步驱动和异步代码。这意味着使用异步 HTTP 客户端，如 `httpx` or `aiohttp` （尽管我们建议缓存客户端以避免加载 SSL 上下文时的 CPU 开销），异步数据库驱动，如 `asyncpg` or `psycopg[async]`，以及异步模型 SDK。对于不可避免的同步库，请将特定调用包装在 `asyncio.to_thread(...)` or `loop.run_in_executor(...)` 中，而不是为整个部署启用此标志。
</Warning>

此环境变量应设置为 `True` if the implementation of a graph/node contains synchronous code. In this situation, the synchronous code will block the serving API event loop, which may cause the API to be unavailable. A symptom of an unavailable API is continuous application restarts due to failing health checks.

<Warning>
当 `BG_JOB_ISOLATED_LOOPS` 启用时，每个后台工作线程在其自己的线程中运行，使用 **独立的 Postgres 连接池**。每个工作线程的连接池大小为 `LANGGRAPH_POSTGRES_POOL_MAX_SIZE // N_JOBS_PER_WORKER`。例如，使用 `LANGGRAPH_POSTGRES_POOL_MAX_SIZE=20` 和 `N_JOBS_PER_WORKER=15`时，每个工作线程只能获得 1 个连接的池。每个工作线程的连接池越小，就越容易发生连接故障，因为单个过期连接会占用池中的很大一部分。如果您启用隔离循环，请确保 `LANGGRAPH_POSTGRES_POOL_MAX_SIZE` 足够大，以便为每个工作线程提供至少几个连接。
</Warning>

默认为 `False`.

## `BG_JOB_MAX_RETRIES`

后台运行在发生可重试故障（例如临时数据库错误、服务器关闭取消）后重试的最大次数。当运行因可重试错误而失败时，它会被放回队列中并从最后一个检查点继续。如果运行超过最大重试次数，则被标记为失败。

默认为 `3`.

## `BG_JOB_SHUTDOWN_GRACE_PERIOD_SECS`

指定服务器在队列收到关闭信号后等待后台作业完成的秒数。超过此时间后，服务器将强制终止。默认为 `180` 秒。最大值为 `3600` 秒。请设置此值以确保作业在关闭期间有足够的时间正常完成。于 `langgraph-api==0.2.16`.

## `BG_JOB_TIMEOUT_SECS`

添加。后台运行的超时时间可以增加。但是，Cloud 部署的基础设施对 API 请求强制执行 1 小时超时限制。这意味着客户端与服务器之间的连接将在 1 小时后超时。这是不可配置的。

后台运行可以执行超过 1 小时，但如果运行时间超过 1 小时，客户端必须重新连接到服务器（例如通过 `POST /threads/{thread_id}/runs/{run_id}/stream`加入流）才能获取运行的输出。

默认为 `86400`.

## `DD_API_KEY`

指定 `DD_API_KEY` （您的 [Datadog API 密钥](https://docs.datadoghq.com/account_management/api-app-keys/)) to automatically enable Datadog tracing for the deployment. Specify other [`DD_*` environment variables](https://ddtrace.readthedocs.io/en/stable/configuration.html) to configure the tracing instrumentation.

## `LANGGRAPH_POSTGRES_POOL_MAX_SIZE`

Beginning with langgraph-api version `0.2.12`, the maximum size of the Postgres connection pool (per replica) can be controlled using the `LANGGRAPH_POSTGRES_POOL_MAX_SIZE` environment variable. By setting this variable, you can determine the upper bound on the number of simultaneous connections the server will establish with the Postgres database.

For example, if a deployment is scaled up to 10 replicas and `LANGGRAPH_POSTGRES_POOL_MAX_SIZE` is configured to `150`, then up to `1500` connections to Postgres can be established. This is particularly useful for deployments where database resources are limited (or more available) or where you need to tune connection behavior for performance or scaling reasons.

When [`BG_JOB_ISOLATED_LOOPS`](#bg_job_isolated_loops) is enabled, the pool is not shared. Instead, each background worker thread creates its own pool with a maximum size of `LANGGRAPH_POSTGRES_POOL_MAX_SIZE / N_JOBS_PER_WORKER`. Keep this in mind when lowering the pool size. A value that works well for a shared pool may result in very small per-worker pools under isolated loops.

Defaults to `150` connections.

## `LS_CHECKPOINT_DELETE`

JSON-valued configuration for deferred checkpoint deletion. When enabled, thread delete and prune operations enqueue checkpoints for background deletion instead of deleting synchronously, moving the I/O off the request hot path. Available in `langgraph-api>=0.8.1`.

<Note>
Only supported with the default PostgreSQL checkpointer backend. Deferred deletes will become the default in a future release.
</Note>

Accepted fields:

- `enabled` (boolean, default `false`): When `true`, thread delete and prune operations enqueue checkpoints into `checkpoint_delete_queue` and return immediately, and the background worker drains the queue.
- `enabledWorkerOnly` (boolean, default `false`): Runs only the background drain worker without enqueuing new entries. Use this to finish draining the queue after rolling `enabled` back to `false`.
- `pollIntervalMs` (integer, default `5000`): How often the worker polls the queue, in milliseconds.
- `batchSize` (integer, default `25`): Number of checkpoint entries the worker dequeues per transaction. Smaller values spread I/O over more time at the cost of longer drain latency.
- `batchSleepMs` (integer, default `500`): How long the worker sleeps between batches when the queue is non-empty, in milliseconds.

Example: `LS_CHECKPOINT_DELETE='{"enabled":true,"batchSize":10,"pollIntervalMs":1000}'`.

Defaults to disabled (synchronous checkpoint deletion).

## `LS_APM_OTEL_ENABLED`

要为您的部署配置 OpenTelemetry APM 跟踪，请设置 `LS_APM_OTEL_ENABLED` to `true` 和 `OTEL_EXPORTER_OTLP_TRACES_ENDPOINT` or `OTEL_EXPORTER_OTLP_ENDPOINT` 到目标跟踪摄取端点。请注意，必须同时设置 `LS_APM_OTEL_ENABLED` 和其他两个导出端点中的一个，才能在服务器版本 `0.7.17`.

指定其他 [`OTEL_*` 环境变量](https://opentelemetry.io/docs/collector/configuration/) 以配置跟踪、日志记录和其他检测功能。

```shell
# If you set LS_APM_OTEL_ENABLED AND (OTEL_EXPORTER_OTLP_TRACES_ENDPOINT or OTEL_EXPORTER_OTLP_ENDPOINT),
# the server starts with OpenTelemetry instrumentation enabled.
LS_APM_OTEL_ENABLED=true
OTEL_EXPORTER_OTLP_TRACES_ENDPOINT=<target trace ingestion endpoint>
OTEL_EXPORTER_OTLP_ENDPOINT=https://otlp.nr-data.net
OTEL_SERVICE_NAME=MY_LANGSMITH_DEPLOYMENT
OTEL_EXPORTER_OTLP_HEADERS=api-key=<YOUR_INGEST_LICENSE_KEY>
LANGSMITH_OTEL_ENABLED=true
# Common OTEL settings
OTEL_ATTRIBUTE_VALUE_LENGTH_LIMIT=4095
OTEL_EXPORTER_OTLP_COMPRESSION=gzip
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE=delta
OTEL_PYTHON_EXCLUDED_URLS=/metrics,/ok,/info
# Optional: OTEL_PYTHON_LOGGING_AUTO_INSTRUMENTATION_ENABLED=true
```

例如，要向 [New Relic 的美国区域](https://docs.newrelic.com/docs/opentelemetry/best-practices/opentelemetry-otlp/)提交 OpenTelemetry 跟踪，请设置以下内容：

```shell
LS_APM_OTEL_ENABLED=true
OTEL_EXPORTER_OTLP_TRACES_ENDPOINT=https://otlp.nr-data.net/v1/traces
OTEL_EXPORTER_OTLP_ENDPOINT=https://otlp.nr-data.net
OTEL_EXPORTER_OTLP_HEADERS=api-key=<YOUR_INGEST_LICENSE_KEY>
```

<Note>
OTel APM 跟踪已在 Agent Server 版本中添加 `0.5.32` ，目前处于 Alpha 阶段。
</Note>

## `REDIS_MAX_CONNECTIONS`

Redis 连接池的最大大小（每个副本）可以通过 `REDIS_MAX_CONNECTIONS` 环境变量来控制。通过设置此变量，您可以确定服务器将与 Redis 实例建立的同时连接数的上限。

例如，如果部署扩展到 10 个副本且 `REDIS_MAX_CONNECTIONS` 配置为 `150`，然后最多 `1500` 个到 Redis 的连接可以被建立。

默认为 `2000`.

## `RESUMABLE_STREAM_TTL_SECONDS`

Redis 中可恢复流数据的生存时间（秒）。

当创建一个 run 并输出被流式传输时，流可以被配置为可恢复的（例如 `stream_resumable=True`）。如果一个流是可恢复的，来自该流的输出会临时存储在 Redis 中。该数据的 TTL 可以通过设置 `RESUMABLE_STREAM_TTL_SECONDS`.

参见 @[Python][RunsClient.stream] 和 [JS/TS](https://langchain-ai.github.io/langgraphjs/reference/classes/sdk_client.RunsClient.html#stream) SDK 以获取有关如何实现可恢复流的更多详细信息。

默认为 `120` seconds.

<Note>
为 `RESUMABLE_STREAM_TTL_SECONDS` 设置一个非常高的值可能导致大量 Redis 内存使用，特别是在有许多并发运行且有大量或频繁的流式输出时。将此值设置为最小值以启用网络中断期间的恢复，并优先使用检查点以实现长期持久性和执行快照。
</Note>
