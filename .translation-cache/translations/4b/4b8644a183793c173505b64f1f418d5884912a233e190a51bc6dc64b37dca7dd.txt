title: 集成测试
description: 通过组织测试、管理密钥、处理不稳定性以及控制成本，使用真实的 LLM API 测试代理。
---

集成测试验证您的代理与模型 API 和外部服务正常协作。与使用伪造和模拟的[单元测试](/oss/langchain/test/unit-testing)不同，集成测试会发起实际的网络调用，以确认组件协同工作、凭证有效且延迟可接受。

由于 LLM 响应具有不确定性，集成测试需要与传统软件测试不同的策略。本指南介绍如何为您的代理组织、编写和运行集成测试。有关为 LangChain 本身贡献代码时的通用测试基础设施，请参阅[代码贡献](/oss/contributing/code#running-tests)。

## 将单元测试和集成测试分开

集成测试速度较慢且需要 API 凭证，因此应与单元测试分开。这样可以每次变更时运行快速单元测试，并将集成测试保留用于 CI 或部署前检查。

:::python
使用 pytest 标记来标记集成测试：

```python
import pytest

@pytest.mark.integration
def test_agent_with_real_model():
    agent = create_agent("claude-sonnet-4-6", tools=[get_weather])
    result = agent.invoke({
        "messages": [HumanMessage(content="What's the weather in SF?")]
    })
    assert len(result["messages"]) > 1
```

配置 pytest 以识别标记并从默认运行中排除集成测试：

<CodeGroup>

</CodeGroup>
```ini pytest.ini
[pytest]
markers =
    integration: tests that call real LLM APIs
addopts = -m "not integration"
```
```toml pyproject.toml
[tool.pytest.ini_options]
markers = [
  "integration: tests that call real LLM APIs"
]
addopts = "-m 'not integration'"
```
</CodeGroup>

显式运行集成测试：

```bash
pytest -m integration
```
:::

:::js
使用文件命名约定来分离集成测试。将集成测试文件命名为 `*.int.test.ts`，并配置 vitest 将其从默认运行中排除：

:::

```ts vitest.config.ts
import { configDefaults, defineConfig } from "vitest/config";

export default defineConfig((env) => {
  if (env.mode === "int") {
    return {
      test: {
        testTimeout: 100_000,
        include: ["**/*.int.test.ts"],
        setupFiles: ["dotenv/config"],
      },
    };
  }

  return {
    test: {
      testTimeout: 30_000,
      exclude: ["**/*.int.test.ts", ...configDefaults.exclude],
    },
  };
});
```

添加脚本到 `package.json`：

```json
{
  "scripts": {
    "test": "vitest",
    "test:integration": "vitest --mode int"
  }
}
```

<fragment>

显式运行集成测试：

</fragment>

```bash
npm run test:integration
```
:::

## 管理 API 密钥

集成测试需要真实的 API 凭证。请从环境变量中加载它们，以确保密钥不会提交到版本控制系统中。

:::python
使用 `conftest.py` fixture 来验证所需的密钥是否可用：

:::

```python
import os
import pytest

@pytest.fixture(autouse=True)
def check_api_keys():
    if not os.environ.get("OPENAI_API_KEY"):
        pytest.skip("OPENAI_API_KEY not set")
```

对于本地开发，将密钥存储在 `.env` 文件中，并使用 [`python-dotenv`](https://pypi.org/project/python-dotenv/) 加载它们：

```bash .env
OPENAI_API_KEY=sk-...
```

```python conftest.py
from dotenv import load_dotenv

load_dotenv()
```
:::

:::js
添加 `dotenv/config` 作为 vitest 配置文件，使环境变量自动从 `.env` 中加载：

```ts vitest.config.ts
export default defineConfig({
  test: {
    setupFiles: ["dotenv/config"],
  },
});
```

```bash .env
OPENAI_API_KEY=sk-...
```

跳过密钥缺失时的测试：

```ts
import { test } from "vitest";

test.skipIf(!process.env.OPENAI_API_KEY)(
  "agent responds with tool call",
  async () => {
    // ...
  }
);
```
:::

<Warning>
    将 `.env` 添加到 `.gitignore` 中以避免提交凭证。在 CI 中，通过提供商的密钥管理功能注入密钥（例如 GitHub Actions secrets）。
</Warning>

## 对结构进行断言，而非内容

LLM 响应在不同运行之间会有所不同。与其对确切的输出字符串进行断言，不如验证响应的结构属性：消息类型、工具调用名称、参数形状和消息数量。

:::python
```python
def test_agent_calls_weather_tool():
    agent = create_agent("claude-sonnet-4-6", tools=[get_weather])
    result = agent.invoke({
        "messages": [HumanMessage(content="What's the weather in SF?")]
    })

    messages = result["messages"]
    tool_calls = [
        tc
        for msg in messages
        if hasattr(msg, "tool_calls")
        for tc in (msg.tool_calls or [])
    ]

    assert any(tc["name"] == "get_weather" for tc in tool_calls)
    assert isinstance(messages[-1], AIMessage)
    assert len(messages[-1].content) > 0
```
:::

:::js
```ts
test("agent calls weather tool", async () => {
  const agent = createAgent({ model: "claude-sonnet-4-6", tools: [getWeather] });
  const result = await agent.invoke({
    messages: [new HumanMessage("What's the weather in SF?")]
  });

  const aiMsg = result.messages.find(
    (m) => AIMessage.isInstance(m) && m.tool_calls?.length
  );
  expect(aiMsg).toContainToolCall({ name: "get_weather" });
  expect(result.messages.at(-1)).toBeAIMessage();
});
```

此示例使用[自定义测试匹配器](#use-custom-test-matchers)。有关设置和完整匹配器参考，请参阅下面的部分。
:::

<Tip>
    要进行更严格的轨迹断言，请使用 [AgentEvals](/oss/langchain/test/evals) 评估器，它支持 `unordered` 和 `superset` 等模糊匹配模式。
</Tip>

:::js

## 使用自定义测试匹配器

`langchain` 附带了[自定义 vitest 匹配器](https://vitest.dev/guide/extending-matchers.html)，使结构化断言更具可读性，并在失败时产生清晰的错误信息。在设置文件中注册一次，它们就会在每次 `expect()` 调用中可用。

### 设置

添加一个 vitest 设置文件，用 LangChain 匹配器扩展 `expect`：

```ts vitest.setup.ts
import { langchainMatchers } from "@langchain/core/testing";

expect.extend(langchainMatchers);
```

在 your vitest 配置中引用 it：

```ts vitest.config.ts
export default defineConfig({
  test: {
    setupFiles: ["vitest.setup.ts"],
  },
});
```

TypeScript 类型是自动包含的，因此无需额外配置即可实现自动补全。

### 检查消息类型

每个消息类都有对应的匹配器：`toBeHumanMessage()`、`toBeAIMessage()`、`toBeSystemMessage()` 和 `toBeToolMessage()`。调用时不传参数仅检查类型，传入字符串则同时匹配内容：

```ts
const response = await agent.invoke({
  messages: [new HumanMessage("What's the weather?")]
});
const lastMessage = response.messages.at(-1);

expect(lastMessage).toBeAIMessage();
expect(lastMessage).toBeAIMessage("It's 72°F and sunny.");
```

传递一个对象来匹配特定字段：

```ts
expect(lastMessage).toBeAIMessage({ name: "weather-bot" });
expect(toolMsg).toBeToolMessage({ tool_call_id: "call_1" });
```

### 断言工具调用

三个匹配器覆盖对 @[`AIMessage`] 的工具调用断言：

```ts
const response = await agent.invoke({
  messages: [new HumanMessage("Weather in SF and NYC?")]
});
const aiMsg = response.messages.find(
  (m) => AIMessage.isInstance(m) && m.tool_calls?.length
);

// Check that specific tool calls are present (order-independent)
expect(aiMsg).toHaveToolCalls([
  { name: "get_weather", args: { city: "San Francisco" } },
  { name: "get_weather", args: { city: "New York" } },
]);

// Check only the count
expect(aiMsg).toHaveToolCallCount(2);

// Check that at least one tool call matches (supports .not)
expect(aiMsg).toContainToolCall({ name: "get_weather" });
expect(aiMsg).not.toContainToolCall({ name: "send_email" });
```

### 对工具消息进行断言

`toHaveToolMessages()` 接收完整的消息数组，并检查其中的 @[`ToolMessage`] 实例，按顺序进行：

```ts
expect(response.messages).toHaveToolMessages([
  { content: "72°F and sunny in San Francisco" },
  { content: "68°F and cloudy in New York" },
]);
```

### 断言中断和结构化响应

`toHaveBeenInterrupted()` 检查 [LangGraph 中断](/oss/langchain/human-in-the-loop) 结果中的 `__interrupt__` 字段。传递一个值来匹配中断载荷：

```ts
const result = await graph.invoke(input);

expect(result).toHaveBeenInterrupted();
expect(result).toHaveBeenInterrupted("confirm_action");
```

`toHaveStructuredResponse()` 检查结果中是否存在 `structuredResponse` 字段。传入一个对象以匹配特定字段：

```ts
expect(result).toHaveStructuredResponse();
expect(result).toHaveStructuredResponse({ name: "Alice", age: 30 });
```

### Matcher 参考

| Matcher | 描述 |
| --- | --- |
| `toBeHumanMessage(expected?)` | 检查值是否为 `HumanMessage`。可选择匹配内容（字符串）或字段（对象）。 |
| `toBeAIMessage(expected?)` | 检查值是否为 `AIMessage`。可选择匹配内容或字段。 |
| `toBeSystemMessage(expected?)` | 检查值是否为 `SystemMessage`。可选择匹配内容或字段。 |
| `toBeToolMessage(expected?)` | 检查值是否为 `ToolMessage`。可选择匹配内容或字段（如 `tool_call_id`）。 |
| `toHaveToolCalls(expected)` | 检查 `AIMessage` 是否包含恰好给定的工具调用（顺序无关）。 |
| `toHaveToolCallCount(n)` | 检查 `AIMessage` 是否恰好有 `n` 个工具调用。 |
| `toContainToolCall(expected)` | 检查 `AIMessage` 是否包含至少一个匹配的工具调用。支持 `.not`。 |
| `toHaveToolMessages(expected)` | 检查消息数组是否按顺序包含给定的 `ToolMessage` 实例。 |
| `toHaveBeenInterrupted(value?)` | 检查结果是否有 `__interrupt__`。可选择匹配中断值。 |
| `toHaveStructuredResponse(expected?)` | 检查结果是否有 `structuredResponse`。可选择匹配特定字段。 |

:::

## 降低成本和延迟

调用 LLM API 的集成测试会产生实际成本。以下做法有助于保持测试套件快速且经济：

- **使用更小的模型**：`gemini-3.1-flash-lite` 或同等模型，用于只需验证工具调用和响应结构的测试。
- **设置 `maxTokens`**：限制响应长度，避免冗长且昂贵的完成输出。
- **限制测试范围**：每个测试验证一个行为。当单轮测试足够时，避免链接多个 LLM 调用的端到端场景。
- **选择性运行**：使用上方（[above](#separate-unit-and-integration-tests)）的测试分离，仅在 CI 或部署前运行集成测试，而非每次保存文件时都运行。

:::python
```python
agent = create_agent(
    "gemini-3.1-flash-lite",
    tools=[get_weather],
    model_kwargs={"max_tokens": 256},
)
```
:::

:::js
```ts
const agent = createAgent({
  model: "gemini-3.1-flash-lite",
  tools: [getWeather],
  modelArgs: { maxTokens: 256 },
});
```
:::

## 录制和回放 HTTP 调用

对于频繁在 CI 中运行的测试，您可以在首次运行时录制 HTTP 交互，并在后续运行中回放它们，而无需发起真实的 API 调用。这样可以消除首次录制后的成本和延迟。

[`vcrpy`](https://pypi.org/project/vcrpy/1.5.2/) 将 HTTP 请求/响应对录制到 YAML "cassette" 文件中。[`pytest-recording`](https://pypi.org/project/pytest-recording/) 插件将其与 pytest 集成。

设置您的 `conftest.py` 以从 cassette 中过滤敏感信息：

```py conftest.py
import pytest

@pytest.fixture(scope="session")
def vcr_config():
    return {
        "filter_headers": [
            ("authorization", "XXXX"),
            ("x-api-key", "XXXX"),
        ],
        "filter_query_parameters": [
            ("api_key", "XXXX"),
            ("key", "XXXX"),
        ],
    }
```

配置您的项目以识别 `vcr` 标记：

<CodeGroup>
```ini pytest.ini
[pytest]
markers =
    vcr: record/replay HTTP via VCR
addopts = --record-mode=once
```
```toml pyproject.toml
[tool.pytest.ini_options]
markers = [
  "vcr: record/replay HTTP via VCR"
]
addopts = "--record-mode=once"
```
</CodeGroup>

<Info>
    `--record-mode=once` 选项在首次运行时记录 HTTP 交互，并在后续运行时回放它们。
</Info>

使用 `vcr` 标记装饰你的测试：

</fragment>

```python
@pytest.mark.vcr()
def test_agent_trajectory():
    agent = create_agent("claude-sonnet-4-6", tools=[get_weather])
    result = agent.invoke({
        "messages": [HumanMessage(content="What's the weather in SF?")]
    })
    assert any(
        tc["name"] == "get_weather"
        for msg in result["messages"]
        if hasattr(msg, "tool_calls")
        for tc in (msg.tool_calls or [])
    )
```

首次运行会发起真实的网络请求，并在 `tests/cassettes/` 目录中生成一个 cassette 文件。后续运行会回放已录制的响应。

<Warning>
    当您修改提示词、添加工具或更改预期轨迹时，已保存的 cassette 将失效，您现有的测试**将会失败**。请删除相应的 cassette 文件并重新运行测试以录制新的交互。
</Warning>
:::
