---

每个 [LangSmith Deployment](/langsmith/deployment) 都包含一个 Postgres 支持的 [store](/oss/langgraph/stores)，用于长期记忆和跨线程数据。默认情况下，store 命名空间在所有调用者之间共享。要为每个用户提供独立的隔离 store，请配置[自定义认证](/langsmith/custom-auth)并添加一个 store [授权处理程序](/langsmith/auth#authorization)，用于重写命名空间以包含已认证用户的身份。

本指南介绍如何在 LangSmith Deployment 中配置该隔离。相同的模式也适用于启用了自定义认证的[自托管](/langsmith/self-hosted)部署。

## 工作原理

Store 项由 **namespace** 元组（例如 `("memories", "preferences")`）和 **key** 组织。与线程和助手不同，store 命名空间由您的应用程序定义，因此授权的工作方式不同：

1. 您的 `@auth.authenticate` 处理程序验证调用者并为每个用户返回唯一的 `identity`。
2. 您的 `@auth.on.store` 处理程序**验证**命名空间以该身份开头，或者**重写**命名空间以添加该身份前缀。
3. 服务器使用验证或重写后的命名空间执行实际读取或写入操作。

用户写入逻辑命名空间 `("memories",)` 时，自动前缀重写实际上会将数据存储在 `("user-123", "memories")`。其他用户无法读取或覆盖该数据，因为他们的请求被限定在 `("user-456", ...)` 范围内。

<Note>
Store 授权处理程序接收一个**可变的** `value` 字典。对 `value["namespace"]` 的更改会在操作中生效。您不需要像线程那样返回元数据过滤器。
</Note>

## 前置条件

- 已配置[自定义认证](/langsmith/set-up-custom-auth)的 LangSmith Deployment。
- 一个 `@auth.authenticate` 处理程序，为每个最终用户返回稳定且唯一的 `identity`。

## 在 LangSmith Deployment 中配置认证

在 [`langgraph.json`](/langsmith/application-structure#configuration-file-concepts) 中将您的部署指向认证模块：

```json
{
  "dependencies": ["."],
  "graphs": {
    "agent": "./src/agent/graph.py:graph"
  },
  "auth": {
    "path": "./src/security/auth.py:auth"
  }
}
```

`auth` 对象必须暴露一个 `langgraph_sdk.Auth` 实例（通常命名为 `auth`）。

## 选择存储授权模式

存储隔离需要一个 `@auth.on.store` 处理器。有两种常见模式效果良好；请根据您希望用户作用域位于何处来选择。

| 模式 | 用户作用域设置位置 | 最佳场景 |
| --- | --- | --- |
| **显式命名空间 + 拒绝** | 应用程序代码将 `user_id` 放在每个命名空间的第一位 | 您已经将用户身份传递到图代码中，或者您希望命名空间反映完整存储路径 |
| **自动前缀重写** | 认证处理器在逻辑命名空间前添加 `ctx.user.identity` | 您希望更简单的代理代码和在 API 层透明的隔离 |

两种模式都使用单一的 `@auth.on.store` 处理器来覆盖所有存储操作（`put`、`get`、`search`、`delete` 和 `list_namespaces`）。只有在需要为每个操作设置不同规则时，才需要特定操作的处理器，如 `@auth.on.store.put`。

<Tip>
如果您使用全局 `@auth.on` 处理器拒绝未处理的请求，请显式注册 `@auth.on.store` 以允许存储操作。请参阅 [Auth 参考文档](https://reference.langchain.com/python/langgraph-sdk/auth/Auth) 了解特定操作的处理器名称。
</Tip>

### 显式命名空间 + 拒绝

您的应用程序代码将用户身份作为每个命名空间的第一段（例如 `("user-123", "memories")`）。认证处理器验证调用者是否与该前缀匹配，并拒绝不匹配的情况：

```python
from langgraph_sdk import Auth

auth = Auth()

# ... your @auth.authenticate handler ...

@auth.on.store
async def authorize_store(
    ctx: Auth.types.AuthContext,
    value: Auth.types.on.store.value,
):
    """Require the user identity as the first namespace segment."""
    namespace = tuple(value["namespace"])
    if not namespace or namespace[0] != ctx.user.identity:
        raise Auth.exceptions.HTTPException(
            status_code=403,
            detail="Not authorized to access this namespace.",
        )
```

此模式使存储布局在你的图代码中显式化。如果已认证用户是 `user-123`，则对命名空间 `("user-456", "memories")` 的请求会立即失败。权衡之处在于，代理中的每个存储调用都必须包含用户前缀，通常来自运行时上下文，例如 `config["configurable"]["langgraph_auth_user_id"]`。

### 自动前缀重写

你的应用代码使用逻辑命名空间，不带用户前缀（例如 `("memories", "preferences")`）。认证处理程序在每个存储操作前添加已认证用户的身份：

```python
from langgraph_sdk import Auth

auth = Auth()

# ... your @auth.authenticate handler ...

@auth.on.store
async def scope_store(
    ctx: Auth.types.AuthContext,
    value: Auth.types.on.store.value,
):
    """Isolate store data per user by rewriting namespaces."""
    namespace = tuple(value["namespace"]) if value.get("namespace") else ()
    if not namespace or namespace[0] != ctx.user.identity:
        namespace = (ctx.user.identity, *namespace)
    value["namespace"] = namespace
```

当客户端调用 `list_namespaces` 不带前缀时，`value["namespace"]` 为空。上面的处理器将空命名空间视为 `(ctx.user.identity,)`，因此用户只能看到自己的命名空间。

此模式使代理代码更简单，因为认证层透明地处理用户隔离。代价是你不能在应用代码中也为命名空间添加前缀，否则你会双重作用域数据并破坏读取。选择一个作用域层：认证重写 **或** 应用级命名空间，不要两者都用。

## 在代理代码中使用命名空间

如果使用自动前缀重写，你的图代码使用逻辑命名空间，不带用户前缀。认证层在请求时自动添加用户作用域。将以下内容放入你的图节点或工具中（例如 `graph.py`）：

```python
from langgraph.store.base import BaseStore

async def save_preference(state: State, *, store: BaseStore):
    await store.aput(
        ("memories", "preferences"),  # logical namespace
        "settings",
        {"theme": "dark"},
    )

async def load_preference(state: State, *, store: BaseStore):
    item = await store.aget(
        ("memories", "preferences"),
        "settings",
    )
    return item.value if item else {}
```

当 User A 调用 store API 时，服务器会将数据持久化到 `("user-a", "memories", "preferences")`。User B 的请求永远不会到达该命名空间。

如果你使用 **explicit namespace + deny**，请改为在代码的每个命名空间中包含用户身份：

```python
from langchain_core.runnables import RunnableConfig
from langgraph.store.base import BaseStore

async def save_preference(state: State, *, store: BaseStore, config: RunnableConfig):
    user_id = config["configurable"]["langgraph_auth_user_id"]
    await store.aput(
        (user_id, "memories", "preferences"),
        "settings",
        {"theme": "dark"},
    )
```

### Deep Agents 与 StoreBackend

如果你使用配合 @[`StoreBackend`] 的 [Deep Agents](/oss/deepagents/overview)，请将你的命名空间工厂与 auth-scoped 布局对齐。当 auth 在前面添加 `ctx.user.identity` 时，在后端使用逻辑命名空间，并让 auth 处理用户隔离：

:::python
```python
from deepagents.backends import StoreBackend

StoreBackend(
    namespace=lambda rt: ("memories",),  # auth adds user identity
)
```
:::

:::js
```typescript
import { StoreBackend } from "deepagents";

new StoreBackend({
  namespace: () => ["memories"],  // auth adds user identity
});
```
:::

或者，如果您通过图内用户进行作用域限定（例如，使用 `rt.server_info.user.identity`），请确保您的 auth handler 不会重复添加命名空间前缀。选择一种作用域层：auth 重写**或**应用级命名空间，不要两者同时使用。

有关命名空间设计的更多信息，请参阅 [Deep Agents 后端](/oss/deepagents/backends#namespace-factories) 和 [用户级内存](/oss/deepagents/memory#user-scoped-memory)。

## 测试 store 隔离

在 `langgraph dev` 运行且 auth handler 中配置了两个测试用户的情况下，验证 store 数据不会在用户之间泄露：

```python
import asyncio

from langgraph_sdk import get_client


async def main():
    alice = get_client(
        url="http://localhost:60058",
        headers={"Authorization": "Bearer user1-token"},
    )
    bob = get_client(
        url="http://localhost:60058",
        headers={"Authorization": "Bearer user2-token"},
    )

    # Alice writes a store item
    await alice.store.put_item(
        ["memories"],
        key="note",
        value={"text": "Alice private note"},
    )

    # Bob cannot read Alice's item
    bob_item = await bob.store.get_item(["memories"], key="note")
    if bob_item is None or bob_item["value"]["text"] != "Alice private note":
        print("✅ Bob correctly denied access to Alice's item")
    else:
        print("❌ Bob should not see Alice's store item")

    # Bob writes his own item at the same logical namespace
    await bob.store.put_item(
        ["memories"],
        key="note",
        value={"text": "Bob private note"},
    )

    alice_item = await alice.store.get_item(["memories"], key="note")
    bob_item = await bob.store.get_item(["memories"], key="note")
    assert alice_item["value"]["text"] == "Alice private note"
    assert bob_item["value"]["text"] == "Bob private note"
    print("✅ Each user sees only their own store data")


if __name__ == "__main__":
    asyncio.run(main())
```

## 与线程隔离结合使用

存储隔离和[线程隔离](/langsmith/resource-auth)解决不同的问题：

| 关注点 | 机制 | 范围 |
| --- | --- | --- |
| 对话历史 | 线程元数据过滤器（`owner`） | 每个线程的检查点和消息 |
| 长期记忆 | 存储命名空间重写 | 跨线程记忆、偏好设置和文档 |

对于多用户代理，请同时配置两者。参见[使对话保持私密](/langsmith/resource-auth)了解线程和运行范围。
