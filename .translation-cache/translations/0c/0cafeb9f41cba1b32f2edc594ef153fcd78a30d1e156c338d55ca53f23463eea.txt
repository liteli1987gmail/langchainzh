## 数据保留

本节介绍数据保留的工作原理以及在 LangSmith 中的定价方式。

### 为什么数据保留很重要

* **隐私**：许多数据隐私法规（如欧洲的 GDPR 或加利福尼亚州的 CCPA）要求组织在个人数据不再用于收集目的时将其删除。设置保留期有助于遵守此类法规。
* **成本**：LangSmith 对数据保留期较短的追踪收取较低的费用。更多信息，请了解如何[强制执行消费限制](/langsmith/billing#enforce-spend-limits)。

<Tip>
在开始发送追踪之前，请规划好您的保留层级。变更仅适用于新的追踪——现有追踪将保持其原始层级。请参阅[更改项目级默认保留期](/langsmith/billing#change-project-level-default-retention)。
</Tip>

### 工作原理

LangSmith 根据数据保留设置两个追踪层级，具有以下特点：

|                      | 基础版             | 扩展版       |
| -------------------- | ---------------- | -------------- |
| **价格**            | [请参阅定价页面](https://www.langchain.com/pricing-langsmith) | [请参阅定价页面](https://www.langchain.com/pricing-langsmith) |
| **保留期限** | 14 天          | 400 天       |

<Note>
企业客户可以按工作区自定义扩展保留期。变更仅适用于新的追踪——现有追踪不受影响。请参阅[自定义扩展保留策略](/langsmith/data-purging-compliance#customize-extended-retention-policy)。
</Note>

**保留期结束后的数据删除**

在指定的保留期后，追踪将不再可通过追踪项目 UI 或 API 访问。与追踪相关的所有用户数据（如输入和输出）将在此后一天内从我们的内部系统中删除。与每个追踪关联的某些元数据可能会被无限期保留，用于分析和计费目的。

### 数据保留自动升级

<Warning>
自动升级可能会影响您的账单。请仔细阅读本节，以充分了解您预计的 LangSmith 追踪费用。
</Warning>

当您对 `基础版` 层级追踪使用某些功能时，其数据保留将自动升级到 `扩展版` 层级。这将增加保留期和追踪成本。

以下是追踪将升级的完整场景列表：

* 通过[手动标注](/langsmith/annotate-traces-inline)、使用[在线评估器](/langsmith/online-evaluations-llm-as-judge)自动评估，或通过 [SDK](/langsmith/attach-user-feedback) 以编程方式向追踪上的任何运行添加**反馈**时。
* **[标注队列](/langsmith/annotation-queues#assign-runs-to-a-single-run-queue)** 从追踪中接收任何运行时。
* **[自动化规则](/langsmith/rules#create-a-rule)** 匹配追踪中的任何运行时。

<Note>
在追踪项目上创建或编辑在线评估器时，您可以选择不让该评估器评分的追踪升级，保持基础保留期。此选项仅在项目的默认保留期为基础层级时可用。更多信息，请参阅[管理评估器追踪保留期](/langsmith/evaluators#manage-evaluator-trace-retention)。
</Note>

**为什么自动升级追踪？**

我们采用自动升级模式有两个原因：

1. 我们认为匹配任何这些条件的追踪本质上比其他追踪更有趣，因此让用户能够将它们保留更长时间是很好的。
2. 从哲学上讲，我们希望对可能没有实质性交互的追踪收取低一个数量级的费用。我们认为自动升级使我们的定价模式与 LangSmith 带来的价值保持一致，只有具有实质性交互的追踪才会以更高的费率收费。

如果您对我们的定价模式有疑问或顾虑，请随时通过 [support.langchain.com](https://support.langchain.com) 联系支持人员并分享您的想法！

**数据保留如何影响下游功能？**

* **标注队列、运行规则和反馈**：使用这些功能的追踪将被[自动升级](#data-retention-auto-upgrades)。
* **监控**：即使基础层级追踪的数据保留期结束，监控标签页仍可继续使用。它由存在超过 30 天的追踪元数据驱动，这意味着即使在 `基础版` 层级追踪上，您的监控图表也将保持准确。
* **数据集**：数据集具有无限的数据保留期。换句话说，如果您将追踪的输入和输出添加到数据集中，它们永远不会被删除。我们建议，如果使用 LangSmith 进行数据收集，您应该利用数据集功能。

### 计费模式

**计费指标**

在您的 LangSmith 发票上，您将看到我们收费的两个指标：

* LangSmith 追踪（基础费用）
* LangSmith 追踪（扩展数据保留升级）。

第一个指标包括所有追踪，无论层级。第二个指标仅计算扩展保留追踪的数量。

**为什么测量所有追踪 + 升级而不是基础和扩展追踪？**

在考虑我们的定价时，一个很自然的问题是为什么不直接在发票上显示 `基础版` 层级和 `扩展版` 层级的追踪数量？

虽然我们理解这会更简单，但它不能很好地处理追踪升级。假设一个 `基础版` 层级的追踪记录于 6 月 30 日，并在 7 月 3 日升级到 `扩展版` 层级。`基础版` 追踪发生在 6 月计费期，但升级发生在 7 月计费期。因此，我们需要能够独立测量这两个事件，以正确地向客户收费。

如果您的追踪被记录为扩展保留追踪，则 `基础版` 和 `扩展版` 指标都将以相同的时间戳记录。

**费用明细**

追踪的基础费用为每追踪 0.05¢。我们将升级定价为扩展保留追踪的价格是基础层级追踪的 10 倍（每追踪 0.50¢），包括两个指标。因此，每次升级费用为 0.45¢。

## 速率限制

LangSmith 有速率限制，旨在确保所有用户服务的稳定性。

为确保访问和稳定性，LangSmith 在以下情况下将返回 HTTP 状态码 429，表明速率或使用限制已超出：

### 我们应用负载均衡器在 1 分钟周期内的临时吞吐量限制

此 429 是由于在 1 分钟窗口内超过每个服务密钥或 PAT 的固定 API 调用次数而导致的。窗口的开始时间会略有不同——不保证在整分钟开始——并且可能根据应用程序部署事件而变化。

收到最大事件数后，我们将返回 429，直到从评估窗口开始达到 60 秒，然后重复此过程。

此 429 由我们的应用程序负载均衡器抛出，是一种适用于所有 LangSmith 用户的机制，与计划层级无关，以确保所有用户服务的连续性。

| 方法            | 端点     | 限制 | 窗口   |
| ----------------- | ------------- | ----- | -------- |
| `DELETE`          | `/sessions*`  | 30    | 1 分钟 |
| `POST` 或 `PATCH` | `/runs*`      | 5000  | 1 分钟 |
| `GET`             | `/runs/:id`   | 30    | 1 分钟 |
| `POST`            | `/feedbacks*` | 5000  | 1 分钟 |
| `*`               | `*`           | 2000  | 1 分钟 |

<Note>
LangSmith SDK 采取措施，通过将来自单个会话 ID 的最多 100 个运行批处理到单个 API 调用中，来最大程度地减少达到这些运行相关端点限制的可能性。
</Note>

### 计划层级每小时追踪事件限制

此 429 是由于达到每小时最大摄入事件数而导致的，并在 UTC 时间每个整小时开始时的固定窗口中进行评估，并在每个新小时开始时重置。

此上下文中，事件是运行的创建或更新。如果运行被创建，然后在同一小时窗口中更新，则算作 2 个事件计入此限制。

此 429 由我们的应用程序抛出，并根据计划层级而有所不同，我们的 Startup/Plus 和 Enterprise 计划层级的组织比专为个人使用而设计的 Free 和 Developer 计划层级具有更高的每小时限制。

| 计划                             | 限制          | 窗口 |
| -------------------------------- | -------------- | ------ |
| Developer（未存档付款方式）   | 50,000 个事件  | 1 小时 |
| Developer（已存档付款方式） | 250,000 个事件 | 1 小时 |
| Startup/Plus                     | 500,000 个事件 | 1 小时 |
| Enterprise                       | 自定义         | 自定义 |

### 计划层级每小时追踪数据摄入限制

此 429 是由于达到追踪输入、输出和元数据的最大数据摄入量而导致的，并在 UTC 时间每个整小时开始时的固定窗口中进行评估，并在每个新小时开始时重置。

通常，输入、输出和元数据在运行创建和更新事件时都会被发送。如果运行在 2.0MB 时被创建，然后在同一小时窗口中更新到 3.0MB，则该限制计入 5.0MB 的存储量。

此 429 由我们的应用程序抛出，并根据计划层级而有所不同，我们的 Startup/Plus 和 Enterprise 计划层级的组织比专为个人使用而设计的 Free 和 Developer 计划层级具有更高的每小时限制。

| 计划                             | 限制  | 窗口 |
| -------------------------------- | ------ | ------ |
| Developer（未存档付款方式）   | 500MB  | 1 小时 |
| Developer（已存档付款方式） | 2.5GB  | 1 小时 |
| Startup/Plus                     | 5.0GB  | 1 小时 |
| Enterprise                       | 自定义 | 自定义 |

### 计划层级每月唯一追踪限制

此 429 是由于达到每月最大摄入追踪数而导致的，并在 UTC 时间每月日历月开始时的固定窗口中进行评估，并在每月开始时重置。

此 429 由我们的应用程序抛出，仅在 Developer 计划层级且没有存档付款方式时适用。

| 计划                           | 限制        | 窗口  |
| ------------------------------ | ------------ | ------- |
| Developer（未存档付款方式） | 5,000 条追踪 | 1 个月 |

### 自行配置的每月使用限制

此 429 是由于达到组织管理员配置的使用限制而导致的，并在 UTC 时间每月日历月开始时的固定窗口中进行评估，并在每月开始时重置。

此 429 由我们的应用程序抛出，并根据组织的配置设置而有所不同。

### 在您的应用程序中处理 429 响应

由于某些 429 响应是临时的，可能会在后续调用时成功，如果您直接在应用程序中调用 LangSmith API，我们建议实施带有指数退避和抖动的重试逻辑。

<Note>
请务必注意，如果您长时间使端点饱和，重试可能无效，因为您的应用程序最终将累积足够大的积压以耗尽所有重试次数。

如果是这种情况，我们希望更具体地讨论您的需求。请通过 [LangSmith 支持](https://support.langchain.com) 联系支持人员，并提供有关应用程序吞吐量需求和示例代码的详细信息，我们可以与您合作，更好地了解最佳方案是修复错误、更改应用程序代码，还是使用不同的 LangSmith 计划。
</Note>

## 使用限制

LangSmith 允许您配置追踪的使用限制。请注意，这些是*使用*限制，而不是*消费*限制，这意味着它们允许您限制某些事件发生的数量，而不是您将花费的总额。

LangSmith 允许您设置两个不同的每月限制，与前面数据保留指南中讨论的计费指标相对应：

* 所有追踪限制
* 扩展数据保留追踪限制

这些限制分别允许您限制总追踪数量和扩展数据保留追踪数量。

### 使用限制的特性

使用限制是近似值，这意味着我们不保证限制的精确性。在极少数情况下，在使用限制开始生效之前，可能会有一个小时间段，其中额外的追踪被处理超过限制阈值。

### 扩展数据保留追踪限制的副作用

扩展数据保留追踪限制有副作用。如果限制已达到，任何可能导致追踪层级自动升级的功能都会变得不可访问。这是因为追踪的自动升级会导致创建另一个扩展保留追踪，而根据限制，这是不允许的。因此，您将无法再：

1. 匹配运行规则
2. 向追踪添加反馈
3. 将运行添加到标注队列

这些功能中的每一个都可能导致自动升级，因此当达到限制时，我们会关闭它们。

### 更新使用限制

使用限制可以从 `使用和计费` 下的 `设置` 页面更新。限制值会被缓存，因此可能需要一两分钟才能应用新的限制。
