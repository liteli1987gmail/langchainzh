Agent 会生成代码、与文件系统交互并运行 shell 命令。由于我们无法预测 Agent 可能执行的操作，因此使其环境保持隔离非常重要，这样它就无法访问凭据、文件或网络。沙箱通过在 Agent 的执行环境与您的主机系统之间创建边界来提供这种隔离。

在 Deep Agents 中，**沙箱是定义 Agent 操作环境的[后端](/oss/deepagents/backends)**。与其他仅暴露文件操作的后端（State、Filesystem、Store）不同，沙箱后端还为 Agent 提供了用于运行 shell 命令的 `execute` 工具。配置沙箱后端后，Agent 将获得：

- 所有标准文件系统工具（`ls`、`read_file`、`write_file`、`edit_file`、`glob`、`grep`）
- 用于在沙箱中运行任意 shell 命令的 `execute` 工具
- 保护主机系统的安全边界

## 为什么使用沙箱？

沙箱用于安全目的。
它们允许代理执行任意代码、访问文件和使用网络，而不会危及您的凭据、本地文件或主机系统。
这种隔离在代理自主运行时至关重要。

沙箱在以下场景特别有用：

- 编码代理：自主运行的代理可以使用 shell、git、克隆仓库（许多提供商提供原生 git API，例如 [Daytona 的 git 操作](https://www.daytona.io/docs/en/git-operations/)），并运行 Docker-in-Docker 来执行构建和测试流水线
- 数据分析代理——加载文件、安装数据分析库（pandas、numpy 等）、运行统计计算，并在安全隔离的环境中创建 PowerPoint 演示文稿等输出

## 生命周期和作用域

大多数应用选择每个[线程](/langsmith/use-threads)一个沙箱（线程作用域），或在同一[助手](/langsmith/assistants)的每个线程之间共享一个沙箱（助手作用域）。

沙箱会消耗资源并产生费用，直到被关闭。请确保在不再使用沙箱时将其关闭。

有关完整生命周期表、异步[图工厂](/langsmith/graph-rebuild)注意事项、TTL 行为、LangGraph 部署连接和客户端示例，请参阅生产环境指南中的[沙箱生命周期](/oss/deepagents/going-to-production#lifecycle)。

### 线程作用域（默认）

每个对话获得自己的沙箱。第一次运行创建它；同一线程上的后续轮次复用它。当线程结束或沙箱 TTL 到期时，环境消失。按以下示例存储沙箱名称或元数据的映射，以便每次运行解析到同一个沙箱。

<Tip>
当用户可能在空闲时间后返回时，请在沙箱上配置 TTL，以便提供商自动删除或归档空闲环境。
</Tip>

## 集成模式

根据代理运行的位置，代理与沙箱集成有两种架构模式。

### 代理在沙箱内模式

代理在沙箱内运行，您通过网络与它通信。您构建一个预装了代理框架的 Docker 或 VM 镜像，在沙箱内运行它，并从外部连接发送消息。

优点：

- ✅ 与本地开发高度一致。
- ✅ 代理和环境之间紧密耦合。

权衡：

- 🔴 API 密钥必须放在沙箱内（安全风险）。
- 🔴 更新需要重新构建镜像。
- 🔴 需要通信基础设施（WebSocket 或 HTTP 层）。

要在沙箱中运行代理，请构建镜像并在其上安装 deepagents。

```dockerfile
FROM python:3.11
RUN pip install deepagents-code
```

然后在沙箱中运行 agent。
要在沙箱中使用 agent，你需要添加额外的基础设施来处理应用程序与沙箱内 agent 之间的通信。

### 将沙箱作为工具的模式

Agent 在你的机器或服务器上运行。当需要执行代码时，它会调用沙箱工具（如 `execute`、`read_file` 或 `write_file`），这些工具会调用提供商的 API 在远程沙箱中运行操作。

优势：

- ✅ 无需重新构建镜像即可即时更新 agent 代码。
- ✅ 更清晰的 agent 状态与执行分离。
    - API 密钥保持在沙箱外部。
    - 沙箱故障不会丢失 agent 状态。
    - 可选择并行运行多个沙箱中的任务。
- ✅ 仅需为执行时间付费。

权衡：

- 🔴 每次执行调用都会产生网络延迟。

:::python

```python Example
from deepagents import create_deep_agent
from deepagents.backends.langsmith import LangSmithSandbox
from dotenv import load_dotenv
from langsmith.sandbox import SandboxClient


load_dotenv()

# Can also do this with AgentCore, Daytona, E2B, Modal, Runloop, or Vercel

    model="google_genai:gemini-3.5-flash",
    backend=backend,
    system_prompt="You are a coding assistant with sandbox access. You can create and run code in the sandbox.",
)

try:
    result = agent.invoke(
        {
            "messages": [
                {
                    "role": "user",
                    "content": "Create a hello world Python script and run it",
                }
            ]
        }
    )
    print(result["messages"][-1].content)
finally:
    client.delete_sandbox(ls_sandbox.name)
```

```typescript Example
import "dotenv/config";
import { createDeepAgent, LangSmithSandbox } from "deepagents";
import { SandboxClient } from "langsmith/sandbox";

// Can also do this with Deno, Daytona, E2B, Modal, or Runloop
const client = new SandboxClient();

    "You are a coding assistant with sandbox access. You can create and run code in the sandbox.",
});

try {
  const result = await agent.invoke({
    messages: [
      {
        role: "user",
        content: "Create a hello world Python script and run it",
      },
    ],
  });
  const lastMessage = result.messages[result.messages.length - 1];
  console.log(
    typeof lastMessage.content === "string"
      ? lastMessage.content
      : String(lastMessage.content),
  );
} finally {
  await client.deleteSandbox(lsSandbox.name);
}
```

## 沙箱工作原理

### 隔离边界

所有沙箱提供商都会保护您的主机系统免受代理的文件系统和 shell 操作的影响。代理无法读取您的本地文件、访问您机器上的环境变量，或干扰其他进程。但是，单独的沙箱**无法**防护以下风险：

- **上下文注入**：控制代理部分输入的攻击者可以指示其在沙箱内运行任意命令。沙箱是隔离的，但代理在其中拥有完全控制权。
- **网络泄露**：除非网络访问被阻止，否则被注入上下文的代理可以通过 HTTP 或 DNS 将数据发送出沙箱。某些提供商支持阻止网络访问（例如 Modal 上的 `blockNetwork: true`）。

请参阅[安全注意事项](#security-considerations)了解如何处理密钥和降低这些风险。

### `execute` 方法

沙箱后端采用简单的架构：提供程序必须实现的唯一方法是 `execute()`，它运行 shell 命令并返回其输出。每个其他文件系统操作（`read`、`write`、`edit`、`ls`、`glob`、`grep`）都由 @[`BaseSandbox`] 基类构建在 `execute()` 之上，该基类构造脚本并通过 `execute()` 在沙箱内运行它们。

```mermaid
graph TB
    subgraph "Agent tools"
        Tools["ls, read_file, ..."]
        execute
    end

    BaseSandbox["BaseSandbox<br/>(uses execute)"] --> Tools
    execute_method["execute()"] --> BaseSandbox
    execute_method --> execute
    Provider["Provider SDK"] --> execute_method

    classDef process fill:#E5F4FF,stroke:#006DDD,stroke-width:2px,color:#030710
    classDef trigger fill:#F6FFDB,stroke:#6E8900,stroke-width:2px,color:#2E3900

    class Tools,execute process
    class BaseSandbox,execute_method process
    class Provider trigger
```

此设计意味着：
- **添加新的 provider 非常简单。** 只需实现 `execute()`——基类会处理其他所有事情。
- **`execute` 工具是有条件可用的。** 每次模型调用时，harness 会检查后端是否实现了 @[`SandboxBackendProtocol`]。如果没有实现，该工具会被过滤掉，agent 永远不会看到它。

当 agent 调用 `execute` 工具时，它提供 `command` 字符串并返回组合的 stdout/stderr、exit code，以及如果输出过大时的截断通知。

您也可以直接在应用代码中调用后端的 `execute()` 方法。

### 文件访问的两个层面

文件进出沙箱有两种不同的方式，理解何时使用每种方式很重要：

**代理文件系统工具**：`read_file`、`write_file`、`edit_file`、`ls`、`glob`、`grep` 和 `execute` 是 LLM 在执行过程中调用的工具。这些通过沙箱内的 `execute()` 运行。代理使用它们来读取代码、写入文件和运行命令，作为其任务的一部分。

**文件传输 API**：您的应用代码调用的 `uploadFiles()` 和 `downloadFiles()` 方法。这些使用提供商的原生文件传输 API（而非 shell 命令），用于在您的宿主机环境和沙箱之间移动文件。使用这些方法可以：

- 在代理运行前，使用源代码、配置或数据**填充沙箱**
- 在代理完成后，**检索产物**（生成的代码、构建输出、报告）
- **预填充代理将需要的依赖项**

## 使用文件

### 初始化沙盒

使用 `uploadFiles()` 在代理运行之前填充沙盒。文件内容以 `Uint8Array` 形式提供：

:::

```typescript
const encoder = new TextEncoder();
const responses = await sandbox.uploadFiles([
  ["src/index.js", encoder.encode("console.log('Hello')")],
  ["package.json", encoder.encode('{"name": "my-app"}')],
]);

// Each response indicates success or failure
for (const res of responses) {
  if (res.error) {
    console.error(`Failed to upload ${res.path}: ${res.error}`);
  }
}
```

### 检索 artifacts

使用 `downloadFiles()` 在代理完成后从沙箱中检索文件：

```typescript
const results = await sandbox.downloadFiles(["src/index.js", "output.txt"]);

const decoder = new TextDecoder();
for (const result of results) {
  if (result.content) {
    console.log(`${result.path}: ${decoder.decode(result.content)}`);
  } else {
    console.error(`Failed to download ${result.path}: ${result.error}`);
  }
}
```

<Note>

## 安全注意事项

沙箱将代码执行与主机系统隔离开来，但它们不能防御**上下文注入**。攻击者如果控制了代理输入的一部分，可以指示代理读取文件、运行命令或从沙箱中窃取数据。这使得沙箱内的凭据尤其危险。

<Warning>
**永远不要将密钥放在沙箱内。** API 密钥、令牌、数据库凭据以及其他通过环境变量、挂载文件或 `secrets` 选项注入到沙箱中的密钥都可能被上下文注入的代理读取和窃取。这同样适用于短期或范围受限的凭据——如果代理可以访问它们，攻击者也可以。
</Warning>

### 安全处理密钥

如果您的代理需要调用需要认证的 API 或访问受保护的资源，您有两个选择：

1. **将密钥放在沙箱外部的工具中。** 定义在您的主机环境中运行（而不是在沙箱内）的工具，并在那里处理身份验证。代理通过名称调用这些工具，但永远看不到凭据。这是推荐的方法。

2. **使用在请求中注入凭据的网络代理。** 某些沙箱提供商支持代理，这些代理拦截来自沙箱的传出 HTTP 请求，并在转发之前附加凭据（例如 `Authorization` 头）。代理永远不会看到密钥，它只是向某个 URL 发出普通请求。这种方法尚未在提供商中广泛采用。

<Warning>
如果您必须将密钥注入沙箱（不推荐），请采取以下预防措施：

- 为**所有**工具调用启用[人工介入](/oss/deepagents/human-in-the-loop)批准，而不仅仅是敏感的工具调用
- 阻止或限制沙箱的网络访问以限制数据泄露路径
- 使用尽可能最小的凭据范围和尽可能短的生存期
- 监控沙箱网络流量以发现意外的出站请求

即使有这些保护措施，这仍然是不安全的解决方法。足够有创意的上下文注入攻击可以绕过输出过滤和 HITL 审查。
</Warning>

### 一般最佳实践

- 在应用程序中执行沙箱输出之前先进行审查
- 不需要时阻止沙箱网络访问
- 使用[中间件](/oss/langchain/middleware)过滤或编辑工具输出中的敏感模式
- 将沙箱内生成的所有内容视为不受信任的输入
