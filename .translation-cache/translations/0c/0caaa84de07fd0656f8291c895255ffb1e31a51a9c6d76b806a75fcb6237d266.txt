代理会生成代码、与文件系统交互并运行 shell 命令。由于我们无法预测代理可能执行的操作，因此确保其环境隔离非常重要，这样它就无法访问凭据、文件或网络。沙箱通过在代理的执行环境与您的主机系统之间创建边界来提供这种隔离。

在 Deep Agents 中， **沙箱是 [后端](/oss/deepagents/backends)** ，用于定义代理操作的环境。与其他后端（State、Filesystem、Store）不同，它们仅暴露文件操作，而沙箱后端还为代理提供了一个 `execute` 工具，用于运行 shell 命令。当您配置沙箱后端时，代理会获得：

- 所有标准文件系统工具（`ls`, `read_file`, `write_file`, `edit_file`, `glob`, `grep`)
- 一个 `execute` 工具，用于在沙箱中运行任意 shell 命令
- 保护您主机系统的安全边界

## 为什么要使用沙箱？

沙箱用于安全目的。
它们允许代理执行任意代码、访问文件和使用网络，而不会危及您的凭据、本地文件或主机系统。
当代理自主运行时，这种隔离至关重要。

沙箱特别适用于：

- 编码代理：自主运行的代理可以使用 shell、git、克隆仓库（许多提供商提供原生 git API，例如， [Daytona 的 git 操作](https://www.daytona.io/docs/en/git-operations/)），并运行 Docker-in-Docker 以进行构建和测试管道
- 数据分析代理——加载文件、安装数据分析库（pandas、numpy 等）、运行统计计算，并在安全隔离的环境中创建 PowerPoint 演示文稿等输出

## 集成模式

根据代理运行的位置，有两种将代理与沙箱集成的架构模式。

### 沙箱中的代理模式

代理在沙箱内运行，您通过网络与其通信。您构建包含预装代理框架的 Docker 或 VM 镜像，在沙箱内运行它，并从外部连接发送消息。

Benefits:

- ✅ 与本地开发高度一致。
- ✅ 代理与环境紧密耦合。

Trade-offs:

- 🔴 API 密钥必须存放在沙箱内（安全风险）。
- 🔴 更新需要重新构建镜像。
- 🔴 需要通信基础设施（WebSocket 或 HTTP 层）。

要在沙箱中运行代理，请构建镜像并在其上安装 deepagents。

```dockerfile
FROM python:3.11
RUN pip install deepagents-code
```

然后在沙箱内运行代理。
要在沙箱内使用代理，您必须添加额外的基础设施来处理应用程序与沙箱内代理之间的通信。

### 沙箱即工具模式

代理在你的机器或服务器上运行。当它需要执行代码时，会调用沙箱工具（例如， `execute`, `read_file`, or `write_file`）这些工具会调用提供商提供的 API，在远程沙箱中执行操作。

Benefits:

- ✅ 无需重新构建镜像，即可即时更新代理代码。
- ✅ 代理状态与执行更清晰地分离。
    - API 密钥保持在沙箱外部。
    - 沙箱故障不会丢失代理状态。
    - 支持在多个沙箱中并行运行任务。
- ✅ 仅按执行时间计费。

Trade-offs:

- 🔴 每次执行调用都有网络延迟。

## 沙箱工作原理

### 隔离边界

所有沙箱提供商都会保护你的主机系统免受代理的文件系统和 shell 操作的影响。代理无法读取你的本地文件、访问你机器上的环境变量或干扰其他进程。然而，仅靠沙箱本身无法 **不能** 防止以下风险：

- **上下文注入**：攻击者若能控制代理输入的一部分，可指示其在沙箱内运行任意命令。沙箱是隔离的，但代理在其中拥有完全控制权。
- **网络数据泄露**：除非网络访问被阻止，否则通过上下文注入的代理可通过 HTTP 或 DNS 将数据从沙箱中发送出去。部分提供商支持阻止网络访问（例如， `blockNetwork: true` 在 Modal 上）。

参见 [安全注意事项](#security-considerations) 了解如何处理密钥和降低这些风险。

### 该 `execute` 方法

沙箱后端采用简单的架构：提供商只需实现一个 `execute()`方法，该方法运行 shell 命令并返回输出。所有其他文件系统操作（`read`, `write`, `edit`, `ls`, `glob`, `grep`）都基于 `execute()` 构建，由 @[`BaseSandbox`] 基类提供，该类构造脚本并通过 `execute()`.

```mermaid
graph TB
    subgraph "Agent tools"
        Tools["ls, read_file, ..."]
        execute
    end

    BaseSandbox["BaseSandbox<br/>(uses execute)"] --> Tools
    execute_method["execute()"] --> BaseSandbox
    execute_method --> execute

    classDef process fill:#E5F4FF,stroke:#006DDD,stroke-width:2px,color:#030710
    classDef trigger fill:#F6FFDB,stroke:#6E8900,stroke-width:2px,color:#2E3900

    class Tools,execute process
    class BaseSandbox,execute_method process
    class Provider trigger
```

在沙箱内运行它们。这一设计意味着：
- **添加新的提供商非常简单。** 只需实现 `execute()`——基类会处理其他所有事情。
- ** `execute` 工具是有条件可用的。** 每次模型调用时，测试工具会检查后端是否实现了 @[`SandboxBackendProtocol`]。如果没有实现，该工具会被过滤掉，代理永远不会看到它。

当代理调用 `execute` 工具时，它会提供一个 `command` string and gets back the combined stdout/stderr, exit code, and a truncation notice if the output was too large.

你也可以在应用代码中直接调用后端的 `execute()` 方法。

### 文件访问的两个层面

文件进出沙盒有两种不同的方式，理解何时使用每种方式很重要：

**代理文件系统工具**: `read_file`, `write_file`, `edit_file`, `ls`, `glob`, `grep`，和 `execute` 是 LLM 在执行期间调用的工具。这些通过 `execute()` 在沙盒内部运行。代理使用它们来读取代码、写入文件和运行命令作为其任务的一部分。

**文件传输 API**： `uploadFiles()` 和 `downloadFiles()` 方法是您的应用程序代码调用的。这些使用提供商的原生文件传输 API（而不是 shell 命令），旨在在您的主机环境和沙盒之间移动文件。使用这些来：
- **填充沙盒** 在代理运行之前填充源代码、配置或数据
- **检索产物** （生成的代码、构建输出、报告）在代理完成后
- **预填充依赖项** 代理将需要的

## 处理文件

### 填充沙盒

使用 `uploadFiles()` 在代理运行之前填充沙盒。文件内容作为 `Uint8Array`:

```typescript
const encoder = new TextEncoder();
const responses = await sandbox.uploadFiles([
  ["src/index.js", encoder.encode("console.log('Hello')")],
  ["package.json", encoder.encode('{"name": "my-app"}')],
]);

// Each response indicates success or failure
for (const res of responses) {
  if (res.error) {
    console.error(`Failed to upload ${res.path}: ${res.error}`);
  }
}
```

### 检索产物

使用 `downloadFiles()` 在代理完成后从沙盒中检索文件：

```typescript
const results = await sandbox.downloadFiles(["src/index.js", "output.txt"]);

const decoder = new TextDecoder();
for (const result of results) {
  if (result.content) {
    console.log(`${result.path}: ${decoder.decode(result.content)}`);
  } else {
    console.error(`Failed to download ${result.path}: ${result.error}`);
  }
}
```

<Note>
在沙盒内部，代理使用自己的文件系统工具（`read_file`, `write_file`）：而不是 `uploadFiles` or `downloadFiles`。这些方法用于您的应用程序代码在您的主机和沙盒之间移动文件。
</Note>

deepagents 沙盒后端支持文件传输 API，用于在您的应用程序和沙盒之间移动文件。

### 填充沙盒

        backend = LangSmithSandbox(sandbox=ls_sandbox)

        backend.upload_files(
            [

### 检索产物

使用 `download_files()` 在代理完成后从沙盒中检索文件：

<Tabs>
    <Tab title="LangSmith">
        ```python
        from langsmith.sandbox import SandboxClient

        from deepagents.backends.langsmith import LangSmithSandbox

        client = SandboxClient()
        ls_sandbox = client.create_sandbox(template_name="deepagents-deploy")
        backend = LangSmithSandbox(sandbox=ls_sandbox)

        results = backend.download_files(["/src/index.py", "/output.txt"])
        for result in results:
            if result.content is not None:
                print(f"{result.path}: {result.content.decode()}")
            else:
                print(f"Failed to download {result.path}: {result.error}")
        ```

## 安全注意事项

沙盒将代码执行与您的主机系统隔离，但它们不能防御 **上下文注入**。控制代理部分输入的攻击者可以指示它读取文件、运行命令或从沙盒内泄露数据。这使得沙盒内的凭证尤其危险。

<Warning>
**永远不要将敏感信息放在沙盒中。** API 密钥、令牌、数据库凭证以及其他注入到沙箱中的密钥（通过环境变量、挂载文件或 `secrets` 选项）可以被上下文注入的代理读取和窃取。即使是短期或范围受限的凭证也适用——如果代理能访问，攻击者也能访问。
</Warning>

### 安全处理密钥

如果您的代理需要调用经过身份验证的 API 或访问受保护的资源，您有两个选择：

1. **将密钥保存在沙箱外部的工具中。** 在您的主机环境中（而非沙箱内部）定义处理身份验证的工具。代理通过名称调用这些工具，但永远不会看到凭证。这是推荐的方法。

2. **使用注入凭证的网络代理。** 某些沙箱提供商支持代理，这些代理会拦截来自沙箱的传出 HTTP 请求并附加凭证（例如， `Authorization` 标头）后再转发。代理永远不会看到密钥——它只是向 URL 发出纯文本请求。这种方法尚未在提供商中广泛可用。

<Warning>
如果您必须将密钥注入沙箱（不推荐），请采取以下预防措施：

- 启用 [human-in-the-loop](/oss/deepagents/human-in-the-loop) 所有 **工具调用的** 审批，而不仅仅是敏感操作
- 阻止或限制沙箱的网络访问以限制数据泄露路径
- 使用尽可能窄的凭证范围和尽可能短的有效期
- 监控沙箱网络流量以发现意外出站请求

即使有这些安全措施，这仍然是不安全的变通方法。足够有创意的上下文注入攻击可以绕过输出过滤和 HITL 审查。
</Warning>

### 通用最佳实践

- 在应用程序中处理沙箱输出之前先进行审查
- 不需要时阻止沙箱网络访问
- 使用 [中间件](/oss/langchain/middleware) 过滤或编辑工具输出中的敏感模式
- 将沙箱内生成的所有内容视为不可信输入
