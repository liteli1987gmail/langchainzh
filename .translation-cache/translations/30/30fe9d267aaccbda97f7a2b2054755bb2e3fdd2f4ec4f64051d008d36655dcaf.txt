断言将审查者的英语标准转化为自动化检查。它们是关于正确答案应该或不应该包含什么的简短、自由格式的声明。你在[单次运行注释队列](/langsmith/annotation-queues#single-run-annotation-queues)中审查运行时编写这些断言，LangSmith 会将每个断言保存到[数据集示例](/langsmith/example-data-format)中。然后任何[离线评估器](/langsmith/evaluation-concepts#offline-evaluations)都可以检查你应用程序的新输出是否满足每个声明。

在以下情况下使用断言：

- 运行的实际输出是错误的，你更愿意描述正确答案的样子，而不是手动编写一个。
- 你想在不离开审查流程的情况下用纯英语捕获验收标准。

<Note>
断言适用于[单次运行注释队列](/langsmith/annotation-queues#single-run-annotation-queues)。[成对队列](/langsmith/annotation-queues#pairwise-annotation-queues)保持不变。断言仅在 LangSmith UI 中可用。
</Note>

<Tip>
[LangSmith Engine](/langsmith/engine#add-offline-examples)可以为标记为重复问题的生产追踪自动提出断言。打开问题的离线示例流程，在将 Engine 提出的断言保存到数据集之前进行审查、编辑或扩展。
</Tip>

## 添加断言

1. 在 [LangSmith UI](https://smith.langchain.com) 中，在左侧边栏导航到 **Annotation Queues**。打开一个单次运行队列并选择一个运行。
2. 在侧边面板中，找到 **Feedback** 下方的 **Assertions** 部分。
3. 点击 **+ Add** 创建断言行。
4. 输入一个总结该声明的 **key**（例如 `must_cite_source`、`must_not_invent_url`）和一句描述该声明的 **comment**。

    key 是自由格式的。`must_` / `must_not_` 前缀只是一种命名约定；LangSmith 不会对它们进行特殊处理。

5. 为你希望捕获的每个标准重复步骤 3 和 4。

    运行编辑器在断言侧面板旁边显示运行的输入和输出。一旦你添加了至少一个断言，运行编辑器的 **Outputs** 面板就会从运行的实际输出切换到你所添加断言的只读预览。此预览会被保存到数据集。运行的原始输出不会被保存，因为断言描述的是正确答案应该包含什么，而不是这个运行产生了什么。

    你可以随时继续编辑运行的 **Inputs**，例如在保存示例之前优化提示。当任何断言存在时，**Outputs** 面板会保持锁定在断言预览上。

6. 在侧边面板页脚中点击 **Add to Dataset & Next**（键盘快捷键：macOS 上为 <kbd>⌘ Enter</kbd>，其他平台上为 <kbd>Ctrl Enter</kbd>）。LangSmith 会将当前运行添加到队列的[默认数据集](/langsmith/annotation-queues#basic-details)，如果没有配置默认数据集，则会提示你选择一个。然后队列会将你移动到下一个运行。

保存的示例的 `outputs` 字段存储为 JSON。例如：

```json
{
  "assertions": [
    {
      "key": "must_cite_source",
      "comment": "The response cites the source URL it is drawing from."
    },
    {
      "key": "must_not_invent_url",
      "comment": "The response does not include URLs that do not appear in the inputs."
    }
  ]
}
```

示例的 `inputs` 字段存储运行时的输入，或者如果您更改了它们，则存储您编辑后的版本。请参阅[示例数据格式](/langsmith/example-data-format)了解已保存示例的完整结构。

## 根据断言进行评估

编写一个[离线评估器](/langsmith/evaluation-concepts#offline-evaluations)，用于读取 `reference_outputs["assertions"]` 中保存的断言，并为每个断言返回一个反馈分数。最小形状为：

```python
def grade_against_assertions(outputs: dict, reference_outputs: dict) -> list[dict]:
    """Return one feedback score per assertion."""
    feedback = []
    for assertion in reference_outputs["assertions"]:
        # Replace with your scoring logic: LLM judge, regex, schema check, and so on.
        score = ...
        feedback.append({"key": assertion["key"], "score": score})
    return feedback
```

如何对每个声明进行评分由你决定。三种模式很常见，可以在一个评估器中组合使用：

- **[LLM-as-a-judge](/langsmith/llm-as-judge)**: 对于每个断言，用应用程序的输出和断言的 `comment` 提示模型，并让它返回一个分数。最适用于声明是主观的或难以机械验证的情况。
- **[基于代码的检查](/langsmith/code-evaluator-ui)**: 对于每个断言，运行一个基于断言 `key` 的确定性检查，例如正则匹配、模式验证或子字符串存在性检查。最适用于声明有明确的、机械化答案的情况。
- **[部分信用评分](/langsmith/multiple-scores)**: 返回一个数值分数（例如在 0.0 到 1.0 之间）而不是布尔值，以进行分级评分，并对满足部分但非全部声明的输出给予"部分信用"。
