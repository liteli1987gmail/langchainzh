## 可观测性

### *我无法在 UI 中创建 API 密钥或管理用户，这是怎么回事？*

* 您可能是在未设置 SSO 的情况下部署了 LangSmith。LangSmith 需要 SSO 来管理用户和 API 密钥。您可以在以下位置找到有关设置 SSO 的更多信息 [配置部分。](/langsmith/self-host-sso)

### *How does load balancing/ingress work*?

* You will need to expose the frontend container/service to your applications/users. This will handle routing to all downstream services.
* 您需要在入口层面终止 SSL。我们建议使用托管服务，如 AWS ALB、GCP Load Balancer 或 Nginx。

### *我们如何对应用程序进行身份验证？*

* 目前，我们的自托管解决方案支持使用 OAuth2.0 和 OIDC 作为身份验证解决方案的 SSO。请注意，我们确实提供无身份验证解决方案，但强烈建议在投入生产之前设置 OAuth。

您可以在以下位置找到有关设置 SSO 的更多信息 [配置部分。](/langsmith/self-host-sso)

### *我可以使用外部存储服务吗？*

* 您可以将 LangSmith 配置为使用所有存储服务的外部版本。在生产环境中，我们强烈建议使用外部存储服务。请查看 [配置部分](/langsmith/self-hosted) 获取更多信息。

### *我的应用程序是否需要出站流量才能正常运行？*

我们的部署只需要少量出站流量（其中大部分可以在您的 VPC 内部）：

* 获取镜像（如果您在镜像自己的镜像，则可能不需要）

* 与任何 LLM 端点通信

* 与您可能已配置的任何外部存储服务通信

* 获取 OAuth 信息

* 订阅指标和操作元数据（如果不在离线模式下运行）

  * 需要出站连接到 `https://beacon.langchain.com`
  * 请参阅 [出站流量](/langsmith/self-host-egress) 获取更多信息

您的 VPC 可以设置规则来限制任何其他访问。注意：我们要求 `X-Organization-Id` 和 `X-Tenant-Id` 标头必须允许传递到后端服务。这些用于确定请求针对的是哪个组织和工作区（前称为"租户"）。

### *应用程序的资源要求？*

* 在 Kubernetes 中，我们建议使用最小 helm 配置，您可以在 [中等规模示例](https://github.com/langchain-ai/helm/blob/main/charts/langsmith/examples/medium_size.yaml)中看到。对于 Docker，我们建议至少 16GB 内存和 4 个 CPU。
* 对于 Postgres，我们建议至少 8GB 内存和 2 个 CPU。
* 对于 Redis，我们建议 4GB 内存和 2 个 CPU。
* 对于 Clickhouse，我们建议 32GB 内存和 8 个 CPU。

### SAML SSO 常见问题

#### *如何更改 SAML SSO 用户的电子邮件地址？*

某些身份提供商会保留原始 `User ID` 通过电子邮件更改，而其他则不会，所以我们建议您遵循以下步骤来避免 LangSmith 中出现重复用户：

1. 从组织中删除该用户（请参阅 [管理用户](/langsmith/set-up-hierarchy#manage-users))
2. 在 IdP 中更改其电子邮件地址
3. 让他们再次通过 SAML SSO 登录 LangSmith - 这将触发通常的 [实时配置](/langsmith/user-management#just-in-time-jit-provisioning) 使用新电子邮件地址的流程

目前不支持通过 SCIM 或其他方式更改具有多个关联登录方法的用户的电子邮件地址。显示此错误消息： `email update not supported with linked login methods`. For example, if a user previously logged in via email/password or Google social login, and then is added with the same email address via SSO, changing their email address is not supported. This applies to both self-hosted and cloud.

#### *我可以更换身份提供商吗？*

通过我们的门户联系 LangChain 支持团队 [https://support.langchain.com](https://support.langchain.com) 获取迁移支持。

#### *如何修复"405 方法不允许"？*

确保您使用的是正确的 ACS URL： [https://auth.langchain.com/auth/v1/sso/saml/acs](https://auth.langchain.com/auth/v1/sso/saml/acs)

### SCIM 常见问题

#### *我可以在不使用 SAML SSO 的情况下使用 SCIM 吗？*

* **云**: 不，云部署中 SCIM 需要 SAML SSO
* **Self-hosted**: 是，SCIM 可与使用客户端密钥身份验证模式的 OAuth 配合使用

#### *如果同时启用 JIT 配置和 SCIM 会怎样？*

JIT 配置和 SCIM 可能会相互冲突。我们建议在启用 SCIM 之前禁用 JIT 配置，以确保一致的用户配置行为。

#### *如何更改用户的角色或工作区访问权限？*

在您的 IdP 中更新用户的组成员身份。变更将根据以下规则同步到 LangSmith [角色优先级规则](/langsmith/user-management#role-precedence).

#### *当用户从所有组中移除时会发生什么？*

用户将根据您的 IdP 取消配置设置从您的 LangSmith 组织中移除。

#### *我可以使用自定义组名称吗？*

可以。如果您的身份提供者支持将备用字段同步到 `displayName` 组属性，您可以使用备用属性（例如 `description`）作为 `displayName` 在 LangSmith 中，并保留身份提供者组名的完整自定义功能。否则，组必须遵循中的特定命名约定 [组命名约定](/langsmith/user-management#group-naming-convention) 部分才能正确映射到 LangSmith 角色和工作区。

您还可以 [配置自定义分隔符](/langsmith/user-management#configure-custom-separator) (e.g., `-`, `_`, `&`）而不是默认的冒号（`:`）来适应对组名称字符有限制的身份提供者。

#### _为什么我的 Okta 集成不工作？_

See Okta's troubleshooting guide here: https://help.okta.com/en-us/content/topics/users-groups-profiles/usgp-group-push-troubleshoot.htm.

### *支持降级吗？*

降级不受官方支持。LangSmith 升级可能包含数据库迁移和其他不向后兼容的变更。如果您需要回滚到之前的版本，请通过 [支持门户](https://support.langchain.com) 联系技术支持获取指导。

## 部署

### 我需要使用 LangChain 才能使用 LangGraph 吗？两者有什么区别？

不需要。LangGraph 是一个用于复杂代理系统的编排框架，比 LangChain 代理更底层且更可控。LangChain 提供了与模型和其他组件交互的标准接口，适用于直接的链式调用和检索流程。

### LangGraph 与其他代理框架有何不同？

其他代理框架可能适用于简单的通用任务，但在满足公司特定需求的复杂任务方面存在不足。LangGraph 提供了更具表现力的框架来处理公司的独特任务，不会将用户限制在单一的黑盒认知架构中。

### LangGraph会影响我的应用程序性能吗？

LangGraph不会给你的代码增加任何开销，并且专门为流式工作流设计。

### LangGraph是开源的吗？它是免费的吗？

是的。LangGraph是一个MIT许可的开源库，可以免费使用。

### LangGraph和LangSmith有什么不同？

LangGraph是一个有状态的编排框架，为代理工作流带来额外的控制。LangSmith是一项用于部署和扩展代理应用程序的服务，提供构建代理用户体验的意见性API，以及集成的开发者UI。

| 功能            | LangGraph（开源）                                   | LangSmith                                                                                     |
|---------------------|-----------------------------------------------------------|--------------------------------------------------------------------------------------------------------|
| 描述         | 面向代理应用的有状态编排框架 | 用于部署LangGraph应用的可扩展基础设施                                           |
| SDK             | Python和JavaScript                                     | Python和JavaScript                                                                                  |
| HTTP API           | 无                                                      | 是的——可用于检索和更新状态或长期记忆，或创建可配置的助手 |
| 流式传输           | 基础                                                     | 专用模式逐token传输消息                                                             |
| 检查点        | 社区贡献                                     | 开箱即用支持                                                                               |
| 持久层   | 自主管理                                              | 使用高效存储的托管Postgres                                                                |
| 部署          | 自主管理                                              | • 云 <br /> • 免费自托管 <br /> • 企业版（付费自托管）                      |
| 可扩展性         | 自主管理                                              | 任务队列和服务器自动扩展                                                                |
| 容错     | 自主管理                                              | 自动重试                                                                                      |

### LangSmith是开源的吗？

不是。LangSmith是专有软件。

有一个免费的自托管版本提供基本功能。云部署选项和自托管部署选项是付费服务。 [联系我们的销售团队](https://www.langchain.com/contact-sales) 了解更多。

更多信息，请参阅我们的 [LangSmith定价页面](https://www.langchain.com/pricing).

### LangGraph是否适用于不支持工具调用的LLM？

可以的！您可以将 LangGraph 与任何 LLM 一起使用。我们使用支持工具调用的 LLM 的主要原因是，这通常是让 LLM 决定下一步做什么的最方便的方式。如果您的 LLM 不支持工具调用，您仍然可以使用它——只需要编写一些逻辑将原始 LLM 字符串响应转换为关于下一步做什么的决策。

### LangGraph 是否支持开源 LLM？

可以的！LangGraph 与底层使用的 LLM 完全无关。我们在大多数教程中使用闭源 LLM 的主要原因是它们无缝支持工具调用，而开源 LLM 通常不支持。但工具调用不是必需的（请参阅 [LangGraph 是否支持不支持工具调用的 LLM？](#does-langgraph-work-with-llms-that-dont-support-tool-calling)），因此您完全可以将 LangGraph 与开源 LLM 一起使用。

### 我可以在不登录 LangSmith 的情况下使用 Studio 吗？

可以的！您可以使用 [Agent Server 的开发版本](/langsmith/local-dev-testing) 在本地运行后端。
这将连接到作为 LangSmith 的一部分托管的 Studio 前端。
如果您设置环境变量 `LANGSMITH_TRACING=false`，则不会向 LangSmith 发送任何追踪。

### 什么是部署运行？

部署运行是通过 LangSmith 部署的 LangGraph 代理的一次端到端调用。节点和子图不会单独计费。对其他 LangGraph 代理的调用（通过 RemoteGraph 或 LangGraph SDK 或直接通过 API）会单独计费，计费到托管被调用代理的部署。人机交互的中断会在恢复时创建一个单独的部署运行。
