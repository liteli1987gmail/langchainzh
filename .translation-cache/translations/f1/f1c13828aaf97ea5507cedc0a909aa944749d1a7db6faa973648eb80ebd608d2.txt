Deep Agents Code 将其配置存储在 `~/.deepagents/` 目录中。主要配置文件包括：

| 文件 | 格式 | 用途 |
|------|--------|---------|
| `config.toml` | TOML | 模型默认值、提供商设置、构造函数参数、配置覆盖、主题、更新设置 |
| `.env` | Dotenv | 全局 API 密钥、密钥和其他环境变量 |
| `hooks.json` | JSON | 外部工具订阅 Deep Agents Code 生命周期事件 |
| `.mcp.json` | JSON | 全局 MCP 服务器定义 |

<Note>
    位于 `~/.deepagents/.state/` 下的文件保存每台机器的 Deep Agents Code 状态，由系统自动管理。
</Note>

---

## 检查配置

 `dcode config` 命令组报告当前生效的配置以及每个值的来源，无需启动会话。这对于确认环境变量或 `config.toml` 设置是否被正确读取，以及在错误报告中分享经过编辑的快照。

| 命令 | 描述 |
|---------|-------------|
| `dcode config show` | 根据实际环境解析每个选项，并 `config.toml`打印实际值及其来源 |
| `dcode config list` （别名 `ls`） | 列出所有可用选项及其类型、默认值和可设置位置，不解析值 |
| `dcode config get <key>` | 显示单个选项的实际值和来源，例如 `dcode config get interpreter.memory_limit_mb` |
| `dcode config path` | 显示磁盘上配置文件的路径（`config.toml`，项目和全局 `.env`, `hooks.json`，以及托管状态文件）以及每个文件是否存在 |

每个选项按以下顺序从第一个已设置的来源解析： `DEEPAGENTS_CODE_`前缀环境变量、规范环境变量， `config.toml`，然后是内置默认值。

所有四个命令都支持 `--json` 以输出机器可读的格式。

<Warning>
    Provider credentials and other secrets are reported as configured / not configured only—their values are never printed by `config show` or `config get`，因此输出可以安全地粘贴到错误报告中。
</Warning>

## 提供商凭证

Deep Agents Code 需要为你使用的每个模型提供商提供 API 密钥。添加密钥的推荐方式是使用 [`/auth`](#use-%2Fauth-recommended) 凭证管理器。对于非交互式运行，可以使用 [`dcode auth`](#manage-credentials-from-the-shell-dcode-auth) 或设置 [环境变量](#environment-variables-ci-and-headless) instead.

如果在多个位置设置了相同的密钥，请参阅 [密钥解析顺序](#key-resolution-order) 以了解哪个优先级更高。

### 使用 `/auth` （推荐）

从任何会话中打开凭证管理器：

```txt
/auth
```

该管理器列出了你的安装中可用的 LLM 提供商，以及 Tavily 网络搜索等非模型服务，并标记已设置密钥的提供商。选择一个提供商来添加或替换其密钥，或删除已存储的密钥。你在此处保存的密钥会在会话之间保持持久化。

<Accordion title="Provider row labels" icon="list-check">
    每一行显示提供商名称及其密钥来源：

    | 标签 | 含义 |
    |-------|---------|
    | `[stored]` 在此管理器中保存的密钥 `/auth` |
    | `[env: VARNAME]` 密钥来自环境变量 `VARNAME` （解析后的名称，例如 `DEEPAGENTS_CODE_OPENAI_API_KEY` or `OPENAI_API_KEY`) |
    | `[missing]` | 未存储密钥且环境变量未设置；选择该行以粘贴一个 |
</Accordion>

该 `/auth` 提示还具有可选的 **基础 URL** 字段。留空使用提供商的默认端点，或设置自定义端点与此密钥配合使用。基础 URL 与密钥一起保存。请参阅 [端点、密钥和网关](#endpoints-keys-and-gateways) 了解端点解析方式，包括与网关的配合。

<Warning>
    存储的基础 URL 不是秘密，可能会被记录；与它配对的密钥绝不会记录。
</Warning>

<Note>
    密钥仅限于此机器上的用户账户——Deep Agents Code 绝不会将它们传输到除配置的提供商 API 之外的任何地方。
</Note>

#### 使用 ChatGPT 登录

选择 `openai_codex` 中的提供商 `/auth` 会启动浏览器登录而非提示输入 API 密钥，使您能够通过 ChatGPT 订阅使用 OpenAI 模型。若要重新认证或退出登录，请选择 `openai_codex` 再次。请参阅 [使用 ChatGPT 登录（Codex 模型）](/oss/deepagents/code/providers) 了解完整流程。

`/auth` 管理 **LLM 提供商** 凭证和 **Tavily 网络搜索密钥**。在此输入的 Tavily 密钥与您的提供商密钥一起存储，并在下次启动时激活网络搜索——请参阅 [使用 Tavily 启用网络搜索](#enable-web-search-with-tavily)。其他工具凭证（如 `LANGSMITH_API_KEY` （追踪））则从环境读取—— [在中设置它们 `~/.deepagents/.env` 或您的 shell](#environment-variables).

### 从 shell 管理凭证（`dcode auth`)

该 `dcode auth` 命令组是的可编写脚本的等价物 `/auth` manager: it manages the same stored credentials without launching the TUI, which makes it usable for dotfile bootstrap, CI/CD, and setting a key on a remote box over SSH. The subcommands mirror the modal's verbs:

| 命令 | 描述 |
|---------|-------------|
| `dcode auth list` （别名 `ls`）| 列出每个已知提供商及其密钥解析位置 |
| `dcode auth status <provider>` | 打印一个提供商的解析来源 |
| `dcode auth set <provider>` | 存储 API 密钥，默认从 stdin 读取 |
| `dcode auth remove <provider>` （别名 `rm`, `delete`）| 删除存储的凭证 |
| `dcode auth path` | 打印凭证存储的解析路径（`auth.json`) |

`set` 从读取密钥 **stdin** 默认情况下，因此永远不会进入 shell 历史记录或 `argv`。通过管道输入密钥，或使用 `--from-env VAR` 从进程环境变量中复制它：

```bash
# Pipe the key in (stdin)
echo "$ANTHROPIC_API_KEY" | dcode auth set anthropic

# Copy it from an existing environment variable
dcode auth set openai --from-env OPENAI_API_KEY
```

<Note>
    `set` 拒绝在交互式终端中运行，这样意外调用不会挂起等待输入——通过 stdin 管道输入密钥或使用 `--from-env VAR`。存储的密钥通过相同的存储作为 `/auth`，因此警告（例如关于文件权限 `auth.json`）会打印到 stderr。
</Note>

移除存储的密钥或打印存储位置：

```bash
dcode auth remove anthropic
dcode auth path
```

<Note>
    `dcode auth set` 仅管理 API 密钥。 `openai_codex` 提供商使用 ChatGPT 浏览器登录而非 API 密钥，因此请运行 [`/auth` 并选择 `openai_codex`](#sign-in-with-chatgpt) 进行登录。 `dcode auth remove openai_codex` 确实会退出登录。
</Note>

### 环境变量（CI 和无头模式）

For non-interactive runs, CI/CD pipelines, or anywhere a TUI isn't available, export the provider's env var in your shell:

```bash
export ANTHROPIC_API_KEY="sk-ant-..."
export OPENAI_API_KEY="sk-..."

# Prefix with DEEPAGENTS_CODE_ to scope a key to Deep Agents Code only,
# leaving a shared key used by other CI steps untouched
export DEEPAGENTS_CODE_OPENAI_API_KEY="sk-..."
```

要将密钥保存在文件中，请在 [`.env` 文件中定义它们](#environment-variables).

### 密钥解析顺序

当提供商的密钥在多个地方设置时，Deep Agents Code 使用以下第一个已设置的：

1. **`DEEPAGENTS_CODE_`前缀的环境变量** — 例如 `DEEPAGENTS_CODE_OPENAI_API_KEY` 作为内联 shell 导出。 [`DEEPAGENTS_CODE_` 前缀](#deepagents_code_-prefix) 是明确的“在 Deep Agents Code 中使用此密钥”覆盖。
2. **应用存储的密钥** — 在 `/auth` 凭据管理器中输入。
3. **纯提供商环境变量** — 例如 `OPENAI_API_KEY`，来自您的 shell 或 `.env` files.

应用存储的密钥优先于同一提供商的纯环境变量密钥，但 `DEEPAGENTS_CODE_`前缀的密钥优先于应用存储的密钥。前缀是一种覆盖已存储密钥的方式，适用于单次运行，而无需清除它：

```bash
# With a key already stored via /auth, a plain env var does not override it.
# dcode still uses the app-stored key for this run:
OPENAI_API_KEY=sk-xxxx dcode -n "..."

# The DEEPAGENTS_CODE_ prefix does override it, for this run only:
DEEPAGENTS_CODE_OPENAI_API_KEY=sk-xxxx dcode -n "..."
```

这种分层设计适用于常见场景：您的机器已经导出了一个纯提供商变量用于其他目的——一个共享的 `OPENAI_API_KEY` 被其他工具、脚本或 CI 使用——您不希望 Deep Agents Code 重用它。应用存储的密钥或 `DEEPAGENTS_CODE_`前缀的变量为 Deep Agents Code 提供其自己的值，同时保持未加前缀的那个不变供其他所有内容使用，因此两者永远不会混淆。

每个提供商的 API 密钥及其端点（`base_url`）作为一对从同一来源解析。请参阅 [端点、密钥和网关](#endpoints-keys-and-gateways).

### 使用 Tavily 启用网络搜索

内置 `web_search` 工具使用 [Tavily](https://tavily.com)。Deep Agents Code 在启动时显示“网络搜索已禁用”通知，直到您提供密钥。您可以将密钥存储在 [`/auth`](#use-%2Fauth-recommended) 凭据管理器中，Tavily 显示为非模型服务，或设置 `TAVILY_API_KEY` 环境变量。


<Tabs>
    <Tab title="Use /auth (recommended)">
        从以下网址获取密钥 [tavily.com](https://tavily.com) （它以 `tvly-`开头；免费套餐足以满足大多数 Deep Agents Code 使用），然后将其存储在凭据管理器中：

        ```txt
        /auth
        ```

        选择 **Tavily** 从列表中并粘贴密钥。您也可以通过“网络搜索已禁用”通知直接进入此提示，选择 **输入 API 密钥**.
    </Tab>

    <Tab title="Set an environment variable">
        <Steps>
            <Step title="Get a key">
                在以下网址注册 [tavily.com](https://tavily.com) 并复制密钥（它以 `tvly-`开头）。免费套餐足以满足大多数 Deep Agents Code 使用。
            </Step>

            <Step title="Add it to your environment">
                将密钥添加到 `~/.deepagents/.env` 以便每个会话都能使用它：

                ```bash title="~/.deepagents/.env"
                TAVILY_API_KEY=tvly-...
                ```

                Shell 导出优先于 `.env` 值（见 [加载顺序和优先级](#loading-order-and-precedence)）。要将某个密钥仅限定于 Deep Agents Code 而不影响其他读取 `TAVILY_API_KEY`的工具，请使用 [`DEEPAGENTS_CODE_` 前缀](#deepagents_code_-prefix): `DEEPAGENTS_CODE_TAVILY_API_KEY=tvly-...`.
            </Step>

            <Step title="Reload or restart">
                在现有会话中，运行 `/reload` 来重新读取 `.env` 文件。在下次启动时，"Web search disabled"通知会消失，代理可以调用 `web_search`.
            </Step>
        </Steps>
    </Tab>
</Tabs>

## 环境变量

除了 shell 导出外，Deep Agents Code 还会从 dotenv 文件读取环境变量，这样您可以将 API 密钥保存在 shell 配置文件之外，避免在 `.env` 文件中重复配置

```bash title="~/.deepagents/.env"
ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...
```

### 加载顺序和优先级

启动时，Deep Agents Code 读取最近的项目 `.env`，通过搜索启动目录并向上遍历其父目录来找到（第一个 `.env` 找到的优先），然后 `~/.deepagents/.env` 作为所有项目的全局备用。一个项目 `.env` 优先于全局配置，且两者都不会覆盖 shell 中已设置的值。运行 `/reload` 可重新读取 `.env` 文件，这样您可以更改密钥而无需重启，shell 值仍优先。这适用于 Deep Agents Code 读取的每个变量（例如， `TAVILY_API_KEY` 或 `DEEPAGENTS_CODE_*` 设置）。提供商 API 密钥有额外的解析规则；请参阅 [提供商凭证](#provider-credentials).

### `DEEPAGENTS_CODE_` 前缀

所有 Deep Agents Code 特定的环境变量都使用 `DEEPAGENTS_CODE_` 前缀（例如， `DEEPAGENTS_CODE_AUTO_UPDATE`, `DEEPAGENTS_CODE_DEBUG`）。请参阅 [环境变量参考](#environment-variable-reference) 获取完整列表。

此前缀还可作为 Deep Agents Code 读取的任何环境变量（包括第三方凭证）的覆盖机制。Deep Agents Code 先检查 `DEEPAGENTS_CODE_{NAME}` ，然后回退到 `{NAME}`:

```bash title="~/.deepagents/.env"
# Give Deep Agents Code its own value, without affecting other tools
DEEPAGENTS_CODE_OPENAI_API_KEY=sk-cli-only

# Or set it empty so Deep Agents Code ignores a key exported in your shell
DEEPAGENTS_CODE_ANTHROPIC_API_KEY=
```

## 配置文件

`~/.deepagents/config.toml` 允许您自定义模型提供商、设置默认值以及向模型构造函数传递额外参数。本节涵盖：

- **默认值**：固定一个 [默认模型](#default-and-recent-model) or [代理](#default-and-recent-agent).
- **提供商设置**：[`[models.providers.<name>]` 表格](#provider-configuration)， [构造函数参数](#model-constructor-params), [重试次数](#retries), [配置文件覆盖](#profile-overrides-advanced)，以及 [向 `/model` 切换器添加模型](#adding-models-to-the-interactive-switcher).
- **自定义端点和提供商**: [自定义基础 URL](#custom-base-url), [OpenAI 或 Anthropic 兼容 API](#compatible-apis)，以及 [任意提供商](#arbitrary-providers).
- **端点和网关**：如何 [API 密钥和基础 URL 如何协同解析](#endpoints-keys-and-gateways)，包括通过托管网关。

### 默认模型和最近使用的模型

```toml
[models]
default = "ollama:qwen3:4b"             # your intentional long-term preference
recent = "google_genai:gemini-3.5-flash"   # last /model switch (written automatically)
```

`[models].default` 始终优先于 `[models].recent`。该 `/model` 命令仅写入 `[models].recent`，因此您配置的默认设置不会被会话中途的切换覆盖。若要移除默认设置，请使用 `/model --default --clear` 或删除 `default` 键。

### 默认代理和最近使用的代理

```toml
[agents]
default = "backend-dev"  # your intentional long-term preference (Ctrl+S in /agents picker)
recent = "frontend-dev"  # last /agents switch (written automatically)
```

`[agents].default` 始终优先于 `[agents].recent`。在 `/agents` 选择器中配合 `Enter` 写入 `recent`；按 `Ctrl+S` 会将高亮行固定为 `default`。再次按 `Ctrl+S` 会清除该行的默认设置。

显式 `-a`/`--agent` 始终覆盖两者，且 `-r`/`--resume` 绕过两者以恢复线程的原始代理。参见 [命令参考](/oss/deepagents/code/overview#command-reference) 了解相关标志。

### Provider 配置

每个 provider 都是 `[models.providers]`:

```toml
[models.providers.<name>]
display_name = "My Provider"
api_key_url = "https://provider.example/keys"
models = ["gpt-4o"]
api_key_env = "OPENAI_API_KEY"
base_url = "https://api.openai.com/v1"
class_path = "my_package.models:MyChatModel"
enabled = true

[models.providers.<name>.params]
temperature = 0
max_tokens = 4096

[models.providers.<name>.params."gpt-4o"]
temperature = 0.7
```

下的 TOML 表。Provider 具有以下配置选项：

<ResponseField name="models" type="string[]" post={["optional"]}>
    要在交互式 `/model` 切换器中显示的模型名称列表，适用于定义为 `<name>`的 provider。对于已附带模型 profile 的 provider，您在此处添加的任何名称都会与内置名称一起显示（适用于尚未添加到包中的新发布模型）。对于 [自定义 provider](#arbitrary-providers)，此列表是切换器中模型的唯一来源。

    此处列出的模型 **会绕过** 任何已应用的基于 profile 的 [过滤条件](/oss/deepagents/code/providers#which-models-appear-in-the-switcher)，始终显示在切换器中。这使得它成为显示因 profile 缺少 `tool_calling` 支持或尚不存在而被排除的模型的推荐方式。

    此键是可选的。您始终可以直接传递任何模型名称到 `/model` or `--model` ，无论它是否出现在切换器中；provider 会在请求时验证该名称。
</ResponseField>

<ResponseField name="api_key_env" type="string" post={["optional"]}>
     **环境变量的** 名称，该环境变量保存 API 密钥（例如 `"OPENAI_API_KEY"`）。Deep Agents Code 在启动时从该环境变量读取凭证，以在创建模型之前验证访问权限。

    大多数聊天模型包会自动从默认环境变量读取。参见 [Provider 参考](/oss/deepagents/code/providers#provider-reference) 表格以了解每个内置 provider 检查的变量名称。对于不在该表格中的 provider，请将其设置为对应的变量名称（参见 `api_key_env` 自定义 provider [）。](#arbitrary-providers)).
</ResponseField>

<ResponseField name="display_name" type="string" post={["optional"]}>
    在授权UI中显示的人类可读提供商名称。用于配置键针对机器优化的任意提供商（例如， `my_gateway`），但其UI标签应包含空格或品牌大写。
</ResponseField>

<ResponseField name="api_key_url" type="string" post={["optional"]}>
    用户创建或管理API密钥的提供商页面URL。 `/auth` 模态框在API密钥输入前链接到此页面。此值是URL，而非凭证。
</ResponseField>

<ResponseField name="base_url" type="string" post={["optional"]}>
    如果支持，覆盖提供商使用的基础URL。请参阅您提供商包的 [参考文档](https://reference.langchain.com/python/integrations/) 了解更多。

    请参阅 [兼容的API](#compatible-apis) 将内置提供商指向线兼容的端点，或 [任意提供商](#arbitrary-providers) 了解通过以下方式配置的一个 `class_path`.
</ResponseField>

<ResponseField name="base_url_env" type="string" post={["optional"]}>
    保存此提供商基础URL的环境变量名称，与 `api_key_env`并行。使用此选项而不是 `base_url` 当端点来自环境而非固定值时——例如因机器或CI作业而不同的网关URL——以便它可以在不编辑的情况下更改 `config.toml` and can take part in endpoint resolution and key/endpoint pairing (see [端点、密钥和网关](#endpoints-keys-and-gateways)）。它还将这些扩展到内置集之外的提供商 [内置集](/oss/deepagents/code/providers#provider-reference)；请参阅 [任意提供商](#arbitrary-providers).

    如果两者都设置，则静态 `base_url` wins:

    ```toml
    [models.providers.example]
    base_url = "https://fixed.example/v1"   # used
    base_url_env = "EXAMPLE_BASE_URL"        # ignored while base_url is set
    ```
</ResponseField>

<ResponseField name="params" type="object" post={["optional"]}>
    转发到模型构造函数的额外关键字参数。扁平键（例如， `temperature = 0`）适用于此提供商提供的每个模型。按模型键入的子表（例如， `[params."gpt-4o"]`）仅覆盖该模型的单个值；合并是浅层的（模型在冲突时获胜）。

    不要放置凭证（例如， `api_key`) in `params`。请使用 [`api_key_env`](#provider-configuration) 指向环境变量。
</ResponseField>

<ResponseField name="profile" type="object" post={["optional"]}>
    （高级）覆盖模型运行时的 [配置文件](/oss/langchain/models#model-profiles) (e.g., `max_input_tokens`）。扁平键适用于此提供商提供的每个模型。按模型键入的子表（例如， `[profile."claude-sonnet-4-5"]`）仅覆盖该模型的单个值；合并是浅层的（模型在冲突时获胜）。这些覆盖在模型创建后应用，因此对上下文限制显示、自动摘要和任何其他读取配置文件的功 [配置文件覆盖](#profile-overrides-advanced) 以获取示例和 `--profile-override` flag.
</ResponseField>

<ResponseField name="class_path" type="string" post={["optional"]}>
    用于 [任意模型](#arbitrary-providers) 提供商。完全限定的Python类 `module.path:ClassName` 格式。设置后，Deep Agents Code直接导入并实例化此类用于提供商 `<name>`。此类必须是 `BaseChatModel` subclass.
</ResponseField>

<ResponseField name="enabled" type="boolean" default="true" post={["optional"]}>
    此提供商是否出现在 `/model` 选择器中。设置为 `false` 以隐藏从已安装包自动发现的提供商（例如，您不想让模型切换器变得混乱的传递依赖项）。您仍然可以直接使用已禁用的提供商通过 `/model provider:model` or `--model`.
</ResponseField>

### 模型构造函数参数

此 [`params` 字段](#provider-configuration) 将额外参数转发给模型构造函数。若要为某个模型设置不同的值，请添加一个以模型为键的子表，这样就不必复制整个提供商配置：

```toml
[models.providers.ollama]
models = ["qwen3:4b", "llama3"]

[models.providers.ollama.params]
temperature = 0
num_ctx = 8192

[models.providers.ollama.params."qwen3:4b"]
temperature = 0.5
num_ctx = 4000
```

使用此配置：

* `ollama:qwen3:4b` 获取 `{temperature: 0.5, num_ctx: 4000}` — 模型覆盖优先。
* `ollama:llama3` 获取 `{temperature: 0, num_ctx: 8192}` — 无覆盖，仅使用提供商级别的参数。

合并是浅层的：模型子表中存在的任何键都会替换提供商级别参数中的相同键，而仅存在于提供商级别的键会被保留。

<Tip>
    对于一次性调整，无需编辑 `config.toml`，请通过 `--model-params` 在启动时或会话中通过 `/model`传递 JSON 对象。CLI 标志的优先级高于配置文件。参见 [模型参数](/oss/deepagents/code/providers#model-parameters) 提供商页面上的语法和提供商特定示例。
</Tip>

### 重试

使用顶级 `[retries]` 部分配置临时模型提供商错误的重试次数。Deep Agents Code 将这些值传递给接受重试计数构造函数参数（kwargs）的提供商集成。如果省略此部分，则应用提供商 SDK 默认值。

```toml
[retries]
max_retries = 2

[retries.fireworks]
max_retries = 3

[retries.anthropic]
max_retries = 0
```

全局 `[retries].max_retries` 值适用于所有支持的提供商。特定提供商的表（如 `[retries.fireworks]`）会覆盖该提供商的全局值。值必须是大于或等于 `0`.

大多数支持的提供商将重试计数作为 `max_retries`接收。某些集成使用不同的构造函数参数（kwargs）。对于任意提供商，或要覆盖已知提供商的已注册参数，请在提供商特定的重试表中设置 `param` ：

```toml
[retries]
max_retries = 2

[retries.my_custom]
param = "retries"
max_retries = 4
```

`param` 必须是有效的 Python 标识符字符串，如 `"max_retries"` or `"retries"`。Deep Agents Code 忽略未设置 `param`的未知提供商，因为传递错误的重试参数（kwargs）可能会破坏模型创建。

`[retries]` 的优先级低于构造函数参数。完整的优先级顺序为：

1. `--max-retries N`，在提供商的已解析重试参数下应用
2. `--model-params` 使用提供商的重试参数，如 `'{"max_retries": N}'` or `'{"retries": N}'`
3. `[models.providers.<provider>.params]` 使用提供商的重试参数
4. `[retries.<provider>].max_retries`
5. `[retries].max_retries`
6. 提供商 SDK 默认值

### 配置文件覆盖（高级）

覆盖模型运行时配置文件中的字段，以更改 Deep Agents Code 解释模型能力的方式。参见 @[`ModelProfile`以获取可覆盖字段的完整列表。最常见的用例是降低 `max_input_tokens` 以提前触发自动摘要——这对于测试或约束上下文使用很有用：

```toml
# Apply to all models from this provider
[models.providers.anthropic.profile]
max_input_tokens = 4096
```

每个模型的子表以与 `params` — 模型级别的值在冲突时优先：

```toml
[models.providers.anthropic.profile]
max_input_tokens = 4096

# This model gets a higher limit
[models.providers.anthropic.profile."claude-sonnet-4-5"]
max_input_tokens = 8192
```

配置文件覆盖在创建后合并到模型的配置文件中。任何读取配置文件的功能——状态栏中的上下文限制显示、自动摘要阈值、功能检查——都会看到覆盖后的值。

<Accordion title="CLI profile overrides with --profile-override" icon="terminal">
    若要在运行时覆盖模型配置文件字段而不编辑配置文件，请通过 `--profile-override`:

    ```bash
    dcode --profile-override '{"max_input_tokens": 4096}'

    # Combine with --model
    dcode --model google_genai:gemini-3.5-flash --profile-override '{"max_input_tokens": 4096}'

    # In non-interactive mode
    dcode -n "Summarize this repo" --profile-override '{"max_input_tokens": 4096}'
    ```

    传递 JSON 对象。这些会合并到配置文件覆盖之上（CLI 优先）。优先级链为：模型默认值 < config.toml 配置 < CLI `--profile-override`.

    `--profile-override` 值在会话中持续存在 `/model` 热切换——切换模型会重新将覆盖应用到新模型。
</Accordion>

### 向交互式切换器添加模型

某些提供商（如 `langchain-ollama`）不捆绑模型配置文件数据（请参阅 [提供商参考](/oss/deepagents/code/providers#provider-reference) 以获取完整列表）。如果是这种情况，交互式 `/model` 切换器将不会列出该提供商的模型。您可以通过定义一个 `models` 列表来填补这一空白：

```toml
[models.providers.ollama]
models = ["gemma4", "qwen3.6", "granite4.1:3b"]
```

该 `/model` 切换器现在将包含一个 Ollama 部分，其中列出了这些模型。

这完全是可选的。您始终可以通过直接指定其完整名称来切换到任何模型：

```txt
/model ollama:qwen3.6:27b
```

<Note>
    当 `langchain-ollama` 已安装且守护进程可访问时，Deep Agents Code 会自动发现本地拉取的模型并将其合并到切换器中——无需 `models` 列表。运行 `/reload` 以在拉取新模型后刷新，或设置 `DEEPAGENTS_CODE_OLLAMA_DISCOVERY=0` 以选择退出。
</Note>

### 自定义基础 URL

某些提供商包接受一个 `base_url` 来覆盖默认端点。例如， `langchain-ollama` 默认为 `http://localhost:11434` 通过底层 `ollama` 客户端。要指向其他地方，请设置 `base_url` 在您的配置中：

```toml
[models.providers.ollama]
base_url = "http://your-host-here:port"
```

请参阅您的提供商的参考文档以获取兼容性信息和附加注意事项。

### 兼容的 API

对于提供与 OpenAI 或 Anthropic 线兼容的 API 的提供商，您可以使用现有的 `langchain-openai` or `langchain-anthropic` 包，方法是指向 `base_url` 提供商的端点：

```toml
[models.providers.openai]
base_url = "https://api.example.com/v1"
api_key_env = "EXAMPLE_API_KEY"
models = ["my-model"]
```

```toml
[models.providers.anthropic]
base_url = "https://api.example.com"
api_key_env = "EXAMPLE_API_KEY"
models = ["my-model"]
```

<Note>
    提供商在官方规范之上添加的任何功能都不会被捕获。如果提供商提供专用的 LangChain 集成包，请优先使用该包。
</Note>

<Warning>
    OpenAI 提供商默认为 [Responses API](https://platform.openai.com/docs/api-reference/responses)，大多数 OpenAI 兼容网关都不实现该 API。如果您的提供商仅支持 Chat Completions API，调用可能会失败。请显式禁用 Responses API：

    ```toml
    [models.providers.openai.params]
    use_responses_api = false
    ```
</Warning>

### 任意提供商

Deep Agents Code 适用于任何可用的工具调用 LLM [LangChain `BaseChatModel`](https://reference.langchain.com/python/langchain_core/language_models/#langchain_core.language_models.BaseChatModel)。内置 [内置提供商](/oss/deepagents/code/providers#provider-reference) 开箱即用；较不常见或内部模型需要更多设置。指向 `class_path` 其 `BaseChatModel` 子类，Deep Agents Code 直接导入并实例化该类。

```toml
[models.providers.my_custom]
display_name = "My Custom Provider"
api_key_url = "https://my-provider.example.com/keys"
class_path = "my_package.models:MyChatModel"
api_key_env = "MY_API_KEY"
base_url = "https://my-endpoint.example.com"

[models.providers.my_custom.params]
temperature = 0
max_tokens = 4096
```

`api_key_env` 和 `base_url` 是可选的。 `display_name` 和 `api_key_url` 自定义提供商名称和显示的密钥获取链接 `/auth`；省略它们以回退到提供商配置密钥和提供商设置文档。要从环境变量读取端点而不是硬编码 `base_url`，请使用 [`base_url_env`](#provider-configuration)；然后它以与内置提供商相同的方式解析并与密钥配对（请参阅 [端点、密钥和网关](#endpoints-keys-and-gateways)).

`class_path` 提供商需要在内部处理自己的身份验证——当你的模型使用自定义身份验证（JWT令牌、自定义头、mTLS等）而不是标准API密钥时很有用：

```toml
[models.providers.xyz]
class_path = "abc.integrations.deepagents:DeepAgentsXYZChat"
models = ["abc-xyz-1"]

[models.providers.xyz.params]
bypass_auth = true
temperature = 0
```

使用此配置，通过以下方式切换到模型 `/model xyz:abc-xyz-1` or `--model xyz:abc-xyz-1`.

<Note>
    Deep Agents Code 需要 **工具调用** 支持。如果你的自定义模型支持工具调用但 Deep Agents Code 不知道，请在其提供商配置文件中声明：

    ```toml
    [models.providers.xyz.profile]
    tool_calling = true
    max_input_tokens = 128000
    ```

    虽然可选，但强烈建议设置 `max_input_tokens` 为模型的上下文窗口。没有它，Deep Agents Code 无法显示上下文的满载程度，自动摘要会回退到固定触发点（约 170,000 个标记），而不是模型窗口的一部分。对于上下文窗口较小的模型，在你达到模型的硬限制之前摘要可能不会运行，因此一旦对话增长，请求就会开始失败。
</Note>

因为 Deep Agents Code 在启动时导入 `class_path` 类，所以定义它的包必须能够从运行 `dcode`的相同环境中导入。内建提供商作为 [安装附加组件](/oss/deepagents/code/providers#quickstart)提供，但自定义或内部包不是。将其安装到 `dcode` 环境中，使用 `--package` flag:

```bash
dcode --install my_package --package
```

在会话中，运行 `/install my_package --package --force`。两者都将包安装到 `dcode`旁边。如果包缺失或无法导入，Deep Agents Code 会跳过该提供商，其模型不会出现在 `/model`.

当你切换到 `my_custom:my-model-v1` 时（通过 `/model` or `--model`），模型名称（`my-model-v1`）作为 `model` kwarg:

```python
MyChatModel(model="my-model-v1", base_url="...", api_key="...", temperature=0, max_tokens=4096)
```

<Warning>
    `class_path` 从你的配置文件中执行任意 Python 代码。这与 `pyproject.toml` 构建脚本具有相同的信任模型——你控制自己的机器。
</Warning>

你的提供商包可以选择在 `_PROFILES` 中提供模型配置文件 `<package>.data._profiles` 来代替在 `models` 键下定义。请参阅 LangChain [模型配置文件](https://github.com/langchain-ai/langchain/tree/master/libs/model-profiles) 了解更多信息。

### 端点、密钥和网关

API 密钥与其发送到的端点必须匹配：端点必须接受该密钥，否则请求可能会失败。Deep Agents Code 将密钥和端点一起解析，因此覆盖其中一个会更新另一个以匹配。例如，如果你用自己的密钥替换网关提供的密钥，Deep Agents Code 也会删除网关端点，这样你的密钥就会直接发送到提供商，而不是发送到会拒绝它的网关。

#### 如何 `base_url` 解析

Deep Agents Code 按以下顺序解析提供商的端点（首个匹配优先）：

1. **`base_url` in `config.toml`** 用于该提供商。
2. **具有 `DEEPAGENTS_CODE_`前缀的端点变量。**
3. **环境中的纯端点变量** （例如， `OPENAI_BASE_URL`).
4. **使用 `/auth` credential.** 保存的端点。此步骤为没有端点变量的提供商应用保存的端点——例如你添加而未声明的提供商 [`base_url_env`](#provider-configuration)的提供商。步骤 2-3 对于这些没有变量可读，因此这里直接使用保存的端点。对于有端点变量的提供商，保存的端点已在步骤 2 或 3 生效（它被写入该变量），所以此步骤不会改变任何内容。无论哪种方式，在 `/auth` applies.
5. **提供程序 SDK 自带的默认端点**，当以上均未设置时。

<Note>
    解析后的端点会作为 `base_url` 构造函数参数传递给模型。
</Note>

与 API 密钥一样， [`DEEPAGENTS_CODE_` 前缀](#deepagents_code_-prefix) 将端点限定于 Deep Agents Code，不会影响其他工具。对于任何其他提供程序，使用 [`base_url_env`](#provider-configuration) 声明名称，端点会以相同方式解析和配对：

```toml
[models.providers.myprovider]
api_key_env = "MYPROVIDER_API_KEY"
base_url_env = "MYPROVIDER_BASE_URL"
models = ["my-model"]
```

字面量 `base_url` 优先于 `base_url_env`，因此只设置您需要的那一个：

```toml
[models.providers.myprovider]
base_url = "https://fixed.example/v1"   # used
base_url_env = "MYPROVIDER_BASE_URL"    # ignored while base_url is set
```

#### 覆盖会保持配对

当您使用 `/auth`存储密钥时，您输入的端点（或提供程序的默认端点，如留空）会与该密钥一起应用。使用空白基础 URL 存储密钥也会清除您环境中已设置的任何端点（例如，您的 shell 导出的网关 `OPENAI_BASE_URL` ），因此您的密钥将发送到提供程序的默认端点而非该网关。

```bash title="Scope both the key and the endpoint to Deep Agents Code"
DEEPAGENTS_CODE_OPENAI_API_KEY=sk-cli-only
DEEPAGENTS_CODE_OPENAI_BASE_URL=https://api.openai.com/v1
```

#### 托管网关

在使用模型网关（例如 LangSmith 网关）配置的环境中，网关通常会导出网关密钥和匹配的端点变量（`OPENAI_BASE_URL`, `ANTHROPIC_BASE_URL`, or `GOOGLE_GEMINI_BASE_URL`）一起导出。Deep Agents Code 默认使用该配对，因此无需配置。

要改用您自己的密钥，请使用 `/auth` 存储它（基础 URL 留空以使用提供程序默认端点，或显式设置），或设置带 `DEEPAGENTS_CODE_` 前缀的密钥和端点。两者都会覆盖网关配对，且不会留下不匹配的端点。

## 技能目录白名单

默认情况下，当 Deep Agents Code 加载技能时，它会验证解析后的技能文件路径是否保持在标准 [技能目录](/oss/deepagents/code/data-locations#skills)之一内。这可以防止技能目录内的符号链接读取这些根目录之外的任意文件。

如果您在非标准位置存储共享技能资产，并使用符号链接从标准技能目录引用它们，则可以将该位置添加到包含白名单。这 **不会** 添加新的技能发现位置：技能仍然只能从标准目录中发现。

<ResponseField name="extra_allowed_dirs" type="string[]" post={["optional"]}>
    添加到技能包含白名单的路径。支持 `~` expansion.

    ```toml
    [skills]
    extra_allowed_dirs = [
        "~/shared-skills",
        "/opt/team-skills",
    ]
    ```
</ResponseField>

或者，将 `DEEPAGENTS_CODE_EXTRA_SKILLS_DIRS` 环境变量设置为冒号分隔的列表：

```bash
export DEEPAGENTS_CODE_EXTRA_SKILLS_DIRS="~/shared-skills:/opt/team-skills"
```

设置环境变量后，它优先于配置文件中的值。更改在 `/reload`.

## 主题

使用 `/theme` 打开交互式主题选择器。在列表中导航以实时预览主题，按 `Enter` 将您的选择持久化到 `config.toml`.

Deep Agents Code 附带许多内置主题。默认主题是 `langchain`，这是一个使用 LangChain 品牌颜色的深色主题。所选主题会持久化保存在 `[ui]`:

```toml
[ui]
theme = "langchain-dark"
```

### 用户自定义主题

在 `[themes.<name>]` 部分的 `config.toml`下定义自定义主题。每个部分需要 `label` （str）。 `dark` （bool）默认为 `false` 如果省略则设为 `true` 用于深色主题。所有颜色字段都是可选的——省略的字段会根据 `dark` flag.

```toml
[themes.my-solarized]
label = "My Solarized"
dark = true
primary = "#268BD2"
warning = "#B58900"

# Theme names with spaces require TOML quoting
[themes."ocean breeze"]
label = "Ocean Breeze"
primary = "#0077B6"
background = "#CAF0F8"
```

用户定义的主题会与内置主题一起显示在 `/theme` selector.

### 覆盖内置主题颜色

要调整内置主题的颜色而不创建新主题，请使用 `[themes.<builtin-name>]` 部分。只读取颜色字段—— `label` 和 `dark` 会从内置主题继承：

```toml
[themes.langchain]
primary = "#FF5500"
```

省略的颜色字段会保留现有的内置值。

对 `[themes.*]` 部分的更改会在 `/reload`.

### 将主题映射到终端

如果在具有不同配色方案的终端之间切换（例如，深色的 iTerm 和浅色的 Apple Terminal），请在 `[ui.terminal_themes]`下将每个终端映射到一个主题。Deep Agents Code 会匹配 shell 的 `TERM_PROGRAM` 并自动应用映射的主题：

```toml
[ui.terminal_themes]
"Apple_Terminal" = "langchain-light"
"iTerm.app" = "langchain"
```

按 `T` 在 `/theme` 选择器中保存当前终端的高亮主题，或运行 `echo $TERM_PROGRAM` 来查找终端标识符并手动添加。

<Accordion title="Advanced: picker shortcuts, resolution order, terminal identifiers">
    #### 选择器快捷键

    在 `/theme` selector:

    - `N` 中切换显示标签和标准注册表项——这些项是 `[ui] theme` 和 `[ui.terminal_themes]` accept.
    - `T` 将高亮主题保存到 `[ui.terminal_themes]` 用于当前 `TERM_PROGRAM`。映射的主题会在选择器中显示标记 `(default)` 。

    #### 常见 `TERM_PROGRAM` 值

    键会与环境变量进行字面匹配——当键包含点或特殊字符时，在 TOML 中需要加引号。

    | 终端 | `TERM_PROGRAM` |
    | --- | --- |
    | Apple Terminal | `Apple_Terminal` |
    | iTerm2 | `iTerm.app` |
    | WezTerm | `WezTerm` |
    | VS Code 集成终端 | `vscode` |
    | Ghostty | `ghostty` |

    #### 解析顺序

    Deep Agents Code 每次启动时使用以下优先级解析主题：

    1. `DEEPAGENTS_CODE_THEME` 环境变量（显式覆盖）。
    2. `[ui.terminal_themes]` 当前 `TERM_PROGRAM`.
    3. `[ui] theme` 的映射 保存的偏好设置（由 `/theme`).
    4. 内置默认值（`langchain`).
</Accordion>

## Auto-update

Deep Agents Code 默认情况下会自动检查并安装更新。

要退出自动更新：

<Tabs>
    <Tab title="Config file">
        ```toml
        [update]
        auto_update = false
        ```
    </Tab>
    <Tab title="Environment variable">
        ```bash
        export DEEPAGENTS_CODE_AUTO_UPDATE=0
        ```
    </Tab>
</Tabs>

环境变量优先于配置文件。

启用时（默认），Deep Agents Code 会在会话开始时检查 PyPI 是否有新版本并自动升级。禁用时，Deep Agents Code 会显示包含适当安装命令的更新提示。

要完全取消自动更新检查：

<Tabs>
    <Tab title="Config file">
        ```toml
        [update]
        check = false
        ```
    </Tab>
    <Tab title="Environment variable">
        ```bash
        export DEEPAGENTS_CODE_NO_UPDATE_CHECK=1
        ```
    </Tab>
</Tabs>

禁用更新检查也会阻止启动时的自动更新安装。

您仍可以随时使用 `/update` 斜杠命令，用于运行按需检查并内联报告成功或失败。

升级后，Deep Agents Code 在下次启动时显示"新增功能"横幅，并附上更新日志链接。

退出会话时，如果在会话期间检测到新版本，系统会显示更新横幅作为提醒。

## 卸载

要移除 `dcode` 和 `deepagents-code` 二进制文件和隔离的工具环境，请运行：

```bash
uv tool uninstall deepagents-code
```

卸载命令不会删除用户配置或会话数据。Deep Agents Code 将这些文件存储在 `~/.deepagents/`，包括 `config.toml`, `hooks.json`，全局的 `.env`和 `.state/` 等内容（如保存的会话和凭据）。要同时删除这些数据，请运行：

```bash
rm -rf ~/.deepagents
```

## 托管部署

 [安装脚本](https://github.com/langchain-ai/deepagents/blob/main/libs/code/scripts/install.sh) 支持以 root 身份运行，面向 macOS MDM 工具（如 Kandji、Jamf 等），这些工具在最小的 root 环境中执行脚本。

当 `id -u` is `0`时，脚本会：

1. 解析真实控制台用户的 `HOME` （通过 `/dev/console` or a `/Users` 目录扫描）
2. `chown`在每个安装步骤后将所有创建的文件分配回目标用户

非 root 安装不受影响：当不以 root 身份运行时，所有 root 特定代码路径都会短路。

### 使用环境变量固定安装版本

安装脚本读取环境变量，允许你固定版本、选择额外组件，以及全 fleet 选择 Python 版本。在管道安装的同一行设置它们：

```bash
# Pin an exact version for reproducible installs across the fleet
curl -LsSf https://langch.in/dcode | DEEPAGENTS_CODE_VERSION="0.1.16" bash
```

</ResponseField>

## 钩子

钩子允许外部程序响应 Deep Agents Code 生命周期事件。在中配置命令 `~/.deepagents/hooks.json` ，每当事件触发时，它会将 JSON 载荷通过管道传递给每个匹配的命令的 stdin。

钩子在后台线程中以即发即忘的方式运行——它们永远不会阻塞 Deep Agents Code，失败会被记录而不会中断您的会话。

### 设置

创建 `~/.deepagents/hooks.json`:

```json
{
  "hooks": [
    {
      "command": ["bash", "-c", "cat >> ~/deepagents-events.log"],
      "events": ["session.start", "session.end"]
    }
  ]
}
```

现在每当会话启动或结束时，Deep Agents Code 都会将事件载荷追加到 `~/deepagents-events.log`.

### 钩子配置

配置文件包含一个单一的 `hooks` 数组。每个条目包含：

<ResponseField name="command" type="list[str]" required>
    要运行的命令和参数。不进行 shell 扩展：如需要请使用 `["bash", "-c", "..."]` （如果需要）。
</ResponseField>

<ResponseField name="events" type="list[str]" post={["optional"]}>
    要订阅的事件名称。省略或留空以接收 **所有** events.
</ResponseField>

```json
{
  "hooks": [
    {
      "command": ["python3", "my_handler.py"],
      "events": ["session.start", "task.complete"]
    },
    {
      "command": ["bash", "log_everything.sh"]
    }
  ]
}
```

上面的第二个钩子没有 `events` 过滤器，因此它接收 Deep Agents Code 发出的每个事件。

### 负载格式

每个钩子命令通过 stdin 接收一个 JSON 对象，包含 `"event"` 键加上事件特定字段：

```json
{
  "event": "session.start",
  "thread_id": "abc123"
}
```

### 事件参考

#### `session.start`

当智能体会话开始时触发（交互和非交互模式）。

<ResponseField name="thread_id" type="string" required>
    会话线程标识符。
</ResponseField>

#### `session.end`

当会话退出时触发。

<ResponseField name="thread_id" type="string" required>
    会话线程标识符。
</ResponseField>

#### `user.prompt`

在交互模式下，当用户提交聊天消息时触发。

无额外字段。

#### `input.required`

当智能体需要人工输入时触发（人机交互中断）。

无额外字段。

#### `permission.request`

当一个或多个工具调用需要用户权限时，在审批对话框之前触发。

<ResponseField name="tool_names" type="list[str]" required>
    请求审批的工具名称。
</ResponseField>

#### `tool.error`

当工具调用返回错误时触发。

<ResponseField name="tool_names" type="list[str]" required>
    出错的工具名称。
</ResponseField>

#### `task.complete`

当智能体完成当前任务时触发（流式循环结束，无进一步中断）。

<ResponseField name="thread_id" type="string" required>
    会话线程标识符。
</ResponseField>

#### `context.compact`

在 Deep Agents Code 压缩（摘要）对话上下文之前触发。

无额外字段。

### 执行模型

- **后台线程**：钩子子进程通过以下方式在线程中运行 `asyncio.to_thread` 以确保主事件循环永远不会被阻塞。
- **并发调度**：当多个钩子匹配一个事件时，它们在线程池中并发运行。
- **5秒超时**：每个命令有5秒的超时时间。超过此时间的命令将被终止。
- **Fire-and-forget**: Errors are caught per-hook and logged at debug/warning level. A failing hook never crashes or stalls Deep Agents Code.
- **延迟加载**：配置文件在第一次事件分派时读取一次，并在会话的剩余时间内进行缓存。
- **无Shell扩展**：命令直接执行（不通过shell）。如需Shell功能（如管道或变量扩展），请使用 `["bash", "-c", "..."]` 包装。

<Accordion title="Log all events to a file">
```json
{
  "hooks": [
    {
      "command": ["bash", "-c", "jq -c . >> ~/.deepagents/hook-events.jsonl"],
      "events": []
    }
  ]

```
</Accordion>

<Accordion title="Python handler">
编写一个从stdin读取JSON负载的处理脚本：

```python title="my_handler.py"
import json
import sys

payload = json.load(sys.stdin)
event = payload["event"]

if event == "session.start":
    print(f"Session started: {payload['thread_id']}", file=sys.stderr)
elif event == "permission.request":
    print(f"Approval needed for: {payload['tool_names']}", file=sys.stderr)
```

```json title="~/.deepagents/hooks.json"
{
  "hooks": [
    {
      "command": ["python3", "my_handler.py"],
      "events": ["session.start", "permission.request"]
    }
  ]
}
```
</Accordion>

### 安全注意事项

钩子遵循与Git钩子或shell别名相同的信任模型——任何可以写入的用户 `~/.deepagents/hooks.json` 都可以执行任意命令。这是设计决定的：

- **无命令注入**：负载数据仅作为JSON流向stdin，从不流向命令行参数。 `json.dumps` 处理转义。
- **默认无Shell**：命令使用 `shell=False`运行，防止Shell注入。
- **配置格式错误**：无效的JSON或意外类型会产生日志警告，而不是安全问题。

    仅从可信来源添加钩子。钩子具有与您的用户账户相同的权限。
</Warning>
