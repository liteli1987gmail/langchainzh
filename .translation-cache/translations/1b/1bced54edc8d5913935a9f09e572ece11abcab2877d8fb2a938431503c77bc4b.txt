以下页面详细描述了一个示例应用，它将 LangChain **深度智能体**完全部署在 [Next.js App](https://nextjs.org/) Router 项目中：流式聊天 UI、子智能体和线程历史记录，全部由 [Agent Streaming Protocol](https://github.com/langchain-ai/agent-protocol/tree/main/streaming) 提供支持，并以 Next.js Route Handlers（HTTP + SSE）的形式实现。无需单独的后端进程。

源码：[deployment-cookbook](https://github.com/langchain-ai/deployment-cookbook/tree/main/js-next) 中的 [`js-next`](https://github.com/langchain-ai/deployment-cookbook/tree/main/js-next)。

## 必需的 API 端点

该应用在 `/api/threads/...` 下暴露 Agent Streaming Protocol。路由处理器位于 `app/api/threads/` 中。

### 最低要求（流式聊天）

以下三个端点足以使用 `@langchain/react` 的 `HttpAgentServerAdapter` 运行单线程流式聊天：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `POST` | `/api/threads/:threadId/commands` | 接受协议命令（`run.start`、…）并启动智能体运行 |
| `POST` | `/api/threads/:threadId/stream` | 运行的协议事件 SSE 流 |
| `GET` / `POST` | `/api/threads/:threadId/state` | 读取和引导检查点线程状态 |

客户端使用 `GET /state`（404 时使用 `POST /state`）引导线程，这样在发送第一条消息之前水合不会导致 404。

### 可选（线程侧边栏）

此示例还为线程历史侧边栏实现了端点。如果你的 UI 不需要多线程管理，可以省略它们：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/threads` | 列出检查点已知的所有线程 |
| `DELETE` | `/api/threads/:threadId` | 删除线程的会话和检查点 |
| `POST` | `/api/threads/:threadId/history` | 分页检查点历史记录（Agent Protocol） |

1. 引导线程状态（`GET`/`POST /state`）。

4. 子代理（`task`）运行发出命名空间事件，作为 `stream.subagents` 暴露。

## 生产环境持久化

开箱即用，代理使用内存中的 `MemorySaver` 检查点（`lib/agent/index.ts`）和进程本地会话映射（`lib/server/registry.ts`）。这适用于本地开发和单实例服务器，但在 Vercel（无服务器、多副本）上，会话状态在冷启动或实例间**不持久**。

对于生产环境，请换入一个[持久化检查点器](/oss/langgraph/checkpointers#checkpointer-libraries)：

| 包 | 后端 |
| --- | --- |
| [`@langchain/langgraph-checkpoint-redis`](https://www.npmjs.com/package/@langchain/langgraph-checkpoint-redis) | Redis（`RedisSaver`） |
| [`@langchain/langgraph-checkpoint-postgres`](https://www.npmjs.com/package/@langchain/langgraph-checkpoint-postgres) | Postgres（`PostgresSaver`） |
| [`@langchain/langgraph-checkpoint-sqlite`](https://www.npmjs.com/package/@langchain/langgraph-checkpoint-sqlite) | SQLite（`SqliteSaver`） |

在 `lib/agent/index.ts` 中将 `MemorySaver` 替换为新的检查点器，并将新的检查点器传递给 `createDeepAgent`。路由处理程序和 `lib/server/threads.ts` 中的辅助函数保持不变。

### Vercel 上的 Redis

Vercel 上常用的一种选择是通过 [Marketplace](https://vercel.com/docs/redis) 使用 Redis（例如 [Upstash Redis](https://vercel.com/marketplace/upstash)）。在你的 Vercel 项目上安装集成；凭证会自动作为环境变量注入。

然后接入 `@langchain/langgraph-checkpoint-redis`：

```ts
import { RedisSaver } from "@langchain/langgraph-checkpoint-redis";

const checkpointer = await RedisSaver.fromUrl(process.env.REDIS_URL!);
```

使用您的 Redis 提供商公开的连接字符串（Upstash 同时提供 REST 和 Redis 协议 URL；检查点器需要 Redis URL）。

您还需要在 `lib/server/registry.ts` 中配置一个共享的会话/重放存储，以便 SSE 重连能够在无服务器调用之间正常工作。检查点器交换是实现持久化线程历史的主要步骤；会话存储则是用于实时运行重放的独立功能。

更多信息，请参阅[检查点器库](/oss/langgraph/checkpointers#checkpointer-libraries)和[添加内存/持久化](/oss/langgraph/add-memory)。

## 本地开发

```bash
cp .env.example .env.local   # set OPENAI_API_KEY
pnpm install
pnpm dev
```

打开 [http://localhost:3000](http://localhost:3000)。

```bash
pnpm build   # production build
pnpm start   # serve the production build
pnpm lint    # eslint
```

## 项目结构

- `lib/agent/`：深度智能体（`createDeepAgent`），包含 `researcher` 和 `math-whiz` 子智能体以及模拟工具。标记为 `server-only`。
- `lib/server/`：协议服务器逻辑：`session.ts`（SSE 运行）、`threads.ts`（检查点备份状态）、`serialize.ts`、`registry.ts`。
- `app/api/threads/`：协议端点的 Route Handlers。
- `lib/chat/threads-client.ts`：浏览器线程引导和侧边栏辅助函数。
- `components/`：聊天 UI（`ChatApp`、`Chat`、`MessageList`、`Subagents`、`ThreadHistory` 等）。
