[Agent Server](/langsmith/agent-server) 是一个用于创建和管理基于代理的应用程序的 API 平台。它提供内置持久化、任务队列，并支持大规模部署、配置和运行助手（代理工作流）。本变更日志记录了 Agent Server 发布的所有重要更新、功能和修复。

## v0.11.0rc4

</Update>

## v0.11.0rc1

### 总体说明
- 包含依赖项和安全维护更新。

### 新功能
- 添加了支持 DeltaChannel 的剪枝，保留状态重建所需的最小祖先检查点，取代了之前拒绝剪枝包含活动增量通道的线程的方法。支持 Postgres、SQLite、DeferredDelete 和内存运行时。
- 为 MongoDB 检查点添加器添加了支持 DeltaChannel 的剪枝，保留状态重建所需的最小祖先检查点和增量通道 blob。
- 添加了可选的 Prometheus 指标抓取支持。设置 `LSD_PROM_METRICS_ENABLED=true` 以在专用 Prometheus 抓取端点（端口 `LSD_PROM_METRICS_PORT`，默认 9464）上暴露 OTel 指标（运行生命周期、延迟、流、工作器 gauge）。配置后，Datadog OTLP 推送可与 Prometheus 同时工作。
- 为一元 core-api RPC 和 Redis 流发布字节添加了可选的 Go 核心速率限制，支持 Redis 驱动的 GCRA 强制执行、影子/强制模式，以及 `LS_RATE_LIMITS` 引导配置和 YAML 覆盖。
- 允许传递自定义证书和密钥文件（`ssl_certfile`、`ssl_keyfile`）以通过 HTTPS 运行开发服务器。

### 修复
- 修复了 JS 图上协议 v2 运行静默失败的问题。Sidecar 因严格的流模式验证拒绝 `streamEvents` 并返回 400，错误被吞掉，运行错误地报告成功但执行了 0 个节点。在 HTTP 边界放宽了流模式验证，现在对非 2xx Sidecar 响应抛出明确错误，而不是掩盖失败。
- 修复了针对 JS sidecar（远程）图的协议 v2 事件流，这些图错误地通过遗留重建路径提供服务。远程图现在为 v2 事件流运行使用 LangGraphJS 原生 v3 流，解决了工具调用不渲染、无头中断从不执行或恢复，以及恢复后最终消息出现 `400: tool_use ids must be unique` 错误的问题。
- 删除运行现在跳过 DeltaChannel 线程的检查点删除，仅移除运行记录。保留存储后续检查点所依赖的增量写入的检查点。使用线程剪枝 API 回收增量通道线程上的检查点存储。
- 修复了 Prometheus 指标导出并对齐了 OpenTelemetry 导出器配置。
</Update>

## v0.10.0rc1

### 总体说明
- v0.10.0rc1 包含安全性和正确性的破坏性变更。更多详情请参阅[安全部分](#security)。
- 包含依赖项和安全维护更新。

### 新功能
- 添加了按 ID 获取 cron 的端点（`GET /runs/crons/{cron_id}`）。

### 修复
- 修复了 Event Streaming v2 运行启动处理，使通过 `config.configurable.checkpoint_id` 提供的检查点重放目标能够被遵守。
- 修复了 Event Streaming v2 `input.respond` 通过 HTTP `POST /commands` 对 postgres 后端上的合法中断返回 `no_such_interrupt` 的问题。
- 修复了一个错误，其中线程先前时间旅行运行的 `checkpoint_map` 会持续存在并污染后续的 `Command(resume=...)`，导致嵌套子图错误地从开始处重放。

### 安全
- **可能具有破坏性** 默认情况下现在拒绝回环 webhook 目标，以修复身份验证绕过原语（[GHSA-2c9q-c2q9-qgqv](https://github.com/langchain-ai/helm/security/advisories/GHSA-2c9q-c2q9-qgqv)）。`webhooks.url.disable_loopback` 策略现在默认为 `true`，阻止相对 URL webhook（通过进程内 ASGI 传输调度并绕过身份验证）以及 localhost / 127.x / ::1 / host.docker.internal 绝对 URL 和任何 DNS 解析为回环范围内的主机名（缓解 DNS 重新绑定）。确实需要回环 webhook 的部署（如带有 localhost webhook 接收器的 `langgraph dev`，或调度到同一服务器上挂载的自定义 FastAPI 路由的生产设置）可以通过在 `langgraph.json` 中设置 `webhooks.url.disable_loopback: false`（或等效的 `LANGGRAPH_WEBHOOKS` JSON 环境变量）来重新启用。只有在您控制回环 webhook 到达的路由时才这样做，因为这些路由在调度时没有身份验证。
- **可能具有破坏性** `POST /runs` 和 `POST /threads/{thread_id}/runs` 现在通过 `assistants.read` 认证事件对附加的助手进行授权（与 cron 创建和直接 GET 匹配），而不是之前使用的带有不完整负载的 `assistants.search` 事件（[GHSA-jfj5-wrj9-63x4](https://github.com/langchain-ai/helm/security/advisories/GHSA-jfj5-wrj9-63x4)）。仅注册了 `@auth.on.assistants.read`（而没有 `.search` 处理器）的部署容易受到跨用户授权绕过的攻击；它们现有的读取处理器现在将在运行创建路径上被调用。作为纵深防御的后续操作，客户端提供的运行/cron 元数据不再从 `Runs.put` 或 `Crons.put` 转发到 `assistants.read` 认证事件负载，inmem/postgres 运行时现在在值结构上达成一致。对于具有自定义身份验证处理器的部署的破坏性变更：(1) 之前在运行创建期间调用的任何 `@auth.on.assistants.search` 处理器将不再在那里被调用——确保您有一个等效的 `@auth.on.assistants.read` 处理器返回相同的所有者风格过滤器；(2) 从运行/cron 创建调用的 `assistants.read` 事件上的 `value["metadata"]` 不再被填充，因此检查或修改它的处理器必须将该逻辑移至 `@auth.on.runs.create_run` / `@auth.on.crons.create`，并依赖返回过滤器进行所有权强制执行。
- 部署现在在服务器启动时看到一条结构化警告，列出每个未覆盖的调度路径以及要复制的默认拒绝代码片段。对于注册了全局 `@auth.on` 处理器的部署或仅使用 `@auth.authenticate` 而没有任何资源级处理器的部署，警告是静默的。

</Update>

## v0.9.0rc1

### 总体说明
- 在 /runs/crons/search 和 /runs/crons/count 中添加了 cron 元数据过滤，匹配助手/线程已提供的元数据过滤行为。
- 为 Postgres 检查点添加器添加了池调优旋钮，用于一次加载大量大检查点的场景。现在可以设置 LANGGRAPH_CHECKPOINTER_POSTGRES_POOL_MIN_SIZE 和 LANGGRAPH_CHECKPOINTER_POSTGRES_POOL_TIMEOUT_SECONDS。
- 修复了 mongo 检查点中 update_state 在线程没有先前检查点时的崩溃问题。
- 包含针对安全漏洞的依赖更新。

### 新功能
#### 增量通道支持
现在支持增量通道，以便检查点可以存储增量状态更新，而不是重复存储完整的通道负载，这有助于处理大型、追加密集的状态（如消息历史）。

要使用，请在图形状态中使用 LangGraph 的 DeltaChannel reducer 模式定义状态通道。
当安装的 langgraph >= 1.2 时启用此行为。
文档：[DeltaChannel 参考](/oss/langgraph/pregel#deltachannel)

#### 事件流 API
正在引入事件流 API，具有统一的事件流接口，用于更丰富的实时运行事件和命令/事件工作流。

功能标志 `FF_V2_EVENT_STREAMING` 可以设置为 true 以启用新的事件流 API。

新端点包括：
- `POST /threads/{thread_id}/stream/events`
- `POST /threads/{thread_id}/commands`
- `WS /threads/{thread_id}/stream/events`

文档：
- [Agent Server API 参考](/langsmith/server-api-ref)
- [LangGraph 事件流参考](/oss/langgraph/event-streaming)

</Update>

## v0.8.0

此次小版本将运行队列轮询从 Postgres 移至 Redis，节省数据库负载并提高性能。

在底层，Agent Server 使用持久化运行队列来管理运行执行。工作者轮询队列以获取新运行并执行它们。以前，队列轮询逻辑通过 Postgres。这可能导致长时间运行的查询，尤其是在高负载下。通过此更新，队列轮询逻辑现在通过 Redis，然后从 Postgres 获取运行详情。这使得队列轮询的热路径显著更快，并减少数据库负载。

这不是破坏性变更，升级时不需要代码更改，但有两点需要注意：
- 在升级后的部署中，队列将切换过去。可能会有一个短暂窗口，线程不是按时间顺序调度的。运行执行顺序在每个线程内仍然有保证。
- **仅限自托管：** Redis 流量可能略有增加。在内部测试中，增加量不大。
</Update>

此小版本更新了流式 API `/join-stream` 和 `/stream` 关于 `last-event-id` 参数的行为，以符合 SSE 规范。之前，传递 last-event-id 会返回该消息以及所有后续消息。今后，这些 API 将仅返回所提供 last-event-id 之后的新消息。例如，对于以下流，之前传递 last-event-id `2` 会返回 id 为 `2` 和 `3` 的消息，但现在将仅返回 id 为 `3` 的消息：

```json
{
    "id": 1,
    "event": "message",
    "data": {
        "content": "Excluded"
    }
},
{
    "id": 2,
    "event": "message",
    "data": {
        "content": "Passed last-event-id"
    }
},
{
    "id": 3,
    "event": "message",
    "data": {
        "content": "Included"
    }
}
```

```json
{
    "checkpointer": {
        "serde": {
            "allowed_json_modules": [
                ["my_agent", "my_file", "SomeType"],
            ]
        }
    }
}
```
