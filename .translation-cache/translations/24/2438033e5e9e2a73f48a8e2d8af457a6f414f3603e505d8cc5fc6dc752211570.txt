[Agent Server](/langsmith/agent-server) 是一个用于创建和管理基于代理的应用程序的 API 平台。它提供内置持久化、任务队列，并支持大规模部署、配置和运行助手（代理工作流）。此更新日志记录了 Agent Server 发布的所有重要更新、功能和修复。

## v0.11.0rc1

### 总体说明
- 包含依赖项和安全维护更新。

### 新功能
- 添加了感知 DeltaChannel 的剪枝，保留状态重建所需的最小祖先检查点，取代了之前拒绝剪枝带有活动增量通道的线程的方法。支持 Postgres、SQLite、DeferredDelete 和内存运行时。
- 为 MongoDB 检查点添加了感知 DeltaChannel 的剪枝，仅保留状态重建所需的最小祖先检查点和增量通道 blob。
- 添加了可选的 Prometheus 指标抓取支持。设置 `LSD_PROM_METRICS_ENABLED=true` 以在专用 Prometheus 抓取端点（默认端口 `LSD_PROM_METRICS_PORT` ）上暴露 OTel 指标（运行生命周期、延迟、流、工作器仪表）。同时配置时，Datadog OTLP 推送可与 Prometheus 一起工作。
- Added opt-in Go core rate limits for unary core-api RPCs and Redis stream publish bytes, with Redis-backed GCRA enforcement, shadow/enforce modes, and `LS_RATE_LIMITS` 使用 YAML 覆盖的引导配置。
- 允许传递自定义证书和密钥文件（`ssl_certfile`, `ssl_keyfile`）以通过 HTTPS 运行开发服务器。

### 修复
- 修复了 JS 图上协议 v2 运行静默失败的问题。边车拒绝了 `streamEvents` 由于严格的流模式验证，sidecar返回400错误，该错误被吞没，运行错误地报告成功但执行了0个节点。放宽了HTTP边界处的流模式验证，现在对非2xx的sidecar响应抛出明确错误，而不是掩盖失败。
- 修复了针对JS sidecar（远程）图的protocol v2事件流，这些图错误地通过旧的重建路径提供。远程图现在使用LangGraphJS的原生v3流进行v2事件流运行，解决了工具调用不渲染、无头中断从不执行或恢复的问题，以及 `400: tool_use ids must be unique` 恢复后最终消息的错误。
- 删除运行现在会跳过使用DeltaChannel的线程的检查点删除，仅删除运行记录。保存后续检查点所依赖的增量写入的检查点会被保留。使用线程修剪API来回收delta-channel线程的检查点存储。
- 修复了Prometheus指标导出，并调整了OpenTelemetry导出器配置。
</Update>

## v0.10.0rc1

### 概述
- v0.10.0rc1包含安全性和正确性的破坏性变更。参阅 [安全部分](#security) 了解更多详情。
- 包含依赖项和安全维护更新。

### 新功能
- 添加了按ID检索cron的端点（`GET /runs/crons/{cron_id}`).

### 修复
- 修复了事件流v2运行启动处理，使通过checkpoint replay targets提供的 `config.configurable.checkpoint_id` 得到尊重。
- 修复了事件流v2 `input.respond` 返回 `no_such_interrupt` 通过HTTP在postgres后端上的合法中断 `POST /commands`.
- 修复了一个bug，其中线程的 `checkpoint_map` 来自先前时间旅行运行的数据会持续存在并污染后续 `Command(resume=...)`，导致嵌套子图错误地从开始处重放。

### 安全
- **可能具有破坏性** Loopback webhook目标现在默认被拒绝以修复身份验证绕过原语（[GHSA-2c9q-c2q9-qgqv](https://github.com/langchain-ai/helm/security/advisories/GHSA-2c9q-c2q9-qgqv)）。 `webhooks.url.disable_loopback` 策略现在默认为 `true`, blocking relative-URL webhooks (which dispatch through the in-process ASGI transport and bypass auth), as well as localhost / 127.x / ::1 / host.docker.internal absolute URLs and any hostname that DNS-resolves into the loopback range (mitigating DNS rebinding). Deployments that legitimately need loopback webhooks (e.g. `langgraph dev` 使用本地主机 webhook 接收器，或分派到挂载在同一服务器上的自定义 FastAPI 路由的生产环境设置）可以通过设置来重新选择加入 `webhooks.url.disable_loopback: false` in `langgraph.json` （或等效的 `LANGGRAPH_WEBHOOKS` JSON 环境变量）。仅在您控制 loopback webhook 到达的路由时执行此操作，因为这些路由在分派时不进行身份验证。
- **可能破坏性** `POST /runs` 和 `POST /threads/{thread_id}/runs` 现在通过`assistants.read` 认证事件（匹配 cron 创建和直接 GET）来授权附加的助手，而不是之前使用的 `assistants.search` 事件（带有不完整的负载（[GHSA-jfj5-wrj9-63x4](https://github.com/langchain-ai/helm/security/advisories/GHSA-jfj5-wrj9-63x4)）。仅注册了 `@auth.on.assistants.read` （且没有 `.search` handler) were vulnerable to a cross-user authorization bypass; their existing read handler will now be invoked on the run-creation path. As a defense-in-depth follow-up, client-supplied run/cron metadata is no longer forwarded into the `assistants.read` 认证事件负载的 `Runs.put` or `Crons.put`, and inmem/postgres runtimes now agree on the value shape. Breaking change for deployments with custom auth handlers: (1) any `@auth.on.assistants.search` 处理程序（在运行创建期间先前调用的）不再在那里被调用——确保您有一个等效的`@auth.on.assistants.read` 处理程序返回相同的所有者风格过滤器；(2) `value["metadata"]` 在 `assistants.read` event invoked from run/cron creation is no longer populated, so handlers that inspected or mutated it must move that logic into `@auth.on.runs.create_run` / `@auth.on.crons.create` 上并依赖返回过滤器以进行所有权强制执行。
- 部署现在在服务器启动时看到结构化警告，列出每个未覆盖的分派路径以及要复制的默认拒绝代码片段。对于注册了全局 `@auth.on` 处理程序或仅使用 `@auth.authenticate` 且没有任何资源级处理程序的部署，警告是静默的。

</Update>

## v0.9.0rc1

### 总体说明
- Added cron metadata filtering in /runs/crons/search and /runs/crons/count, matching metadata filter behavior already available for assistants/threads.
- 添加了 Postgres 检查点池调优旋钮，用于一次加载大量大型检查点的情况。LANGGRAPH_CHECKPOINTER_POSTGRES_POOL_MIN_SIZE 和 LANGGRAPH_CHECKPOINTER_POSTGRES_POOL_TIMEOUT_SECONDS 现在可以设置了。
- 修复了 mongo 检查点中更新_状态时的崩溃，当时线程没有先前的检查点。
- 包含安全漏洞的依赖项更新。

### 新功能
#### 增量通道支持
现在支持增量通道，因此检查点可以存储增量状态更新，而不是重复存储完整的通道负载，这有助于处理大型、追加密集型状态（如消息历史记录）。

要使用，请在图形状态中使用 LangGraph 的 DeltaChannel 简化器模式定义状态通道。
当安装的 langgraph >= 1.2 时启用此行为。
Docs: [DeltaChannel 参考](/oss/langgraph/pregel#deltachannel)

#### 事件流 API
Event streaming APIs are being introduced, with a unified event-streaming surface intended for richer real-time run events and command/event workflows.

功能标志 `FF_V2_EVENT_STREAMING` 可设置为 true 以启用新的事件流 API。

新端点包括：
- `POST /threads/{thread_id}/stream/events`
- `POST /threads/{thread_id}/commands`
- `WS /threads/{thread_id}/stream/events`

Docs:
- [Agent Server API 参考](/langsmith/server-api-ref)
- [LangGraph 事件流参考](/oss/langgraph/event-streaming)

</Update>

## v0.8.0

此次小版本将运行队列轮询从 Postgres 迁移到 Redis，减少数据库负载并提高性能。

在底层，Agent Server 使用持久化运行队列来管理运行执行。Worker 从队列中轮询新运行并执行它们。以前，队列轮询逻辑通过 Postgres。这在高负载下可能导致长时间运行的查询。通过此更新，队列轮询逻辑现在通过 Redis，然后从 Postgres 获取运行详情。这使得队列轮询的热路径大幅加快，并减少了数据库负载。

这不是破坏性更改，升级时不需要代码更改，但有几件事需要注意：
- 在升级后的部署中，队列将切换。可能会有一小段时间线程被非按序调度。但每个线程内的运行执行顺序仍是有保证的。
- **仅限自托管：** Redis 流量可能会略有增加。在内部测试中，增加幅度很小。
</Update>

## v0.6.0

此补丁版本更新了流式传输 API `/join-stream` 和 `/stream` 与 `last-event-id` 参数的行为以与 SSE 规范保持一致。之前，传递 last-event-id 会返回该消息以及任何后续消息。从现在开始，这些 API 将仅返回提供 last-event-id 之后的新消息。例如，对于以下流，之前传递 last-event-id `2` 会返回 id 为 `2` 和 `3`的消息，但现在将仅返回 id 为 `3`:

```json
{
    "id": 1,
    "event": "message",
    "data": {
        "content": "Excluded"
    }
},
{
    "id": 2,
    "event": "message",
    "data": {
        "content": "Passed last-event-id"
    }
},
{
    "id": 3,
    "event": "message",
    "data": {
        "content": "Included"
    }
}
```

此更新还包括一些修复，包括一个暴露运行流中意外内部事件的错误。


</Update>

## v0.5.0

此 minor 版本现在要求 langgraph-checkpoint 版本高于 3.0，以防止早期版本的 langgraph-checkpoint 库中存在反序列化漏洞。
该 `langgraph-checkpoint` 库与 `langgraph` minor 版本 0.4、0.5、0.6 和 1.0 兼容。

此版本移除了对使用 "json" 类型保存的有效载荷的反序列化默认支持，这从未成为默认选项。
默认情况下，对象使用 msgpack 进行序列化。在某些不常见的情况下，有效载荷使用较旧的 "json" 模式进行序列化。如果这些有效载荷包含自定义 Python 对象，则除非您提供一个 `serde` config:

```json
{
    "checkpointer": {
        "serde": {
            "allowed_json_modules": [
                ["my_agent", "my_file", "SomeType"],
            ]
        }
    }
}
```
</Update>
