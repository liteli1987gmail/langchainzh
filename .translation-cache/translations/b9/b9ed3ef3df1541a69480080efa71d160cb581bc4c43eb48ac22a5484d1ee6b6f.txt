LangChain 拥有庞大的集成生态系统，整合了各种外部资源，如本地和远程文件系统、API 和数据库。这些集成使开发者能够创建多功能应用程序，将 LLMs 的强大能力与访问、交互和操作外部资源的能力相结合。

## 最佳实践

在构建此类应用程序时，开发者应记住遵循良好的安全实践：

* [**限制权限**](https://en.wikipedia.org/wiki/Principle_of_least_privilege): 将权限范围严格限定在应用程序的必需范围内。授予过多或过于宽泛的权限可能会引入严重的安全漏洞。为避免此类漏洞，可考虑使用只读凭证、限制对敏感资源的访问、采用沙箱技术（如在容器中运行）、指定代理配置以控制外部请求等，根据应用程序的具体需求来实施。
* **预判潜在滥用**: 与人类一样，大型语言模型（LLMs）也会犯错。因此应始终假设任何系统访问或凭证都可能被以允许的任何方式使用。例如，若数据库凭证允许删除数据，最安全的做法是假设任何能使用这些凭证的 LLM 都可能实际执行删除操作。
* [**深度防御**](https://en.wikipedia.org/wiki/Defense_in_depth_(computing)): 任何安全技术都不是完美的。微调和良好的链设计可以降低但无法消除大型语言模型（LLM）出错的可能性。最好将多层次安全方法结合使用，而非依赖任何单一防御层来确保安全。例如：同时使用只读权限和沙箱来确保 LLM 只能访问明确允许其使用的数据。

不遵循这些原则的风险包括但不限于：

* 数据损坏或丢失。
* 对机密信息的未授权访问。
* 关键资源的性能或可用性受损。

带有缓解策略的示例场景：

* 用户可能请求具有文件系统访问权限的代理删除不应删除的文件或读取包含敏感信息的文件内容。为了缓解此风险，应限制代理只能使用特定目录，并只允许其读取或写入安全的文件。进一步可考虑通过在容器中运行来对代理进行沙箱化。
* 用户可能请求具有外部 API 写权限的代理向该 API 写入恶意数据或从该 API 删除数据。为了缓解此风险，应为代理提供只读 API 密钥，或限制其只能使用已具备抗滥用能力的端点。
* 用户可能请求具有数据库访问权限的代理删除表或修改模式。为了缓解此风险，应将凭证限制为只能访问代理需要的表，并考虑颁发只读凭证。

如果您正在构建访问文件系统、API 或数据库等外部资源的应用程序，请考虑与您公司的安全团队沟通，了解如何最佳地设计和保护您的应用程序。

## 报告开源安全漏洞

请使用以下流程报告与 LangChain 开源项目相关的安全漏洞：

1. **提交安全公告** 在存在漏洞的 GitHub 仓库的"Security（安全）"选项卡上。
2. **发送电子邮件**至 `security@langchain.dev`，通知我们您已提交安全漏洞报告以及提交的是哪个仓库。

在报告漏洞之前，请查看上面的[最佳实践](#最佳实践)，以了解我们认为什么是安全漏洞与开发者责任。

### 漏洞赏金 eligibility

我们欢迎所有 LangChain 库的安全漏洞报告。但是，我们可能仅针对以下软件包中的漏洞提供临时漏洞赏金：

* 由 LangChain 团队拥有和维护的核心库：`langchain-core`、`langchain`（v1）、`langgraph` 以及相关的检查点包（或其 JavaScript 等效包）
* 由 LangChain 团队维护的热门集成（如 `langchain-openai`、`langchain-anthropic` 等，或其 JavaScript 等效包）

漏洞必须存在于库代码本身中，而非示例代码或示例应用程序中。

我们欢迎所有其他 LangChain 软件包的报告，并将处理有效的安全问题，但不会为超出此范围的软件包颁发漏洞赏金。这包括已归档的 `langchain-community`，由于其社区驱动的性质，不符合漏洞赏金 eligibility，但我们将接受并处理相关报告。

### 超出范围的目标

以下内容不在安全漏洞报告的范围内：

* **langchain-experimental**: 此归档仓库用于实验代码，不在安全报告的范围内（请参阅[软件包警告](https://pypi.org/project/langchain-experimental/)）。
* **示例和示例应用程序**: 示例代码和演示应用程序不在安全报告的范围内。
* **带有安全通知的文档代码**: 这将根据具体情况决定，但可能不在范围内，因为代码已带有供开发者遵循的指南，以使其应用程序安全。
* **与 LangSmith 相关的仓库或 API**: 请参阅下面的[报告 LangSmith 漏洞](#reporting-langsmith-vulnerabilities)。
