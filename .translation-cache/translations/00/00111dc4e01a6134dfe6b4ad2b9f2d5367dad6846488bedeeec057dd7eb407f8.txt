最强大的基于LLM的应用程序是复杂的问答（Q&A）聊天机器人，它们通过为LLM提供对一组数据的结构化访问来增强LLM的能力。
这些数据可能是私有数据、最新数据或不属于LLM训练数据的数据。
这些应用程序使用一种称为检索增强生成（Retrieval Augmented Generation）的技术，或称[RAG](/oss/langchain/retrieval/)。

本教程将引导你构建一个能够回答长篇非结构化文本问题的应用：

1. **[索引内容](#index-your-content)**：创建从数据源摄取数据并进行索引的管道。
2. **[RAG agent](#rag-agent)**：一种通用实现，搜索索引内容并将相关上下文传递给LLM。
3. **[RAG chain](#rag-chain)**：一种两步实现，每次查询使用一次LLM调用。这是处理简单查询的快速有效方法。

本教程使用 Lilian Weng 的 [LLM Powered Autonomous Agents](https://lilianweng.github.io/posts/2023-06-23-agent/) 博客文章作为示例。

在完成本教程的过程中，使用 [LangSmith](/langsmith/observability) 来[追踪](/langsmith/trace-with-langchain)检索和生成过程。

## 为内容建立索引

在索引步骤中，您将获取源内容并将其 _分块_ 转换为数值表示。这种数值表示能够捕捉分块的语义含义。将这些数值表示与文档分块的映射关系存储在 `VectorStore` 中，可以让您在用户基于其自身的数值表示发送查询时，高效地检索相关内容。

索引通常遵循以下四个步骤：

1. **[加载](#load-documents)**：将数据源加载到 @[`Document`] 对象中。
2. **[分割](#split-documents)**：使用[文本分割器](/oss/integrations/splitters)将大型 `Document` 分割成更小的分块。这对于索引数据和向模型传递数据都很有用，因为大型分块更难搜索，而且要么无法适应模型的有限上下文窗口，要么会使用超出必要数量的 tokens。
3. **[嵌入](#select-an-embeddings-model)**：[嵌入模型](/oss/integrations/embeddings) 将每个分块转换为捕获其含义的数值向量，使您能够对内容进行相似性搜索。
4. **[存储](#store-chunks-and-embeddings-in-vectorstore)**：使用 [VectorStore](/oss/integrations/vectorstores) 为分块及其嵌入建立索引以便检索。

![index_diagram](/images/rag_indexing.png)

在以下步骤中，您将设置摄取源内容所需的组件。

下一步是检索和生成：在运行时给定一个用户问题，从索引中拉取相关文本块，并将其传递给模型以生成答案。RAG应用通常分两个阶段实现该流程：

1. **检索**：给定用户输入，从存储中使用[检索器](/oss/integrations/retrievers)检索相关文本块。
2. **生成**：使用包含问题和检索数据的[模型](/oss/langchain/models)生成答案。

![retrieval_diagram](/images/rag_retrieval_generation.png)

本教程逐步讲解该流程的两种实现：一种是在需要时调用搜索工具的[RAG智能体](#rag-agent)，另一种是始终一次检索并在单次模型调用中回答的[RAG链](#rag-chain)。

## RAG 链

在您创建的 [RAG agent](#rag-agent) 中，您允许 LLM 自行决定是否生成 [tool call](/oss/langchain/models#tool-calling) 来帮助回答用户查询。这是一个良好的通用解决方案，但也有一些权衡：

| ✅ 优势                                                                         | ⚠️ 缺点                                                                         |
|-----------------------------------------------------------------------------|----------------------------------------------------------------------------|
| **仅在需要时搜索**：LLM 可以处理问候语、跟进问题和简单查询，而不会触发不必要的搜索。 | **两次推理调用**：当执行搜索时，需要一次调用来生成查询，另一次来产生最终响应。 |
| **上下文感知搜索查询**：通过将搜索作为带有 `query` 输入的工具，LLM 可以生成包含对话上下文的查询。 | **控制力降低**：LLM 可能会跳过实际需要的搜索，或者在不必要时发出额外搜索。 |
| **允许多次搜索**：LLM 可以为单个用户查询执行多次搜索以提供支持。 |                                                                            |

另一种常见方法是两步链，在这种方法中，您始终运行搜索（可能使用原始用户查询），并将结果作为上下文纳入单个 LLM 查询中。这导致每次查询一次推理调用，用灵活性换取降低的延迟。

在这种方法中，我们不再循环调用模型，而是进行单次传递。

您可以通过从代理中移除工具，并将检索步骤合并到自定义提示词中来实现此链：

:::python
<RagCreateChainPy />

<RagReturnSourceDocumentsPy />
:::
:::js

<RagReturnSourceDocumentsJs />
:::

## 安全性注意事项

<Warning>
RAG 应用容易受到 **间接提示注入** 的影响。检索到的文档可能包含类似指令的文本（例如"以 JSON 格式响应"或"忽略之前的指令"）。由于检索到的上下文与系统提示共享相同的上下文窗口，模型可能会无意中遵循嵌入在数据中的指令，而不是您预期的提示。

例如，本教程中索引的博客文章包含描述 [Auto-GPT](https://lilianweng.github.io/posts/2023-06-23-agent/#case-studies) JSON 响应格式的文本。如果用户查询检索到该块，模型可能会输出 JSON 而不是自然语言答案。
</Warning>

为了缓解这个问题：

1. **使用防御性提示**：明确指示模型仅将检索到的上下文作为数据处理，并忽略其中的任何指令。本教程中的提示包含此类说明。
2. **使用分隔符包装上下文**：使用清晰的结构标记（例如 XML 标签如 `<context>...</context>`）将检索到的数据与指令分离，使模型更容易区分它们。
3. **验证响应**：检查模型输出是否符合预期格式（例如纯文本），并妥善处理意外格式。

没有万无一失的缓解措施——这是当前 LLM 架构的固有局限性，其中指令和数据共享相同的上下文窗口。要了解更多，请参阅 [prompt injection](https://simonwillison.net/series/prompt-injection/) 研究。
