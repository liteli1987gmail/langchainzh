在本教程中，我们将构建一个能够使用 LangGraph 回答 SQL 数据库相关问题的自定义 agent。

LangChain 提供内置 [agent](/oss/langchain/agents) 实现，使用 [LangGraph](/oss/langgraph/overview) 原语实现。如果需要更深入的定制，可以直接在 LangGraph 中实现 agent。本指南演示了 SQL agent 的示例实现。关于实践介绍，请参阅 [使用更高级的 LangChain 抽象构建 SQL agent](/oss/langchain/sql-agent).

<Warning>
构建 SQL 数据库问答系统需要执行模型生成的 SQL 查询。这存在固有风险。请确保您的数据库连接权限始终尽可能窄地限定于 agent 的需求范围。这将减轻（但不能完全消除）构建模型驱动系统的风险。
</Warning>

该 [预构建 agent](/oss/langchain/sql-agent) 让我们可以快速入门，但我们依赖系统提示来约束其行为——例如，我们指示 agent 始终以 "list tables" 工具开始，并在执行查询之前始终运行 query-checker 工具。

通过自定义 agent，我们可以在 LangGraph 中实现更高程度的控制。在这里，我们实现了一个简单的 ReAct-agent 设置，具有用于特定工具调用的专用节点。我们将使用与预构建 agent 相同的 [state]。

### 概念

我们将涵盖以下概念：

- [工具](/oss/langchain/tools) 用于读取 SQL 数据库
- LangGraph [Graph API](/oss/langgraph/graph-api)，包括 state、nodes、edges 和条件 edges。
- [Human-in-the-loop](/oss/langgraph/interrupts) 流程

sql_db_list_tables: Input is an empty string, output is a comma-separated list of tables in the database.

sql_db_schema: Input to this tool is a comma-separated list of tables, output is the schema and sample rows for those tables.
    Be sure that the tables actually exist by calling sql_db_list_tables first!
    Example Input: table1, table2, table3

sql_db_query: Input to this tool is a detailed and correct SQL query, output is a result from the database.
    If the query is not correct, an error message will be returned.
    If an error is returned, rewrite the query, check the query, and try again.
    If you encounter an issue with Unknown column 'xxxx' in 'field list', use sql_db_schema to query the correct table fields.

## 6. 实现人工介入审查

在执行SQL查询之前检查代理的SQL查询以防止任何意外操作或低效问题是明智的做法。

在这里我们利用LangGraph的 [human-in-the-loop](/oss/langgraph/interrupts) 功能在执行SQL查询之前暂停运行并等待人工审查。使用LangGraph的 [持久层](/oss/langgraph/persistence)，我们可以无限期暂停运行（或至少在持久层存活期间）。
