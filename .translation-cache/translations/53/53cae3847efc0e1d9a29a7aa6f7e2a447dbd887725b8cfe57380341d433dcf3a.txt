默认情况下，LangSmith 会从我们的公共 Docker 注册表拉取镜像。但是，如果您在没有互联网访问的环境中运行 LangSmith，或者您想使用私有 Docker 注册表，可以将镜像镜像到您自己的注册表，然后配置 LangSmith 安装以使用这些镜像。

## 前提条件

* 有权访问您的 Kubernetes 集群/机器可以访问的 Docker 注册表。
* 在本地机器或可以访问 Docker 注册表的机器上安装 Docker。
* 一个可以运行 LangSmith 的 Kubernetes 集群。

## 镜像镜像

为了方便起见，我们提供了一个脚本来帮助您镜像镜像。您可以在 [LangSmith Helm Chart 仓库](https://github.com/langchain-ai/helm/blob/main/charts/langsmith/scripts/mirror_langsmith_images.sh) 中找到该脚本

要使用该脚本，您需要使用以下命令运行该脚本，指定您的注册表和平台：

```bash
bash mirror_images.sh <your-registry> [<platform>]
```

其中 `<your-registry>` 是您的 Docker registry 的 URL（例如 `myregistry.com`），`<platform>` 是您使用的平台（例如 `linux/amd64`、`linux/arm64` 等）。如果您不指定平台，则默认为 `linux/amd64`。

例如，如果您的 registry 是 `myregistry.com`，您的平台是 `linux/arm64`，并且您想要使用镜像的最新版本，则运行：

```bash
bash mirror_langsmith_images.sh --registry myregistry --platform linux/arm64 --version 0.10.66
```

请注意，此脚本将假定您已安装 Docker 并且已通过身份验证登录到您的镜像仓库。它还会将镜像推送到指定的镜像仓库，并保持与原始镜像相同的仓库/标签。

或者，您也可以手动拉取、镜像和推送镜像。您需要镜像的镜像可以在 LangSmith Helm Chart 的 `values.yaml` 文件中找到。可在此处找到：[LangSmith Helm Chart values.yaml](https://github.com/langchain-ai/helm/blob/main/charts/langsmith/values.yaml#L14)

以下是使用 Docker 镜像镜像的示例：

```bash
# Pull the images from the public registry
docker pull langchain/langsmith-backend:latest
docker tag langchain/langsmith-backend:latest <your-registry>/langsmith-backend:latest
docker push <your-registry>/langsmith-backend:latest
```

你需要对每个需要镜像的镜像重复此操作。

## 配置

镜像完成后，你需要配置 LangSmith 安装以使用这些镜像。可以通过修改 LangSmith Helm Chart 安装的 `values.yaml` 文件来实现。将 tag 替换为你要使用的版本，例如，撰写本文档时的最新版本 `0.10.66`。

```yaml
images:
  imagePullSecrets: [] # Add your image pull secrets here if needed
  registry: "" # Set this to your registry URL if you mirrored all images to the same registry using our script. Then you can remove the repository prefix from the images below.
  aceBackendImage:
    repository: "(your-registry)/langchain/langsmith-ace-backend"
    pullPolicy: IfNotPresent
    tag: "0.10.66"
  backendImage:
    repository: "(your-registry)/langchain/langsmith-backend"
    pullPolicy: IfNotPresent
    tag: "0.10.66"
  frontendImage:
    repository: "(your-registry)/langchain/langsmith-frontend"
    pullPolicy: IfNotPresent
    tag: "0.10.66"
  hostBackendImage:
    repository: "(your-registry)/langchain/hosted-langserve-backend"
    pullPolicy: IfNotPresent
    tag: "0.10.66"
  operatorImage:
    repository: "(your-registry)/langchain/langgraph-operator"
    pullPolicy: IfNotPresent
    tag: "6cc83a8"
  platformBackendImage:
    repository: "(your-registry)/langchain/langsmith-go-backend"
    pullPolicy: IfNotPresent
    tag: "0.10.66"
  playgroundImage:
    repository: "(your-registry)/langchain/langsmith-playground"
    pullPolicy: IfNotPresent
    tag: "0.10.66"
  postgresImage:
    repository: "(your-registry)/postgres"
    pullPolicy: IfNotPresent
    tag: "14.7"
  redisImage:
    repository: "(your-registry)/redis"
    pullPolicy: IfNotPresent
    tag: "7"
  clickhouseImage:
    repository: "(your-registry)/clickhouse/clickhouse-server"
    pullPolicy: Always
    tag: "24.8"
```

## Fleet 和 Insights 的额外镜像

如果您正在使用 Fleet 或 Insights，LangGraph 操作员会为每个部署动态创建 Redis 和 PostgreSQL (pgvector) Pod。这些 Pod 使用操作员模板中定义的镜像，需要单独配置。

您必须镜像这些额外的镜像：
- `docker.io/redis:7`
- `docker.io/pgvector/pgvector:pg15`

然后在您的 `values.yaml` 中覆盖操作员模板以使用您的镜像：

```yaml
operator:
  templates:
    redis: |
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: ${service_name}
        namespace: ${namespace}
      spec:
        replicas: 1
        selector:
          matchLabels:
            app: ${service_name}
        template:
          metadata:
            labels:
              app: ${service_name}
          spec:
            enableServiceLinks: false
            containers:
            - name: redis
              image: (your-registry)/redis:7
              ports:
              - containerPort: 6379
                name: redis
              livenessProbe:
                exec:
                  command:
                  - redis-cli
                  - ping
                initialDelaySeconds: 30
                periodSeconds: 10
              readinessProbe:
                tcpSocket:
                  port: 6379
                initialDelaySeconds: 10
                periodSeconds: 5
    db: |
      apiVersion: apps/v1
      kind: StatefulSet
      metadata:
        name: ${service_name}
      spec:
        serviceName: ${service_name}
        selector:
          matchLabels:
            app: ${service_name}
        persistentVolumeClaimRetentionPolicy:
          whenDeleted: Delete
          whenScaled: Retain
        template:
          metadata:
            labels:
              app: ${service_name}
          spec:
            containers:
            - name: postgres
              image: (your-registry)/pgvector/pgvector:pg15
              ports:
              - containerPort: 5432
              command: ["docker-entrypoint.sh"]
              args:
                - postgres
                - -c
                - max_connections=${max_connections}
              env:
              - name: PGDATA
                value: /var/lib/postgresql/data/pgdata
              volumeMounts:
              - name: postgres-data
                mountPath: /var/lib/postgresql/data
            enableServiceLinks: false
        volumeClaimTemplates:
        - metadata:
            name: postgres-data
          spec:
            accessModes: ["ReadWriteOnce"]
            resources:
              requests:
                storage: "${storage_gi}Gi"
```

将 `(your-registry)` 替换为您的镜像仓库 URL。模板变量（`${service_name}`、`${namespace}`、`${max_connections}`、`${storage_gi}`）在运行时由 operator 替换，必须保持原样。

配置完成后，您需要更新 LangSmith 安装。您可以按照我们的升级指南操作：[升级 LangSmith](/langsmith/self-host-upgrades)。如果升级成功，您的 LangSmith 实例现在应该正在使用来自 Docker 镜像仓库的镜像。

## 验证镜像签名

<Note>
镜像签名功能**从 v15 开始可用**（LangSmith 应用版本 `0.15.x` 及更高版本）。`v14-stable` 及更早版本的镜像未经过签名，无法使用以下步骤进行验证。
</Note>

`docker.io/langchain/*` 上的稳定版 LangSmith 镜像在发布时使用无密钥的 [Sigstore/Cosign](https://docs.sigstore.dev/cosign/overview/) 进行签名，签名身份绑定到特定的 GitHub Actions 工作流、运行和提交。因此，签名不仅证明镜像的真实性，还证明它是由 `langchain-ai/langchainplus` 中运行的 stable-branch 发布流水线生成的。您可以在拉取或镜像化镜像之前验证签名，也可以在镜像化后再次验证以确认您镜像的摘要与我们签名的内容一致。

安装 `cosign`（[安装指南](https://docs.sigstore.dev/cosign/system_config/installation/)），然后验证任意标签：

```bash
cosign verify \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --certificate-identity-regexp 'https://github\.com/langchain-ai/langchainplus/\.github/workflows/release_self_hosted_on_version_bump\.yaml@refs/heads/v[0-9]+-stable' \
  docker.io/langchain/langsmith-backend:<tag>
```

成功的验证确认：

- 签名上的 cosign 声明有效。
- 证书链到 Sigstore 根并记录在 [Rekor](https://docs.sigstore.dev/rekor/overview/) 透明度日志中。
- 签名证书通过 GitHub Actions OIDC 颁发给 stable-branch 发布工作流。

通过替换仓库名称，相同命令适用于任何已发布的镜像（`langsmith-frontend`、`langsmith-go-backend`、`agent-builder-deep-agent`、`langsmith-clio`、`langsmith-polly`、`agent-builder-tool-server`、`agent-builder-trigger-server`、`hosted-langserve-backend`、`langsmith-playground`、`langsmith-ace-backend`，以及它们的 `*-fips` 变体）。

### 固定到特定版本

要进行更严格的验证（例如，固定到单个稳定分支或特定提交），请删除正则表达式并提供精确的证书身份。每个签名的证书还包含工作流运行 ID 和提交 SHA 作为主题备用名称扩展，这样您可以约束到特定的版本：

```bash
cosign verify \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --certificate-identity 'https://github.com/langchain-ai/langchainplus/.github/workflows/release_self_hosted_on_version_bump.yaml@refs/heads/v15-stable' \
  docker.io/langchain/langsmith-backend:0.15.9
```

要检查证书的声明（工作流运行、提交、运行器），请下载证明并解码嵌入的证书：

```bash
cosign download attestation docker.io/langchain/langsmith-backend:<tag>
```

### 验证 SBOM 证明

发布的镜像还包含签名的 SPDX 软件物料清单 (SBOM) 证明，每个架构在镜像索引中各有一个。证明附加在每个架构的子摘要上，而不是多架构标签上，因此对裸标签运行 `cosign verify-attestation` 会报告 `no matching attestations`。请先解析子摘要，然后验证每个摘要。

列出某个标签的每个架构摘要：

```bash
docker buildx imagetools inspect --raw docker.io/langchain/langsmith-backend:<tag> \
  | jq -r '.manifests[] | select(.platform.os == "linux") | .digest + "  " + .platform.architecture'
```

验证其中一个摘要上的 SBOM 证明：

```bash
cosign verify-attestation \
  --type spdxjson \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --certificate-identity-regexp 'https://github\.com/langchain-ai/langchainplus/\.github/workflows/release_self_hosted_on_version_bump\.yaml@refs/heads/v[0-9]+-stable' \
  docker.io/langchain/langsmith-backend@<digest>
```

成功的验证提供与镜像签名相同的保证：证明由 stable-branch 发布工作流生成，其声明记录在 [Rekor](https://docs.sigstore.dev/rekor/overview/) 透明度日志中。

要提取 SPDX 文档本身，请解码已验证的证明载荷：

```bash
cosign verify-attestation \
  --type spdxjson \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --certificate-identity-regexp 'https://github\.com/langchain-ai/langchainplus/\.github/workflows/release_self_hosted_on_version_bump\.yaml@refs/heads/v[0-9]+-stable' \
  docker.io/langchain/langsmith-backend@<digest> \
  | jq -r '.payload' | base64 -d | jq '.predicate'
```

解码后的谓词是一个标准的 SPDX 2.3 文档，列出了该镜像中的每个包。
