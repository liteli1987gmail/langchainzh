### 快速修复：提交错误修复

对于简单的错误修复，您可以立即开始：

<Steps>
    <Step title="复现问题">
        在克隆仓库之前，请确保您能够可靠地复现该错误。这有助于确认问题，并为您的修复提供起点。维护者和其他贡献者应该能够根据您的描述复现问题，而无需额外的设置或修改。
    </Step>

    <Step title="派生仓库">
        :::python
        将 [LangChain](https://github.com/langchain-ai/langchain)、[LangGraph](https://github.com/langchain-ai/langgraph) 或 [Deep Agents](https://github.com/langchain-ai/deepagents) 仓库派生到您的<Tooltip tip="如果您派生到组织账户，维护者将无法进行编辑，这可能会大大延迟接受。">个人 GitHub 账户</Tooltip>
        :::
        :::js
        将 [LangChain](https://github.com/langchain-ai/langchainjs)、[LangGraph](https://github.com/langchain-ai/langgraphjs) 或 [Deep Agents](https://github.com/langchain-ai/deepagentsjs) 仓库派生到您的<Tooltip tip="如果您派生到组织账户，维护者将无法进行编辑，这可能会大大延迟接受">个人 GitHub 账户</Tooltip>
        :::
    </Step>

    <Step title="克隆并设置">
        :::python

        ```bash
        git clone https://github.com/your-username/name-of-forked-repo.git

        # For instance, for LangChain:
        git clone https://github.com/parrot123/langchain.git
        ```

        ```bash
        # Inside your repo, initialize environment and install dependencies
        uv venv && source .venv/bin/activate
        uv sync --all-groups

        # or, to install a specific group only:
        uv sync --group test
        ```

如果之前没有安装过，你需要安装 [`uv`](https://docs.astral.sh/uv/)
        :::
        :::js

        ```bash
        git clone https://github.com/your-username/name-of-forked-repo.git

        # For instance, for LangChain:
        git clone https://github.com/parrot123/langchainjs.git

        # For LangGraph:
        git clone https://github.com/parrot123/langgraphjs.git
        ```

        ```bash
        # Inside your repo, install dependencies
        pnpm install
        # Create a build for all packages to resolve workspace dependencies
        pnpm build
        ```
:::
    </Step>

    <Step title="Create a branch">
        为你的修复创建一个新分支。这有助于保持更改的有序性，并使你以后提交拉取请求更加容易。

        ```bash
        git checkout -b your-username/short-bugfix-name
        ```
</Step>

    <Step title="编写失败测试">
        添加[单元测试](#test-writing-guidelines)，这些测试在没有你的修复时会失败。这使我们能够验证 bug 是否已解决，并防止回归
    </Step>

    <Step title="进行更改">
        在遵循我们的[代码质量标准](#code-quality-standards)的同时修复 bug。进行解决问题所需的**最小更改**。我们强烈鼓励贡献者在开始编码之前在 issue 上评论。例如：

        > *"我很想处理这个问题。我打算的方法是 [...简要描述...]。这是否符合维护者的期望？"*

        30 秒的评论通常可以防止在最初方法错误时浪费精力。
    </Step>

    :::js
    <Step title="运行构建">
        运行构建命令以确保包仍然正确构建
        ```bash
        pnpm build
        # or build a specific workspace package
        pnpm --filter @langchain/core build
        ```
</Step>
    :::

    <Step title="验证修复">
        确保测试通过且没有引入回归问题。在提交 PR 之前，确保所有测试在本地通过

        :::python

        ```bash
        make format
        make lint
        make test

        # For bugfixes involving integrations, also run:
        make integration_tests
        # (You may need to set up API testing credentials)
        ```
        :::
        :::js

        ```bash
        pnpm lint
        pnpm test

        # For bugfixes involving integrations, also run:
        pnpm test:int

        # Or run tests in a specific workspace package
        cd libs/langchain-core
        pnpm test
        pnpm lint

        # Or run tests for a specific package from the root of the repo
        pnpm --filter @langchain/core test
        pnpm --filter @langchain/core lint
        ```
<Step title="记录变更">
        如果行为有变化，请更新文档字符串和/或内联注释
    </Step>

    <Step title="提交拉取请求">
        请遵循提供的 PR 模板。如适用，请使用[关闭关键字](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/linking-a-pull-request-to-an-issue#linking-a-pull-request-to-an-issue-using-a-keyword)引用你要修复的问题（例如 `Fixes #ISSUE_NUMBER`），以便在你的 PR 合并时自动关闭该问题。
    </Step>
</Steps>

## 贡献指南

在开始为 LangChain 项目做贡献之前，请花点时间思考一下你这样做的原因。如果你唯一的目的是在简历上添加一个"首次贡献"（或者你只是想快速获得成功），那么参加训练营或在线教程可能更适合你。

为开源项目做贡献需要时间和精力，但这也可以帮助你成为更好的开发者并学习新技能。然而，重要的是要知道，这可能比参加训练营课程更难、更慢。也就是说，如果你愿意花时间把事情做好，那么为开源做贡献是值得的！

### 向后兼容性

<Warning>
    禁止对公共 API 进行破坏性更改，关键安全修复除外。

    有关主版本发布的详细信息，请参阅我们的[版本控制策略](/oss/versioning)。
</Warning>

通过以下方式维护兼容性：

<AccordionGroup>
    <Accordion title="稳定接口">
        **始终保留**：

        - 函数签名和参数名
        - 类接口和方法名
        - 返回值结构和类型
        - 公共 API 的导入路径
    </Accordion>

    <Accordion title="安全变更">
        **可接受的修改**：

        :::python
        - 添加新的可选参数
        :::
        :::js
        - 添加新的可选参数/类型参数
        :::
        - 向类添加新方法
        - 在不改变行为的情况下提升性能
        - 添加新模块或函数
    </Accordion>

    <Accordion title="在做出更改之前">
        - **这会破坏现有用户代码吗？**
        - 检查你的目标是否公开
        :::python
        - 如果需要，是否在 `__init__.py` 中导出？
        :::
        - 测试中是否有现有的使用模式？
    </Accordion>
</AccordionGroup>

### 新功能

我们对新功能保持高标准。我们通常不接受外部贡献者提出的新核心抽象，除非有一个现有的议题证明对此有迫切需求。这同样适用于基础设施和依赖项的变更。

通常，功能贡献要求包括：

<Steps>
    <Step title="设计讨论">
        开启一个议题，描述：

        - 你要解决的问题
        - 提议的 API 设计
        - 预期的使用模式
    </Step>

    <Step title="实现">
        - 遵循现有代码模式
        - 包含全面的测试和文档
        - 考虑安全影响
    </Step>

    <Step title="集成考虑">
        - 这如何与现有功能交互？
        - 是否会影响性能？
        - 这是否引入了新的依赖项？

        我们将拒绝可能导致安全漏洞或报告的功能。
    </Step>
</Steps>

### 安全指南

<Warning>
    安全至关重要。切勿引入漏洞或不安全的模式。
</Warning>

安全检查清单：

<AccordionGroup>
    <Accordion title="输入验证">
        - 验证并清理所有用户输入
        - 在模板和查询中正确转义数据
        :::python
        - 切勿在用户数据上使用 `eval()`、`exec()` 或 `pickle`，因为这可能导致任意代码执行漏洞
        :::
        :::js
        - 切勿使用 `eval()`，因为这可能导致任意代码执行漏洞
        :::
    </Accordion>

    <Accordion title="错误处理">
        - 使用特定的异常类型
        - 不要在错误消息中暴露敏感信息
        - 实现适当的资源清理
    </Accordion>

    <Accordion title="依赖项">
        - 避免添加硬依赖
        - 保持可选依赖最小化
        - 审查第三方包的安全问题
    </Accordion>
</AccordionGroup>

## 开发工作流

### Pre-commit hooks

:::python
[LangChain](https://github.com/langchain-ai/langchain) 和 [Deep Agents](https://github.com/langchain-ai/deepagents) 仓库包含 [pre-commit](https://pre-commit.com/) hooks，会在每次提交前自动运行格式化、linting 和验证检查。请从仓库根目录安装它们：


</fragment>

```bash
pip install pre-commit  # or: uv tool install pre-commit
pre-commit install
```

Hooks 强制执行：
- 不允许直接提交到受保护分支
- YAML 和 TOML 语法验证
- 修复尾随空格和文件末尾
- 智能引号和非标准空格规范化
- 每个包的 `make format` 和 `make lint`
:::

### 运行测试

<Info>
    目录路径是相对于你正在工作的包的。
</Info>

我们倾向于在可能的情况下优先使用单元测试而不是集成测试。单元测试在每次 pull request 时运行，因此应该快速且可靠。集成测试按计划运行，需要更多设置，因此应该保留用于确认与外部服务的接口点。

#### 单元测试

:::python
**位置**: `tests/unit_tests/`
:::
:::js
**位置**: `src/tests/FILENAME_BEING_TESTED.test.ts`
:::

单元测试涵盖不需要调用外部 API 的模块化逻辑。如果你添加了新逻辑，应该添加一个单元测试。在单元测试中，检查预处理/后处理并模拟外部依赖。

**要求**：
- 不允许网络调用
- 测试所有代码路径，包括边界情况
- 使用 mock 来处理外部依赖

运行单元测试：

:::python
```bash
make test

# Or directly:
uv run --group test pytest tests/unit_tests

# To run a specific test:
TEST_FILE=tests/unit_tests/test_imports.py make test
```
:::
:::js
```bash
# Run the entire test suite
pnpm test

# Or run a specific test file
pnpm test src/tests/FILENAME_BEING_TESTED.test.ts

# Or run a specific test function
pnpm test -t "the test that should be run"
```
:::

#### 集成测试

:::python
**位置**：`tests/integration_tests/`
:::
:::js
**位置**：`src/tests/FILENAME_BEING_TESTED.int.test.ts`
:::

集成测试涵盖了需要调用外部 API 的逻辑（通常是与其他服务的集成）。

集成测试需要访问外部服务/提供商 API（这可能需要付费），因此默认情况下不会运行。

并非每次代码更改都需要集成测试，但请记住，作为审查流程的一部分，我们会单独要求/运行集成测试。

**要求**：
- 测试与外部服务的真实集成
- 使用环境变量存储 API 密钥
- 如果凭据不可用，请优雅地跳过

运行集成测试：
:::python
```bash
make integration_tests

# Or directly:
uv run --group test --group test_integration pytest --retries 3 --retry-delay 1 tests/integration_tests

# To run a specific test:
TEST_FILE=tests/integration_tests/test_openai.py make integration_tests
```
:::
:::js
```bash
pnpm test:int
```
:::

### 代码质量标准

### 依赖项

LangChain 包区分**硬依赖（hard dependencies）**和**可选依赖（optional dependencies）**，以保持包的轻量化，并最大限度地减少用户的安装开销。

<Tabs>
    <Tab title="可选依赖">
        几乎所有新依赖都应该是可选的。在以下情况下使用可选依赖：

        - 该依赖仅用于特定的集成或功能
        - 用户可以在没有此依赖的情况下有意义地使用该包
        - 该依赖较大或有很多传递依赖

        **要求：**

        - 未安装该依赖的用户必须能够**导入**你的代码，且没有任何副作用（无警告、无错误、无异常）
        - **不会**修改 `pyproject.toml` 和 `uv.lock`

        **添加可选依赖：**

        1. 将依赖添加到相应的测试依赖文件（例如 `extended_testing_deps.txt`）
        2. 添加一个单元测试，至少尝试导入新代码。理想情况下，单元测试使用轻量级 fixture 来测试代码逻辑。
        3. 对任何需要该依赖的单元测试使用 `@pytest.mark.requires("package_name")` 装饰器。
    </Tab>

    <Tab title="硬依赖">
        硬依赖在用户安装包时会自动安装。仅在以下情况下使用硬依赖：

        - 没有该依赖，包根本无法正常运行
        - 该依赖较小且传递依赖最少
        - 没有合理的方法使该功能成为可选

        <Warning>
            添加硬依赖会增加所有用户的安装时间和潜在的版本冲突。

            维护者会仔细审查硬依赖的添加！
        </Warning>

        **添加硬依赖：**

        1. 创建一个 issue 或讨论，解释为什么该依赖必须是硬依赖而非可选依赖
        2. 将依赖添加到 `pyproject.toml` 的相应部分
        3. 运行 `uv lock` 更新锁文件
        4. 包含覆盖新功能的综合测试
    </Tab>
</Tabs>
:::
