使用 LangSmith 在一个地方运行、追踪、比较和评估 Agent 成本，以 Harbor 作为执行层。Harbor 是一个在沙箱环境中评估和优化 Agent 与语言模型的框架，由 [Terminal-Bench](https://www.tbench.ai) 的创建者开发。它在隔离的容器中运行每个试验，因此您可以同时并行化评估和部署到多个环境。

LangSmith 在三个层面与 Harbor 集成：

- **LangSmith 评估**：使用 `--plugin langsmith` 将每个 Harbor 任务记录为 LangSmith 实验。
- **Deep Agents**：使用 `--agent langgraph` 将 LangGraph 或 Deep Agents 应用作为 Harbor Agent 运行。
- **沙箱**：使用 `--env langsmith` 在 LangSmith 沙箱上运行每个 Harbor 试验。

本页介绍 LangSmith 特定的 Harbor 参数。完整的 CLI，请运行 `harbor run --help` 或参阅 [Harbor 文档](https://harborframework.com/docs)。

## 前置条件

- 一个 [LangSmith 账户](https://smith.langchain.com)和一个 [API 密钥](/langsmith/create-account-api-key)。
- 带有 `pip` 的 Python 环境。
- 用于 Agent 调用模型的提供商 API 密钥，例如 `ANTHROPIC_API_KEY`。

### 安装

安装带有 `langsmith` 扩展的 Harbor。该扩展包含 `harbor-langsmith` 包，LangSmith 插件、环境和 Agent 都会使用它：

```bash
pip install "harbor[langsmith]"
```

### 身份验证

Harbor 使用您的 LangSmith 凭据进行身份验证。设置一个 API 密钥：

```bash
export LANGSMITH_API_KEY="<LANGSMITH_API_KEY>"
```

或者，选择一个 [LangSmith SDK 配置文件](/langsmith/profile-configuration) 而不是导出密钥：

```bash
export LANGSMITH_PROFILE=prod
```

## 快速入门

将 Harbor 任务记录到 LangSmith 作为实验：

```bash
harbor run -d "terminal-bench@2.0" \
  --agent <agent> \
  --model <provider:model> \
  --plugin langsmith
```

将 `<agent>` 替换为 Harbor agent，将 `<provider:model>` 替换为 `provider:model` 格式的模型，已安装的 `langchain-*` provider 可以解析该模型，例如 `anthropic:claude-opus-4-8`。运行 `harbor run --help` 列出可用的 agent，或参阅 [Deep Agents](#deep-agents) 获取完整的 `langgraph` 运行。

打开 [Datasets & Experiments](/langsmith/manage-datasets)，选择 Harbor 同步的数据集，例如 `terminal-bench@2.0`，然后打开 Experiments 标签页查看运行结果。

## LangSmith 评估

LangSmith 插件会将每个 Harbor 作业记录到 LangSmith，因此你可以在 Datasets & Experiments 下查看和比较结果。该插件适用于任何 Harbor agent，不仅限于 Deep Agents。使用 `--plugin langsmith` 启用它。[Quickstart](#quickstart) 展示了基本调用方式，本节涵盖插件记录的内容及其配置方法。

选择一个能够追踪到 LangSmith 的 agent，以捕获完整的 agent 追踪信息以及实验数据。如果 agent 不追踪到 LangSmith，插件仍会创建数据集和包含结果与反馈的实验，但不会包含 agent 追踪。

当你需要消除歧义时，请传递完整的导入路径，而不是简短的插件名称：

```bash
harbor run ... --plugin harbor_langsmith:LangSmithPlugin
```

该插件需要 `LANGSMITH_API_KEY`。

### 查看插件记录的内容

任务运行时，插件通过 API 向 LangSmith 写入数据：

- **Dataset**：从任务同步一个参考数据集。默认名称来自数据集或任务，例如 `terminal-bench@2.0`。每个任务成为一个示例，其输入包括任务名称、指令和任务 ID。
- **Experiment**：为每个任务创建一个实验，命名为 `<name>-<job-id-prefix>`，并关联到参考数据集。
- **Runs**：为每次试验创建一个根运行，包含任务名称、指令、代理和模型的输入，以及环境、代理和验证阶段的孩子运行。
- **Feedback**：为每个验证器奖励键附加一个反馈分数，如 `reward`，并在试验抛出异常时添加 `harbor_error` 反馈。
- **Outputs**：记录每个试验运行的 token 数量（`input`、`cache`、`output`）和运行成本（`cost_usd`）。

### 在 LangSmith 中查看结果

在 LangSmith 中打开 [Datasets & Experiments](/langsmith/manage-datasets)，选择插件同步的数据集（如 `terminal-bench@2.0`），然后打开 Experiments 标签页。每个 Harbor 任务显示为一个实验，您可以通过 `reward` 和 `harbor_error` 反馈、每次运行记录的 token 数量和成本以及延迟来 [对比实验](/langsmith/analyze-an-experiment)。

### 配置插件输入

插件首先从构造函数关键字参数读取每个输入，然后回退到环境变量。使用环境变量设置输入：

- **`HARBOR_LANGSMITH_DATASET`**：数据集名称。默认为从任务派生的名称。
- **`HARBOR_LANGSMITH_EXPERIMENT`**：实验基础名称。默认为任务名称。
- **`LANGSMITH_ENDPOINT`**：LangSmith API 端点。默认为 `https://api.smith.langchain.com`。
- **`LANGSMITH_WORKSPACE_ID`**：目标工作区。
- **`HARBOR_LANGSMITH_SYNC_DATASET`**：设置为 `false` 可禁用数据集和示例同步。
- **`HARBOR_LANGSMITH_FAIL_FAST`**：设置为 `true` 可在 LangSmith API 错误时抛出异常而非继续运行任务。

或在命令行中使用 `--pk` 将相同的输入设置为插件 kwargs，或在任务配置文件的 `kwargs:` 下设置。kwargs 映射到构造函数选项：`dataset_name`、`experiment_name`、`endpoint`、`api_key`、`workspace_id`、`sync_dataset` 和 `fail_fast`。

## Deep Agents

`langgraph` 代理将 LangGraph 应用程序（如 Deep Agent）作为 Harbor 代理运行。使用 `--agent langgraph` 选择它。Harbor 会将您的项目暂存到沙箱中，安装其依赖项，然后在容器内为每次试验运行图。

设置您的 LangSmith 和模型凭据，然后运行 Harbor。`harbor run` 是 `harbor job start` 的别名，它会构建任务、启动环境并运行 LangGraph 代理：

```bash
export LANGSMITH_PROFILE=prod
export LANGSMITH_TRACING=true
export LANGSMITH_PROJECT=harbor-deepagents
export FIREWORKS_API_KEY="<FIREWORKS_API_KEY>"

harbor run \
  -t hello-world/hello-world \
  --agent langgraph \
  --model fireworks:accounts/fireworks/models/glm-5p2 \
  --ak project_path=./deep-agent \
  --ak graph=deep_agent
```

### 选择评估对象

任务是一个具有固定布局的目录：`task.toml` 用于配置，`instruction.md` 用于提示，`environment/` 用于存放构建沙箱的 Dockerfile，`tests/` 用于存放写入奖励的验证器。数据集是多个这样的任务目录。

任务或数据集可以是本地或远程的：将 Harbor 指向你自己的任务目录文件夹，或从 Harbor 的注册表中拉取一个。

三个输入选择作业运行的任务：

- **`-t org/name[@ref]`**：注册表中的单个任务。远程任务通过注册表查找获取，然后克隆到 `~/.cache/harbor/tasks` 中的固定提交。
- **`-d name@version`**：整个基准数据集，即多个任务。每个任务从注册表解析并克隆到缓存中。
- **`-p <dir>`**：一个任务或多个任务根文件夹的本地路径。本地路径就地读取，无需下载和缓存副本。

使用 `-i` 和 `-x`（glob 包含和排除）筛选选定的任务，并用 `-l` 限制数量。

任务目录的布局如下：

```text
hello-world/
├── task.toml         # timeouts, CPU, and memory
├── instruction.md    # the prompt given to the agent
├── environment/
│   └── Dockerfile    # image the sandbox is built from
├── tests/
│   ├── test.sh       # writes the reward to /logs/verifier/reward.txt
│   └── test_state.py # the assertions
└── solution/         # optional, used only by the oracle agent
```

数据集是任务目录的目录：

```text
terminal-bench/
├── hello-world/      # each subdirectory is a full task
├── fix-bug/          # (task.toml + instruction.md + environment/ + tests/)
└── parse-csv/
```

### 配置代理

使用 `--ak` 传递代理 kwargs：

- **`--agent langgraph`**：选择 LangGraph 代理。
- **`--model <provider:model>`**：要运行的模型。没有默认值，因此此参数必填。代理使用 @[init_chat_model] 解析它，因此它必须可以被已安装的 `langchain-*` 提供程序解析，格式为 `provider:model`，例如 `anthropic:claude-opus-4-8`。`provider/model` 值会被规范化为 `provider:model`。模型来自 `configurable['model']` 或 `HARBOR_MODEL` 环境变量，如果无法解析或缺失会抛出 `ValueError`。
- **`--ak project_path=<dir>`**：包含 `langgraph.json` 的本地目录。
- **`--ak graph=<name>`**：在 `langgraph.json` 中要运行的图。
- **`--ak config=<file>`**：位于 `project_path` 内的配置文件名，用于声明图。默认为 `langgraph.json`。
- **`--ak configurable='{...}'`**：LangGraph 每轮运行配置，传递给 `config["configurable"]`，并在调用时由图读取。常用键包括 `model`、`model_kwargs` 和 `cwd`。
- **`--ak model_kwargs='{...}'`**：`configurable` 中嵌套的 `model_kwargs` 键的简写形式，例如 `{"temperature": 0, "max_tokens": 8000}`。
- **`--ak dependency_overrides='[...]'`**：代理虚拟环境的 pip 包。此列表替换 `langgraph.json` 中声明的依赖项，允许你无需编辑项目即可固定或切换版本，例如 `'["deepagents==0.1.5"]'`。

### 指向代理和依赖的 langgraph.json

代理从 `project_path` 中的 `langgraph.json` 文件加载图。该文件声明了图的入口点和 Harbor 在沙箱虚拟环境中安装的 pip 依赖项：

```json
{
  "dependencies": [
    "deepagents>=0.6.10,<0.7.0",
    "langchain-anthropic>=1.4.6,<1.5.0",
    "langchain-openai>=1.3.0,<1.4.0"
  ],
  "graphs": {
    "deep_agent": "./agent.py:make_graph",
    "research_agent": "./agent.py:make_research_graph"
  }
}
```

该项目暴露了两个图，通过 `--ak graph` 选择。两者都使用 @[create_deep_agent] 构建 Deep Agent，仅在输入上有所不同：

- **`deep_agent`** 解析为 `make_graph`，一个仅使用模型的 Deep Agent。
- **`research_agent`** 解析为 `make_research_graph`，带有研究系统提示词的相同 Deep Agent。

每个图都将 `--model` 的模型（从 `configurable.model` 读取）传递给 `create_deep_agent`，它通过 `init_chat_model()` 解析：

```python
from deepagents import create_deep_agent


def make_graph(config):
    return create_deep_agent(model=config["configurable"]["model"])


def make_research_graph(config):
    return create_deep_agent(
        model=config["configurable"]["model"],
        system_prompt="You are a research assistant.",
    )
```

读取 `configurable.model` 的工厂函数可以让图保持模型无关，但你也可以在图内硬编码模型，使其始终运行同一个模型。对于固定模型，请将 `langgraph.json` 指向已编译的图而不是工厂函数：

```python
from deepagents import create_deep_agent

graph = create_deep_agent(model="fireworks:accounts/fireworks/models/glm-5p2")
```

### 在沙箱中运行 Agent

Harbor 在 trial 容器内运行整个 Agent。

<Accordion title="单个 trial 生命周期">

1. **解析和准备**：`harbor run` 将标志解析为作业配置。作业工厂解析并缓存任务，验证环境资源限制，并在任何 trial 运行之前解析指标。缓存仅适用于远程任务，因此 `-p` 本地任务就地读取。
2. **展开**：Harbor 从 `n_attempts × tasks × agents` 构建 trial 列表，然后按 `-n` 限制并发运行 trial，并使用 `-r` 重试。并发是按 trial 进行的，因此不同的任务、Agent 和尝试一起运行，每个都在自己的沙箱中。
3. **创建 trial**：trial 加载缓存的任务，从 `project_path`、`graph` 和 `model` 构建 LangGraph Agent，并构造环境而不启动它。
4. **启动环境**：环境启动并启动容器。对于 Docker 环境，这会构建或重用镜像并运行容器。
5. **安装 Agent**：Harbor 在容器中创建虚拟环境，上传 `project_path`，并在容器内 pip 安装 `langgraph.json` 依赖项。
6. **运行和验证**：Harbor 通过 LangGraph 运行器在容器内运行图，然后运行 `tests/test.sh`，该脚本将奖励写入 `/logs/verifier/reward.txt`。
7. **完成**：Harbor 停止并删除容器，并写入 trial 结果。作业将所有 trial 结果聚合为一个作业结果。

</Accordion>

有关构建 Deep Agents 的更多信息，请参阅 [Deep Agents 文档](/oss/deepagents/overview)。

## 沙箱

`langsmith` Harbor 环境在 LangSmith 沙箱上运行每个 trial。使用 `--env langsmith` 选择它，以便在 LangSmith 基础设施上执行 Harbor 作业，与其他沙箱提供商一起使用。每个 trial 获得自己的沙箱，Harbor 在 trial 完成后将其删除。

### 运行评估

运行 Harbor 作业并使用 `--env langsmith` 选择 LangSmith 环境：

```bash
harbor run -d "<org/name>" \
  --model "<model>" \
  --agent "<agent>" \
  --env langsmith \
  -n "<n-parallel-trials>"
```

Harbor 每个试用创建一个 LangSmith 沙盒，并在其中运行 agent 和验证器。

### 配置沙盒环境

LangSmith 环境从文件系统快照启动每个沙盒。在您的 Harbor 任务中提供以下其中一项：

- **预构建镜像**：在 `task.toml` 中设置 `[environment].docker_image`。Harbor 从该镜像重用或创建快照。
- **现有快照**：传递 `environment.kwargs.snapshot_name` 以从您已创建的[快照](/langsmith/sandbox-snapshots)启动。
- **Dockerfile**：包含 `environment/Dockerfile`。Harbor 使用[从 Dockerfile 构建流程](/langsmith/sandbox-snapshots#build-a-snapshot-from-a-dockerfile)从中构建快照，使用任务 `environment/` 目录作为构建上下文。

使用 environment kwargs 调整沙盒生命周期，通过命令行以 `--ek` 传递：

```bash
harbor run -d "<org/name>" \
  --model "<model>" \
  --agent "<agent>" \
  --env langsmith \
  -n "<n-parallel-trials>" \
  --ek idle_ttl_seconds=0 \
  --ek delete_after_stop_seconds=7200
```

- **`idle_ttl_seconds`**：在指定的秒数后停止空闲沙箱。设置为 `0` 可禁用空闲超时。
- **`delete_after_stop_seconds`**：在指定的秒数后删除已停止的沙箱。

## 故障排除

- **作业启动失败并出现身份验证错误**：请确认已设置 `LANGSMITH_API_KEY`，或 `LANGSMITH_PROFILE` 指向已配置的配置文件。
- **代理为模型抛出 `ValueError`**：请以 `provider:model` 格式传递 `--model`，并安装匹配的 `langchain-*` provider 包，以便 `init_chat_model()` 能够解析它。
