Fleet 为您提供跨组织扩展代理的控制层：分层权限、凭证管理、人工介入监督以及代理操作的审计跟踪。

## 权限与共享

Fleet 在两个维度上提供对每个代理的细粒度控制：**谁可以访问**以及**他们可以做什么**。

- **谁**：与个人用户或整个工作空间共享。
- **什么**：三个权限级别：
  - **克隆** — 复制并自定义代理
  - **运行** — 使用但不能修改
  - **编辑** — 完全访问权限，可以修改指令、工具和设置

您可以分层组合这些权限。授予核心团队编辑权限，向更广泛的组织共享仅运行权限，并可随时撤销。

有关设置说明，请参阅[更改代理的访问权限](/langsmith/fleet/manage-agent-settings#change-access-to-the-agent)。

## 代理身份与凭证

Fleet 提供两种凭证模型，用于控制代理如何与外部工具进行身份验证：

- **固定凭证（"Claws"）**：代理使用单一凭证集，无论谁运行它。适用于共享资源代理（如团队 Slack 机器人），因为每个人都通过同一账户进行交互。
- **用户凭证（"Assistants"）**：代理代表调用它的个人用户行事。每个用户通过 OAuth 使用自己的账户进行身份验证。适用于用户具有不同访问级别的工具，如个人电子邮件助手。

每个代理都可进行配置，因此您可以为每个用例选择合适的模型。

有关设置说明，请参阅[代理身份](/langsmith/fleet/agent-identity)。

## 工具访问控制

Fleet 为工具提供分层访问控制，涵盖**自定义 MCP 服务器**（用户添加的，工作空间范围的）和**内置集成**（平台提供的，如 Gmail、Slack 和 GitHub）：

- **[基于角色的访问控制 (RBAC)](#role-based-permissions)**：在角色级别控制访问。
- **[基于属性的访问控制 (ABAC)](#attribute-based-access-control)**：在 RBAC 基础上添加每个资源的细粒度。
- **[工作空间集成策略](#workspace-integration-policy)**：为内置集成提供管理员控制的启用/禁用开关。

<Note>
工具访问控制是一项企业功能。如果您对此功能感兴趣，请[联系我们的销售团队](https://www.langchain.com/contact-sales)。
</Note>

### 基于角色的权限

基于角色的访问控制 (RBAC) 根据用户的角色授予或拒绝访问工作空间中的所有 MCP 服务器和集成。在**设置 > 角色**中配置角色。

以下权限适用于 MCP 服务器和集成：

| 权限 | 描述 |
|------|------|
| `mcp-servers:read` | 发现并列出 MCP 服务器和集成 |
| `mcp-servers:invoke` | 执行 MCP 服务器和集成中的工具，包括 OAuth 连接/断开连接 |
| `mcp-servers:create` | 创建新的 MCP 服务器配置 |
| `mcp-servers:update` | 修改 MCP 服务器配置 |
| `mcp-servers:delete` | 删除 MCP 服务器配置 |

<Note>
拥有 `mcp-servers:read` 和 `mcp-servers:invoke` 权限的角色可以查看和使用工作空间中的所有 MCP 服务器和集成。
</Note>

有关 RBAC 的更多信息，请参阅[基于角色的访问控制](/langsmith/rbac)。

#### 创建具有工具权限的角色

<Steps>
  <Step title="打开角色设置">
    导航到**设置 > 角色**并点击**创建角色**。
  </Step>
  <Step title="配置 MCP 服务器权限">
    展开**MCP 服务器**部分并选择要包含的权限。例如，为需要使用工具但不需要管理服务器配置的用户授予`读取`和`调用`权限。
  </Step>
  <Step title="分配角色">
    在**设置 > 成员**中为工作空间中的用户分配角色。
  </Step>
</Steps>

### 基于属性的访问控制

基于属性的访问控制 (ABAC) 在 RBAC 基础上添加资源级别的细粒度。管理员可以为单个 MCP 服务器或集成添加标签，并创建基于这些标签授予或限制访问的策略。

ABAC 对工具使用两种资源类型：

| 资源类型 | 适用于 |
|----------|--------|
| `mcp_server` | 添加到工作空间的自定义 MCP 服务器 |
| `fleet_integration` | 内置集成（Gmail、Slack、GitHub 等） |

<Note>
没有 `mcp-servers:*` RBAC 权限的角色仍然可以通过 ABAC 允许策略授予对特定标签资源的访问权限（例如仅限 Notion 和 Gmail）。相反，拥有广泛 RBAC 访问权限的角色可以通过 ABAC 拒绝策略被限制访问特定资源。
</Note>

有关策略结构、运算符和通过 API 管理策略的详细信息，请参阅[基于属性的访问控制](/langsmith/abac)。

### 工作空间集成策略

内置集成有额外的控制层：工作空间级别的启用/禁用开关，从**设置 > 集成 > 访问控制**管理。这充当管理员控制的基础线，在每个用户的 RBAC 和 ABAC 之前运行。

如果集成在工作空间级别被禁用，则无论其角色或 ABAC 策略如何，没有任何用户可以访问它。

<Note>
访问控制页面仅对管理员用户可见（需要 `workspaces:manage` 权限）。
</Note>

### 策略评估顺序

这三个层按顺序评估。评估顺序在自定义 MCP 服务器和内置集成之间略有不同：

**自定义 MCP 服务器：**

```
ABAC deny → RBAC → ABAC allow
```

**内置集成：**

```
Workspace policy gate → ABAC deny → RBAC → ABAC allow
```

## 可观测性与审计跟踪

Fleet 中的代理操作被捕获到结构化的 [LangSmith 追踪](/langsmith/observability) 中，包括工具调用、决策和输出。您可以检查、搜索和导出追踪。

结合代理身份和权限，追踪可以告诉您哪个代理执行了操作、代表谁、使用什么凭据，以及每个步骤执行了什么操作。

## 人工介入监督

Fleet 提供了一个[集中式收件箱](https://smith.langchain.com/agents/inbox)，用于审查所有代理的操作。您可以配置代理在执行特定操作前暂停并请求批准，然后从一个地方进行审查、批准、编辑或拒绝。

有关设置说明，请参阅[人工介入](/langsmith/fleet/essentials#human-in-the-loop)。
