无头工具让你的智能体可以调用那些实际执行必须在用户应用而非服务器端完成的工具。智能体仍然看到的是正常的工具模式，但实现位于前端，在那里可以访问浏览器 API，如 IndexedDB、地理位置、剪贴板、画布或文件选择器。

这种模式在数据应该保留在本地设备上时特别有用。此页面上的 playground 示例使用了一个由 IndexedDB 支持的小型浏览器内存工具包，外加一个完全在客户端运行的地理位置工具。

## headless 工具的工作原理

从高层次来看，headless 工具将工具模式与浏览器专用实现分离。

:::python
1. 在 agent 上注册一个工具，该工具立即调用 `interrupt()` 以将执行推迟到前端。
2. 在前端定义中镜像相同的工具名称和参数字段。
3. 在前端中使用 `.implement(...)` 实现匹配的 tools，并通过 `useStream({ tools: [...] })` 传递它们。
4. 当 agent 调用匹配的 tool 时，客户端处理该操作，并使用工具结果恢复中断的运行。
:::

:::js
1. 在 agent 上注册一个仅模式的工具定义。
2. 在前端中使用 `.implement(...)` 实现匹配的 tool。
3. 将这些实现传递给 `useStream({ tools: [...] })`。
4. 当 agent 发出匹配的 tool 调用时，客户端运行它，并使用工具结果恢复中断的运行。

<Tip>
将工具定义和实现保持在独立的模块中。在 agent 和前端之间共享定义，
以保持工具名称和模式的一致性，然后将仅限浏览器的代码保留在仅客户端的 `impl` 模块中。
</Tip>
:::

## 在 agent 上注册工具

playground 定义了一组遵循相同模式的小型客户端工具：agent 暴露工具模式，
而前端处理实际执行。

:::python
在服务器上定义普通工具，这些工具立即调用 `interrupt()`，然后在
前端的 `tools.ts` 文件中镜像相同的工具名称和参数字段。

</:::python>

```python agent.py
from typing import Any

from langchain import create_agent
from langchain.tools import ToolRuntime, tool
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import interrupt
from pydantic import BaseModel


class MemoryPutInput(BaseModel):
    key: str
    value: Any


class MemoryGetInput(BaseModel):
    key: str


class GeolocationGetInput(BaseModel):
    save: bool = True


def _interrupt_for_client(
    tool_name: str,
    args: dict[str, Any],
    runtime: ToolRuntime,
) -> Any:
    return interrupt({
        "type": "tool",
        "tool_call": {
            "id": runtime.tool_call_id,
            "name": tool_name,
            "args": args,
        },
    })


@tool(
    "memory_put",
    description="Store a memory in the user's browser.",
    args_schema=MemoryPutInput,
)
def memory_put(key: str, value: Any, runtime: ToolRuntime) -> Any:
    return _interrupt_for_client(
        "memory_put",
        {"key": key, "value": value},
        runtime,
    )


@tool(
    "memory_get",
    description="Look up a memory stored in the user's browser.",
    args_schema=MemoryGetInput,
)
def memory_get(key: str, runtime: ToolRuntime) -> Any:
    return _interrupt_for_client("memory_get", {"key": key}, runtime)


@tool(
    "geolocation_get",
    description="Get the user's current location from the browser.",
    args_schema=GeolocationGetInput,
)
def geolocation_get(runtime: ToolRuntime, save: bool = True) -> Any:
    return _interrupt_for_client(
        "geolocation_get",
        {"save": save},
        runtime,
    )

agent = create_agent(
    model="openai:gpt-5.5",
    tools=[memory_put, memory_get, geolocation_get],
    checkpointer=MemorySaver(),
)
```

每个工具都会以一个前端可以处理的结构化负载进行中断，然后在运行恢复时返回提供的值。在客户端上镜像相同的工具名称和 schema，以便前端附加实现。


## 实现浏览器行为

将仅客户端行为放在单独的模块中，并通过 `.implement(...)` 附加。实际的 playground 包含一个更完整的 IndexedDB 存储，具有搜索、列表、过期和删除操作。以下示例展示更高层次的相同结构：

```ts impl.ts
import {
  geolocationGet as geolocationGetDefinition,
  memoryGet as memoryGetDefinition,
  memoryPut as memoryPutDefinition,
} from "./tools";

async function saveMemory(key: string, value: unknown) {
  localStorage.setItem(`agent-memory:${key}`, JSON.stringify(value));
}

async function getMemory(key: string) {
  const value = localStorage.getItem(`agent-memory:${key}`);
  return value ? JSON.parse(value) : null;
}

export const memoryPut = memoryPutDefinition.implement(async ({ key, value }) => {
  await saveMemory(key, value);
  return { success: true, key };
});

export const memoryGet = memoryGetDefinition.implement(async ({ key }) => {
  const value = await getMemory(key);
  return value === null ? { found: false, key } : { found: true, key, value };
});

export const geolocationGet = geolocationGetDefinition.implement(
  async ({ save = true }) => {
    const position = await new Promise<GeolocationPosition>((resolve, reject) =>
      navigator.geolocation.getCurrentPosition(resolve, reject),
    );

    const location = {
      latitude: position.coords.latitude,
      longitude: position.coords.longitude,
      accuracy: position.coords.accuracy,
    };

    if (save) {
      await saveMemory("user_location", location);
    }

    return location;
  },
);
```

## 将实现接入 `useStream`

将已实现的工具传递给 `useStream`。当 agent 发出匹配的工具调用时，hook 会运行客户端实现并为你恢复运行。

:::python

定义一个与你的 agent 状态 schema 匹配的 TypeScript 接口，并将其作为类型参数传递给 `useStream`，以获得对状态值的类型安全访问：

```ts types.ts
export interface AgentState {
  messages: BaseMessage[];
}
```

:::js

## 渲染工具活动内联

Playground 会将每个内存或地理位置操作渲染为独立的卡片，并在输入框附近保留一个小的内存统计面板。关键步骤是将 `stream.toolCalls` 中的每个条目与其触发的 AI 消息进行匹配：

```tsx
import type { ToolCallWithResult, DefaultToolCall } from "@langchain/react";

function Message({ message, toolCalls }: {
  message: AIMessage,
  toolCalls: ToolCallWithResult[]
}) {
  const messageToolCalls = toolCalls.filter((tc) =>
    message.tool_calls?.some((call) => call.id === tc.call.id),
  );

  return (
    <div>
      {message.text && <p>{message.text}</p>}
      {messageToolCalls.map((tc) => (
        <HeadlessToolCard key={tc.call.id} toolCall={tc} />
      ))}
    </div>
  );
}
```

这在与 [Tool calling](/oss/langchain/frontend/tool-calling) 中更丰富的 UI 模式配合使用时效果特别好，每个工具结果可以渲染为专用卡片，而不是原始 JSON。

## 使用场景

在以下情况下，使用无头工具效果最佳：

- IndexedDB 或 `localStorage` 中的本地内存
- 设备 API，如地理位置、剪贴板、摄像头或文件选择器
- Canvas、audio 或其他仅限浏览器的渲染原语
- 应保留在用户设备上的隐私敏感数据
- 需要直接访问内存中前端状态的 UI 操作

## 最佳实践

- 保持工具小而类型化。优先选择多个窄工具而非一个通用的"运行任意浏览器代码"工具。
- 返回 JSON 可序列化的结果。不要尝试返回 DOM 节点、文件句柄或其他不可序列化的浏览器对象。
- 共享定义，分离实现。代理和客户端应就工具名称和模式达成一致，但只有客户端应加载浏览器 API。
- 在 UI 中显示工具状态。使用 `stream.toolCalls` 和 `onTool` 来显示待处理、成功和错误状态。
- 根据需要添加审核。对于敏感的客户端操作，请将此模式与 [Human-in-the-loop](/oss/langchain/frontend/human-in-the-loop) 结合使用。
