[Agent Server](/langsmith/agent-server) 是一个用于创建和管理基于代理的应用程序的 API 平台。它提供内置持久化、任务队列，并支持大规模部署、配置和运行助手（代理工作流）。此更新日志记录了 Agent Server 发布的所有重要更新、功能和修复。

## v0.11.0rc1

### 一般说明
- 包含依赖项和安全维护更新。

### 新功能
- 添加了 DeltaChannel 感知剪枝，仅保留状态重建所需的最小祖先检查点，取代了之前拒绝剪枝活跃增量通道线程的方法。支持 Postgres、SQLite、DeferredDelete 和内存运行时。
- 为 MongoDB 检查点程序添加了 DeltaChannel 感知剪枝，仅保留状态重建所需的最小祖先检查点和增量通道 blob。
- 添加了可选的 Prometheus 指标抓取支持。设置 `LSD_PROM_METRICS_ENABLED=true` 以在专用 Prometheus 抓取端点（默认端口 `LSD_PROM_METRICS_PORT` ）上公开 OTel 指标（运行生命周期、延迟、流、worker 指标）。同时配置 Datadog OTLP 推送时，Datadog OTLP 推送仍可继续工作。
- Added opt-in Go core rate limits for unary core-api RPCs and Redis stream publish bytes, with Redis-backed GCRA enforcement, shadow/enforce modes, and `LS_RATE_LIMITS` bootstrap 配置，支持 YAML 覆盖。
- 允许传递自定义证书和密钥文件（`ssl_certfile`, `ssl_keyfile`）以通过 HTTPS 运行开发服务器。

### 修复
- 修复了协议 v2 在 JS 图上静默失败的运行。sidecar 拒绝了 `streamEvents` 由于严格的流模式验证导致 400 错误，错误被吞没，运行错误地报告成功但执行了 0 个节点。在 HTTP 边界放宽了流模式验证，现在对非 2xx sidecar 响应抛出明确错误，而不是掩盖失败。
- 修复了针对 JS sidecar（远程）图的协议 v2 事件流，这些图错误地通过遗留重建路径提供服务。远程图现在使用 LangGraphJS 原生 v3 流进行 v2 事件流运行，解决了工具调用不渲染、无头中断从不执行或恢复的问题，以及 `400: tool_use ids must be unique` 恢复后最终消息的错误。
- 删除运行现在跳过使用 DeltaChannel 的线程的检查点删除，仅删除运行记录。保留后续检查点依赖的存储增量写入的检查点。使用线程修剪 API 回收 delta-channel 线程上的检查点存储。
- 修复了 Prometheus 指标导出并对齐了 OpenTelemetry 导出器配置。
</Update>

## v0.10.0rc1

### 概述
- v0.10.0rc1 包含安全性和正确性的破坏性变更。请参阅 [安全部分](#security) 了解更多详情。
- 包含依赖项和安全维护更新。

### 新功能
- 添加了按 ID 检索 cron 的端点（`GET /runs/crons/{cron_id}`).

### 修复
- 修复了事件流 v2 运行启动处理，使通过 `config.configurable.checkpoint_id` 提供的检查点重放目标现在会被正确遵守。
- 修复了事件流 v2 `input.respond` 返回 `no_such_interrupt` 对于通过 HTTP 的 postgres 后端上的合法中断 `POST /commands`.
- 修复了一个错误，其中线程的 `checkpoint_map` 会从之前的时间旅行运行中持久化并污染后续的 `Command(resume=...)`，导致嵌套子图错误地从开始处重放。

### 安全
- **可能破坏兼容性** 现在默认拒绝 loopback webhook 目标以修复身份验证绕过原语（[GHSA-2c9q-c2q9-qgqv](https://github.com/langchain-ai/helm/security/advisories/GHSA-2c9q-c2q9-qgqv)）。 `webhooks.url.disable_loopback` 策略现在默认为 `true`, blocking relative-URL webhooks (which dispatch through the in-process ASGI transport and bypass auth), as well as localhost / 127.x / ::1 / host.docker.internal absolute URLs and any hostname that DNS-resolves into the loopback range (mitigating DNS rebinding). Deployments that legitimately need loopback webhooks (e.g. `langgraph dev` 使用本地主机 Webhook 接收器的设置，或分派到挂载在同一服务器上的自定义 FastAPI 路由的生产设置，可以通过设置 `webhooks.url.disable_loopback: false` in `langgraph.json` （或等效的 `LANGGRAPH_WEBHOOKS` JSON 环境变量）。仅在您控制循环 Webhook 到达的路由时执行此操作，因为这些路由在分派时不进行身份验证。
- **可能破坏性** `POST /runs` 和 `POST /threads/{thread_id}/runs` 现在通过`assistants.read` 认证事件（匹配 cron 创建和直接 GET）进行授权，而不是之前使用的 `assistants.search` 带有不完整负载的事件（[GHSA-jfj5-wrj9-63x4](https://github.com/langchain-ai/helm/security/advisories/GHSA-jfj5-wrj9-63x4)）。仅注册了 `@auth.on.assistants.read` （且无 `.search` handler) were vulnerable to a cross-user authorization bypass; their existing read handler will now be invoked on the run-creation path. As a defense-in-depth follow-up, client-supplied run/cron metadata is no longer forwarded into the `assistants.read` 来自 `Runs.put` or `Crons.put`, and inmem/postgres runtimes now agree on the value shape. Breaking change for deployments with custom auth handlers: (1) any `@auth.on.assistants.search` 处理程序的认证事件负载在运行创建期间不再被调用——确保您有等效的`@auth.on.assistants.read` 处理程序返回相同的所有者风格过滤器；(2) `value["metadata"]` 在 `assistants.read` event invoked from run/cron creation is no longer populated, so handlers that inspected or mutated it must move that logic into `@auth.on.runs.create_run` / `@auth.on.crons.create` 并依赖返回过滤器进行所有权强制执行。
- 部署现在会在服务器启动时看到结构化警告，列出每个未覆盖的分派路径以及要复制的默认拒绝代码片段。对于注册了全局 `@auth.on` 处理程序或仅使用 `@auth.authenticate` 而不使用任何资源级处理程序的部署，此警告为静默。

</Update>

## v0.9.0rc1

### 概述
- Added cron metadata filtering in /runs/crons/search and /runs/crons/count, matching metadata filter behavior already available for assistants/threads.
- 为一次性加载大量大型检查点的情况添加了 Postgres 检查点池调优旋钮。LANGGRAPH_CHECKPOINTER_POSTGRES_POOL_MIN_SIZE 和 LANGGRAPH_CHECKPOINTER_POSTGRES_POOL_TIMEOUT_SECONDS 现在可以设置。
- 修复了更新_MongoDB 检查点中当线程没有先前检查点时的崩溃问题。
- 包含针对安全漏洞的依赖项更新。

### 新功能
#### Delta 通道支持
现在支持 Delta 通道，以便检查点可以存储增量状态更新，而不是重复存储完整通道负载，这有助于处理大型、追加密集型的状态（如消息历史）。

要使用，请在图形状态中定义具有 LangGraph DeltaChannel 化简器模式的通道。
当安装的 langgraph >= 1.2 时启用此行为。
Docs: [DeltaChannel 参考](/oss/langgraph/pregel#deltachannel)

#### 事件流式传输 API
Event streaming APIs are being introduced, with a unified event-streaming surface intended for richer real-time run events and command/event workflows.

功能标志 `FF_V2_EVENT_STREAMING` 可以设置为 true 以启用新的事件流式传输 API。

新的端点包括：
- `POST /threads/{thread_id}/stream/events`
- `POST /threads/{thread_id}/commands`
- `WS /threads/{thread_id}/stream/events`

Docs:
- [Agent Server API 参考](/langsmith/server-api-ref)
- [LangGraph 事件流式传输参考](/oss/langgraph/event-streaming)

</Update>

## v0.8.0

此小版本将运行队列轮询从 Postgres 移至 Redis，节省了数据库负载并提升了性能。

在底层，Agent Server 使用持久化运行队列来管理运行执行。Worker 轮询队列获取新运行并执行它们。之前，队列轮询逻辑通过 Postgres。这可能导致长时间运行的查询，尤其是在高负载下。通过此更新，队列轮询逻辑现在通过 Redis，然后从 Postgres 获取运行详情。这使得队列轮询的热路径大大加快，并减少了数据库的负载。

这不是破坏性更改，升级时无需代码更改，但有几件事需要注意：
- 在升级后的部署中，队列将切换。可能会有一段时间线程被非时间顺序调度。每个线程内的运行执行顺序仍有保证。
- **仅限自托管：** Redis 流量可能略有增加。在内部测试中，增加幅度很小。
</Update>

## v0.6.0

此小版本更新了流式API `/join-stream` 和 `/stream` 关于 `last-event-id` 参数的行为以符合SSE规范。以前，传递last-event-id会返回该消息以及后续的所有消息。今后，这些API将仅返回所提供的last-event-id之后的新消息。例如，对于以下流，之前传递last-event-id为 `2` 会返回id为 `2` 和 `3`的消息，但现在只会返回id为 `3`:

```json
{
    "id": 1,
    "event": "message",
    "data": {
        "content": "Excluded"
    }
},
{
    "id": 2,
    "event": "message",
    "data": {
        "content": "Passed last-event-id"
    }
},
{
    "id": 3,
    "event": "message",
    "data": {
        "content": "Included"
    }
}
```

此版本还包含一些修复，包括修复了运行流中暴露意外内部事件的bug。


</Update>
