[Agent Server](/langsmith/agent-server) 是一个用于创建和管理基于代理的应用程序的 API 平台。它提供内置持久化、任务队列，并支持大规模部署、配置和运行助手（代理工作流）。此更新日志记录了 Agent Server 版本的所有重要更新、功能和修复。

## v0.11.0rc1

### 总体说明
- 包含依赖项和安全维护更新。

### 新功能
- 新增 DeltaChannel 感知剪枝功能，仅保留状态重建所需的最少祖先检查点，替代之前拒绝剪枝包含活跃增量通道的线程的行为。支持 Postgres、SQLite、DeferredDelete 和内存运行时。
- MongoDB 检查点程序新增 DeltaChannel 感知剪枝功能，仅保留状态重建所需的最少祖先检查点和增量通道 blob。
- 新增可选的 Prometheus 指标采集支持。将 `LSD_PROM_METRICS_ENABLED=true` 设置为在专用 Prometheus 采集端点上暴露 OTel 指标（运行生命周期、延迟、流、worker 仪表），端口为 `LSD_PROM_METRICS_PORT`（默认为 9464）。配置两者时，Datadog OTLP 推送可与 Prometheus 同时工作。
- 新增可选的 Go 核心速率限制功能，用于 unary core-api RPC 和 Redis stream 发布字节，采用 Redis 后端的 GCRA 强制执行，支持 shadow/enforce 模式，以及 `LS_RATE_LIMITS` 引导配置（可使用 YAML 覆盖）。
- 允许传递自定义证书和密钥文件（`ssl_certfile`、`ssl_keyfile`）以通过 HTTPS 运行开发服务器。

### 修复
- 修复了 JS 图形上协议 v2 运行静默失败的问题。sidecar 因严格的流模式验证拒绝了 `streamEvents` 并返回 400，错误被吞掉，导致运行虚假报告成功且执行了 0 个节点。现在放宽了 HTTP 边界处的流模式验证，对非 2xx sidecar 响应抛出明确错误，而不是掩盖失败。
- 修复了针对 JS sidecar（远程）图形的协议 v2 事件流问题，这些图形之前被错误地通过遗留重构路径提供。远程图形现在使用 LangGraphJS 原生 v3 流进行 v2 事件流运行，解决了工具调用不渲染、无头中断从不执行或恢复，以及恢复后最终消息出现 `400: tool_use ids must be unique` 错误的问题。
- 删除运行现在跳过 DeltaChannel 线程的检查点删除，仅移除运行记录。保留存储后续检查点所依赖的增量写入的检查点。使用线程剪枝 API 回收 delta-channel 线程上的检查点存储。
- 修复了 Prometheus 指标导出并对齐了 OpenTelemetry 导出器配置。
</Update>

## v0.10.0rc1

### 总体说明
- v0.10.0rc1 包含安全和正确性方面的破坏性变更。有关详细信息，请参阅[安全部分](#security)。
- 包含依赖和安全维护更新。

### 新功能
- 添加了按 ID 获取 cron 的端点（`GET /runs/crons/{cron_id}`）。

### 修复
- 修复了事件流 v2 运行启动处理，以便通过 `config.configurable.checkpoint_id` 提供的检查点重放目标能够被正确遵守。
- 修复了事件流 v2 `input.respond` 在通过 HTTP `POST /commands` 使用 Postgres 后端时对合法中断返回 `no_such_interrupt` 的问题。
- 修复了一个 bug，其中线程之前时间旅行运行的 `checkpoint_map` 会持续存在并污染后续的 `Command(resume=...)`，导致嵌套子图错误地从开始处重放。

### 安全
- **可能具有破坏性** 现在默认拒绝环回 webhook 目标，以修复身份验证绕过原语（[GHSA-2c9q-c2q9-qgqv](https://github.com/langchain-ai/helm/security/advisories/GHSA-2c9q-c2q9-qgqv)）。`webhooks.url.disable_loopback` 策略现在默认为 `true`，阻止相对 URL webhook（通过进程内 ASGI 传输调度并绕过身份验证）以及 localhost / 127.x / ::1 / host.docker.internal 绝对 URL，以及任何 DNS 解析到环回范围的主机名（缓解 DNS 重绑定）。确实需要环回 webhook 的部署（例如使用 localhost webhook 接收器的 `langgraph dev`，或调度到同一服务器上挂载的自定义 FastAPI 路由的生产设置）可以通过在 `langgraph.json` 中设置 `webhooks.url.disable_loopback: false`（或等效的 `LANGGRAPH_WEBHOOKS` JSON 环境变量）来重新启用。只有在您控制环回 webhook 到达的路由时才执行此操作，因为这些路由在调度时不进行身份验证。
- **可能具有破坏性** `POST /runs` 和 `POST /threads/{thread_id}/runs` 现在通过 `assistants.read` 身份验证事件附加授权的助手（与 cron 创建和直接 GET 一致），而不是之前使用的带有不完整载荷的 `assistants.search` 事件（[GHSA-jfj5-wrj9-63x4](https://github.com/langchain-ai/helm/security/advisories/GHSA-jfj5-wrj9-63x4)）。仅注册了 `@auth.on.assistants.read`（而没有 `.search` 处理程序）的部署容易受到跨用户授权绕过的影响；它们现有的读取处理程序现在将在运行创建路径上被调用。作为纵深防御的后续措施，客户端提供的运行/cron 元数据不再从 `Runs.put` 或 `Crons.put` 转发到 `assistants.read` auth 事件载荷中，inmem/postgres 运行时现在在值形状上达成一致。对于具有自定义身份验证处理程序的部署的破坏性变更：(1) 之前在运行创建期间调用的任何 `@auth.on.assistants.search` 处理程序不再在那里被调用——确保您有一个等效的 `@auth.on.assistants.read` 处理程序返回相同的所有者风格过滤器；(2) 从运行/cron 创建调用的 `assistants.read` 事件上的 `value["metadata"]` 不再被填充，因此检查或修改它的处理程序必须将该逻辑移动到 `@auth.on.runs.create_run` / `@auth.on.crons.create` 中，并依靠返回过滤器进行所有权强制执行。
- 部署现在在服务器启动时看到结构化警告，列出每个未覆盖的调度路径以及要复制的默认拒绝代码片段。对于注册了全局 `@auth.on` 处理程序或仅使用 `@auth.authenticate` 而没有任何资源级处理程序的部署，警告是静默的。

</Update>

## v0.9.0rc1

### 总体说明
- 在 /runs/crons/search 和 /runs/crons/count 中添加了 cron 元数据过滤，匹配助手/线程已有的元数据过滤行为。
- 为一次加载大量大型检查点的情况添加了 Postgres 检查点池调优旋钮。现在可以设置 LANGGRAPH_CHECKPOINTER_POSTGRES_POOL_MIN_SIZE 和 LANGGRAPH_CHECKPOINTER_POSTGRES_POOL_TIMEOUT_SECONDS。
- 修复了 mongo 检查点中线程没有先前检查点时 update_state 的崩溃。
- 包含安全漏洞的依赖更新。

### 新功能
#### Delta 通道支持
现在支持 Delta 通道，因此检查点可以存储增量状态更新，而不是重复存储完整通道载荷，这有助于处理大型、追加密集的状态（如消息历史）。

要使用，请在图状态中使用 LangGraph 的 DeltaChannel reducer 模式定义状态通道。
当安装的 langgraph >= 1.2 时启用此行为。
文档：[DeltaChannel 参考](/oss/langgraph/pregel#deltachannel)

#### 事件流 API
正在引入事件流 API，具有统一的 events 流表面，旨在提供更丰富的实时运行事件和命令/事件工作流。

功能标志 `FF_V2_EVENT_STREAMING` 可设置为 true 以启用新的事件流 API。

新端点包括：
- `POST /threads/{thread_id}/stream/events`
- `POST /threads/{thread_id}/commands`
- `WS /threads/{thread_id}/stream/events`

文档：
- [Agent Server API 参考](/langsmith/server-api-ref)
- [LangGraph 事件流参考](/oss/langgraph/event-streaming)

</Update>

## v0.8.0

此小版本将运行队列轮询从 Postgres 迁移到 Redis，节省数据库负载并提高性能。

在底层，Agent Server 使用持久化运行队列来管理运行执行。工作进程轮询队列获取新运行并执行它们。之前，队列轮询逻辑通过 Postgres。这在高负载下可能导致长查询。通过此更新，队列轮询逻辑现在通过 Redis，然后从 Postgres 获取运行详情。这使得队列轮询的热门路径大大加快，并减少数据库负载。

这不是破坏性更改，升级时不需要代码更改，但有两点需要注意：
- 升级后立即部署时，队列将进行切换。可能会有一个短暂的时间窗口，线程会按非时间顺序调度。每个线程内的运行执行顺序仍然有保证。
- **仅限自托管：** Redis 流量可能略有增加。在内部测试中，增加量很小。
</Update>

## v0.5.0

此 minor 版本现在需要 langgraph-checkpoint 3.0 之后的版本，以防止早期版本 langgraph-checkpoint 库中的反序列化漏洞。
`langgraph-checkpoint` 库与 `langgraph` minor 版本 0.4、0.5、0.6 和 1.0 兼容。

此版本移除了对使用"json"类型保存的有效载荷的反序列化默认支持，而这从未是默认选项。
默认情况下，对象使用 msgpack 进行序列化。在某些不常见的情况下，有效载荷可能使用较旧的"json"模式进行序列化。如果这些有效载荷包含自定义 Python 对象，除非您提供 `serde` 配置，否则它们将无法再被反序列化：

```json
{
    "checkpointer": {
        "serde": {
            "allowed_json_modules": [
                ["my_agent", "my_file", "SomeType"],
            ]
        }
    }
}
```
