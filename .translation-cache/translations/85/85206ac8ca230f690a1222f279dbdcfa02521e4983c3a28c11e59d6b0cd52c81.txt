LangSmith 自托管版通过 OAuth2.0 和 OIDC 提供 SSO。这将把身份验证委托给您的身份提供商（IdP）来管理对 LangSmith 的访问。

我们的实现支持几乎所有符合 OIDC 标准的服务，但有一些例外。配置完成后，您将看到如下登录界面：

![LangSmith UI with OAuth SSO](/langsmith/images/langsmith-ui-sso.png)

## 概述

<Note>
您可以将使用[基本认证](/langsmith/self-host-basic-auth)的安装升级到此模式，但无法将使用[无认证](/langsmith/authentication-methods#none)的安装升级到此模式。升级时，只需移除基本认证配置并添加如下所示的必需配置参数即可。之后用户将只能通过 OAuth 登录。**为了在升级后保持访问权限，您必须能够使用之前通过基本认证登录过的电子邮件地址通过 OAuth 登录。**
</Note>

<Warning>
LangSmith 目前不支持在自托管中从 SSO 切换到基本认证模式。我们也不支持在带客户端密钥的 OAuth 模式和不带客户端密钥的 OAuth 模式之间切换，反之亦然。最后，我们不支持同时启用基本认证和 OAuth。在启用 OAuth 时，请确保禁用基本认证配置。
</Warning>

## 使用客户端密钥（推荐）

默认情况下，LangSmith 自托管版支持使用 `Client Secret` 的 `Authorization Code` 流程。在此版本的流程中，您的客户端密钥安全地存储在 LangSmith 中（而不是前端），并用于身份验证和建立认证会话。

### 前置要求

* 您必须是自托管用户，且使用 Enterprise 计划。
* 您的 IdP 必须支持使用 `Client Secret` 的 `Authorization Code` 流程。
* 您的 IdP 必须支持使用外部 discovery/issuer URL。我们将使用它来获取 IdP 的必要路由和密钥。
* 您必须向 LangSmith 提供 `OIDC`、`email` 和 `profile` 作用域。我们使用这些作用域来获取用户必要的用户信息和电子邮件。
<Note>

LangSmith SSO 仅支持通过 `https` 访问。
</Note>

### 配置

* 您需要在 IdP 中设置回调 URL 为 `https://<host>/api/v1/oauth/custom-oidc/callback`，其中 `host` 是您为 LangSmith 实例配置的域名或 IP。这是 IdP 在用户完成身份验证后将用户重定向到的位置。
* 要在注销时终止 IdP 会话（使用户必须重新认证），请在 IdP 中将您的 LangSmith URL（例如 `https://<host>`）注册为**注销后重定向 URI**（有时称为"登出重定向 URI"），然后在您的环境变量中设置 `OAUTH_IDP_LOGOUT_ENABLED=true`（通过 Helm 中的 `commonEnv` 或 Docker Compose 中的 `.env` 文件）。
* 您需要在 `values.yaml` 文件中提供 `oauthClientId`、`oauthClientSecret`、`hostname` 和 `oauthIssuerUrl`。这是您配置 LangSmith 实例的地方。
* 如果您**尚未**配置带客户端密钥的 OAuth，或者您只有个人组织，则必须提供一个电子邮件地址来分配为新配置 SSO 组织的初始组织管理员。如果您是从基本认证升级的，则您的现有组织将被重复使用。

<CodeGroup>


</fragment>

```yaml Helm
config:
  authType: mixed
  hostname: https://langsmith.example.com
  initialOrgAdminEmail: test@email.com # Set this if required
  oauth:
    enabled: true
    oauthClientId: <YOUR CLIENT ID>
    oauthClientSecret: <YOUR CLIENT SECRET>
    oauthIssuerUrl: <YOUR DISCOVERY URL>
    oauthScopes: "email,profile,openid"
```

```bash Docker
# In your .env file
AUTH_TYPE=mixed
INITIAL_ORG_ADMIN_EMAIL=test@email.com
LANGSMITH_URL=https://langsmith.example.com
OAUTH_CLIENT_ID=your-client-id
OAUTH_CLIENT_SECRET=your-client-secret
OAUTH_ISSUER_URL=https://your-issuer-url
OAUTH_SCOPES=email,profile,openid
```

</CodeGroup>

### 会话时长控制

<Note>
本节中的所有环境变量都适用于 `platform-backend` 服务，可通过 Helm 中的 `platformBackend.deployment.extraEnv` 添加。
</Note>

- 默认情况下，会话时长由身份提供商返回的身份令牌的过期时间控制
- 大多数设置应使用刷新令牌，以支持将会话时长延长至超过身份令牌的过期时间，最长至 `OAUTH_SESSION_MAX_SEC`，这可能需要通过添加 `offline_access` 作用域来实现，可在 Helm 中添加到 `oauthScopes`，或在 Docker 中添加到 `OAUTH_SCOPES`
- `OAUTH_SESSION_MAX_SEC`（默认 1 天）可被覆盖，最长一周（`604800`）
- 对于不支持刷新令牌的身份提供商设置，将 `OAUTH_OVERRIDE_TOKEN_EXPIRY="true"` 设置为 `true` 将以 `OAUTH_SESSION_MAX_SEC` 作为会话时长，忽略身份令牌过期时间

### 覆盖 sub 声明

在某些场景下，可能需要覆盖身份提供商使用的 `sub` 声明来源。
例如，在 SCIM 中，解析后的 `sub` 声明和 SCIM `externalId` 必须匹配才能使登录成功。
如果对 `sub` 声明的来源属性和/或 SCIM `externalId` 有限制，请设置 `ISSUER_SUB_CLAIM_OVERRIDES` 环境变量来选择使用哪个 OIDC JWT 声明作为 `sub`。

如果颁发者 URL **以** 此配置中某个 URL 开头，则从指定的字段名中获取 `sub` 声明。
例如，使用以下配置，具有颁发者 `https://idp.yourdomain.com/application/uuid` 的令牌将使用 `customClaim` 值作为 `sub`：

```
ISSUER_SUB_CLAIM_OVERRIDES='{"https://idp.yourdomain.com": "customClaim"}'
```

如果未设置，此配置在将 Azure Entra ID 用作标识提供者时会使用 `oid` 声明作为默认值：

```
ISSUER_SUB_CLAIM_OVERRIDES='{"https://login.microsoftonline.com/": "oid", "https://sts.windows.net/": "oid", "https://login.microsoftonline.us/": "oid", "https://login.partner.microsoftonline.cn/": "oid"}'
```

### SSO 群组同步

<Note>
自托管版本的 SSO 群组同步需要 LangSmith chart 版本 **0.15.0-rc.3**（应用版本 **0.15.2rc1**）或更高版本。
</Note>

[SSO 群组同步](/langsmith/user-management#sso-groups-sync-alternative) 允许 LangSmith 根据 OIDC 令牌中的群组成员身份声明分配组织和项目角色，作为 [SCIM](/langsmith/user-management#set-up-scim-for-your-organization) 的更简单替代方案。在自托管部署中，您必须配置 IdP 在 OIDC ID 令牌中包含群组信息，LangSmith 才能读取这些信息。

**IdP 端配置（因提供商而异）：**

1. 配置您的 IdP 应用程序在 OIDC ID 令牌中发出群组成员身份声明。源属性和生成的声明名称因 IdP 而异。常见示例包括 `groups`、`roles` 或自定义声明名称。LangSmith 不规定源属性。
2. 根据您的 IdP，您可能需要将额外的 scope（通常是 `groups`）添加到 `oauthScopes` 中才能接收声明。请查阅您的 IdP 文档，了解所需的 scope 以及在令牌中包含群组成员身份所需的任何额外配置。
3. 群组名称必须遵循 [SCIM 命名规范](/langsmith/user-management#group-naming-convention)（例如 `LS:Organization Admin`、`LS:Organization User:prod:Editor`）。分隔符通过 [`scim_group_name_separator`](/langsmith/user-management#configure-custom-separator) 与 SCIM 共享。

**Helm 配置：**

如果您的 IdP 需要额外的 OIDC scope 才能在令牌中包含群组（通常是 `groups`），请将其添加到 `oauthScopes`：

<CodeGroup>

```yaml Helm
config:
  authType: mixed
  hostname: https://langsmith.example.com
  oauth:
    enabled: true
    oauthClientId: <YOUR CLIENT ID>
    oauthClientSecret: <YOUR CLIENT SECRET>
    oauthIssuerUrl: <YOUR DISCOVERY URL>
    oauthScopes: "email,profile,openid,groups"
```

```bash Docker
# In your .env file
AUTH_TYPE=mixed
LANGSMITH_URL=https://langsmith.example.com
OAUTH_CLIENT_ID=your-client-id
OAUTH_CLIENT_SECRET=your-client-secret
OAUTH_ISSUER_URL=https://your-issuer-url
OAUTH_SCOPES=email,profile,openid,groups
```

</CodeGroup>

具体的作用域名称（`groups`、`roles` 等）取决于你的 IdP。请查阅你的 IdP 的 OIDC 文档。

**LangSmith 端配置：**

每个提供商的 SSO Groups Sync 设置存储在 SSO 提供商记录中，并通过 LangSmith UI 或 [API](/langsmith/reference) 切换（不是通过 Helm values）。一旦你的 IdP 发出 groups 声明，请在 UI 中配置 SSO Groups Sync：进入 **Settings** → **Members and roles** → **SSO Configuration** → **SSO Groups Sync**。或者按照主要 SSO Groups Sync 文档中的说明通过 API 配置（请参阅 [SSO Groups Sync 主文档](/langsmith/user-management#sso-groups-sync-alternative)）。**Groups claim 字段**中配置的声明名称必须与你 IdP 发出的声明相匹配。

### Google Workspace IdP 设置

你可以使用 Google Workspace 作为单点登录 (SSO) 提供商，使用 [OAuth2.0 和 OIDC](https://developers.google.com/identity/openid-connect/openid-connect)，无需 PKCE。

<Note>
你必须拥有组织的 Google Cloud Platform (GCP) 账户的管理员级别访问权限才能创建新项目，或有权限为现有项目创建和配置 OAuth 2.0 凭证。我们建议创建一个新项目来管理访问权限，因为每个 GCP 项目只有一个 OAuth 同意屏幕。
</Note>

1. 创建一个新的 GCP 项目，请参阅 Google 文档主题[创建和管理项目](https://cloud.google.com/resource-manager/docs/creating-managing-projects)

2. 创建项目后，在 Google API Console 中打开[凭证](https://console.developers.google.com/apis/credentials)页面（确保左上角的项目是正确的）

3. 创建新凭证：`Create Credentials → OAuth client ID`

4. 选择 `Web application` 作为 `Application type`，并输入应用程序名称，例如 `LangSmith`

5. 在 `Authorized Javascript origins` 中输入你的 LangSmith 实例域名，例如 `https://langsmith.yourdomain.com`

6. 在 `Authorized redirect URIs` 中输入你的 LangSmith 实例域名，后跟 `/api/v1/oauth/custom-oidc/callback`，例如 `https://langsmith.yourdomain.com/api/v1/oauth/custom-oidc/callback`

7. 点击 `Create`，然后下载 JSON 或复制并保存 `Client ID`（以 `.apps.googleusercontent.com` 结尾）和 `Client secret` 到安全的地方。**如果需要，以后可以访问这些信息**。

8. 从左侧导航菜单中选择 `OAuth consent screen`

   1. 选择应用程序类型为 `Internal`。**如果选择 `Public`，任何拥有 Google 账户的人都可以登录。**
   2. 输入描述性的 `Application name`。此名称会在用户登录时显示在同意屏幕上。例如，使用 `LangSmith` 或 `<organization_name> SSO for LangSmith`。
   3. 验证 Google API 的 Scopes 只列出了 email、profile 和 openid 作用域。单点登录只需要这些作用域。如果授予额外的作用域，会增加暴露敏感数据的风险。

9. （可选）控制你组织内谁可以访问 LangSmith：[https://admin.google.com/ac/owl/list?tab=configuredApps](https://admin.google.com/ac/owl/list?tab=configuredApps)。有关更多详细信息，请参阅 [Google 的文档](https://support.google.com/a/answer/7281227?hl=en&fl=1&sjid=9554153972856467090-NA)。

10. 配置 LangSmith 使用此 OAuth 应用程序。以下是用于 Kubernetes 配置的 `config` 值示例：

    1. `oauthClientId`：`Client ID`（以 `.apps.googleusercontent.com` 结尾）
    2. `oauthClientSecret`：`Client secret`
    3. `hostname`：你的 LangSmith 实例域名，例如 `https://langsmith.yourdomain.com`（无尾部斜杠）
    4. `oauthIssuerUrl`：`https://accounts.google.com`
    5. `oauth.enabled`：`true`
    6. `authType`：`mixed`

## 不使用客户端密钥 (PKCE)（已弃用）

如果可能的话，我们建议使用 `Client Secret` 运行（之前我们不支持此功能）。但是，如果您的 IdP 不支持此功能，您可以使用 `Authorization Code with PKCE` 流程。

此流程*不需要* `Client Secret`。有关替代工作流程，请参阅[使用客户端密钥](#with-client-secret-recommended)。

### 要求

使用 LangSmith 的 OAuth SSO 有几个要求：

* 您的 IdP 必须支持 `Authorization Code with PKCE` [流程](https://www.oauth.com/oauth2-servers/pkce)（例如，Google 不支持此流程，但请参阅[上文](#with-client-secret-recommended)了解 Google 支持的替代配置）。这通常在您的 OAuth 提供商中显示为配置"单页应用程序 (SPA)"
* 您的 IdP 必须支持使用外部 discovery/issuer URL。我们将使用它来获取您 IdP 的必要路由和密钥。
* 您必须向 LangSmith 提供 `OIDC`、`email` 和 `profile` 范围。我们使用这些来获取用户必要的用户信息和电子邮件。
* 您需要在 IdP 中设置回调 URL 为 `http://<host>/oauth-callback`，其中 host 是您为 LangSmith 实例配置域名或 IP。这是您的 IdP 在用户身份验证后将用户重定向到的位置。
* 您需要在 `values.yaml` 文件中提供 `oauthClientId` 和 `oauthIssuerUrl`。这是您配置 LangSmith 实例的地方。

<CodeGroup>

```yaml Helm
config:
  oauth:
    enabled: true
    oauthClientId: <YOUR CLIENT ID>
    oauthIssuerUrl: <YOUR DISCOVERY URL>
```

```bash Docker
# In your .env file
AUTH_TYPE=oauth
OAUTH_CLIENT_ID=your-client-id
OAUTH_ISSUER_URL=https://your-issuer-url
```

</CodeGroup>
