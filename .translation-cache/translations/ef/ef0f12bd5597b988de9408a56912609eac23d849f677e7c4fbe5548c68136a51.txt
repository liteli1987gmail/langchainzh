<Note>
`RubricMiddleware` 需要 `deepagents>=0.6.5`。目前处于[**测试版**](/oss/versioning)；API 可能在未来发生变化。
</Note>

某些智能体任务有明确的"完成"定义，仅靠工作模型无法在第一次就可靠地达到：符合正确音节模式的俳句、所有测试通过的代码重构、涵盖所有要求章节的报告。`RubricMiddleware` 允许你将*什么是"完成"*声明为评分标准，并让智能体**自我评估和迭代**，直到满足评分标准（或达到配置的最大迭代次数）。

**LLM-as-a-judge** 是一种模式，即由一个语言模型根据预定义的标准评估另一个模型的输出。在 [LangSmith 评估](/langsmith/evaluation-concepts#llm-as-judge) 中，LLM-as-a-judge 评估器离线批量对应用输出进行评分。`RubricMiddleware` 在运行时应用相同的模式：在深度智能体生成输出后，专门的评分模型根据你的评分标准审查对话记录，并推动修订，直到每项标准都通过（或达到配置的迭代上限）。

当深度智能体完成推理后，LLM-as-a-judge 评分子智能体会审查输出并返回判定结果。如果返回 `needs_revision`，则将每项标准的反馈重新注入对话，并让智能体再次运行。循环在 `satisfied`、`max_iterations_reached`、`failed` 或 `grader_error` 时终止。

```mermaid
graph LR
    Start[User invokes<br/>with rubric] --> Agent[Deep agent]
    Agent --> Grader{Grader<br/>verdict}

    Grader --> |satisfied| Done[Finish execution]
    Grader --> |failed| Done
    Grader --> |grader_error| Done
    Grader --> |needs_revision| Cap{Iterations < <br/> max_iterations?}

    Cap --> |yes| Inject[Re-prompt deep agent with per-criterion feedback]
    Cap --> |no| Done

    Inject --> Agent

    classDef trigger fill:#F6FFDB,stroke:#6E8900,stroke-width:2px,color:#2E3900
    classDef process fill:#E5F4FF,stroke:#006DDD,stroke-width:2px,color:#030710
    classDef decision fill:#FDF3FF,stroke:#7E65AE,stroke-width:2px,color:#504B5F
    classDef alert fill:#F8E8E6,stroke:#B27D75,stroke-width:2px,color:#634643

    class Start trigger
    class Agent,Inject process
    class Grader,Cap decision

```

## 配置中间件

在调用 `create_deep_agent` 时，将 `RubricMiddleware` 添加到 `middleware` 列表中：

<RubricConfigurePy />

| 参数 | 必填 | 默认值 | 描述 |
| --- | --- | --- | --- |
| `model` | 是 | `None` | LLM-as-a-judge 评分子代理使用的聊天模型。接受 `"provider:model-id"` 字符串或 `BaseChatModel` 实例。通常比深度代理的工作模型更小或更便宜。 |
| `system_prompt` | 否 | 内置评分器提示词 | 自定义评分指令。回退到默认系统提示词，教导评分器判决格式及其可用的工具。 |
| `tools` | 否 | `None` | 评分器在做出判决前可以调用的工具，用于收集证据（运行测试、计算 token、读取文件）。如果不提供工具，评分器仅根据对话记录进行推理。 |
| `max_iterations` | 否 | `3` | 每次评分标准尝试的评分器迭代次数硬上限。最大输入值为 20。当达到上限仍未获得 `satisfied` 判决时，代理将以 `max_iterations_reached` 状态终止。 |
| `on_evaluation` | 否 | `None` | 可选回调，在每次评分迭代后（无论使用 `invoke()`、`stream()` 还是 `stream_events()`）调用，包含每个 `RubricEvaluation`。可用于日志记录、自定义指标、评估数据集或 UI 更新。 |

## 调用时传递评分标准

在调用状态上传递 `rubric` 字符串以启动自评循环。使用 `invoke()` 进行单个阻塞调用，或使用 [`stream_events(..., version="v3")`](/oss/langchain/event-streaming) 配合 [`CustomTransformer`](/oss/langchain/event-streaming#custom-updates) 接收 `stream.custom` 上实时发生的评分事件：

        <RubricInvokePy />
    </Tab>
    <Tab title="stream_events()">
        <RubricStreamPy />

        评分标准评分在 `stream.custom` 上发出以下自定义事件：

        | 事件 | 触发时机 | 载荷字段 |
        | --- | --- | --- |
        | `rubric_evaluation_start` | 评分器运行之前。 | <ul><li>`type`：事件名称</li><li>`grading_run_id`：在一次评分标准尝试中所有事件共享</li><li>`iteration`：当前评分运行的从零开始的索引</li></ul> |
        | `rubric_evaluation_end` | 评分器返回后或评分器异常后。 | <ul><li>`type`：事件名称</li><li>`grading_run_id`：在一次评分标准尝试中所有事件共享</li><li>`iteration`：当前评分器遍历的从零开始的索引</li><li>`result`：本次遍历的最终判决</li><li>`explanation`：评分器的摘要</li><li>`criteria`：每项标准的判决</li></ul> |
    </Tab>
</Tabs>

### 评分标准判决

当深度代理完成推理并产生输出后，LLM-as-a-judge 评分子代理会根据评分标准审查输出，并产生以下判决之一：

| 状态 | 含义 | 是否回环？ |
| --- | --- | --- |
| `satisfied` | 评分标准中的每项标准都通过。 | 否 |
| `needs_revision` | 至少一项标准未通过；评分器反馈被注入，代理再次运行。 | 是 |
| `max_iterations_reached` | 评分器仍希望进行修改，但已达到 `max_iterations`。 | 否 |
| `failed` | 评分器判定评分标准格式不正确或无法根据对话记录进行评估。 | 否 |
| `grader_error` | LLM-as-a-judge 评分子代理本身抛出异常（提供商超时、凭据缺失、格式错误的结构化响应等）。 | 否 |

## 观察迭代进度

`on_evaluation` 是一个回调函数，在每次评分迭代后（无论你调用 `invoke()` 还是 `stream_events()`）触发，包含评分器的判决。如果你没有从 `stream.custom`（使用 `CustomTransformer`）读取评分事件，也没有使用 LangSmith 对运行进行跟踪，这是检查评分过程中发生了什么的主要方式。

<RubricOnEvaluationPy />

中间件在每次[评分器遍历](#grader-pass-events)后用 `RubricEvaluation` 字典调用你的函数。`RubricEvaluation` 字典包含：

| 字段 | 类型 | 描述 |
| --- | --- | --- |
| `grading_run_id` | `str` | 在一次评分标准尝试中所有评估共享的标识符。当调用者提供不同的 `rubric` 时，或在得到最终判决后再次使用相同的 `rubric` 调用时，新的运行开始。 |
| `iteration` | `int` | 该运行中当前评分器遍历的从零开始的索引。 |
| `result` | `str` | 本次遍历的评分器判决：`satisfied`、`needs_revision`、`failed` 或 `grader_error`。 |
| `explanation` | `str` | 评分器的自由格式摘要。在基础设施故障时，包含异常类型和消息。 |
| `criteria` | `list` | 每项标准的判决。每个条目为 `{name, passed: true}` 或 `{name, passed: false, gap}`，其中 `gap` 是针对未通过标准的可操作反馈。 |

### 评分器遍历事件

| 事件 | 描述 |
| ----- | ------------- |
| **成功评分** | 每次遍历触发一次，包括中间的 `needs_revision` 判决和最终的 `satisfied` 或 `failed` 判决。<br /><br /> 当评分器返回 `needs_revision` 但已达到 `max_iterations` 时，回调仍然接收 `result: "needs_revision"`（评分器的判决）。运行的最终状态是私有状态 `_rubric_status` 上的 `max_iterations_reached`，而不是评估记录上的。在 `invoke` 完成后检查 `_rubric_status`，或者同时读取 `_rubric_evaluations` 的最后一条和 `_rubric_iterations`，以在达到上限时进行分支处理。 |
| **评分器异常** | 使用 `result: "grader_error"`、从异常派生的解释和空的 `criteria` 列表触发。 |
| **回调中的错误** | 异常被记录并抑制。评分循环继续。不要使用 `on_evaluation` 来强制控制流（例如，抛出异常来停止代理）。 |

## 在调用之间持久化评分标准

单个 `agent.invoke()` 或 `agent.stream_events()` 调用会运行评分标准循环直到完成，并以最终判决结束：`satisfied`、`failed` 或 `max_iterations_reached`。

要将评分标准延续到后续调用，请附加一个[检查点保存器](/oss/langgraph/checkpointers#checkpoints)，并在调用时传递相同的 `thread_id`。在这些情况下，相同的 `rubric` 会在未来的 `invoke()` 或 `stream_events()` 调用中保持有效，直到你在输入中传入新的 `rubric`。

中断（`KeyboardInterrupt`、`asyncio.CancelledError`）会从 `agent.invoke()` 和 `agent.stream_events()` 中未捕获地传播。在检查点保存的线程上，下次使用相同 `rubric` 的调用会恢复进行中的评分运行。
