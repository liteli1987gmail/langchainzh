---
title: 向后兼容性
description: 在生产环境中更新 LangGraph 图代码而不中断进行中的运行。
---

生产环境中的软件需要不断变化。新需求、bug 修复和重构最终都会应用到你的图代码中。因为 LangGraph 针对已为现有线程[持久化](/oss/langgraph/persistence)的状态运行最新部署的图，你发布的每个更改实际上都是相对于现有检查点的向后兼容 API 变更。

与将运行绑定到其启动时代码版本的 workflow 引擎不同，LangGraph 立即将最新图应用到*每个*线程，无论是新线程还是从检查点恢复的线程。这很方便：bug 修复会传播到进行中的对话和代理，无需额外操作。但这也意味着你必须考虑每个更改如何与在先前版本代码下启动的运行交互。

有三种兼容性问题是需要注意的，按你将遇到的大致顺序排列：

1. [技术兼容性](#technical-compatibility)：最常见；新代码必须仍能针对现有 State 加载和执行。
2. [业务兼容性](#business-compatibility)：较少见；现有运行应该继续遵循旧业务逻辑，即使代码已更改。
3. [非确定性](#non-determinism)：仅适用于 [Functional API](/oss/langgraph/functional-api)。

<Tip>
有关运行时默认可支持哪些图拓扑和状态更改的简短总结，请参阅 [图迁移](/oss/langgraph/graph-api#graph-migrations)。本页其余部分涵盖了你可以在更改超出支持范围时应用的模式。
</Tip>

## 技术兼容性

技术兼容性等同于微服务中的 API 破坏性变更。这里的"API"是你图代码与已为现有线程[持久化](</oss/langgraph/checkpointers#checkpointer-libraries)的数据之间的契约。当线程恢复时，LangGraph 反序列化保存的状态，按名称将数据分派给节点，并期望节点返回符合状态 schema 的值。

常见的技术性破坏：

- **重命名或删除节点**，而线程在或即将进入该节点时暂停，例如在 @[`interrupt`] 处或通过仍路由到旧名称的检查点条件边。恢复时，LangGraph 无法按其保存的名称找到节点，运行失败。恢复运行的起点是执行停止的节点开头，因此缺失的节点没有恢复的地方。
- **重命名或删除 State 键**，而旧检查点仍包含该键或下游节点仍读取该键。
- **收紧 State 字段**，例如将 `Optional` 字段改为必需、缩窄类型或添加没有默认值的新必需字段。现有检查点将不满足新 schema。

边拓扑本身在检查点中*不*被持久化。在仍存在的节点之间添加、删除或重路由边对于进行中的线程是安全的。根据[图迁移](/oss/langgraph/graph-api#graph-migrations)摘要，唯一可能中断被中断线程的拓扑更改是重命名或删除节点。

### 推荐的模式

:::python

- 将新状态字段添加为 `NotRequired`（或 `Optional[...] = None`），以便旧检查点仍能通过验证：

  ```python
  from typing import NotRequired
  from typing_extensions import TypedDict

  class State(TypedDict):
      messages: list
      summary: NotRequired[str]  # [!code ++]
  ```

- 将删除视为弃用。即使没有节点读取该字段，也请在状态上保留该字段定义至少一个排水周期，以便现有检查点继续加载。
- 通过*先添加后删除*进行重命名。在旧字段或节点旁边添加新字段或节点，在弃用窗口期内双向写入或路由到两者，一旦确认没有运行中的线程依赖它后删除旧的。
- 让节点函数对未知键保持容错。`TypedDict` 在运行时忽略额外的键，因此旧代码版本留下的状态不会引发错误，除非节点显式读取缺失的键。
- 使用[时间旅行](/oss/langgraph/use-time-travel)和 @[`graph.get_state`][get_state] 在预发布部署中对新代码进行现有线程的抽查，然后再推出。

:::

:::js

- 将新状态字段标记为可选（`z.string().optional()` 或 `.nullish()`），以便旧检查点仍能验证。
- 将删除视为弃用：在状态上保留该字段至少一个排水周期，以便现有检查点继续加载。
- 通过先添加后删除进行重命名：在旧字段或节点旁边添加新字段或节点，在弃用窗口期内双向写入或路由到两者，一旦没有运行中的线程依赖它后删除旧的。
- 使用[时间旅行](/oss/langgraph/use-time-travel)和 @[`graph.getState`][get_state] 在预发布部署中对新代码进行现有线程的抽查，然后再推出。

:::

### 检测运行中的线程

在删除节点、重命名 State 键或进行旧线程无法容受的更改之前，你需要知道是否有任何线程当前停在你即将丢弃的代码版本上。LangGraph 本身不维护线程状态的搜索索引，因此答案取决于你的图运行在哪里。

**如果你部署到 [LangSmith](/langsmith/deployment)。** 使用 Agent Server 的线程搜索按状态过滤。`status` 字段接受 `idle`、`busy`、`interrupted` 和 `error`，因此你可以批量查询 `interrupted` 或 `busy` 线程，可选择使用元数据过滤器缩小范围。请参阅[按线程状态过滤](/langsmith/use-threads#filter-by-thread-status)和[列出线程](/langsmith/use-threads#list-threads)。

**LangGraph 运行的所有地方。** 使用 [LangSmith 追踪](/oss/langgraph/observability) 监控生产中哪些节点正在被进入和退出。这是节点或状态字段在任何活动代码路径中不再可到达的最可靠信号。

**当你已有 `thread_id` 时。** 直接检查该单个线程：

:::python

- @[`graph.get_state(config)`][get_state] 返回最新的检查点，包括线程暂停的节点和任何待处理的打断。
- @[`graph.get_state_history(config)`][get_state_history] 返回该线程的完整时间顺序检查点列表。

:::

:::js

- @[`graph.getState(config)`][get_state] 返回最新的检查点，包括线程暂停的节点和任何待处理的打断。
- @[`graph.getStateHistory(config)`][get_state_history] 返回该线程的完整时间顺序检查点列表。

:::

如有疑问，请保留弃用的节点或字段，直到 Agent Server 线程列表和追踪都显示它不再有任何活动。

## 业务兼容性

有时更改在技术上是有效的（每个现有检查点仍能加载，每个节点仍能解析），但新图的*含义*与旧图不同。新行为对新线程是正确的，但你不想将其追溯应用到在旧逻辑下启动的线程。

例如，假设你的图运行 `intake → triage → respond`，你决定在 `triage` 和 `respond` 之间插入新的 `policy_check` 步骤：

- 已通过 `triage` 的线程应继续直接到 `respond`（旧流程）。
- 新线程应运行完整的新流程。

推荐的模式是在线程开始时在状态上记录相关的*行为版本*，然后使用[条件边](/oss/langgraph/graph-api#conditional-edges)对其进行分支：

:::python

```python
from typing import NotRequired
from typing_extensions import TypedDict

from langgraph.graph import END, START, StateGraph


class State(TypedDict):
    request: str
    flow_version: NotRequired[int]
    response: NotRequired[str]


def intake(state: State) -> dict:
    # Stamp new threads with the current flow version. Existing threads
    # that resume past `intake` keep whatever value was already saved.
    return {"flow_version": state.get("flow_version", 2)}


def triage(state: State) -> dict: ...
def policy_check(state: State) -> dict: ...
def respond(state: State) -> dict: ...


def after_triage(state: State) -> str:
    if state.get("flow_version", 1) >= 2:
        return "policy_check"
    return "respond"


builder = StateGraph(State)
builder.add_node("intake", intake)
builder.add_node("triage", triage)
builder.add_node("policy_check", policy_check)
builder.add_node("respond", respond)
builder.add_edge(START, "intake")
builder.add_edge("intake", "triage")
builder.add_conditional_edges("triage", after_triage, ["policy_check", "respond"])
builder.add_edge("policy_check", "respond")
builder.add_edge("respond", END)

graph = builder.compile()
```

:::

在 `triage` 后恢复的旧线程会从其保存的状态中读取 `flow_version`（或回退到 v1 默认值），并跳过 `policy_check`。新线程从 `intake` 开始，被标记为 `flow_version=2`，并运行新的路径。一旦所有 v1 线程完成后，你可以移除版本标志和条件边。

只有*在线程启动时*设置版本此模式才能生效，在任何需要版本化的分支之前设置。在之后设置意味着现有线程在需要时不会有该设置。

## 非确定性

本类别仅适用于 [Functional API](/oss/langgraph/functional-api) 以及 [Graph API](/oss/langgraph/graph-api) **节点**内部的 [**tasks**](/oss/langgraph/functional-api#task) 或 @[`interrupt`] 调用。普通的 Graph API **节点**在恢复时会 [从头重新运行节点函数](/oss/langgraph/graph-api#re-execution-and-idempotency)；请将副作用设计为幂等的，但除非你在该**节点**中使用了 **tasks** 或 @[`interrupt`]，否则不需要保留任务调用顺序。

Functional API **entrypoint** 会被编译成单个**节点**，当运行恢复时从头重放 entrypoint 主体，并使用缓存的 @[`@task`][task] 结果跳过已完成的操作。两类变更会破坏此模型：

- **在恢复点之前添加、移除或重新排序 `@task` 调用或 @[`interrupt`] 调用**。LangGraph 会根据重放中的位置将缓存的结果和恢复值与调用进行匹配，因此位置移动可能导致错误的缓存值被重放到不同的调用上。
- **在 `@task` 之外引入非确定性操作**，例如 `time.time()`、`random.random()`，或在 entrypoint 主体中内联的网络调用。在重放时这些会产生与首次运行不同的值，可能改变控制流。

有关更深入的处理和示例，请参阅 Functional API 指南中的 [确定性](/oss/langgraph/functional-api#determinism) 和 [常见陷阱](/oss/langgraph/functional-api#common-pitfalls)。

如果需要对有进行中运行的 `@entrypoint` 进行重大代码更改，最安全的选项是：

- 让进行中的运行在部署更改之前耗尽。
- 将任何新逻辑包装在新的 `@task` 中，以便其结果被独立检查点化。
- 在 `langgraph.json` 中使用新的图名称注册新的 entrypoint 来处理新行为，并将新线程路由到该 entrypoint。
