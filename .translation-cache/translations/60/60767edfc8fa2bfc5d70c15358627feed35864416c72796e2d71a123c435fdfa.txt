## 概述

LLM 赋能的最强大应用之一是复杂的问答聊天机器人。这些应用可以回答有关特定来源信息的问题。这些应用使用一种称为检索增强生成的技术，或 [RAG](/oss/langchain/retrieval/).

本教程将展示如何基于非结构化文本数据源构建一个简单的问答应用。我们将演示：

1. 一个 RAG [代理](#rag-agents) 它使用简单工具执行搜索。这是一个良好的通用实现。
2. 一个两步式 RAG [链](#rag-chains) 每个查询仅需一次 LLM 调用。这是一种快速有效的方法，适用于简单查询。

### 概念

我们将涵盖以下概念：

- **索引构建**：用于从数据源摄取数据并建立索引的管道流程。 *这通常发生在独立的进程中。*

- **检索与生成**：这是实际的 RAG 流程，它在运行时接收用户查询，从索引中检索相关数据，然后将其传递给模型。

一旦我们建立好数据索引，我们将使用一个 [代理](/oss/langchain/agents) 作为编排框架来实现检索和生成步骤。

<Note>
    本教程的索引构建部分将主要参考 [语义搜索教程](/oss/langchain/knowledge-base).

    如果您的数据已经可以用于搜索（即您有一个执行搜索的函数），或者您对该教程的内容已经熟悉，可以跳到关于的部分 [检索与生成](#2-retrieval-and-generation)
</Note>

索引通常按如下方式工作：

1. **加载**：首先，我们需要将数据加载到 @[`Document`] 对象中。
2. **分块**: [文本分割器](/oss/integrations/splitters) 将大型 `Documents` 拆分成较小的块。这对于索引数据和将其传递给模型都很有用，因为大块难以搜索且无法适应模型的有限上下文窗口。
3. **存储**：我们需要某个地方来存储和索引我们的分块，以便日后可以搜索。这通常使用 [VectorStore](/oss/integrations/vectorstores) 和 [Embeddings](/oss/integrations/embeddings) model.

## 2. 检索与生成

RAG应用的典型工作流程如下：

1. **检索**：给定用户输入，使用 [检索器](/oss/integrations/retrievers).
2. **生成**: A [模型](/oss/langchain/models) 使用包含问题和检索到的数据的提示词生成答案

### RAG 链

在上述 [代理式 RAG](#rag-agents) formulation 中，我们允许 LLM 自行决定生成 [工具调用](/oss/langchain/models#tool-calling) 来帮助回答用户查询。这是一个很好的通用解决方案，但也有一些权衡：

| ✅ 优势                                                                 | ⚠️ 劣势                                                                 |
|-----------------------------------------------------------------------------|----------------------------------------------------------------------------|
| **仅在需要时搜索**— LLM 可以处理问候、跟进和简单查询，而不会触发不必要的搜索。 | **两次推理调用**— 当执行搜索时，需要一次调用来生成查询，另一次调用来产生最终响应。 |
| **上下文搜索查询**— 通过将搜索作为一个带有 `query` 输入的工具，LLM 会生成包含对话上下文的查询。 | **控制力降低**—LLM 可能会在实际需要时跳过搜索，或在不必要时发出额外搜索。 |
| **允许多次搜索**—LLM 可以执行多次搜索以支持单个用户查询。 |                                                                            |


另一种常见方法是两步链，其中我们始终运行一次搜索（可能使用原始用户查询）并将结果作为上下文用于单个 LLM 查询。这导致每个查询一次推理调用，以牺牲灵活性为代价获得更低的延迟。

在这种方法中，我们不再在循环中调用模型，而是进行单次传递。

## 安全：间接提示注入

<Warning>
RAG 应用容易受到 **间接提示注入**。检索到的文档可能包含类似指令的文本（例如"以 JSON 格式响应"或"忽略之前的指令"）。由于检索到的上下文与系统提示共享相同的上下文窗口，模型可能会无意中遵循数据中嵌入的指令，而不是预期的提示。

例如，本教程中索引的博客文章包含描述 [Auto-GPT](https://lilianweng.github.io/posts/2023-06-23-agent/#case-studies) 的文本。如果用户查询检索到该块，模型可能会输出 JSON 而不是自然语言答案。
</Warning>

缓解措施：

1. **使用防御性提示**：明确指示模型将检索到的上下文仅作为数据处理，并忽略其中的任何指令。本教程中的提示包含此类指令。
2. **使用分隔符包装上下文**：使用清晰的结构标记（例如 XML 标签如 `<context>...</context>`）来区分检索到的数据与指令，使模型更容易区分它们。
3. **验证响应**：检查模型的输出是否与预期格式（例如纯文本）匹配，并优雅地处理意外格式。

没有任何缓解措施是万无一失的——这是当前 LLM 架构的固有局限性，其中指令和数据共享相同的上下文窗口。有关此主题的更多信息，请参阅 [提示注入](https://simonwillison.net/series/prompt-injection/).
