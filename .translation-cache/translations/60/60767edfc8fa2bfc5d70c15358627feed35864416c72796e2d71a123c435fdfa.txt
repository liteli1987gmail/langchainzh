## 概述

LLM 最强大的应用之一是复杂的问答（Q&A）聊天机器人。这些应用可以回答关于特定来源信息的问题。这些应用使用一种称为检索增强生成（Retrieval Augmented Generation）的技术，或称 [RAG](/oss/langchain/retrieval/)。

本教程将展示如何基于非结构化文本数据源构建一个简单的问答应用程序。我们将演示：

1. 一个使用简单工具执行搜索的 RAG [agent](#rag-agents)。这是一个良好的通用实现。
2. 一个两步 RAG [chain](#rag-chains)，每个查询只需一次 LLM 调用。这是一种针对简单查询的快速有效方法。

### 概念

我们将涵盖以下概念：

- **索引**：用于从源获取数据并建立索引的管道。*这通常发生在单独的进程中。*

- **检索与生成**：实际的 RAG 过程，在运行时获取用户查询并从索引中检索相关数据，然后将其传递给模型。

一旦数据被索引，我们将使用一个 [agent](/oss/langchain/agents) 作为我们的编排框架来实现检索和生成步骤。

<Note>
    本教程的索引部分将主要遵循[语义搜索教程](/oss/langchain/knowledge-base)。

    如果您的数据已经可用于搜索（即您有一个执行搜索的函数），或者您对该教程的内容已经熟悉，可以直接跳到[检索与生成](#2-retrieval-and-generation)部分。
</Note>

索引通常按以下步骤进行：

1. **加载**：首先，我们需要将数据加载到 @[`Document`] 对象中。
2. **分割**：[文本分割器](/oss/integrations/splitters)将大型 `Documents` 分割成更小的块。这对于索引数据和将其传递给模型都很有用，因为大块更难搜索，也无法适应模型的有限上下文窗口。
3. **存储**：我们需要将分割后的数据存储和索引到某个地方，以便后续搜索。这通常通过 [VectorStore](/oss/integrations/vectorstores) 和 [Embeddings](/oss/integrations/embeddings) 模型来实现。

## 2. 检索与生成

RAG 应用的常见工作流程如下：

1. **检索**：给定用户输入，使用[检索器](/oss/integrations/retrievers)从存储中检索相关片段。
2. **生成**：[模型](/oss/langchain/models)使用包含问题和检索数据的提示词来生成答案。

![retrieval_diagram](/images/rag_retrieval_generation.png)

现在让我们编写实际的应用逻辑。我们希望创建一个简单的应用程序，它接收用户问题，搜索与该问题相关的文档，将检索到的文档和初始问题传递给模型，然后返回答案。

我们将演示：

1. 使用简单工具执行搜索的 RAG [智能体](#rag-agents)。这是一种通用性强的实现。
2. 每个查询仅使用一次 LLM 调用的两步式 RAG [链](#rag-chains)。这是一种简单查询的快速有效方法。

### RAG 链

在上述 [agentic RAG](#rag-agents) 公式中，我们允许 LLM 自行决定是否生成 [tool call](/oss/langchain/models#tool-calling) 来帮助回答用户查询。这是一种不错的通用解决方案，但存在一些权衡：

| ✅ 优势                                                                 | ⚠️ 劣势                                                                 |
|-----------------------------------------------------------------------------|----------------------------------------------------------------------------|
| **仅在需要时搜索**——LLM 可以更好地处理问候、跟进和简单查询，而不会触发不必要的搜索。 | **两次推理调用**——当执行搜索时，需要一次调用来生成查询，另一次来生成最终响应。 |
| **上下文搜索查询**——通过将搜索作为具有 `query` 输入的工具，LLM 可以生成包含对话上下文的自己的查询。 | **控制力降低**——LLM 可能会跳过实际需要的搜索，或发出不必要的额外搜索。 |
| **允许多次搜索**——LLM 可以针对单个用户查询执行多次搜索。 |                                                                            |


另一种常见方法是两步链式处理，其中我们总是运行搜索（可能使用原始用户查询）并将结果作为上下文纳入单个 LLM 查询。这实现了每次查询仅需一次推理调用，以牺牲灵活性为代价降低了延迟。

在此方法中，我们不再循环调用模型，而是进行单次处理。

## 安全：间接提示词注入

<Warning>
RAG 应用容易受到**间接提示词注入**的影响。检索到的文档可能包含类似指令的文本（例如"以 JSON 格式响应"或"忽略之前的指令"）。由于检索到的上下文与您的系统提示词共享相同的上下文窗口，模型可能会无意中遵循数据中嵌入的指令，而不是您原本的意图。

例如，本教程中索引的博客文章包含描述 [Auto-GPT](https://lilianweng.github.io/posts/2023-06-23-agent/#case-studies) JSON 响应格式的文本。如果用户查询检索到该内容块，模型可能会输出 JSON 而不是自然语言答案。
</Warning>

为了缓解这个问题：

1. **使用防御性提示词**：明确指示模型将检索到的上下文仅视为数据，并忽略其中的任何指令。本教程中的提示词包含此类指令。
2. **使用分隔符包裹上下文**：使用清晰的结构标记（例如 XML 标签如 `<context>...</context>`）将检索到的数据与指令分开，使模型更容易区分两者。
3. **验证响应**：检查模型的输出是否符合预期格式（例如纯文本），并优雅地处理意外的格式。

没有任何缓解措施是万无一失的——这是当前 LLM 架构的固有局限性，因为指令和数据共享相同的上下文窗口。有关此话题的更多信息，请参阅 [prompt injection](https://simonwillison.net/series/prompt-injection/) 相关研究。
