<Note>
本节适用于使用 LangSmith JS SDK 0.2.0 及更高版本的用户。如果您在无服务器环境中使用 LangChain.js 或 LangGraph.js 进行追踪，请参阅[本指南](https://js.langchain.com/docs/how_to/callbacks_serverless)。
</Note>

追踪 JavaScript 函数时，LangSmith 默认会在后台追踪运行，以避免增加延迟。在无服务器环境中，执行上下文可能会突然终止，因此在函数完成之前确保所有追踪数据正确刷新非常重要。

为确保发生这种情况，您可以：

* 将名为 `LANGSMITH_TRACING_BACKGROUND` 的环境变量设置为 `"false"`。这将导致您的追踪函数在返回之前等待追踪完成。
  * 请注意，这与 LangChain.js 中的[环境变量](https://js.langchain.com/docs/how_to/callbacks_serverless)命名不同，因为 LangSmith 可以独立于 LangChain 使用。
* 将自定义客户端传递到追踪的运行中，并 `await` `client.awaitPendingTraceBatches();` 方法。

以下是结合使用 `awaitPendingTraceBatches` 和 [`traceable`](/langsmith/annotate-code) 方法的示例：

```typescript
import { Client } from "langsmith";
import { traceable } from "langsmith/traceable";
const langsmithClient = new Client({});
const tracedFn = traceable(
  async () => {
    return "Some return value";
  },
  {
    client: langsmithClient,
  }
);
const res = await tracedFn();
await langsmithClient.awaitPendingTraceBatches();
```

## 高并发场景下的速率限制[ ](#rate-limits-at-high-concurrency "Direct link to rate limits at high concurrency")

默认情况下，LangSmith 客户端会批处理操作（与您被追踪的运行执行同步），每隔几毫秒发送一个新批次。

这在大多数情况下都能很好地工作，但如果您被追踪的函数运行时间较长且并发量非常高，您可能也会遇到与总体请求数相关的速率限制。

如果您遇到与此相关的速率限制错误，可以尝试在客户端中设置 `manualFlushMode: true`，如下所示：

```typescript
import { Client } from "langsmith";
const langsmithClient = new Client({  manualFlushMode: true,});
const myTracedFunc = traceable(
  async () => {
    // Your logic here...
  },
  { client: langsmithClient }
);
```

然后像这样手动调用 `client.flush()`，在您的无服务器函数关闭之前：

```typescript
try {
  await myTracedFunc();
} finally {
  await langsmithClient.flush();
}
```

请注意，在您调用 `.flush()` 之前，这将阻止运行记录显示在 LangSmith UI 中。
