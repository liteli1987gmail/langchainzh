以下页面详细说明了一个示例应用，该应用在 [Deno Deploy](https://deno.com/deploy) 上部署 LangChain **深度代理**：流式聊天 UI、子代理和线程历史，全部由实现为 Hono 服务器上 HTTP + SSE 路由处理程序的 [Agent Streaming Protocol](https://github.com/langchain-ai/agent-protocol/tree/main/streaming) 支持。React 前端是一个 Vite SPA（从 Next.js 示例移植而来）；Deno 从单一的 `main.ts` 入口点提供构建的静态资源和 API。

这是将 Next.js 示例移植到 Deno + Hono，展示如何在 Deno Deploy 上运行相同的代理栈，而不是 Vercel。

源代码：[`js-deno`](https://github.com/langchain-ai/deployment-cookbook/tree/main/js-deno)（部署 cookbook 中）。

## 部署到 Deno Deploy

<Steps>

<Step title="创建 Deno Deploy 项目">

Fork 或克隆 [`langchain-ai/deployment-cookbook`](https://github.com/langchain-ai/deployment-cookbook)。在 [Deno Deploy 仪表板](https://dash.deno.com/) 中，创建一个链接到此仓库的新项目。

</Step>

<Step title="配置构建设置">

- 将**根目录**设置为 `js-deno`。
- 将**构建命令**设置为 `deno task build:client`（将 Vite SPA 构建到 `dist/`）。
- 将**入口点**设置为 `main.ts`。
- 在项目环境变量中添加 `OPENAI_API_KEY`。

</Step>

<Step title="部署">

从仪表板部署。Deno 的构建环境会运行构建命令，因此 `dist/` 是在云端生成的，无需提交。

</Step>

</Steps>

或者，使用内置的 `deno deploy` CLI（Deno 2.x）。[`deno.json`](https://github.com/langchain-ai/deployment-cookbook/blob/main/js-deno/deno.json) 中的 `deploy` 块设置了 `org`/`app`。将这些更改为您自己的（或传递 `--org`/`--app` 标志来覆盖它们）。

```bash
cd js-deno

# First time only: create the app
deno deploy create --org <your-org> --app <your-app> --source local --region us --entrypoint main.ts

# Set your OpenAI key
deno deploy env add OPENAI_API_KEY <your-key> --org <your-org> --app <your-app>

# Build the client, deploy to production, and clean up dist/
deno task deploy
```

`deno task deploy` 运行 `deno task build:client && deno deploy --prod`，然后运行 `rm -rf dist`。必须在本地构建，因为 `deno deploy --source local` 会上传你的工作目录（不包括 `.gitignore` 中的文件），并且**不会**运行构建命令。构建命令仅在连接 GitHub 的应用中运行。

<Warning>
CLI `--source local` 流程有两个特别需要注意的问题：

- **`dist/` 不能被 gitignore。** 上传工具会遵守 `.gitignore` 规则，因此在上传窗口期间，新构建的 `dist/` 必须可见，否则所有非 `/api` 路由都会返回 **404**。仓库根目录的 `.gitignore` 会忽略所有 `dist`，所以 `js-deno/.gitignore` 通过 `!dist/` 和 `!dist/**` 重新包含它。`deno task deploy` 流程会在上传后删除 `dist/`，因此尽管没有被忽略，它不会残留在 `git status` 中。
- **不要使用 `deploy.include` 列表。** 存在一个 Deno Deploy bug，添加 `include` 会导致构建将入口点解析为 `src/main.ts` 并失败。请依赖默认的基于 `.gitignore` 的上传方式。
</Warning>

（可选）通过添加 [`.env.example`](https://github.com/langchain-ai/deployment-cookbook/blob/main/js-deno/.env.example) 中的变量来启用 LangSmith 追踪。

## 必需的 API 端点

该应用在 `/api/threads/...` 下暴露了 Agent Streaming Protocol。路由处理程序位于 `server/routes.ts` 中，镜像了 `js-next/app/api/threads/` 中的 Next.js 处理程序。

### 最小集（流式聊天）

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `POST` | `/api/threads/:threadId/commands` | 接受协议命令（`run.start`、…）并启动 agent 运行 |
| `POST` | `/api/threads/:threadId/stream` | 运行的 SSE 协议事件流 |
| `GET` / `POST` | `/api/threads/:threadId/state` | 读取和引导检查点线程状态 |

### 可选（侧边栏）

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/threads` | 列出检查点已知的所有线程 |
| `DELETE` | `/api/threads/:threadId` | 删除线程的会话和检查点 |
| `POST` | `/api/threads/:threadId/history` | 分页检查点历史（Agent Protocol） |

## Deno 后端工作原理

本示例作为**单个 Deno 进程**运行：

- **`main.ts`**：`Deno.serve` + Hono 应用。挂载 `/api` 路由并从 `dist/` 提供 Vite 构建的 SPA 服务。
- **`server/routes.ts`**：Hono 的 Agent Streaming Protocol 路由定义。
- **`server/session.ts`**：`LocalThreadSession`：在 LangGraph `StreamChannel` 中缓冲协议事件，使用 `matchesSubscription` 过滤，并通过 SSE `ReadableStream` 将匹配的帧分发出。
- **`server/threads.ts`**：在 LangGraph SDK 线路格式中提供基于检查点的 `getState` / `updateState` / `getHistory` 辅助方法。
- **`server/registry.ts`**：进程本地的单例，拥有 agent 和每个线程 ID 对应的一个会话。
- **`server/agent/`**：与 Next.js 示例相同的 `createDeepAgent` 编排器（researcher + math-whiz 子 agent，mock 工具）。

Deno Deploy 为每个 isolate 运行各自独立的内存 `MemorySaver` 检查点器。若要在生产环境中实现跨 isolate 的持久化，可以替换为[持久化检查点器](/oss/langgraph/checkpointers#checkpointer-libraries)（Postgres、Redis 等）。路由处理程序和 `server/threads.ts` 辅助方法保持不变。

## 生产环境持久化

默认情况下，agent 使用内存中的 `MemorySaver` 检查点器（`server/agent/index.ts`）和进程本地的会话映射（`server/registry.ts`）。这适用于本地开发和单 isolate 部署，但在 Deno Deploy（多 isolate、冷启动）环境下，跨实例的对话状态**不具备持久性**。

将 `server/agent/index.ts` 中的 `MemorySaver` 替换为持久化检查点器（如 `@langchain/langgraph-checkpoint-postgres` 或 `@langchain/langgraph-checkpoint-redis`）。同时还需要一个共享的会话/回放存储，以便 SSE 重连能够在不同 isolate 之间正常工作。

## 本地开发

你需要 [Deno](https://deno.com/) 2.x 和 [pnpm](https://pnpm.io/) 来运行客户端。

```bash
cp .env.example .env   # set OPENAI_API_KEY
export $(grep -v '^#' .env | xargs)   # load env for Deno

# Terminal 1 — API + static (after first client build)
deno task build:client   # first time only
deno task dev

# Terminal 2 — Vite dev server with HMR (proxies /api to :8000)
cd client && pnpm install && pnpm dev
```

打开 [http://localhost:5173](http://localhost:5173) 进行热重载开发。Vite 开发服务器会将 `/api` 代理到 Deno 服务器的 8000 端口。

如需本地生产环境模拟运行（单服务器，无 HMR）：

```bash
deno task build:client
deno task start
```

打开 [http://localhost:8000](http://localhost:8000)。

## 项目结构

- `main.ts`: Deno Deploy 入口点（`Deno.serve` + Hono）。
- `server/agent/`: 深度智能体（`createDeepAgent`），包含子智能体和模拟工具。
- `server/`: 协议服务器逻辑：`session.ts`、`threads.ts`、`serialize.ts`、`registry.ts`、`routes.ts`。
- `client/`: Vite + React SPA（与 Next.js 示例相同的 UI）。
- `dist/`: Vite 构建输出，由 Deno 提供服务（由 `deno task build:client` 生成）。
