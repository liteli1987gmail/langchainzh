此页面涵盖 LangSmith 中的用户管理功能，包括访问控制、身份验证和自动用户配置：

- [设置访问控制](#set-up-access-control): 配置基于角色的访问控制 (RBAC) 以管理工作区内的用户权限，包括创建自定义角色并将其分配给用户。
- [SAML SSO（企业计划）](#set-up-saml-sso-for-your-organization): 为使用 SAML 2.0 的企业客户设置单点登录身份验证，包括热门身份提供商的配置。
- [SCIM 用户配置（企业计划）](#set-up-scim-for-your-organization): 使用 SCIM 在您的身份提供商和 LangSmith 之间自动配置和取消配置用户。

## 设置访问控制

<Note>
RBAC（基于角色的访问控制）仅面向企业客户提供。如果您对此功能感兴趣，请[联系我们的销售团队](https://www.langchain.com/contact-sales)。其他计划默认为所有用户使用 [`Admin` 角色](/langsmith/administration-overview)。
</Note>

<Check>
在设置访问控制之前，您可能会发现阅读[管理概述](/langsmith/administration-overview)页面很有帮助。
</Check>

LangSmith 依靠 RBAC 来管理[工作区](/langsmith/administration-overview#workspaces)内的用户权限。这允许您控制谁可以访问您的 LangSmith 工作区以及他们可以在其中做什么。具有 `workspaces:manage` 权限的用户可以管理工作区设置，具有 `workspaces:manage-members` 权限的用户可以添加、删除和更新工作区成员。内置的 Workspace Admin 角色包含这两个权限。

有关工作区角色及其权限的完整参考，请参阅[基于角色的访问控制](/langsmith/rbac#workspace-roles)指南。有关每个角色可以执行的特定操作，请参阅[组织和操作参考](/langsmith/organization-workspace-operations)。

### 创建角色

默认情况下，LangSmith 附带一组系统角色：

- `Admin`：对工作区内的所有资源拥有完全访问权限。
- `Viewer`：对工作区内的所有资源拥有只读访问权限。
- `Editor`：拥有除工作区管理（添加/删除用户、更改角色、配置服务密钥）之外的所有权限。

如果这些不符合您的访问模型，`Organization Admins` 可以创建自定义角色来满足您的需求。

要创建角色，请导航到[组织设置页面](https://smith.langchain.com/settings)中 **Members and roles** 部分的 **Roles** 选项卡。请注意，您创建的新角色可以在组织内的所有工作区中使用。

单击 **Create Role** 按钮创建新角色。将打开 **Create role** 表单。

![Create Role](/langsmith/images/create-role.png)

为要控制访问权限的不同 LangSmith 资源分配权限。

### 为用户分配角色

设置好角色后，您可以将其分配给用户。要为用户分配角色，请导航到[组织设置页面](https://smith.langchain.com/settings)中 **Workspaces** 部分的 `Workspace members` 选项卡。

每个用户都有一个 **Role** 下拉菜单，您可以使用它来分配角色。

![Assign Role](/langsmith/images/assign-role.png)

您还可以使用给定角色邀请新用户。

![Invite User](/langsmith/images/invite-user.png)

## 为您的组织设置 SAML SSO

单点登录 (SSO) 功能**可供企业云客户**使用，通过单一身份验证源访问 LangSmith。这允许管理员集中管理团队访问，并使信息更加安全。

LangSmith 的 SSO 配置基于 SAML（安全断言标记语言）2.0 标准构建。SAML 2.0 支持将身份提供商 (IdP) 连接到您的组织，以获得更简单、更安全的登录体验。

SSO 服务允许用户使用一组凭据（例如，姓名或电子邮件地址和密码）访问多个应用程序。该服务仅为用户已获得权限的所有应用程序对最终用户进行一次身份验证，并在用户在同一次会话中切换应用程序时消除进一步提示。SSO 的好处包括：

- 为组织所有者简化跨系统的用户管理。
- 使组织能够执行自己的安全策略（例如，MFA）。
- 消除最终用户记住和管理多个密码的需要。通过允许多个应用程序在单一访问点登录来简化最终用户体验。

### 即时 (JIT) 配置

LangSmith 在使用 SAML SSO 时支持即时配置。这允许通过 SAML SSO 登录的人员自动作为成员加入组织和所选工作区。有关管理 JIT 配置和用户邀请的详细信息，请参阅[在 SSO 组织中管理用户访问](/langsmith/jit-invite-sso)。

<Note>
JIT 配置仅对新用户运行，即通过[不同登录方式](/langsmith/authentication-methods#cloud)尚无相同电子邮件地址访问该组织的用户。
</Note>

### 登录方式和访问

完成组织的 SAML SSO 配置后，用户将能够通过 SAML SSO 以及[其他登录方式](/langsmith/authentication-methods#cloud)（如用户名/密码或 Google 身份验证）登录：

- 通过 SAML SSO 登录时，用户只能访问配置了 SAML SSO 的相应组织。
- SAML SSO 作为唯一登录方式的用户没有[个人组织](/langsmith/administration-overview#organizations)。
- 通过任何其他方式登录时，用户可以访问配置了 SAML SSO 的组织以及他们所属的任何其他组织。

### 仅强制使用 SAML SSO

<Note>
不支持在强制仅使用 SAML SSO 的组织中进行用户邀请。初始工作区成员资格和角色由 [JIT 配置](/langsmith/jit-invite-sso#jit-provisioning) 决定，之后的更改可以在 UI 中管理。
为了在自动化用户管理方面获得更大的灵活性，LangSmith 支持 SCIM。
</Note>

要确保用户只能使用 SAML SSO 登录访问组织，请勾选 **Login via SSO only** 复选框并点击 **Save**。完成后，通过非 SSO 登录方式访问组织的用户需要使用 SAML SSO 重新登录。可以通过取消选中该复选框并点击 **Save** 来将此设置切换回允许所有登录方式。

<Note>
您必须通过 SAML SSO 登录才能将此设置更新为 `Only SAML SSO`。这是为了确保 SAML 设置有效，并避免将用户锁定在组织之外。
</Note>

有关故障排除，请参阅 [SAML SSO 常见问题](/langsmith/faq#saml-sso-faqs)。如果在设置 SAML SSO 时遇到问题，请通过 [support.langchain.com](https://support.langchain.com) 联系 LangChain 支持团队。

### 先决条件

<Note>
SAML SSO 适用于[企业计划](https://www.langchain.com/pricing-langsmith)上的组织。请[联系销售](https://www.langchain.com/contact-sales)了解更多。
</Note>

- 您的组织必须在企业计划上。
- 您的身份提供商 (IdP) 必须支持 SAML 2.0 标准。
- 只有 [`Organization Admins`](/langsmith/organization-workspace-operations#sso-and-authentication) 可以配置 SAML SSO。

有关将 SCIM 与 SAML 结合使用进行用户配置和取消配置的操作说明，请参阅 [SCIM 设置](#set-up-scim-for-your-organization)。

1. 在您的 IdP 中：使用以下详细信息配置 SAML 应用程序，然后复制步骤 3 的元数据 URL 或 XML。

   <Note>
   以下 URL 取决于您的组织是在 GCP US、GCP EU、GCP APAC 还是 AWS US 云区域。确保选择正确的链接。
   </Note>

   1. 单一登录 URL（或 ACS URL）：
      <SaasRegionUrls prefix="auth" suffix="/auth/v1/sso/saml/acs" />

   2. 受众 URI（或 SP 实体 ID）：
      <SaasRegionUrls prefix="auth" suffix="/auth/v1/sso/saml/metadata" />
   3. Name ID 格式：电子邮件地址。
   4. 应用程序用户名：电子邮件地址。
   5. 必需声明：`sub` 和 `email`。

2. 在 LangSmith 中：转到 **Settings** -> **Members and roles** -> **SSO Configuration**。填写所需信息并提交以激活 SSO 登录：

   1. 填写 `SAML metadata URL` 或 `SAML metadata XML`。
   2. 选择 `Default workspace role` 和 `Default workspaces`。通过 SSO 登录的新用户将被添加到具有所选角色的指定工作区。

      - `Default workspace role` 和 `Default workspaces` 是可编辑的。更新后的设置将仅适用于新用户，不适用于现有用户。
      -（即将推出）`SAML metadata URL` 和 `SAML metadata XML` 是可编辑的。这通常仅在加密密钥轮换/过期或元数据 URL 发生变化但仍在使用相同 IdP 时才需要。

### Supabase 属性映射

<Note>
Supabase 属性映射是一个[仅限云](/langsmith/cloud)的功能。[自托管](/langsmith/self-hosted)部署直接在 IdP 中配置 SAML/OIDC 属性——请参阅[使用 OAuth2.0 和 OIDC 设置 SSO](/langsmith/self-host-sso)。
</Note>

LangSmith 云使用 [Supabase](/langsmith/cloud) 作为 SAML SSO 后端。Supabase 自动将一小组标准 SAML 属性（如 `email` 和 `sub`）传递到用户的 JWT。您 IdP 发出的任何其他非标准 SAML 属性（例如，[SSO 群组同步](#sso-groups-sync-alternative)的 `groups`）必须通过 Supabase 显式转发，然后 LangSmith 才能读取。

**属性流程 (1:1)：**

1. **IdP**：发出带有配置名称的 SAML 属性（例如，`groups`）。
2. **Supabase**：仅当属性名称出现在 SSO 提供商的 **Supabase Attribute Mapping** 表中时，才将属性转发到用户的 JWT。标准属性自动转发；非标准属性除非明确列出，否则会被丢弃。
3. **LangSmith**：按名称读取 JWT 声明（例如，[SSO 群组同步](#sso-groups-sync-alternative)的 **Groups claim field** 的值）。

属性名称端到端保留：IdP 属性名称、Supabase Attribute Mapping 条目和下游 LangSmith 设置都使用相同的字符串。

#### 配置

在 **Settings** → **Members and roles** → **SSO Configuration** 中，向下滚动到 **Supabase Attribute Mapping** 部分，为每个要转发的非标准属性添加一行：

| 列 | 描述 |
| --- | --- |
| **Attribute name** | IdP 发出的 SAML 属性名称。必须与 LangSmith 下游期望的 JWT 声明名称匹配（对于 SSO 群组同步，这匹配 **Groups claim field** 值）。 |
| **Array** | 如果属性是多值的（字符串列表），请勾选此项。对于标量（单值）属性，请保持未选中状态。示例：对于 `groups` 勾选此项；对于 `full_name` 保持未选中。 |

单击 **Add row** 添加每个其他属性，然后 **Save**。空的映射表意味着没有非标准属性流向 JWT。

### Entra ID (Azure)

有关更多信息，请参阅 Microsoft 的[文档](https://learn.microsoft.com/en-us/entra/identity/enterprise-apps/add-application-portal-setup-sso)。

<div id="create-application-entra-id"></div>
**步骤 1：创建新的 Entra ID 应用程序集成**

1. 使用特权角色（例如，`Global Administrator`）登录 [Azure 门户](https://portal.azure.com/#home)。在左侧导航窗格中，选择 `Entra ID` 服务。

2. 导航到 **Enterprise Applications**，然后选择 **All Applications**。

3. 点击 **Create your own application**。

4. 在 **Create your own application** 窗口中：

   1. 输入应用程序名称（例如，`LangSmith`）。
   2. 选择 **Integrate any other application you don't find in the gallery (Non-gallery)**。

5. 点击 **Create**。

**步骤 2：配置 Entra ID 应用程序并获取 SAML 元数据**

1. 打开您创建的企业应用程序。

2. 在左侧导航中，选择 **Manage** > **Single sign-on**。

3. 在单点登录页面上，点击 **SAML**。

4. 更新 **Basic SAML Configuration**：

   1. `Identifier (Entity ID)`:
      <SaasRegionUrls prefix="auth" suffix="/auth/v1/sso/saml/metadata" />

   2. `Reply URL (Assertion Consumer Service URL)`:
      <SaasRegionUrls prefix="auth" suffix="/auth/v1/sso/saml/acs" />

   3. 将 `Relay State`、`Logout Url` 和 `Sign on URL` 留空。
   4. 点击 **Save**。

5. 使用 **Namespace**：`http://schemas.xmlsoap.org/ws/2005/05/identity/claims` 确保存在必需声明：

   1. `sub`: `user.objectid`。
   2. `emailaddress`: `user.userprincipalname` 或 `user.mail`（如果使用后者，请确保所有用户在 `Contact Information` 下的 `Email` 字段都已填写）。
   3.（可选）有关 SCIM，请参阅[设置文档](/langsmith/user-management)中有关 `Unique User Identifier (Name ID)` 的具体说明。

6. 在基于 SAML 的登录页面上的 **SAML Certificates** 下，复制 **App Federation Metadata URL**。

**步骤 3：设置 LangSmith SSO 配置**

使用上一步的元数据 URL，按照[初始配置](#initial-configuration)中 `填写所需信息` 步骤的说明进行操作。

**步骤 4：验证 SSO 设置**

1. 在 Entra ID 中将应用程序分配给用户/组：

   1. 选择 **Manage** > **Users and groups**。

   2. 点击 **Add user/group**。

   3. 在 **Add Assignment** 窗口中：

      1. 在 **Users** 下，点击 **None Selected**。
      2. 搜索要分配给企业应用程序的用户，然后点击 **Select**。
      3. 确认用户已选中，然后点击 **Assign**。

2. 让用户通过 **SSO Configuration** 页面中的唯一登录 URL 登录，或转到 **Manage** > **Single sign-on** 并选择 **Test single sign-on with (application name)**。

### Google

有关更多信息，请参阅 Google 的[文档](https://support.google.com/a/answer/6087519)。

**步骤 1：创建和配置 Google Workspace SAML 应用程序**

1. 确保您使用具有适当权限的管理员账户登录。

2. 在管理控制台中，转到 **Menu** -> **Apps** -> **Web and mobile apps**。

3. 点击 **Add App**，然后点击 **Add custom SAML app**。

4. 输入应用程序名称，可选择上传图标。点击 **Continue**。

5. 在 Google Identity Provider 详细信息页面上，下载 **IDP metadata** 并为步骤 2 保存。点击 **Continue**。

6. 在 `Service Provider Details` 窗口中，输入：

   1. `ACS URL`:
      <SaasRegionUrls prefix="auth" suffix="/auth/v1/sso/saml/acs" />

   2. `Entity ID`:
      <SaasRegionUrls prefix="auth" suffix="/auth/v1/sso/saml/metadata" />

   3. 将 `Start URL` 和 `Signed response` 复选框留空。
   4. 将 `Name ID` 格式设置为 `EMAIL`，并将 `Name ID` 保留为默认值（`Basic Information > Primary email`）。
   5. 点击 `Continue`。

7. 使用 `Add mapping` 确保存在必需声明：
   1. `Basic Information > Primary email` -> `email`

**步骤 2：设置 LangSmith SSO 配置**

使用上一步的 `IDP metadata` 作为元数据 XML，按照[初始配置](#initial-configuration)中 `填写所需信息` 步骤的说明进行操作。

**步骤 3：在 Google 中启用 SAML 应用程序**

1. 在 `Menu -> Apps -> Web and mobile apps` 下选择 SAML 应用程序。

2. 点击 `User access`。

3. 启用服务：

   1. 要为组织中的每个人启用服务，点击 `On for everyone`，然后点击 `Save`。

   2. 要为组织单位启用服务：

      1. 在左侧，选择组织单位，然后点击 `On`。
      2. 如果服务状态设置为 `Inherited`，并且您希望保留更新的设置（即使父设置发生变化），点击 `Override`。
      3. 如果服务状态设置为 `Overridden`，请点击 `Inherit` 恢复到与其父级相同的设置，或点击 `Save` 以保留新设置（即使父设置发生变化）。

   3. 要为跨组织单位或组织单位内的用户集启用服务，请选择一个访问组。有关详细信息，请转到[使用组自定义服务访问](https://support.google.com/a/answer/9050643)。

4. 确保用户用于登录 LangSmith 的电子邮件地址与用于登录 Google 网域的电子邮件地址一致。

**步骤 4：验证 SSO 设置**

让有访问权限的用户通过 **SSO Configuration** 页面中的唯一登录 URL 登录，或在 Google 的 SAML 应用程序页面中点击 **TEST SAML LOGIN**。

### Okta

#### 支持的功能

- IdP 发起 SSO（单点登录）
- SP 发起 SSO
- 即时配置
- 仅强制 SSO

#### 配置步骤

有关更多信息，请参阅 Okta 的[文档](https://help.okta.com/en-us/content/topics/apps/apps_app_integration_wizard_saml.htm)。

**步骤 1：创建和配置 Okta SAML 应用程序**

<div id="via-okta-integration-network">
    <b>通过 Okta Integration Network（推荐）</b>
</div>

1. 登录 [Okta](https://login.okta.com/)。
1. 在右上角，选择 Admin。从管理区域看不到该按钮。
1. 选择 `Browse App Integration Catalog`。
1. 找到并选择 LangSmith 应用程序。
1. 在应用程序概述页面上，选择 Add Integration。
1. 将 `ApiUrlBase` 留空。
1. 填写 `AuthHost`：
    <SaasRegionUrls prefix="auth" protocol={false} />

1.（可选，如果也计划使用 [SCIM](#set-up-scim-for-your-organization)）填写 `LangSmithUrl`：
    <SaasRegionUrls prefix="api.smith" protocol={false} />
1. 在 Application Visibility 下，保持复选框未选中。
1. 选择 Next。
1. 选择 `SAML 2.0`。
1. 填写 `Sign-On Options`：
   - `Application username format`: `Email`
   - `Update application username on`: `Create and update`
   - `Allow users to securely see their password`: 保持**未选中**。
1. 从 **Sign On Options** 页面复制 **Metadata URL** 以供下一步使用。

**通过自定义应用程序集成**

<Warning>
SCIM 与此配置方法不兼容。请参阅[**通过 Okta Integration Network**](#via-okta-integration-network)。
</Warning>

1. 以管理员身份登录 Okta，然后进入 **Okta Admin console**。

2. 在 **Applications** > **Applications** 下，点击 **Create App Integration**。

3. 选择 **SAML 2.0**。

4. 输入 `App name`（例如，`LangSmith`）和可选的 **App logo**，然后点击 **Next**。

5. 在 **Configure SAML** 页面中输入以下信息：

   1. `Single sign-on URL`（`ACS URL`）。保持 `Use this for Recipient URL and Destination URL` 选中：
      <SaasRegionUrls prefix="auth" suffix="/auth/v1/sso/saml/acs" />

   2. `Audience URI (SP Entity ID)`:
      <SaasRegionUrls prefix="auth" suffix="/auth/v1/sso/saml/metadata" />

   3. `Name ID format`：**Persistent**。
   4. `Application username`: `email`。
   5. 将其余字段留空或设置为其默认值。
   6. 点击 **Next**。

6. 点击 **Finish**。

7. 从 **Sign On** 页面复制 **Metadata URL** 以供下一步使用。

**步骤 2：设置 LangSmith SSO 配置**

使用上一步的元数据 URL，按照[初始配置](#initial-configuration)中 **填写所需信息** 步骤的说明进行操作。

**步骤 3：在 Okta 中将用户分配给 LangSmith**

1. 在 **Applications** > **Applications** 下，选择步骤 1 创建的 SAML 应用程序。
2. 在 **Assignments** 选项卡下，点击 **Assign**，然后选择 **Assign to People** 或 **Assign to Groups**。
3. 进行所需的选择，然后 **Assign** 和 **Done**。

**步骤 4：验证 SSO 设置**

让有访问权限的用户通过 `SSO Configuration` 页面中的唯一登录 URL 登录，或让用户从他们的 Okta 仪表板中选择应用程序。

#### SP 发起 SSO

配置服务提供商发起的 SSO 后，用户可以使用唯一登录 URL 登录。您可以在 LangSmith UI 中的 **Organization members and roles** 下的 **SSO configuration** 中找到此 URL。

## 为您的组织设置 SCIM

<Note>
在寻找比 SCIM 更轻量级的替代方案，不需要 IdP 管理员参与推送群组？请参阅下面的[SSO 群组同步](#sso-groups-sync-alternative)——它在登录时直接从 SSO 令牌读取群组成员资格，并重用相同的命名约定。
</Note>

跨域身份管理系统 (SCIM) 是一个开放标准，允许自动化用户配置。使用 SCIM，您可以自动在 LangSmith [组织和](/langsmith/administration-overview)工作区中配置和取消配置用户，使用户访问与您组织的身份提供商保持同步。

<Note>
SCIM 适用于[企业计划](https://www.langchain.com/pricing)上的组织。[联系销售](https://www.langchain.com/contact-sales)了解更多。

SCIM 适用于 Helm chart 版本 0.10.41（应用程序版本 0.10.108）及更高版本。

SCIM 支持仅限 API（请参阅下面的说明）。
</Note>

SCIM 消除了手动用户管理的需要，并确保用户访问始终与您组织的身份系统保持同步。这允许：

- **自动化用户管理**：根据用户在 IdP 中的状态，自动在 LangSmith 中添加、更新和删除用户。
- **减少管理开销**：无需在多个系统上手动管理用户访问。
- **提高安全性**：离开组织的用户会自动从 LangSmith 中取消配置。
- **一致的访问控制**：用户属性和群组成员资格在系统之间同步。
- **扩展团队访问控制**：高效管理具有多个工作区和自定义角色的大型团队。
- **角色分配**：为用户组选择特定的[组织角色](/langsmith/rbac#organization-roles)和[工作区角色](/langsmith/rbac#workspace-roles)。

### 要求

#### 先决条件

- 您的组织必须在企业计划上。
- 您的身份提供商 (IdP) 必须支持 SCIM 2.0。
- 只有[组织管理员](/langsmith/administration-overview#organization-roles)可以配置 SCIM。
- 对于云客户：必须能够为您的组织[配置 SAML SSO](#set-up-saml-sso-for-your-organization)。
- 对于自托管客户：必须启用[带客户端密钥的 OAuth](/langsmith/self-host-sso#with-client-secret-recommended) 身份验证模式。
- 对于自托管客户，必须允许从身份提供商到 LangSmith 的网络流量：
  - Microsoft Entra ID 支持将 IP 范围列入白名单或基于代理的解决方案来提供连接。
    （[详细信息](https://learn.microsoft.com/en-us/entra/identity/app-provisioning/use-scim-to-provision-users-and-groups#ip-ranges)）。
  - Okta 支持将 IP 或域列入白名单（[详细信息](https://help.okta.com/en-us/content/topics/security/ip-address-allow-listing.htm)）
    或基于代理的解决方案（[详细信息](https://help.okta.com/en-us/content/topics/provisioning/opp/opp-main.htm)）来提供连接。

<Note>
SCIM 连接通常需要 HTTP/1.1 或更高版本。如果您的客户端使用 HTTP/1.0，可能会遇到 `426 Upgrade Required` 错误。
</Note>

#### 角色优先级

当用户属于同一工作区的多个群组时，适用以下优先级：

1. **组织 Admin 组**拥有最高优先级。这些组中的用户在所有工作空间中都将具有 `Admin` 角色。
2. **最近创建的工作空间特定组**优先于其他工作空间组。

<Note>
当某个组被删除或用户从某个组中移除时，他们的访问权限会根据其剩余的组成员资格进行更新，遵循优先级规则。

SCIM 组成员资格会覆盖手动分配的角色或通过实时配置 (JIT) 分配的角色。建议禁用 JIT 配置以避免冲突。更多详情，请参阅[在 SSO 组织中管理用户访问权限](/langsmith/jit-invite-sso#scim-integration)。
</Note>

#### 邮箱验证

仅在云端，通过 SCIM 创建新用户会向该用户触发一封电子邮件。
他们必须通过点击电子邮件中的链接来验证其邮箱地址。
链接在 24 小时后过期，如需要，可以通过 SCIM 删除并重新添加用户来重新发送。

### 属性和映射

#### 组命名约定

<Warning>
通过 SCIM 不支持重命名组。组名称是永久性的，因为它们必须与 LangSmith 中的角色名称和/或工作空间名称相匹配。
</Warning>

组成员资格按照特定的命名约定映射到 LangSmith 工作空间成员资格和工作空间角色。默认情况下，组件之间的分隔符是冒号 (`:`)，但您可以为组织[配置自定义分隔符](#configure-custom-separator)。

<Note>
您可以在组名称的 **组织角色名称** 部分省略空格。这有助于那些不允许组名称中出现空格的身份提供商。例如，LangSmith 接受 `OrganizationAdmins` 和 `OrganizationUser` 作为 `Organization Admins` 和 `Organization User` 的等效名称。这种灵活性仅适用于组织角色名称标记。工作空间名称和工作空间角色名称将空格视为字面字符，因此省略空格的变体与带空格的对应名称不匹配。
</Note>

**组织 Admin 组**

格式：`<optional_prefix>Organization Admin` 或 `<optional_prefix>Organization Admins`

示例：

- `LS:Organization Admins`
- `LS:OrganizationAdmins`（省略了空格——对于不允许组名称中出现空格的身份提供商很有用）
- `Groups-Organization Admins`
- `Organization Admin`

**工作空间特定组**

格式：`<optional_prefix><org_role_name><separator><workspace_name><separator><workspace_role_name>`

分隔符默认为 `:`（冒号）。支持的分隔符有：`:`（冒号）、`-`（连字符）、`_`（下划线）、` `（空格）、`&`（和号）。

使用默认冒号分隔符的示例：

- `LS:Organization User:Production:Annotators`
- `LS:OrganizationUser:Production:Annotators`（在角色名称标记中省略了空格）
- `Groups-Organization User:Engineering:Developers`
- `Organization User:Marketing:Viewers`

使用连字符分隔符的示例：

- `LS-Organization User-Production-Annotators`
- `LS-OrganizationUser-Production-Annotators`（在角色名称标记中省略了空格）
- `Organization User-Engineering-Developers`

<Note>
如果工作空间名称包含分隔符字符（例如，使用 `-` 作为分隔符的工作空间 `my-team`），LangSmith 将自动尝试所有可能的分割方式来找到有效的工作空间和角色组合。
</Note>

#### 配置自定义分隔符

要为组织更改 SCIM 组名称分隔符，请使用 `PATCH /api/v1/orgs/current/info` [端点](/langsmith/smith-api/orgs/update-current-organization-info)。对于区域 SaaS 部署，请将请求发送到区域主机上的相同路径（`eu.api.smith.langchain.com`、`apac.api.smith.langchain.com` 或 `aws.api.smith.langchain.com`）：

```bash
curl -X PATCH $LANGCHAIN_ENDPOINT/api/v1/orgs/current/info \
  -H "X-Api-Key: $LANGCHAIN_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"scim_group_name_separator": "-"}'
```

分隔符必须是单个字符，且为以下之一：`:`（冒号）、`-`（连字符）、`_`（下划线）、` `（空格）或 `&`（& 符号）。默认为 `:`（冒号）。

<Note>
更改分隔符不会重命名现有的 SCIM 组。如果你更改了分隔符，你还必须更新身份提供者中的组名以使用新的分隔符。
</Note>

### 步骤 1 - 配置 SAML SSO（仅限云端）

[SAML SSO](#set-up-saml-sso-for-your-organization) 配置有两种场景：

1. 如果你的组织已经配置了 SAML SSO，你应该跳过初始添加应用的步骤（[从 Okta Integration Network 添加应用](#add-application-okta-oin) 或 [创建新的 Entra ID 应用集成](#create-application-entra-id)），因为你已经配置了应用，只需要启用预配。
1. 如果你首次配合 SCIM 配置 SAML SSO，请先按照说明 [设置 SAML SSO](#set-up-saml-sso-for-your-organization)，_然后_按照此处的说明启用 SCIM。

#### NameID 格式

LangSmith 使用 SAML NameID 来识别用户。NameID 是 SAML 响应中的必需字段，且不区分大小写。

NameID 必须：

1. 每个用户唯一。
2. 是一个永不更改的持久值，例如随机生成的唯一用户 ID。
3. 每次登录尝试时完全匹配。它不应依赖用户输入。

NameID 不应该是邮箱地址或用户名，因为邮箱地址和用户名更可能随时间更改，而且可能区分大小写。

NameID 格式必须是 `Persistent`，除非你使用的字段（如邮箱）需要不同的格式。

### 步骤 2 - 禁用 JIT 预配

在启用 SCIM 之前，请禁用 [实时 (JIT) 预配](/langsmith/jit-invite-sso#jit-provisioning) 以防止自动和手动用户预配之间的冲突。

#### 为云端禁用 JIT

使用 `PATCH /orgs/current/info` [端点](/langsmith/smith-api/orgs/update-current-organization-info)。对于区域 SaaS 部署，请将请求发送到区域主机的相同路径（`eu.api.smith.langchain.com`、`apac.api.smith.langchain.com` 或 `aws.api.smith.langchain.com`）：

```bash
curl -X PATCH $LANGCHAIN_ENDPOINT/orgs/current/info \
  -H "X-Api-Key: $LANGCHAIN_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"jit_provisioning_enabled": false}'
```

#### 为自托管禁用 JIT

从 LangSmith chart 版本 **0.11.14** 开始，您可以为使用 SSO 的自托管组织禁用 JIT 配置。禁用方法，设置以下值：

```yaml
commonEnv:
  - name: SELF_HOSTED_JIT_PROVISIONING_ENABLED
    value: "false"
```

### Step 3 - 生成 SCIM Bearer Token

<Note>
在自托管环境中，下面的完整 URL 可能如下所示 `https://langsmith.yourdomain.com/api/v1/platform/orgs/current/scim/tokens`（不带子域名，请注意 `/api/v1` 路径前缀）或 `https://langsmith.yourdomain.com/subdomain/api/v1/platform/orgs/current/scim/tokens`（带子域名）- 请参阅[入口文档](/langsmith/self-host-ingress)了解更多详情。
</Note>

为您的组织生成 SCIM Bearer Token。此令牌将用于您的 IdP 对 SCIM API 请求进行身份验证。请确保正确设置环境变量，例如：

```bash
curl -X POST $LANGCHAIN_ENDPOINT/v1/platform/orgs/current/scim/tokens \
  -H "X-Api-Key: $LANGCHAIN_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"description": "Your description here"}'
```

请注意，SCIM Bearer Token 值在此请求的响应之外不可用。这些额外的端点存在：

- `GET /v1/platform/orgs/current/scim/tokens`
- `GET /v1/platform/orgs/current/scim/tokens/{scim_token_id}`
- `PATCH /v1/platform/orgs/current/scim/tokens/{scim_token_id}`（仅支持 `description` 字段）
- `DELETE /v1/platform/orgs/current/scim/tokens/{scim_token_id}`

### SSO 组同步（替代方案）

<Note>
SSO 组同步适用于已配置 SAML SSO（云）或 OIDC（自托管）的 [Enterprise 计划](/langsmith/pricing-plans) 组织。[联系销售](https://www.langchain.com/contact-sales)了解更多。
</Note>

SSO 组同步是 [SCIM](#set-up-scim-for-your-organization) 的更简单替代方案，适用于无法或不想配置 SCIM 组推送的组织。它不是按单独的同步间隔将组从您的 IdP 推送到 LangSmith，而是在登录时直接从 SSO 令牌中的可配置声明读取组成员资格，并使用与 SCIM 相同的[命名约定](#group-naming-convention)应用组织级别和工作区级别的角色分配。

这是一种成熟的做法，被 GitLab（SAML Group Links）、Grafana（Team Sync）、HashiCorp Vault（`groups_claim`）和 Atlassian（JIT 组分配）使用。

#### 何时使用 SSO 组同步与 SCIM

SSO 组同步和 SCIM 技术上可以共存（每个仅管理标记有其自身预配方法的身份），但我们建议**每个组织选择一种机制**，而不是同时使用两者，以避免混淆优先级行为。

| | SSO 组同步 | SCIM |
| --- | --- | --- |
| **同步触发器** | 每次 SSO 登录时 | IdP 主动推送（约 1 小时周期） |
| **IdP 管理员参与** | 极少，只需在 SSO 令牌中包含组 | 必需，配置 SCIM 预配应用程序 |
| **取消预配** | 延迟到下次登录 | 通过 IdP 推送近乎实时 |
| **命名约定** | 重用 [SCIM 约定](#group-naming-convention) | [SCIM 约定](#group-naming-convention) |
| **自定义分隔符** | 重用组织级别的 [`scim_group_name_separator`](#configure-custom-separator) | [`scim_group_name_separator`](#configure-custom-separator) |

当 IdP 管理员参与最少且可接受响应式（登录时）同步时，选择 **SSO 组同步**。当需要主动预配/取消预配和近乎实时的组成员资格更新时，选择 **SCIM**。

#### 配置

1. 在您的 IdP 中：将用户的组成员资格添加到 SSO 令牌声明中（默认声明名称：`groups`）。组名必须遵循 [SCIM 命名约定](#group-naming-convention)。
2. 在 LangSmith 中：进入 **Settings** → **Members and roles** → **SSO Configuration** → **SSO Groups Sync** 并配置以下设置：

   | 设置 | 描述 |
   | --- | --- |
   | **Enable SSO Groups Sync** | 根据 SSO 令牌中的组成员资格自动分配工作区角色。 |
   | **Groups claim field**（默认 `groups`） | SSO 令牌中包含组成员资格的声明名称。 |
   | **Sync workspace/role assignments** | 在每次 SSO 登录时根据组名更新工作区成员资格和角色。 |
   | **Require matching group to sign in** | 如果 SSO 令牌不包含与命名约定匹配的组，则阻止登录。 |

您还可以通过向 SSO 设置端点发送 `PATCH` 请求来通过 API 配置这些设置：

```bash
curl -X PATCH $LANGCHAIN_ENDPOINT/api/v1/orgs/current/sso-settings/$SSO_PROVIDER_ID \
  -H "X-Api-Key: $LANGCHAIN_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "sso_groups_enabled": true,
    "sso_groups_claim_field": "groups",
    "sso_groups_role_sync_enabled": true,
    "sso_groups_required": false
  }'
```

<Warning>
禁用 SSO 组同步不会移除现有访问权限。通过 SSO 组配置的用户将保留其当前访问权限，直至下次登录。
</Warning>

#### 配置您的 IdP 以发出 groups SAML 属性（云端）

<Note>
本节仅适用于**企业云**版本。自托管客户直接在 OIDC IdP 中配置组，请参阅[自托管 SSO 上的 SSO 组同步](/langsmith/self-host-sso#sso-groups-sync)。
</Note>

要使用户的组成员资格在登录时对 LangSmith 可见，您需要做两件事：

1. 配置您的 IdP SAML 应用程序以发出多值组属性。
2. 在 [Supabase 属性映射](#supabase-attribute-mapping) 中添加匹配的条目，以便该属性通过 JWT 传递（勾选**数组**）。

**要求：**

- IdP 属性名称（例如 `groups`）必须与 **Supabase 属性映射**条目和 **Groups 声明字段**值（默认 `groups`）匹配。
- 属性必须是**多值的**（字符串列表），而不是单个带分隔符的字符串。如果您的 IdP 仅支持单值属性，您需要为每个组发出一个属性语句。
- 每个值必须是遵循 [SCIM 命名约定](#group-naming-convention) 的组名称。
- 只有名称符合约定的组才会被处理。LangSmith 会忽略不符合其命名约定的组，例如组织范围的目录组或应用分配组。您无需在 IdP 端筛选这些组——发出所有组，LangSmith 会跳过不相关的组。

**各 IdP 设置：**

<Tabs>
<Tab title="Okta">

在 LangSmith SAML 应用程序中：

1. **Directory** → **Profile Editor** → 选择 LangSmith 应用程序的用户配置文件。
2. 添加名为 `groups` 的自定义属性，**类型**为 `string array`。
3. **Sign On** → 编辑 SAML 设置并添加属性语句：
   - **Name**：`groups`
   - **Name format**：`Unspecified`（或 `Basic`）
   - **Filter**：`Matches regex`，使用 `.*` 发送所有组，或使用更严格的正则表达式（例如 `^LS:.*`）限制为带 LangSmith 前缀的组。

</Tab>
<Tab title="Entra ID (Azure)">

在 LangSmith 企业应用程序中：

1. **Single sign-on** → **Attributes & Claims** → **Add a group claim**。
2. 选择要发出的组（通常为 **Groups assigned to the application**）。
3. 将 **Source attribute** 设置为 `Cloud-only group display names`，以便发送组名称（必须符合[命名约定](#group-naming-convention)）而不是对象 ID。
4. 将声明 **Name** 设置为 `groups`（或您配置的 **Groups claim field** 值），不设置命名空间。

</Tab>
<Tab title="Google Workspace">

Google 的 SAML SSO 本身不会将 Google Group 成员资格作为 SAML 属性发出。要将 SSO 组同步与 Google Workspace 配合使用，您必须：

- 通过目录同步工具管理组成员资格，该工具将组公开为 SAML 属性，或
- 改用 [SCIM](#set-up-scim-for-your-organization)，它支持从 Google Workspace 推送组。

</Tab>
</Tabs>

#### 组命名示例

组名称遵循 [SCIM 命名约定](#group-naming-convention)。`<workspace_role>` 段接受内置角色和[自定义工作区角色](/langsmith/rbac#custom-roles)的名称。

| 用途 | 示例组名称 |
| --- | --- |
| 组织管理员（在所有工作区授予工作区管理员） | `LS:Organization Admins` |
| `Production` 中的工作区管理员 | `LS:Organization User:Production:Admin` |
| `Engineering` 中的工作区编辑者 | `LS:Organization User:Engineering:Editor` |
| `Marketing` 中的工作区查看者 | `LS:Organization User:Marketing:Viewer` |
| `Production` 中的自定义角色 `Annotators` | `LS:Organization User:Production:Annotators` |

#### 行为

- **命名约定**：组名称遵循与 SCIM 相同的格式（例如，`LS:Organization Admins` 用于组织管理员，`LS:Organization User:Production:Editor` 用于工作区范围）。有关完整格式，请参阅[组命名约定](#group-naming-convention)。分隔符按组织通过 [`scim_group_name_separator`](#configure-custom-separator) 配置，与 SCIM 共享。
- **格式错误的组名称**：不符合约定的组名称会被静默跳过（记录）并且不会阻止有效组的登录。
- **登录门控**：当启用**Require matching group to sign in**且 SSO 令牌包含零个匹配组时，登录被阻止。
- **优先级**：SSO 组同步不会修改 SCIM 来源、手动分配或 JIT 配置的成员资格。它完全掌控自己的分配，并根据令牌中的组成员资格在每次登录时替换它们。
- **组织管理员传播**：如果用户从其组获得组织管理员角色，他们将在所有工作区被授予工作区管理员（与 SCIM 行为相同）。

#### 注意事项

- **取消配置延迟**：与 SCIM（主动推送）不同，SSO 组同步仅在登录时更新。从 IdP 中移除的组中的用户在下次 LangSmith 登录前会保留其现有工作区访问权限。**Require matching group to sign in** 门控通过在下一次登录时完全阻止用户来缓解此问题。
- **无回溯同步**：更改角色映射或启用该功能不会更新现有用户，直至他们再次登录。
- **需要命名约定**：客户必须按照 SCIM 约定命名其 IdP 组。如果您的 IdP 组遵循不同的命名策略，基于 `description` 映射的 SCIM（请参阅[组属性](#group-attributes)）可能是更好的选择。
