我们欢迎对 LangChain 文档的贡献，包括新功能、[集成](/oss/contributing/publish-langchain)以及对现有文档的改进。

## 快速开始 - 本地开发

要运行文档的本地预览：

```bash
git clone https://github.com/langchain-ai/docs.git
```

```bash
cd docs
```

```bash
make install
```

```bash
make dev
```

这将启动一个热重载开发服务器，访问地址为 `http://localhost:3000`。编辑 `src/` 中的文件即可即时查看更改。

<Tip>
    **使用 AI 编码代理？** 安装 [LangChain Skills](https://github.com/langchain-ai/langchain-skills) 来提升代理在 LangChain 生态任务中的表现，然后点击页面右上角的"复制页面"按钮，将原始内容粘贴到你的代理中，让它自动为你配置环境。
</Tip>

<Tip>
如果本地预览遇到问题，请尝试运行 `mint update` 以确保你使用的是最新版本的 Mintlify。
</Tip>

<Accordion title="前提条件">
    **必需：**
    - Python 3.13+
    - [uv](https://docs.astral.sh/uv/) - Python 包管理器
    - [Node.js](https://nodejs.org/en) 和 npm
    - [Make](https://www.gnu.org/software/make/)
    - [Git](https://git-scm.com/)

    **可选：**
    - [markdownlint-cli](https://github.com/igorshubovych/markdownlint-cli) - `npm install -g markdownlint-cli`
    - [Mintlify MDX VSCode 扩展](https://www.mintlify.com/blog/mdx-vscode-extension)
</Accordion>

## 编辑文档

<Accordion title="GitHub 快速编辑">
    对于拼写错误或小幅修改，可直接在 GitHub 上编辑，无需本地设置：

    1. 点击任意页面底部的 **在 GitHub 上编辑此页面**。
    1. Fork 到你的个人账户。
    1. 在 GitHub 的网页编辑器中进行更改。
    1. 创建 pull request。
</Accordion>

<Note>
    **仅编辑 `src/` 中的文件**-- `build/` 目录是自动生成的。
</Note>

1. 按照我们的[编写规范](#writing-standards)编辑 `src/` 中的文件。
1. 提交前运行[质量检查](#run-quality-checks)。
1. 创建 pull request 进行审核。

<Note>
    所有 pull request 必须链接到一个 issue 或讨论，且解决方案已获得维护者的批准。详见我们的 [pull request 要求](/oss/contributing/overview#pull-request-requirements)。
</Note>

<Accordion title="创建可分享的预览构建（仅限 LangChain 团队）">
    当你创建或更新 PR 时，系统会自动生成一个[预览分支/ID](https://github.com/langchain-ai/docs/actions/workflows/create-preview-branch.yml)，并会在 PR 上留下一条包含该 ID 的评论。

    1. 从评论中复制预览分支的 ID
    1. 在 [Mintlify 仪表板](https://dashboard.mintlify.com/langchain-5e9cc07a/langchain-5e9cc07a?section=previews) 中点击 **Create preview deployment**
    1. 输入预览分支的 ID，然后点击 **Create deployment**
    1. 选择该预览并点击 **Visit** 查看

    如需使用最新更改重新部署，请在仪表板上点击 **Redeploy**。
</Accordion>

### 运行质量检查

提交更改前，请确保你的代码通过格式化和代码检查：

```bash
# Check broken links
make broken-links

# Format code automatically
make format

# Check for linting issues
make lint

# Fix markdown issues
make lint_md_fix

# Run tests to ensure your changes don't break existing functionality
make test
```

更多详情，请参阅 `README` 中的[可用命令](https://github.com/langchain-ai/docs?tab=readme-ov-file#available-commands)部分。

<Important>
    所有拉取请求都会自动通过 CI/CD 检查。相同的代码规范和格式化标准将被强制执行，如果这些检查失败，拉取请求将无法合并。
</Important>

## 文档类型

所有文档都属于以下四个类别之一：

<CardGroup cols={2}>

    <Card title="操作指南" icon="tool" href="#how-to-guides">
        面向任务的指导，面向已知晓目标的用户。
    </Card>

    <Card title="概念指南" icon="bulb" href="#conceptual-guides">
        提供更深入理解和见解的解释性内容。
    </Card>

    <Card title="参考" icon="book" href="#reference">
        API 和实现细节的技术描述。
    </Card>

    <Card title="教程" icon="school" href="#tutorials">
        引导用户通过实践活动来建立理解的课程。
    </Card>

</CardGroup>

<Note>
    在适用的情况下，所有文档都必须包含 Python 和 JavaScript/TypeScript 内容。更多详情，请参阅[共存 Python 和 JavaScript/TypeScript 内容](#co-locate-python-and-javascript%2Ftypescript-content)部分。
</Note>

### 操作指南

操作指南是面向任务的指导，面向已知晓目标的用户。操作指南的示例请参阅 [LangChain](/oss/langchain/overview) 和 [LangGraph](/oss/langgraph/overview) 标签页。

<AccordionGroup>
    <Accordion title="特点">
        - **任务导向**：专注于特定任务或问题
        - **分步骤**：将任务分解为更小的步骤
        - **实践导向**：提供具体的示例和代码片段
    </Accordion>

    <Accordion title="技巧">
        - 专注于**如何做**而不是**为什么**
        - 使用具体的示例和代码片段
        - 将任务分解为更小的步骤
        - 链接到相关的概念指南和参考文档
    </Accordion>

    <Accordion title="示例">
        - [消息](/oss/langchain/messages)
        - [工具](/oss/langchain/tools)
        - [流式处理](/oss/langgraph/streaming)
    </Accordion>
</AccordionGroup>

### 概念指南

概念指南抽象地介绍核心概念，提供深入的理解。

<AccordionGroup>
    <Accordion title="特点">
        - **理解导向**：解释事情为何如此运作
        - **全局视角**：比其他类型更高更广的视角
        - **设计导向**：解释决策和权衡
        - **上下文丰富**：使用类比和比较
    </Accordion>

    <Accordion title="技巧">
        - 专注于**"为什么"**而不是"如何做"
        - 提供功能使用不一定需要的补充信息
        - 可以使用类比并参考替代方案
        - 避免混入过多参考内容
        - 链接到相关的教程和操作指南
    </Accordion>

    <Accordion title="示例">
        - [内存](/oss/concepts/memory)
        - [上下文](/oss/concepts/context)
        - [图形 API](/oss/langgraph/graph-api)
        - [函数式 API](/oss/langgraph/functional-api)
    </Accordion>
</AccordionGroup>

### 参考

参考文档包含详细的底层信息，精确描述存在哪些功能以及如何使用。

<CardGroup cols={2}>
    <Card title="Python 参考" href="https://reference.langchain.com/python/" icon="brand-python" arrow />
    <Card title="JavaScript/TypeScript 参考" href="https://reference.langchain.com/javascript/" icon="brand-javascript" arrow />
</CardGroup>

一份好的参考文档应该：
- 描述存在的内容（所有参数、选项、返回值）
- 内容全面且结构清晰，便于查阅
- 作为技术细节的权威来源

<AccordionGroup>
    <Accordion title="为参考文档做贡献">
        [reference.langchain.com](https://reference.langchain.com/python/) 上的生成 API 参考文档是在此仓库之外构建和部署的。要报告该处的错误、缺失的包或损坏的页面，请[提交参考文档问题](https://github.com/langchain-ai/docs/issues/new?template=04-reference-docs.yml)。
    </Accordion>

    <Accordion title="LangChain 参考文档最佳实践">
        - **保持一致**；遵循提供商特定文档的现有模式
        - 包含基本用法（代码片段）和常见边缘情况/失败模式
        - 注明功能所需的特定版本
    </Accordion>

    <Accordion title="何时创建新的参考文档">
        - 新集成或提供商需要专门的参考页面
        - 复杂的配置选项需要详细说明
        - API 变更引入新参数或行为
        - 社区经常询问特定功能的用法
    </Accordion>
</AccordionGroup>

### 教程

教程是较长的分步骤指南，逐层递进，引导用户通过特定的实践活动来建立理解。教程通常位于[学习](/oss/learn)标签页。

<Note>
    我们通常不会合并来自外部贡献者的新教程，除非有迫切需要。如果您认为某个主题在文档中缺失或覆盖不足，请[提交新问题](https://github.com/langchain-ai/docs/issues)。
</Note>

<AccordionGroup>
    <Accordion title="特点">
        - **实践导向**：专注于通过实践活动建立理解。
        - **分步骤**：将活动分解为更小的步骤。
        - **动手实践**：提供顺序执行且可运行的代码片段。
        - **补充性质**：提供功能使用不一定需要的额外上下文和信息。
    </Accordion>

    <Accordion title="技巧">
        - 代码片段应该按顺序排列，如果用户按步骤执行则应能正常运行。
        - 为活动提供一些上下文，但链接到相关的概念指南和参考文档以获取更详细的信息。
    </Accordion>

    <Accordion title="示例">
        - [语义搜索](/oss/langchain/knowledge-base)
        - [RAG 智能体](/oss/langchain/rag)
    </Accordion>
</AccordionGroup>

### 将 Python 和 JavaScript/TypeScript 内容放在一起

所有文档在可能的情况下都必须用 Python 和 JavaScript/TypeScript 两种语言编写。为此，我们使用一种自定义的内联语法来区分应单独或同时出现在不同语言中的部分：

```mdx
\:::python
Python-specific content. In real docs, the preceding backslash (before `python`) is omitted.
\:::

\:::js
JavaScript/TypeScript-specific content. In real docs, the preceding backslash (before `js`) is omitted.
\:::

Content for both languages (not wrapped)
```

这将生成两个输出（每个语言各一个），分别位于 `/oss/python/concepts/foo.mdx` 和 `/oss/javascript/concepts/foo.mdx`。每个输出的页面都需要添加到 `/src/docs.json` 文件中以包含在导航中。

<Note>
    我们不希望缺乏对等性而阻止贡献。如果某个功能仅在一种语言中可用，则可以仅在该语言中提供文档，直到另一种语言跟进。在这种情况下，请包含一个说明，指出该功能在另一种语言中尚不可用。

    如果您在 Python 和 JavaScript/TypeScript 之间翻译内容时需要帮助，请在 [社区 Slack](https://www.langchain.com/join-community) 中提问，或在您的 PR 中标记维护者。
</Note>

## 质量标准

### 通用指南

<AccordionGroup>
    <Accordion title="避免重复">
        多个页面涵盖相同的内容会使维护变得困难并造成混淆。每个概念或功能应该只有一个规范页面。请链接到其他指南，而不是重新解释。
    </Accordion>

    <Accordion title="经常链接">
        文档部分不是孤立存在的。经常链接到其他部分，让用户能够了解不熟悉的主题。这包括链接到 API 参考和概念部分。
    </Accordion>

    <Accordion title="保持简洁">
        采用少即是多的方法。如果另一个部分已有良好的解释，请链接到它而不是重新解释，除非您的内容提出了新的角度。
    </Accordion>
</AccordionGroup>

### 可访问性要求

确保文档对所有用户都可访问：

- 使用标题和列表构建内容，以便于快速浏览
- 使用具体、可操作的链接文本，而不是“点击这里”
- 为所有图像和图表包含描述性的 alt 文本

### 交叉引用

使用一致的交叉引用来连接文档和 API 参考文档。

**从文档到 API 参考：**

使用 `@[]` 语法链接到 API 参考页面：

```mdx
See \@[`ChatAnthropic`] for all configuration options.

The \@[`bind_tools`][ChatAnthropic.bind_tools] method accepts...
```

构建管道会根据当前的语言范围（Python 或 JavaScript）将这些转换为正确的 Markdown 链接。例如，`\@[ChatAnthropic]` 会变成指向 Python 或 JS API 参考页面的链接，具体取决于正在构建的文档版本，**但前提是 `link_map.py` 文件中存在相应的条目！** 详见下文。

<Accordion title="自动链接的工作原理">

`@[]` 语法由 [`handle_auto_links.py`](https://github.com/langchain-ai/docs/blob/main/pipeline/preprocessors/handle_auto_links.py) 处理。它会在 [`link_map.py`](https://github.com/langchain-ai/docs/blob/main/pipeline/preprocessors/link_map.py) 中查找链接键，其中包含 Python 和 JavaScript 范围的字典映射。

**支持的格式：**

| 语法 | 结果 |
|--------|--------|
| `\@[ChatAnthropic]` | 显示文本为 "ChatAnthropic" 的链接 |
| `` \@[`ChatAnthropic`] `` | 显示文本为 `` `ChatAnthropic` `` (代码格式) 的链接 |
| `\@[text][ChatAnthropic]` | 显示文本为 "text"，链接键为 `ChatAnthropic` 的链接 |
| `\\@[ChatAnthropic]` | 转义：渲染为字面量 `\@[ChatAnthropic]`（无链接——本页使用的就是这个！） |

**添加新链接：**

如果链接在映射中未找到，它将保持不变输出到最终结果中。添加新的自动链接：

1. 打开 `pipeline/preprocessors/link_map.py`
2. 在 `LINK_MAPS` 中向相应的范围（`python` 或 `js`）添加条目
3. 键是 `\@[key]` 或 `\@[text][key]` 中使用的链接名称，值是相对于参考主机的路径

</Accordion>

:::python
**从 API 参考存根到开源文档：**

发布的 Python API 参考中的跨链接和深度锚点是在此仓库外部生成的。如果从 reference.langchain.com 到 docs.langchain.com 的链接有误或已过时，请[提交问题](https://github.com/langchain-ai/docs/issues/new?template=04-reference-docs.yml)，并附上源 URL 和目标 URL。
:::

### 本地化

当某个功能在两个 SDK 中都存在时，请将 [Python 和 JavaScript/TypeScript 的文档放在一起](#co-locate-python-and-javascript%2Ftypescript-content)。如果目前仅支持一种语言，请确保该功能及其引用仅对该语言可见。

### 代码内文档

示例必须正确、在可行的情况下可以复制粘贴使用，并且在提交 pull request 之前经过**测试**。请清楚地标记不可运行的代码片段（例如伪代码或说明性片段）。
