多代理系统协调专门的组件来处理复杂的工作流程。但是，并非每个复杂任务都需要这种方法——一个配备正确工具（有时是动态工具）和提示的单代理通常可以实现类似的结果。

<Tip>
要使用内置的多代理支持，请使用 [Deep Agents](/oss/deepagents/overview)：这是一个基于 LangChain 构建的高级框架，包含[子代理](/oss/deepagents/subagents)、[技能](/oss/deepagents/skills)、规划、虚拟文件系统和上下文管理。
</Tip>

## 为什么需要多代理？

当开发者说他们需要"多代理"时，他们通常是在寻求以下一种或多种能力：

- <Icon icon="brain" /> **上下文管理**：提供专门知识而不会使模型的上下文窗口过载。如果上下文是无限的且延迟为零，你可以将所有知识放入单个提示中——但由于事实并非如此，你需要选择性呈现相关信息的模式。
- <Icon icon="users" /> **分布式开发**：允许不同团队独立开发和维护能力，将其组合成具有清晰边界的大型系统。
- <Icon icon="git-branch" /> **并行化**：为子任务生成专门的工作线程并并发执行以获得更快的结果。

当单个代理拥有太多[工具](/oss/langchain/tools)而做出糟糕的使用决策时、当任务需要具有广泛上下文（长提示和领域特定工具）的专门知识时，或者当你需要强制执行顺序约束以在满足某些条件后解锁能力时，多代理模式特别有价值。

<Tip>
多代理设计的核心是**[上下文工程](/oss/langchain/context-engineering)**——决定每个代理看到什么信息。你的系统质量取决于确保每个代理能够访问适合其任务的正确数据。
</Tip>

## 模式

以下是构建多代理系统的主要模式，每个模式适用于不同的用例：

| 模式 | 工作原理 |
|--------------|--------------|
| [**子代理**](/oss/langchain/multi-agent/subagents) | 主代理将子代理作为工具进行协调。所有路由都通过主代理，它决定何时以及如何调用每个子代理。 |
| [**交接**](/oss/langchain/multi-agent/handoffs) | 行为根据状态动态变化。工具调用更新状态变量，触发路由或配置更改，切换代理或调整当前代理的工具和提示。 |
| [**技能**](/oss/langchain/multi-agent/skills) | 按需加载专门提示和知识。单个代理保持控制权，同时根据需要从技能加载上下文。 |
| [**路由器**](/oss/langchain/multi-agent/router) | 路由步骤对输入进行分类并将其定向到一个或多个专门代理。结果被合成为组合响应。 |
| [**自定义工作流**](/oss/langchain/multi-agent/custom-workflow) | 使用 [LangGraph](/oss/langgraph/overview) 构建定制的执行流程，混合确定性逻辑和代理行为。将其他模式作为工作流中的节点嵌入。 |

### 选择模式

使用此表将你的需求与正确的模式进行匹配：

<div className="compact-first-col">

| 模式 | 分布式开发 | 并行化 | 多跳 | 直接用户交互 |
|---------|:-----------------------:|:---------------:|:----------:|:-----------------------:|
| [**子代理**](/oss/langchain/multi-agent/subagents) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐ |
| [**交接**](/oss/langchain/multi-agent/handoffs) | - | - | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| [**技能**](/oss/langchain/multi-agent/skills) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| [**路由器**](/oss/langchain/multi-agent/router) | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | - | ⭐⭐⭐ |

</div>

- **分布式开发**：不同团队能否独立维护组件？
- **并行化**：多个代理能否并发执行？
- **多跳**：该模式是否支持串行调用多个子代理？
- **直接用户交互**：子代理能否直接与用户对话？

<Tip>
    你可以混合模式！例如，**子代理**架构可以调用工具来调用自定义工作流或路由器代理。子代理甚至可以使用**技能**模式按需加载上下文。可能性是无限的！
</Tip>

## 性能比较

不同模式具有不同的性能特征。了解这些权衡有助于你根据延迟和成本需求选择正确的模式。

**关键指标：**
- **模型调用**：LLM 调用次数。调用越多 = 延迟越高（特别是串行时）且每次请求的 API 成本越高。
- **处理的 Token**：所有调用的总[上下文窗口](/oss/langchain/context-engineering)使用量。Token 越多 = 处理成本越高且可能达到上下文限制。

### 重复请求

> **第 1 轮：** "买咖啡"
> **第 2 轮：** "再买一杯咖啡"

用户在同一个对话中重复相同的请求。

<div className="compact-first-col">

| 模式 | 第 2 轮调用 | 总计（两轮） | 最佳匹配 |
|---------|:------------:|:------------------:|:--------:|
| [**子代理**](/oss/langchain/multi-agent/subagents) | 4 | 8 | |
| [**交接**](/oss/langchain/multi-agent/handoffs) | 2 | 5 | ✅ |
| [**技能**](/oss/langchain/multi-agent/skills) | 2 | 5 | ✅ |
| [**路由器**](/oss/langchain/multi-agent/router) | 3 | 6 | |

</div>

<Tabs>
  <Tab title="子代理">
    **再次 4 次调用 → 共 8 次**

    - 子代理**按设计是无状态的**——每次调用遵循相同的流程
    - 主代理维护对话上下文，但子代理每次都重新开始
    - 这提供了强大的上下文隔离，但会重复整个流程
  </Tab>

  <Tab title="交接">
    **2 次调用 → 共 5 次**

    - 咖啡代理**从第 1 轮开始仍然处于活动状态**（状态持久化）
    - 无需交接——代理直接调用 `buy_coffee` 工具（调用 1）
    - 代理响应用户（调用 2）
    - **通过跳过交接节省 1 次调用**
  </Tab>

  <Tab title="技能">
    **2 次调用 → 共 5 次**

    - 技能上下文**已在对话历史中加载**
    - 无需重新加载——代理直接调用 `buy_coffee` 工具（调用 1）
    - 代理响应用户（调用 2）
    - **通过重用已加载的技能节省 1 次调用**
  </Tab>

  <Tab title="路由器">
    **再次 3 次调用 → 共 6 次**

    - 路由器**是无状态的**——每次请求都需要 LLM 路由调用
    - 第 2 轮：路由器 LLM 调用 (1) → 牛奶代理调用 buy_coffee (2) → 牛奶代理响应 (3)
    - 可以通过包装为有状态代理的工具来优化
  </Tab>
</Tabs>

**关键洞察：** 有状态模式（交接、技能）在重复请求上节省 40-50% 的调用。子代理保持每次请求的一致成本——这种无状态设计提供了强大的上下文隔离，但代价是重复的模型调用。

### 多领域

> **用户：** "比较 Python、JavaScript 和 Rust 在 Web 开发中的优劣"

每个语言代理/技能包含约 2000 个 token 的文档。所有模式都可以进行并行工具调用。

| 模式 | 模型调用 | 总 Token | 最佳匹配 |
|---------|:-----------:|:------------:|:--------:|
| [**子代理**](/oss/langchain/multi-agent/subagents) | 5 | ~9K | ✅ |
| [**交接**](/oss/langchain/multi-agent/handoffs) | 7+ | ~14K+ | |
| [**技能**](/oss/langchain/multi-agent/skills) | 3 | ~15K | |
| [**路由器**](/oss/langchain/multi-agent/router) | 5 | ~9K | ✅ |

<Tabs>
  <Tab title="子代理">
    **5 次调用，约 9K Token**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-subagents.png" alt="子代理多领域：并行执行的 5 次调用" />
    </Frame>

    每个子代理在**隔离状态**下工作，只包含其相关上下文。总计：**9K Token**。
  </Tab>

  <Tab title="交接">
    **7+ 次调用，约 14K+ Token**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-handoffs.png" alt="交接多领域：7+ 次串行调用" />
    </Frame>

    交接**串行执行**——无法并行研究三种语言。不断增长的对话历史增加了开销。总计：**约 14K+ Token**。
  </Tab>

  <Tab title="技能">
    **3 次调用，约 15K Token**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-skills.png" alt="技能多领域：累积上下文的 3 次调用" />
    </Frame>

    加载后，**每次后续调用都会处理所有 6K Token 的技能文档**。子代理由于上下文隔离整体处理了 67% 更少的 Token。总计：**15K Token**。
  </Tab>

  <Tab title="路由器">
    **5 次调用，约 9K Token**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-router.png" alt="路由器多领域：并行执行的 5 次调用" />
    </Frame>

    路由器使用 **LLM 进行路由**，然后并行调用代理。类似于子代理但有明确的路由步骤。总计：**9K Token**。
  </Tab>
</Tabs>

**关键洞察：** 对于多领域任务，具有并行执行能力的模式（子代理、路由器）效率最高。技能调用次数较少，但由于上下文累积导致 Token 使用量高。交接在这里效率低下——必须串行执行，无法利用并行工具调用来同时咨询多个领域。
