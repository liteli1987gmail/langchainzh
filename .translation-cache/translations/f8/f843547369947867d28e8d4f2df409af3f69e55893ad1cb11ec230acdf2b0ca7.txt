多智能体系统协调专门的组件来处理复杂的工作流程。然而，并非每个复杂任务都需要这种方案——一个配备了合适工具（有时是动态工具）和提示的单一智能体通常也能达到类似的效果。

<Tip>
如需使用内置的多智能体支持，请使用 [Deep Agents](/oss/deepagents/overview)：一个基于 LangChain 构建的高级框架，内置 [子智能体](/oss/deepagents/subagents), [技能](/oss/deepagents/skills)、规划、虚拟文件系统及上下文管理功能。
</Tip>

## 为什么需要多智能体？

当开发者说需要"多智能体"时，他们通常是在寻求以下一项或多项能力：

- <Icon icon="brain" /> **上下文管理**：提供专门知识而不至于撑爆模型的上下文窗口。如果上下文是无限的且延迟为零，你可以将所有知识一股脑塞进一个提示——但事实并非如此，因此你需要通过特定模式来选择性地呈现相关信息。
- <Icon icon="users" /> **分布式开发**：让不同团队能够独立开发和维护各自的能力，并通过明确的边界将它们组合成一个更大的系统。
- <Icon icon="git-branch" /> **并行化**：为子任务生成专门的执行者并并发运行它们，以获得更快的成果。

当单一智能体拥有过多 [工具](/oss/langchain/tools) 导致决策质量下降时，当任务需要专门知识配合大量上下文（长提示和领域特定工具）时，或者当你需要强制执行顺序约束以在满足特定条件后才能解锁某些能力时，多智能体模式尤其有价值。

<Tip>
多智能体设计的核心是 **[上下文工程](/oss/langchain/context-engineering)**——即决定每个智能体能看到什么信息。你的系统质量取决于能否确保每个智能体都能获取到与其任务相关的正确数据。
</Tip>

## 模式

以下是构建多智能体系统的主要模式，每种模式适用于不同的用例：

| 模式 | 工作原理 |
|--------------|--------------|
| [**子智能体**](/oss/langchain/multi-agent/subagents) | 一个主智能体将子智能体作为工具进行协调。所有路由都通过主智能体，它决定何时以及如何调用每个子智能体。 |
| [**交接**](/oss/langchain/multi-agent/handoffs) | 行为根据状态动态变化。工具调用会更新一个状态变量，触发路由或配置变更，从而切换智能体或调整当前智能体的工具和提示。 |
| [**技能**](/oss/langchain/multi-agent/skills) | 按需加载的专门提示和知识。一个单一智能体保持控制权，同时根据需要从技能中加载上下文。 |
| [**路由器**](/oss/langchain/multi-agent/router) | 一个路由步骤对输入进行分类，并将其导向一个或多个专门的智能体。结果被综合成一个组合响应。 |
| [**自定义工作流**](/oss/langchain/multi-agent/custom-workflow) | 使用 [LangGraph](/oss/langgraph/overview)构建定制的执行流程，融合确定性逻辑和智能体行为。将其他模式作为工作流中的节点嵌入。 |

### 选择合适的模式

使用下表将你的需求与合适的模式进行匹配：

<div className="compact-first-col">

| 模式 | 分布式开发 | 并行化 | 多轮交互 | 直接用户交互 |
|---------|:-----------------------:|:---------------:|:----------:|:-----------------------:|
| [**子代理**](/oss/langchain/multi-agent/subagents) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐ |
| [**交接**](/oss/langchain/multi-agent/handoffs) | - | - | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| [**技能**](/oss/langchain/multi-agent/skills) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| [**路由器**](/oss/langchain/multi-agent/router) | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | - | ⭐⭐⭐ |

</div>

- **分布式开发**：不同的团队能否独立维护组件？
- **并行化**：多个代理能否并发执行？
- **Multi-hop**：该模式是否支持串行调用多个子代理？
- **直接用户交互**：子代理能否直接与用户对话？

<Tip>
    您可以混合使用模式！例如， **子代理** 架构可以调用工具来调用自定义工作流或路由器代理。子代理甚至可以使用 **技能** 模式按需加载上下文。可能性是无限的！
</Tip>

## 性能比较

不同的模式具有不同的性能特征。了解这些权衡有助于您为延迟和成本需求选择合适的模式。

**关键指标：**
- **模型调用**：LLM 调用次数。调用越多 = 延迟越高（特别是串行情况下）和每次请求的 API 成本越高。
- **处理的 Token 数**：所有调用的总 [上下文窗口](/oss/langchain/context-engineering) 使用量。Token 越多 = 处理成本越高和潜在的上下文限制。

### 重复请求

> **回合 1：** “买咖啡”
> **回合 2：** “再次买咖啡”

用户在同一个对话中重复相同的请求。

<div className="compact-first-col">

| 模式 | 回合 2 调用次数 | 总计（两个回合） | 最佳适用场景 |
|---------|:------------:|:------------------:|:--------:|
| [**Subagents**](/oss/langchain/multi-agent/subagents) | 4 | 8 | |
| [**Handoffs**](/oss/langchain/multi-agent/handoffs) | 2 | 5 | ✅ |
| [**Skills**](/oss/langchain/multi-agent/skills) | 2 | 5 | ✅ |
| [**Router**](/oss/langchain/multi-agent/router) | 3 | 6 | |

</div>

<Tabs>
  <Tab title="Subagents">
    **再次 4 次调用 → 总计 8 次**

    - Subagents 的设计 **本质上是无状态的**——每次调用都遵循相同的流程
    - 主代理维护对话上下文，但子代理每次都重新开始
    - 这提供了强大的上下文隔离，但会重复完整的流程
  </Tab>

  <Tab title="Handoffs">
    **2 次调用 → 总计 5 次**

    - 咖啡代理 **仍处于活动状态** 从回合 1 开始（状态持续）
    - 无需交接——代理直接调用 `buy_coffee` 工具（第 1 次调用）
    - 代理向用户做出响应（第 2 次调用）
    - **通过跳过交接节省 1 次调用**
  </Tab>

  <Tab title="Skills">
    **2 次调用 → 总计 5 次**

    - 技能上下文 **已加载** 在对话历史中
    - 无需重新加载——代理直接调用 `buy_coffee` 工具（第 1 次调用）
    - 代理向用户做出响应（第 2 次调用）
    - **通过复用已加载的技能节省 1 次调用**
  </Tab>

  <Tab title="Router">
    **再次 3 次调用 → 总计 6 次**

    - Router 是 **无状态的**——每个请求都需要一次 LLM 路由调用
    - 回合 2：Router LLM 调用（1）→ 牛奶代理调用购买_咖啡（2）→ 牛奶代理做出响应（3）
    - 可以通过在有状态代理中封装为工具来优化
  </Tab>
</Tabs>

**关键洞察：** 有状态模式（Handoffs、Skills）在重复请求上节省 40-50% 的调用。Subagents 保持每次请求的一致成本——这种无状态设计提供了强大的上下文隔离，但代价是重复的模型调用。

### Multi-domain

> **User:** “对比 Python、JavaScript 和 Rust 在 Web 开发中的表现”

Each language agent/skill contains ~2000 tokens of documentation. All patterns can make parallel tool calls.

| 模式 | 模型调用次数 | 总 token 数 | 最佳适用场景 |
|---------|:-----------:|:------------:|:--------:|
| [**Subagents**](/oss/langchain/multi-agent/subagents) | 5 | ~9K | ✅ |
| [**Handoffs**](/oss/langchain/multi-agent/handoffs) | 7+ | ~14K+ | |
| [**Skills**](/oss/langchain/multi-agent/skills) | 3 | ~15K | |
| [**Router**](/oss/langchain/multi-agent/router) | 5 | ~9K | ✅ |

<Tabs>
  <Tab title="Subagents">
    **5 次调用，约 9K token**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-subagents.png" alt="Subagents multi-domain: 5 calls with parallel execution" />
    </Frame>

    每个子代理在 **隔离环境中运行** 仅使用其相关上下文。总计： **9K tokens**.
  </Tab>

  <Tab title="Handoffs">
    **7+ 调用，约 14K+ tokens**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-handoffs.png" alt="Handoffs multi-domain: 7+ sequential calls" />
    </Frame>

    Handoffs 执行 **顺序执行**—无法并行研究三种语言。不断增长的对话历史带来开销。总计： **约 14K+ tokens**.
  </Tab>

  <Tab title="Skills">
    **3 次调用，约 15K tokens**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-skills.png" alt="Skills multi-domain: 3 calls with accumulated context" />
    </Frame>

    加载之后， **每次后续调用都会处理全部 6K token 的技能文档**。子代理因上下文隔离而减少处理 67% 的 token。总计： **15K tokens**.
  </Tab>

  <Tab title="Router">
    **5 次调用，约 9K tokens**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-router.png" alt="Router multi-domain: 5 calls with parallel execution" />
    </Frame>

    路由器使用 **LLM 进行路由**，然后并行调用代理。类似于子代理模式，但有明确的路由步骤。总计： **9K tokens**.
  </Tab>
</Tabs>

**关键洞察：** 对于多领域任务，具有并行执行能力的模式（子代理、路由器）效率最高。技能模式调用次数较少，但因上下文累积导致 token 使用量高。Handoffs 模式效率较低——必须顺序执行，无法利用并行工具调用同时咨询多个领域。

### 总结

以下是各模式在三种场景下的对比：
