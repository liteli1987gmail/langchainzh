<Note>
**自托管版本要求**：访问告警需要 Helm chart 版本 **0.10.3** 或更高版本。
</Note>

LLM 应用的有效可观测性需要主动检测故障、性能下降和回归。LangSmith 的告警功能有助于识别以下关键问题：

- 来自模型提供商的 API 速率限制违规。
- 应用延迟增加。
- 影响反馈分数的应用变更，反映最终用户体验。
- LLM 使用产生的意外成本激增。

LangSmith 中的告警按项目范围划分，需要为每个受监控的项目单独配置。

<Tip>
告警可以 [路由到](#step-4-configure-notification-channel) PagerDuty、Dynatrace 或通过 webhook 路由到任何 HTTP 端点。 **Webhook** 选项卡包含 [示例配方](#example-recipes) ，用于向 Slack、Microsoft Teams 和电子邮件发送告警。
</Tip>

按照以下步骤配置告警。

## 步骤 1：导航到创建告警

在 [UI](https://smith.langchain.com)中，导航到要配置告警的 Tracing 项目。点击页面右上角的 **告警** 图标，查看该项目的现有告警并设置新告警。

## 步骤 2：选择指标类型

LangSmith 提供以下指标的阈值告警：

| 指标类型 | 描述 | 使用场景 |
| ------------------ | ----------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **运行计数**      | 跟踪时间窗口内的 [运行](/langsmith/observability-concepts#runs) 总数。| 监控管道是否以预期量产生运行，并在意外下降时告警。|
| **成本**           | 跟踪时间窗口内的运行总成本。| 监控 LLM 支出，在成本超过预期阈值时告警。需要配置 [成本跟踪](/langsmith/cost-tracking) 。|
| **错误**   | 跟踪具有错误状态的运行。按错误总数或错误百分比（错误运行占总运行的比例）告警。| 监控应用故障，或在错误率超过可接受阈值时告警。|
| **反馈分数** | 测量平均反馈分数。| 跟踪 [最终用户反馈](/langsmith/attach-user-feedback) or [和](/langsmith/online-evaluations-llm-as-judge) 以告警回归问题。|
| **延迟**        | 测量平均运行执行时间。| 跟踪应用延迟以告警峰值和性能瓶颈。|

此外，对于 **错误** 和 **延迟**，您可以使用筛选构建器对以下字段堆叠条件，例如 **状态**, **运行类型**, **标签**，以及 **错误**。例如，您可以将错误告警限定为符合条件的运行 **状态** is `error`, **运行类型** is `llm`, **标签** is `support_agent`，以及 **错误** 匹配 `RateLimitExceeded`.

## 步骤 3：定义告警条件

告警条件由以下几个部分组成：

- **聚合方法**：平均值、百分比或计数。
- **比较运算符**: `>=`, `<=`，或超过阈值。
- **阈值**：触发告警的数值。
- **聚合窗口**：指标计算的时间段（选择 5 分钟或 15 分钟）。
- **反馈键** （仅限反馈分数告警）：要监控的特定反馈指标。

## 步骤 4：配置通知渠道

<Tip>
 **Webhook** 选项卡包含 [示例配置](#example-recipes) ，用于向 Slack、Microsoft Teams 和电子邮件发送告警。
</Tip>

<Tabs>
  <Tab title="PagerDuty">
    使用 PagerDuty 的 [Events API v2](https://developer.pagerduty.com/docs/events-api-v2-overview)将 PagerDuty 配置为通知渠道。此集成允许关键的 LLM 应用问题触发 PagerDuty 事件，从而通过您的事件管理流程实现快速响应。

    **前提条件**

    - 拥有管理员访问权限的活跃 PagerDuty 账户
    - PagerDuty 中适当的服务级权限

    如果使用 LangSmith 的自定义部署，请确保没有防火墙设置阻止 LangSmith 服务的出口流量。

    ### 1. 在 PagerDuty 中创建服务

    1. 登录您的 PagerDuty 账户
    2. 导航至 **服务 → 服务目录**
    3. 点击 **+ 新建服务**
    4. 填写以下字段：
       - **名称**：提供描述性名称（例如"LangSmith 监控"）
       - **描述**：添加有关被监控应用程序的详细信息
       - **升级策略**：选择适当的团队升级策略
       - **集成类型**：选择"Events API V2"
    5. 点击 **添加服务** 创建服务

    ### 2. 获取集成密钥

    创建服务后，获取集成密钥：

    1. 从 **服务目录**中，找到并点击您新创建的服务
    2. 选择 **集成** 选项卡
    3. 找到"Events API V2"集成
    4. 复制 **集成密钥** （一个32个字符的字母数字字符串）

    ![PagerDuty 集成密钥位置](/langsmith/images/pager-duty.png)

    ### 3. 配置 LangSmith 警报与 PagerDuty

    <Info>
    要在警报触发后一小时内再次接收相同警报，您必须在 PagerDuty 中解决该警报创建的活动事件。
    </Info>

    ![PagerDuty 设置](/langsmith/images/pagerduty-setup.png)

    1. 在 LangSmith 中设置警报的通知部分，选择 **PagerDuty**
    2. 点击密钥图标将集成密钥保存为工作区密钥，或选择现有的工作区密钥。作为最佳实践，我们建议将集成密钥保存为工作区密钥，而不是直接添加。这将允许您在工作区的多个警报中重复使用同一个密钥。
    3. 配置其他通知选项：
       - **严重程度**：映射到 PagerDuty 事件优先级
    4. 点击发送测试警报 **发送测试警报**
    5. 验证 PagerDuty 是否触发了事件，并包含相关的 LangSmith 警报信息

    ### 故障排除

    如果 PagerDuty 中未创建事件：

    - 验证 LangSmith 中输入的集成密钥是否正确
    - 确保 PagerDuty 服务处于活动状态且不在维护模式
    - 检查您的 PagerDuty 账户是否已启用 Events API v2
    - 如果 PagerDuty 中缺少警报触发器，请检查预期触发器是否发生在同一警报规则上次触发后一小时内，以及上次警报创建的事件是否仍处于打开状态。
    - 如果 LangSmith 实例位于防火墙后面，请检查网络连接

    ### 其他资源

    - [PagerDuty Events API v2 文档](https://developer.pagerduty.com/docs/events-api-v2/overview/)
    - [PagerDuty 集成指南](https://support.pagerduty.com/docs/services-and-integrations)
  </Tab>
  <Tab title="Dynatrace">
    使用 Dynatrace 的配置 Dynatrace 作为通知通道 [Events API v2](https://docs.dynatrace.com/docs/dynatrace-api/environment-api/events-v2/post-event)此集成将 LangSmith 告警事件发送到您的 Dynatrace 环境，从而能够与您更广泛的基础设施监控进行关联。

    **前提条件**

    - 一个活跃的 Dynatrace 环境（SaaS 或托管版本）。
    - 一个具有 `events.ingest` scope.

    如果您使用的是自定义 [部署](/langsmith/self-hosted) 的 LangSmith，请确保没有防火墙设置阻止 LangSmith 服务的出站流量。

    ### 1. 在 Dynatrace 中创建 API 令牌

    1. 登录到您的 Dynatrace 环境。
    1. 导航至 **访问令牌**.
    1. 点击 **生成新令牌**.
    1. 提供一个描述性名称（例如 "LangSmith 告警"）。
    1. 在 **作用域**下，搜索并启用 `events.ingest` （摄取事件）。
    1. 点击 **生成令牌**.
    1. 复制生成的令牌并安全存储。该令牌仅显示一次。

    ### 2. 获取您的 Dynatrace 环境 URL

    您的 Dynatrace 环境 URL 遵循以下格式：

    ```
    https://{your-environment-id}.live.dynatrace.com
    ```

    您可以在登录 Dynatrace 时的浏览器 URL 栏中找到您的环境 ID。

    ### 3. 配置 LangSmith 告警与 Dynatrace 的集成

    1. 在 **通知设置** 中，为您的 LangSmith 告警设置选择 **Dynatrace**.
    1. 输入您的 Dynatrace 环境 URL。
    1. 点击密钥图标将 API 令牌保存为工作区密钥，或选择现有的工作区密钥。作为最佳实践，建议将 API 令牌保存为工作区密钥，而不是直接添加。这允许您在工作区的多个告警中重复使用同一个令牌。
    1. 配置其他通知选项：
       - **事件类型**：选择 Dynatrace 事件类型（例如 `CUSTOM_ALERT`, `ERROR_EVENT`)
    1. 通过点击发送测试告警 **发送测试通知**.
    1. 验证事件是否出现在您的 Dynatrace 环境中。

    ### 故障排除

    如果事件未出现在 Dynatrace 中：

    - 验证 API 令牌具有 `events.ingest` 作用域且未过期。
    - 确保环境 URL 正确且包含您的环境 ID。
    - 确认 `Authorization` 标头格式使用 `Api-Token` （而非 `Bearer`).
    - 检查您的 Dynatrace 环境是否处于活跃状态且可访问。
    - 如果您的 LangSmith 实例位于防火墙后，请检查网络连接。

    ### 其他资源

    - [Dynatrace Events API v2 文档](https://docs.dynatrace.com/docs/dynatrace-api/environment-api/events-v2/post-event)
    - [Dynatrace 访问令牌](https://docs.dynatrace.com/docs/manage/access-control/access-tokens)
  </Tab>
  <Tab title="Webhook">
    Webhook 通过在触发告警条件时发送 HTTP POST 请求，实现与自定义服务和第三方平台的集成。使用 webhook 可将告警数据转发到工单系统、聊天应用程序或自定义监控解决方案。

    **前提条件**

    - 一个可以接收 HTTP POST 请求的端点
    - 接收服务的适当身份验证凭据（如果需要）

    ### 1. 准备您的接收端点

    在 LangSmith 中配置 webhook 之前，请确保您的接收端点：

    - 接受 HTTP POST 请求
    - 可以处理 JSON 负载
    - 可从外部服务访问
    - 具有适当的身份验证机制（如果需要）

    如果使用 LangSmith 的自定义部署，请确保没有防火墙设置阻止 LangSmith 服务的出口流量。

    ### 2. 配置 webhook 参数

    在 **监控** 部分， [LangSmith UI](https://smith.langchain.com) 的 **告警** 标签页中，点击 **+ 告警** 来创建一个新告警。

    在 **通知设置** 部分，使用以下参数完成 webhook 配置：

    **必填字段**

    - **URL**：接收端点的完整 URL
      - Example: `https://api.example.com/incident-webhook`

    **可选字段**

    - **请求头**：随 webhook 请求发送的 JSON 键值对
      - 常见请求头包括：
        - `Authorization`：用于身份验证令牌
        - `Content-Type`：通常设置为 `application/json` （默认）
        - `X-Source`：将来源标识为 LangSmith
      - 如果没有请求头，请使用 `{}`

    - **请求体模板**：自定义发送到您端点的 JSON 负载
      - 默认值：LangSmith 发送定义的负载，并以下列额外的键值对附加到负载中：
        - `project_name`：触发的告警名称
        - `alert_rule_id`：用于标识 LangSmith 告警的 UUID。可用作 webhook 服务中的去重键。
        - `alert_rule_name`：告警规则的名称。
        - `alert_rule_type`: The type of alert (as of 04/01/2025 all alerts are of type `threshold`).
        - `alert_rule_attribute`：与告警规则关联的属性 - `error_count`, `feedback_score`, `latency`, or `cost`.
        - `triggered_metric_value`：触发阈值时指标的值。
        - `triggered_threshold`：触发告警的阈值。
        - `timestamp`：触发告警的时间戳。

    <Info>
    LangSmith 不会对请求体执行模板替换。上述自动填充的字段作为顶级键合并到传出 JSON 中，与您配置的主体并列。占位符语法如 `{alert_rule_name}` 原样发送到接收服务。只有当接收方本身可以从传入的 JSON 中提取字段时（例如 Power Automate Workflow、AWS Lambda 或自定义 HTTP 处理程序），它才会解析为实际值。
    </Info>

    ### 3. 测试 Webhook

    点击 **发送测试警报** 以发送 webhook 通知，确保通知按预期工作。

    ### 故障排除

    如果 webhook 通知未送达：

    - 验证 webhook URL 是否正确且可访问
    - 确保所有身份验证标头格式正确
    - 检查您的接收端点是否接受 POST 请求
    - 检查端点日志中收到但被拒绝的请求
    - 验证您的自定义负载模板是有效的 JSON 格式

    ### 安全注意事项

    - 为您的 webhook 端点使用 HTTPS
    - 为您的 webhook 端点实施身份验证
    - 考虑在标头中添加共享密钥以验证 webhook 来源
    - 在处理传入的 webhook 请求之前先验证它们

    ### 示例配方

    <Accordion title="Configure Slack notifications via webhook">
      以下是一个配置 LangSmith 警报以使用 [`chat.postMessage`](https://api.slack.com/methods/chat.postMessage) API.

      **前提条件**

      - 访问 Slack 工作区。
      - 一个用于设置警报的 LangSmith 项目。
      - 创建 Slack 应用程序的权限。

      **第 1 步：创建 Slack 应用**

      1. 访问 [Slack API Applications 页面](https://api.slack.com/apps).
      2. 点击 **Create New App**.
      3. 选择 **From scratch**.
      4. 提供一个 **App Name** （例如 "LangSmith Alerts"）。
      5. 选择要安装该应用的工作区。
      6. 点击 **Create App**.

      **第 2 步：配置机器人权限**

      1. 在 Slack 应用配置的左侧边栏中，点击 **OAuth & Permissions**.
      2. 向下滚动到 **Bot Token Scopes** 在 **Scopes** 下，点击 **Add an OAuth Scope**.
      3. 添加以下作用域：
         - `chat:write` （以应用身份发送消息）。
         - `chat:write.public` （向应用不在的频道发送消息）。
         - `channels:read` （查看基本频道信息）。

      **第 3 步：将应用安装到您的工作区**

      1. 向上滚动到 **OAuth & Permissions** page.
      2. 点击 **安装到工作区**.
      3. 查看权限并点击 **允许**.
      4. 复制 **机器人用户 OAuth 令牌** 显示的内容（以 `xoxb-`).

      **步骤 4：将机器人添加到 Slack 频道**

      将机器人添加到您想要接收警报的特定频道。您可以通过在消息字段中提及机器人来将其添加到 Slack 频道（例如， `@botname`).

      您还需要频道 ID 才能在 LangSmith 中配置 webhook 警报。您可以通过打开频道详情 > 关于来找到频道 ID。

      **步骤 5：在 LangSmith 中配置 webhook 警报**

      1. 在 LangSmith 中，导航到您的项目。
      2. 选择 **警报 → 创建警报**.
      3. 定义您的警报指标和条件。
      4. 在通知部分，选择 **Webhook**.
      5. 使用以下设置配置 webhook：

      **Webhook URL**

      ```json
      https://slack.com/api/chat.postMessage
      ```

      **请求头**
      <Note>替换 `xoxb-your-token-here` 为您的机器人的用户 OAuth 令牌</Note>

      ```json
      {
        "Content-Type": "application/json",
        "Authorization": "Bearer xoxb-your-token-here"
      }
      ```

      **请求正文模板**
      <Note>必须填写 `{channel_id}` 从第4步中找到的值。 <br /><br />剩余字段： `alert_name`, `project_name` 和 `project_url` （可选）向警报消息添加额外的上下文。您可以在 `project_url` 在浏览器的URL栏中找到。复制该部分，但不包括任何查询参数。</Note>

      ```json
      {
        "channel": "{channel_id}",
        "text": "{alert_name} triggered for {project_name}",
        "blocks": [
          {
            "type": "section",
            "text": {
              "type": "mrkdwn",
              "text": "🚨{alert_name} has been triggered"
            }
          },
          {
            "type": "section",
            "text": {
              "type": "mrkdwn",
              "text": "Please check the following link for more information:"
            }
          },
          {
            "type": "section",
            "text": {
              "type": "mrkdwn",
              "text": "<{project-url}|View in LangSmith>"
            }
          }
        ]
      }
      ```

      6. 点击 **保存** 以激活webhook配置。

      **步骤6：测试集成**

      1. 在LangSmith警报配置中，点击 **测试警报**.
      2. 检查您指定的Slack频道以获取测试通知。
      3. 验证消息包含预期的告警信息。

      **（可选）步骤 7：在请求正文中链接到告警预览**

      创建告警后，您可以选择在其 Webhook 请求正文中链接到告警预览。

      ![告警预览窗格](/langsmith/images/alert-preview-pane.png)

      配置步骤如下：

      1. 保存您的告警。
      2. 在告警表中找到您保存的告警并点击它。
      3. 复制显示的 URL。
      4. 点击“编辑告警”。
      5. 用复制的告警预览 URL 替换现有的项目 URL。
    </Accordion>

    <Accordion title="Configure Microsoft Teams notifications via webhook">
      以下是通过 [工作流应用](https://support.microsoft.com/en-us/office/create-incoming-webhooks-with-workflows-for-microsoft-teams-8ae491c7-0394-4861-ba59-055e33f75498) （Power Automate）配置 LangSmith 告警以向 Microsoft Teams 频道发送通知的示例。推荐使用此方法，因为它可以从流程内的传入 JSON 中提取字段，因此自动填充的 LangSmith 告警字段可以在 Teams 消息中正确呈现。

      <Note>
      Microsoft 的旧版 Office 365 传入 Webhook 连接器即将停用。请使用 Workflows 应用进行新的集成。
      </Note>

      **前置要求**

      - 拥有 Microsoft Teams 工作区的访问权限，并具有添加 Workflows 的权限。
      - 一个用于设置告警的 LangSmith 项目。

      **步骤 1：在 Teams 中创建 Workflow**

      1. 在 Microsoft Teams 中，导航到您希望接收告警的频道。
      2. 点击 **...** 频道名称旁边的（更多选项）菜单。
      3. 选择 **Workflows**.
      4. 搜索并选择 **收到 webhook 请求时发布到频道** template.
      5. 登录以确认连接，然后点击 **下一步**.
      6. 确认要发布告警的团队和频道，然后点击 **添加工作流**.
      7. 复制生成的 **HTTP POST URL**— 在 LangSmith 中使用此 URL。

      **步骤 2：在 Power Automate 中自定义消息（可选）**

      默认工作流将原始 JSON 正文作为卡片发布。若要格式化告警详情，请在 Power Automate 中编辑流程：

      1. 打开 [Power Automate 门户](https://make.powerautomate.com) 并编辑您创建的工作流。
      2. 点击 **在聊天或频道中发布卡片** action.
      3. 在 **Adaptive Card** 字段中，使用 `triggerOutputs()?['body/alert_rule_name']`, `triggerOutputs()?['body/project_name']`, `triggerOutputs()?['body/triggered_metric_value']`, `triggerOutputs()?['body/triggered_threshold']`, `triggerOutputs()?['body/timestamp']`，以及 `triggerOutputs()?['body/alert_rule_url']`.
      4. 保存流程。

      **步骤 3：在 LangSmith 中配置 webhook 告警**

      1. 在 LangSmith 中，导航到您的项目。
      2. 选择 **告警 → 创建告警**.
      3. 定义您的告警指标和条件。
      4. 在通知部分，选择 **Webhook**.
      5. 使用以下设置配置 webhook：

      **Webhook URL**

      粘贴您的 Teams Workflow 中的 HTTP POST URL：

      ```
      https://prod-XX.westus.logic.azure.com:443/workflows/.../triggers/manual/paths/invoke?...
      ```

      **请求头**

      ```json
      {
        "Content-Type": "application/json"
      }
      ```

      **请求正文模板**

      LangSmith 会自动将自动填充的告警字段（`alert_rule_name`, `project_name`, `triggered_metric_value`, `triggered_threshold`, `timestamp`, `alert_rule_url`等）合并到请求正文中作为顶级 JSON 键。Power Automate 直接从传入的有效负载中读取这些字段，因此空正文即可：

      ```json
      {}
      ```

      6. 点击 **保存** 以激活 webhook 配置。

      **步骤 4：测试集成**

      1. 在 LangSmith 告警配置中，点击 **发送测试告警**.
      2. 检查您指定的 Teams 频道中是否收到测试通知。
      3. 验证卡片包含预期的告警信息。

      **参考实现**

      有关将 LangSmith Webhook 载荷（阈值告警、运行规则和通用事件）转换为格式化 Teams 自适应卡片的工作示例，请参阅 [langsmith-teams-webhook](https://github.com/langchain-samples/langsmith-teams-webhook) 示例仓库。该示例作为 Teams Workflow URL 前端的小型 Python 服务运行，可避免自定义 Power Automate 流程本身。
    </Accordion>

    <Accordion title="Configure email notifications via webhook">
      以下是配置 LangSmith 告警以使用 [SendGrid 的邮件发送 API](https://docs.sendgrid.com/api-reference/mail-send/mail-send)发送电子邮件通知的示例。您可以使用任何提供 HTTP API 的事务性邮件服务商（例如 Mailgun、Amazon SES、Postmark）。

      **前提条件**

      - 拥有已验证发件人身份的 SendGrid 账户。
      - 具有 **邮件发送** permissions.
      - 权限的 SendGrid API 密钥。

      **步骤 1：创建 SendGrid API 密钥**

      1. 登录您的 [SendGrid 控制台](https://app.sendgrid.com).
      2. 导航到 **设置 → API 密钥**.
      3. 点击 **创建 API 密钥**.
      4. 选择 **受限访问** 并启用 **邮件发送 → 完全访问**.
      5. 点击 **创建并查看**，复制密钥，并将其安全存储。

      **步骤 2：验证您的发件人邮箱**

      1. 在 SendGrid 中，导航至 **设置 → 发件人身份验证**.
      2. 完成以下任一操作 **域名身份验证** （推荐）或 **单发件人验证** 针对您要发送的地址。

      **步骤 3：在 LangSmith 中配置 Webhook 警报**

      1. 在 LangSmith 中，导航至您的项目。
      2. 选择 **警报 → 创建警报**.
      3. 定义您的警报指标和条件。
      4. 在通知部分，选择 **Webhook**.
      5. 使用以下设置配置 Webhook：

      **Webhook URL**

      ```
      https://api.sendgrid.com/v3/mail/send
      ```

      **请求头**

      <Note>替换 `SG.your-api-key-here` 为您的 SendGrid API 密钥。</Note>

      ```json
      {
        "Content-Type": "application/json",
        "Authorization": "Bearer SG.your-api-key-here"
      }
      ```

      **请求正文模板**

      <Note>
      替换 `alerts@your-company.com` 为您的已验证发件人地址， `oncall@your-company.com` 为收件人地址。SendGrid 不会从任意顶级 JSON 键中提取字段，因此此示例使用固定的主题和正文。要在电子邮件中包含警报特定值，请通过中间件（如 Power Automate 流、AWS Lambda 或 Zapier webhook）路由 LangSmith Webhook，该中间件读取传入的有效负载并渲染 SendGrid 请求。
      </Note>

      ```json
      {
        "personalizations": [
          {
            "to": [
              {
                "email": "oncall@your-company.com"
              }
            ],
            "subject": "LangSmith alert triggered"
          }
        ],
        "from": {
          "email": "alerts@your-company.com",
          "name": "LangSmith Alerts"
        },
        "content": [
          {
            "type": "text/plain",
            "value": "A LangSmith alert was triggered. Open your LangSmith workspace to view the alert details, including the project, metric value, threshold, and timestamp."
          }
        ]
      }
      ```

      6. 点击 **保存** 以激活 webhook 配置。

      **步骤 4：测试集成**

      1. 在 LangSmith 告警配置中，点击 **发送测试告警**.
      2. 检查收件人的收件箱中是否收到测试通知。
      3. 验证邮件包含预期的告警信息。

      **使用其他邮件服务商**

      相同的模式也适用于其他接受静态身份验证标头的交易邮件 API。更改 **Webhook URL** 和 **标头** 以匹配您的服务商：

      | 服务商 | Webhook URL | 身份验证标头格式 |
      |----------|-------------|--------------------|
      | Mailgun | `https://api.mailgun.net/v3/{your-domain}/messages` | `Authorization: Basic <base64(api:<key>)>` |
      | Postmark | `https://api.postmarkapp.com/email` | `X-Postmark-Server-Token: <token>` |

      调整 **请求正文模板** 以匹配每个服务商预期的负载格式。Amazon SES 不直接兼容，因为 SES API 需要每个请求的 AWS SigV4 签名，无法表示为静态标头。要使用 SES，请通过中间件路由（例如，具有 HTTP 触发器的 Lambda 函数）。
    </Accordion>
