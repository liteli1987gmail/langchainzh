仓库结构

LangChain

LangChain 被组织为一个包含多个包的单体仓库：

核心包

  • - **langchain**（位于 libs/langchain/）：包含 chains、agents 和检索逻辑的主包
  • - **langchain-core**（位于 libs/core/）：基础接口和核心抽象

合作伙伴包

位于 libs/partners/，这些是针对特定集成的独立版本包。例如： - **langchain-openai**：OpenAI 集成 - **langchain-anthropic**：Anthropic 集成 - **langchain-google-genai**：Google Generative AI 集成

许多合作伙伴包位于外部仓库中。请查看集成列表了解详情。

支持包

  • - **langchain-text-splitters**：文本分割工具
  • - **langchain-standard-tests**：集成的标准测试套件

LangGraph

LangGraph 被组织为一个包含多个 Python 包的单体仓库：

核心包

- **langgraph**（位于 libs/langgraph/）：用于构建有状态多角色 agents 的核心框架 - **langgraph-prebuilt**（位于 libs/prebuilt/）：用于创建和运行 agents 和工具的高级 API </section>

Checkpoint 包

- **langgraph-checkpoint**（位于 libs/checkpoint/）：checkpoint saver 的基础接口 - **langgraph-checkpoint-postgres**（位于 libs/checkpoint-postgres/）：Postgres 实现 - **langgraph-checkpoint-sqlite**（位于 libs/checkpoint-sqlite/）：SQLite 实现 </section>

SDK 和 CLI

- **langgraph-sdk**（位于 libs/sdk-py/）：Agent Server API 的 Python SDK - **langgraph-cli**（位于 libs/cli/）：官方命令行界面 </section>

Deep Agents

Deep Agents 被组织为一个包含多个 Python 包的单体仓库：

核心包

- **deepagents**（位于 libs/deepagents/）：用于构建具有规划、文件系统和子代理能力的深度 agents 的核心框架 - **deepagents-code**（位于 libs/code/）：Deep Agents Code — 具有对话恢复、网络搜索和沙箱功能的交互式终端界面 - **deepagents-cli**（位于 libs/cli/）：用于将 agents 部署到 LangSmith Deployments 的部署工具（deepagents deploy、deepagents init、deepagents dev） </section>

集成包

- **deepagents-harbor**（位于 libs/harbor/）：带有 LangSmith 追踪的 Harbor 集成 - **deepagents-acp**（位于 libs/acp/）：Agent Client Protocol 集成 </section>

LangChain 和 Deep Agents 仓库包含 pre-commit hooks，会在每次提交前自动运行格式化、linting 和验证检查。请从仓库根目录安装它们：

运行测试

信息

目录路径是相对于你正在工作的包的。

我们倾向于在可能的情况下优先使用单元测试而不是集成测试。单元测试在每次 pull request 时运行，因此应该快速且可靠。集成测试按计划运行，需要更多设置，因此应该保留用于确认与外部服务的接口点。

单元测试

**位置**: tests/unit_tests/

单元测试涵盖不需要调用外部 API 的模块化逻辑。如果你添加了新逻辑，应该添加一个单元测试。在单元测试中，检查预处理/后处理并模拟外部依赖。

**要求**： - 不允许网络调用 - 测试所有代码路径，包括边界情况 - 使用 mock 来处理外部依赖

运行单元测试：

make test

# Or directly:
uv run --group test pytest tests/unit_tests

# To run a specific test:
TEST_FILE=tests/unit_tests/test_imports.py make test

集成测试

**位置**：tests/integration_tests/

集成测试涵盖了需要调用外部 API 的逻辑（通常是与其他服务的集成）。

集成测试需要访问外部服务/提供商 API（这可能需要付费），因此默认情况下不会运行。

并非每次代码更改都需要集成测试，但请记住，作为审查流程的一部分，我们会单独要求/运行集成测试。

**要求**： - 测试与外部服务的真实集成 - 使用环境变量存储 API 密钥 - 如果凭据不可用，请优雅地跳过

运行集成测试：

make integration_tests

# Or directly:
uv run --group test --group test_integration pytest --retries 3 --retry-delay 1 tests/integration_tests

# To run a specific test:
TEST_FILE=tests/integration_tests/test_openai.py make integration_tests

文档

**必需**：所有公共函数都需要遵循 Google 风格文档字符串。

**指导原则**：文档字符串描述"是什么"；本网站的文档解释"如何做"和"为什么"。

| 内容类型 | 位置 | 用途 | |----------|------|------| | 参数类型 | 函数签名 | 自动生成 API 参考文档 | | 参数描述 | 文档字符串 | 自动生成 API 参考文档 | | 返回类型和异常 | 文档字符串 | API 参考文档 | | 最小使用示例 | 文档字符串 | 展示基本实例化模式 | | 功能教程 | 本网站 | 深入的使用指南 | | 端到端示例 | 本网站 | 真实使用场景 | | 概念解释 | 本网站 | 帮助理解和使用背景 |

**文档字符串应包含：**

1. 类/函数的单行简要说明 2. 本网站教程、指南和使用模式的链接 3. 带类型和描述的参数文档 4. 返回值说明 5. 可能抛出的异常 6. 必要时提供展示基本实例化/使用的最小示例
