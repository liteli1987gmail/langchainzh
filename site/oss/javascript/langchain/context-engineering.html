概述


构建 Agent（或任何 LLM 应用）的难点在于使其足够可靠。虽然它们可能在原型阶段工作正常，但在实际使用场景中往往会失败。

### Agent 为何会失败？

当 Agent 失败时，通常是因为 Agent 内部的 LLM 调用采取了错误的操作/没有做出我们期望的行为。LLM 失败的原因有以下两个：

1. 底层 LLM 能力不足
2. 没有向 LLM 传递"正确"的上下文

大多数情况下——实际上第二个原因才是导致 Agent 不可靠的根本所在。

**上下文工程** 是以正确的格式提供正确的信息和工具，以便 LLM 能够完成任务。这是 AI 工程师的首要工作。这种"正确"上下文的缺失是实现更可靠 Agent 的最大障碍，而 LangChain 的 Agent 抽象层专门设计用于促进上下文工程。

### Agent 循环

一个典型的 Agent 循环包含两个主要步骤：

1. **模型调用** - 使用提示词和可用工具调用 LLM，返回响应或执行工具的请求
2. **工具执行** - 执行 LLM 请求的工具，返回工具结果

### 你可以控制的内容

要构建可靠的 Agent，你需要控制 Agent 循环中每个步骤发生的事情，以及步骤之间发生的事情。

| 上下文类型 | 你控制的内容 | 瞬态还是持久态 |
|--------------|------------------|-------------------------|
| **[模型上下文](#model-context)** | 模型调用的输入内容（指令、消息历史、工具、响应格式） | 瞬态 |
| **[工具上下文](#tool-context)** | 工具可以访问和产生的内容（对状态、存储、运行时上下文的读写） | 持久态 |
| **[生命周期上下文](#life-cycle-context)** | 模型调用和工具调用之间发生的事情（摘要、护栏、日志记录等） | 持久态 |


  
### 瞬态上下文


    LLM 在单次调用中看到的内容。你可以修改消息、工具或提示词，而不会更改状态中保存的内容。
  

  
### 持久态上下文


    跨轮次保存到状态中的内容。生命周期钩子和工具写入会永久修改此上下文。
  



### 数据源

在整个过程中，你的 Agent 会访问（读取/写入）不同的数据源：

| 数据源 | 也称为 | 范围 | 示例 |
|-------------|---------------|-------|----------|
| **运行时上下文** | 静态配置 | 对话级别 | 用户 ID、API 密钥、数据库连接、权限、环境设置 |
| **状态** | 短期记忆 | 对话级别 | 当前消息、上传的文件、认证状态、工具结果 |
| **存储** | 长期记忆 | 跨对话 | 用户偏好、提取的洞察、记忆、历史数据 |

### 工作原理

LangChain [中间件](/oss/javascript/langchain/middleware) 是在底层使开发者使用 LangChain 进行上下文工程变得实用的机制。

中间件允许你钩入 Agent 生命周期的任何步骤并：

* 更新上下文
* 跳转到 Agent 生命周期中的不同步骤

在本指南中，你将频繁看到中间件 API 作为上下文工程手段的使用。


模型上下文


控制每个模型调用的输入内容——指令、可用的工具、使用的模型以及输出格式。这些决策直接影响可靠性和成本。


    
### 系统提示词


        开发者向 LLM 提供的基础指令。
    

    
### 消息


        发送给 LLM 的完整消息列表（对话历史）。
    

    
### 工具


        Agent 可用于执行操作的实用工具。
    

    
### 模型


        实际要调用的模型（包括配置）。
    

    
### 响应格式


        模型最终响应的模式规范。
    



所有这些类型的模型上下文都可以从**状态**（短期记忆）、**存储**（长期记忆）或**运行时上下文**（静态配置）中获取。

### 系统提示词

系统提示词设定了 LLM 的行为和能力。不同的用户、上下文或对话阶段需要不同的指令。成功的 Agent 会利用记忆、偏好和配置来为当前对话状态提供正确的指令。

### 响应格式

结构化输出将非结构化文本转换为经过验证的结构化数据。在提取特定字段或返回供下游系统使用的数据时，自由格式的文本是不够的。

工具上下文


工具的特殊之处在于它们既读取也写入上下文。

在最基本的情况下，当工具执行时，它接收 LLM 的请求参数并返回工具消息。工具完成其工作并产生结果。

工具还可以为模型获取重要信息，使其能够执行和完成任务。

### 读取

大多数实际工具需要的不仅仅是 LLM 的参数。它们需要用于数据库查询的用户 ID、用于外部服务的 API 密钥，或当前的会话状态来做出决策。工具从状态、存储和运行时上下文读取以访问此信息。

生命周期上下文


控制核心代理步骤**之间**发生的事情——拦截数据流以实现跨领域关注点，如摘要生成、护栏和日志记录。

正如您在[模型上下文](#model-context)和[工具上下文](#tool-context)中所见，[中间件](/oss/javascript/langchain/middleware)是使上下文工程变得实用的机制。中间件允许您挂钩到代理生命周期的任何步骤，并可以：

1. **更新上下文** - 修改状态和存储以保留更改、更新对话历史或保存洞察
2. **跳入生命周期** - 根据上下文移动到代理循环中的不同步骤（例如，如果满足条件则跳过工具执行，使用修改后的上下文重复模型调用）

### 示例：摘要生成

最常见的生命周期模式之一是当对话历史过长时自动压缩对话历史。与[模型上下文](#messages)中所示的临时消息修剪不同，摘要生成**持久更新状态**——用摘要永久替换旧消息，该摘要会被保存以供所有未来轮次使用。

LangChain 为此提供了内置中间件：



:::js
```typescript


const agent = createAgent({
  model: "gpt-5.5",
  tools: [...],
  middleware: [
    summarizationMiddleware({
      model: "gpt-5.4-mini",
      trigger: { tokens: 4000 },
      keep: { messages: 20 },
    }),
  ],
});
```
当对话超出 token 限制时，`SummarizationMiddleware` 会自动：

1. 使用单独的 LLM 调用来总结较早的消息
2. 用摘要消息替换它们（在 State 中永久替换）
3. 保持最近的消息完整以保留上下文

总结后的对话历史会被永久更新——后续轮次将看到摘要而非原始消息。

最佳实践


1. **从简单开始** - 从静态提示词和工具开始，仅在需要时才添加动态内容
2. **增量测试** - 每次添加一个上下文工程功能
3. **监控性能** - 跟踪模型调用、token 使用量和延迟
4. **使用内置中间件** - 利用 [`SummarizationMiddleware`](/oss/javascript/langchain/middleware#summarization)、[`LLMToolSelectorMiddleware`](/oss/javascript/langchain/middleware#llm-tool-selector) 等
5. **记录你的上下文策略** - 明确说明传递了哪些上下文以及原因
6. **理解瞬态与持久化的区别**：模型上下文变更是瞬态的（每次调用），而生命周期上下文变更会持久化到 state
