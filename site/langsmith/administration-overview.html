本概述涵盖了与在 LangSmith 中管理用户、组织、工作区和应用程序相关的主题。

资源层级结构

组织

组织是 LangSmith 中用户的逻辑分组，用于定义适用于其所有工作区的共享设置。这些设置管理组织范围的关注事项，而非工作区内的单个项目。常见的组织级配置包括用户管理、单点登录（SSO）、OAuth 提供商配置、自定义角色创建、计费和使用量跟踪。通常，每个公司对应一个组织。一个组织可以拥有多个工作区。更多详情，请参阅设置指南。

首次登录时，系统会自动为您创建一个个人组织。如果您想与他人协作，可以创建一个单独的组织并邀请团队成员加入。您的个人组织和共享组织之间存在一些重要区别：

功能个人共享
最大工作区数1因计划而异（请参阅定价页面）
协作无法邀请用户可以邀请用户
计费：付费计划仅开发者计划所有其他计划均可使用

工作区

信息

工作区以前称为租户。在过渡期间，某些代码和 API 可能仍会引用旧名称。

工作区是组织内用户和资源的逻辑分组。工作区通常用于隔离团队或业务单元，提供项目和关联资源之间的分离。工作区分离了资源的信任边界和访问控制。用户在工作区级别被授予权限，这些权限决定了他们对该工作区中资源的访问权限，包括跟踪项目、数据集、标注队列和提示语。关于设置详情，请参阅设置指南，关于权限详情，请参阅工作区（RBAC）。

我们建议为组织中的每个团队创建一个单独的工作区。要进一步组织资源，可以使用应用程序在工作区内对资源进行分组。有关基于团队隔离要求的不同工作区组织模型，请参阅工作负载隔离。

应用程序

应用程序是工作区内资源的逻辑分组。应用程序通常是智能体，但您可以将其用于团队内的任何项目。应用程序通过仅显示当前上下文中关联的应用程序资源来保持 UI 的整洁。

应用程序构建在资源标签之上，可用于使用 ABAC 控制资源访问。

在 LangSmith UI 的主导航侧边栏中切换应用程序。使用侧边栏顶部的应用程序下拉菜单选择一个应用程序。

任何资源都可以在不关联应用程序的情况下创建。这些资源将在选择所有应用程序选项时可见。

资源

资源是用于构建、运行和观察应用程序及智能体的具体实体，例如跟踪项目、提示语、数据集和部署。资源的作用域限定在特定应用程序内。

资源标签

资源标签允许您在工作区内进一步隔离资源，以便与 ABAC 一起使用。每个标签是一个键值对，您可以将其分配给资源。

LangSmith 资源标签与云服务（如 AWS）中的标签非常相似。

在 LangSmith UI 中导航至设置，在侧边栏中选择资源标签页面。

用户管理和 RBAC

用户

用户是能够访问 LangSmith 的人。用户可以是一个或多个组织及这些组织内工作区的成员。

组织成员在设置页面的成员和角色下管理。

工作区成员在工作区页面的设置下管理。

API 密钥

警告

我们已于 2024 年 10 月 22 日终止对以 ls 为前缀的传统 API 密钥的支持，转而支持个人访问令牌（PAT）和服务密钥。我们要求所有新集成使用 PAT 和服务密钥。以 ls 为前缀的 API 密钥将于 2024 年 10 月 22 日起不再工作。

过期日期

创建 API 密钥时，您可以选择设置过期日期。为密钥添加过期日期可以增强安全性并最大程度降低未经授权访问的风险。例如，您可以为需要临时提升访问权限的任务设置过期日期。

默认情况下，密钥永不过期。一旦过期，API 密钥将不再有效，无法重新激活或修改其过期时间。

个人访问令牌（PAT）

个人访问令牌（PAT）用于向 LangSmith API 进行请求认证。它们由用户创建，作用域限定在用户级别。PAT 将拥有与创建它的用户相同的权限。我们建议不要使用这些令牌来认证来自应用程序的请求，而应将其用于与 LangSmith API 交互的个人脚本或工具。如果与 PAT 关联的用户被从组织中移除，该 PAT 将不再工作。

PAT 以 lsv2_pt_ 为前缀

服务密钥

服务密钥类似于 PAT，但用于代表服务账户向 LangSmith API 进行请求认证。只有管理员可以创建服务密钥。我们建议将它们用于需要与 LangSmith API 交互的应用程序/服务，例如 LangGraph 智能体或其他集成。服务密钥可以限定在单个工作区、多个工作区或整个组织，并且可用于向其有权访问的工作区的 LangSmith API 进行请求认证。

服务密钥以 lsv2_sk_ 为前缀

警告

使用 X-Tenant-Id 标头指定目标工作区。

  • 使用 PAT 时：如果省略此标头，请求将在与密钥关联的默认工作区上运行。
  • 使用组织级服务密钥时：访问工作区级资源时必须包含 X-Tenant-Id 标头。如果没有该标头，请求将失败并返回 403 Forbidden 错误。
提示

有关如何创建服务密钥或个人访问令牌，请参阅设置指南

组织角色

组织角色与企业功能工作区 RBAC 不同，用于多个工作区的上下文中。您的组织角色决定了您的工作区成员资格特征和您的组织级权限。

所选的组织角色也会影响工作区成员资格，如下所述：

  • 组织管理员 授予完全访问权限，可管理所有组织配置、用户、计费和工作区。
  • 组织管理员对组织中的所有工作区拥有 Admin 访问权限。
  • 组织用户 可以查看组织信息，但无法在组织级别执行任何写操作。组织用户可以创建个人访问令牌。
  • 组织用户可以被添加到部分工作区，并照常分配工作区角色（如果启用了 RBAC），这些角色指定工作区级别的权限。
  • 组织查看者 等同于组织用户，但无法创建个人访问令牌。（对于自托管版本，在 Helm chart 0.11.25+ 版本中可用）。

工作区角色（RBAC）

提示

RBAC（基于角色的访问控制）是一项仅适用于企业客户的功能。如果您对此功能感兴趣，请联系我们的销售团队。其他计划默认对所有用户使用管理员角色。

角色用于定义用户在个工作区内拥有的权限集。有三个内置系统角色无法编辑：

  • 工作区管理员 对工作区内的所有资源拥有完全访问权限。
  • 工作区编辑者 除工作区管理（添加/移除用户、更改角色、配置服务密钥）外，拥有所有完整权限。
  • 工作区查看者 对工作区内的所有资源拥有只读访问权限。

组织管理员 还可以创建/编辑具有不同资源特定权限的自定义角色。

您可以在组织设置 > 成员和角色下管理角色，并选择角色选项卡。

最佳实践

环境隔离

使用资源标签通过默认标签键 Environment 和不同的环境值（如 dev、staging、prod）来组织资源。我们不建议使用单独的工作区进行环境隔离，因为资源无法跨工作区共享，这会阻止您在环境之间提升资源（如提示语）。

提示

资源标签与提交标签用于提示语管理

虽然两种类型的标签都可以使用 dev、staging 和 prod 等环境术语，但它们用途不同：

  • 资源标签（Environment: prod）：使用这些标签来组织和过滤工作区中的资源。将资源标签应用于跟踪项目、数据集和其他资源（包括提示语），以便按环境对它们进行分组，从而在 UI 中启用过滤功能。
  • 提交标签（prod 标签）：使用这些标签来管理代码引用的提示语版本。提交标签是指向提示语历史记录中特定提交的标签。当代码通过标签名称拉取提示语时（例如 client.pull_prompt("prompt-name:prod")），它会检索该标签当前指向的提交。要将提示语从 staging 提升到 prod，请移动提交标签以指向所需版本。

资源标签组织哪些资源属于某个环境。提交标签让您可以控制代码引用提示语的哪个版本，而无需更改代码本身。

使用和计费

数据保留

本节介绍数据保留的工作方式以及在 LangSmith 中的定价方式。

为什么保留很重要

  • 隐私：许多数据隐私法规（如欧洲的 GDPR 或加州的 CCPA）要求组织在个人数据不再需要用于收集目的时将其删除。设置保留期有助于遵守此类法规。
  • 成本：LangSmith 对低数据保留的跟踪收费较低。有关更多信息，了解如何强制执行支出限制。
建议

在开始发送跟踪之前规划您的保留层。变更仅适用于新跟踪——现有跟踪保持其原始层。请参阅更改项目级默认保留。

工作原理

LangSmith 根据数据保留有两个级别的跟踪，具有以下特征：

基础扩展
价格请参阅定价页面请参阅定价页面
保留期14 天400 天
提示

企业客户可以按工作区自定义扩展保留期。变更仅适用于新跟踪——现有跟踪不受影响。请参阅自定义扩展保留策略。

保留期结束后的数据删除

在指定的保留期后，跟踪在跟踪项目 UI 或 API 中将不再可访问。跟踪关联的所有用户数据（例如输入和输出）将在此后一天内从我们的内部系统中删除。每个跟踪的一些元数据可能会无限期保留，用于分析和计费目的。

数据保留自动升级

警告

自动升级可能会影响您的账单。请仔细阅读本节以充分了解您估计的 LangSmith 跟踪成本。

当您对 基础 级跟踪使用某些功能时，其数据保留将自动升级到 扩展 级。这将增加保留期和跟踪成本。

触发跟踪升级的完整场景列表如下：

  • 反馈被添加到跟踪中的任何运行上（或线程中的任何跟踪上），无论是通过手动标注、使用在线评估器自动进行，还是通过 SDK以编程方式进行。
  • 标注队列接收来自跟踪的任何运行。
  • 自动化规则匹配跟踪中的任何运行。

为什么自动升级跟踪？

我们采用跟踪自动升级模式有两个原因：

  1. 我们认为匹配这些条件中任何一个的跟踪从根本上比其他跟踪更有趣，因此让用户能够将它们保留更长时间是好事。
  2. 从哲学上讲，我们希望对可能没有有意义交互的跟踪收取低一个数量级的费用。我们认为自动升级使我们的定价模型与 LangSmith 带来的价值保持一致，只有有意义交互的跟踪才会以更高的费率收费。

如果您对我们的定价模式有疑问或顾虑，请随时通过 support.langchain.com 联系支持团队并分享您的想法！

数据保留如何影响下游功能？

  • 标注队列、运行规则和反馈：使用这些功能的跟踪将被自动升级。
  • 监控：监控选项卡即使在基础级跟踪的数据保留期结束后仍能继续工作。它由存在超过 30 天的跟踪元数据提供支持，这意味着您的监控图表即使在 基础 级跟踪上也能保持准确。
  • 数据集：数据集具有无限的数据保留期。换句话说，如果您将跟踪的输入和输出添加到数据集中，它们永远不会被删除。我们建议如果您使用 LangSmith 进行数据收集，请利用数据集功能。

计费模式

计费指标

在您的 LangSmith 发票上，您将看到我们收费的两个指标：

  • LangSmith 跟踪（基础收费）
  • LangSmith 跟踪（扩展数据保留升级）。

第一个指标包括所有跟踪，无论级别如何。第二个指标仅计算扩展保留跟踪的数量。

为什么测量所有跟踪 + 升级而不是基础和扩展跟踪？

在考虑我们的定价时，一个自然的问题是为什么不直接在发票上显示 基础 级和 扩展 级跟踪的数量？

虽然我们理解这会更简单，但它不能很好地适应跟踪升级。考虑一个 6 月 30 日记录的 基础 级跟踪，以及 7 月 3 日升级到 扩展 级的跟踪。基础 级跟踪发生在 6 月计费期，但升级发生在 7 月计费期。因此，我们需要能够独立测量这两个事件，以便正确地向客户收费。

如果您的跟踪记录为扩展保留跟踪，则 基础 和 扩展 指标都将以相同的时间戳记录。

速率限制

LangSmith 有速率限制，旨在确保所有用户的服务稳定性。

为确保访问和稳定性，在以下情况下，LangSmith 将响应 HTTP 状态码 429，表明超出了速率或使用限制：

应用程序负载均衡器上 1 分钟期间的临时吞吐量限制

此 429 是由于在 1 分钟窗口内基于每个服务密钥或 PAT 超过了固定数量的 API 调用而导致的。窗口的开始会略有不同——不能保证从时钟分钟的开始开始——并且可能因应用程序部署事件而异。

收到最大事件数后，我们将返回 429，直到达到评估窗口开始后的 60 秒，然后重复此过程。

此 429 由我们的应用程序负载均衡器抛出，是为所有 LangSmith 用户独立实施的一种机制，不受计划等级影响，以确保所有用户的服务连续性。

方法端点限制窗口
DELETE/sessions*301 分钟
POST 或 PATCH/runs*50001 分钟
GET/runs/:id301 分钟
POST/feedbacks*50001 分钟
**20001 分钟
提示

LangSmith SDK 采取措施通过将来自单个会话 ID 的最多 100 个运行批量到一个 API 调用中来最大程度地降低达到这些限制的可能性。

计划级每小时跟踪事件限制

此 429 错误是因为达到了每小时引入事件的最大数量，评估方式为从 UTC 时间每小时开始时启动的固定时间窗口，并在每小时整点重置。

此上下文中的事件是指运行的创建或更新。如果在同一个每小时窗口内创建了一个运行后又对其进行了更新，则计为 2 个事件。

此错误由我们的应用程序抛出，且因方案等级而异。我们的 Startup/Plus 和 Enterprise 方案等级的组织的每小时限制高于我们的免费版和开发者版方案等级，后者专为个人使用设计。

使用量限制

LangSmith 允许您在追踪上配置使用量限制。请注意，这些是使用量限制，而非消费限制，这意味着它们允许您限制某些事件的发生次数，而非您将花费的总额。

LangSmith 允许您设置两种不同的每月限制，与上述数据保留指南中讨论的计费指标一致：

  • 所有追踪限制
  • 扩展数据保留追踪限制

这些限制分别允许您限制总追踪数量和扩展数据保留追踪数量。

使用量限制的特性

使用量限制是近似值，这意味着我们不保证限制的精确性。在极少数情况下，可能会有一小段时间在额外追踪被处理后才会应用使用量限制。

扩展数据保留追踪限制的副作用

扩展数据保留追踪限制具有副作用。如果已达到限制，任何可能导致追踪层级自动升级的功能都将无法访问。这是因为追踪的自动升级将导致另一个扩展保留追踪被创建，而根据限制这不应被允许。因此，您将无法再：

  1. 匹配运行规则
  2. 向追踪添加反馈
  3. 将运行添加到标注队列

这些功能中的每一项都可能触发自动升级，因此当达到限制时我们会将其关闭。

更新使用量限制

使用量限制可以从"使用量和计费"下的"设置"页面更新。限制值会被缓存，因此可能需要一两分钟新限制才会生效。
