提示

  • Fleet 允许您在 LangSmith 中直接创建、部署和管理 AI 代理，无需编写代码。

先决条件

安装基础 LangSmith 平台

在继续之前，请按照 Kubernetes 安装指南安装基础 LangSmith 平台。 </section>

安装 KEDA

运行以下命令在集群上安装 KEDA：

    helm repo add kedacore https://kedacore.github.io/charts
    helm upgrade --install keda kedacore/keda --namespace keda --create-namespace
信息

KEDA 会根据队列大小自动扩展部署系统。

配置入口

为您的 LangSmith 实例配置入口、网关或 Istio。所有代理将作为 Kubernetes 服务部署在此入口之后。请参阅设置入口。您必须在langsmith_config.yaml中提供 hostname。 </section>

验证集群容量

确保您的集群有足够的可用容量来部署多个实例。建议使用集群自动扩缩容器。 </section>

验证存储

确保您的集群上有有效的动态 PV 配置器或 PV 可用。 </section>

    kubectl get storageclass

至少应有一个 StorageClass 具有 PROVISIONER 值（不是 kubernetes.io/no-provisioner）并标记为 (default)，否则必须先配置一个。 <section class="mdx-block">

验证出口流量

确保可以访问 https://beacon.langchain.com 的出口流量。请参阅出口流量文档。 </section>

启用 LangSmith 部署

组件

启用 LangSmith 部署会在您的集群中配置以下资源：

  • listener：监听控制平面中部署的更改，并创建或更新下游 CRD。
  • LangGraphPlatform CRD：管理 LangSmith 部署的实例。
  • operator：处理对 LangSmith CRD 的更改。
  • host-backend：控制平面。

启用此功能

要启用 LangSmith 部署，请更新您的 langsmith_config.yaml：

在配置中启用部署

在您的 langsmith_config.yaml 中，启用 deployment 选项。您还必须配置有效的入口。 </section>

    config:
      deployment:
        enabled: true
提示

从 v0.12.0 开始，langgraphPlatform 选项已弃用。对于 v0.12.0 之后的任何版本，请使用 config.deployment。

(可选) 配置镜像镜像

如果需要将镜像镜像到私有仓库，请在 langsmith_config.yaml 中配置 hostBackendImage 和 operatorImage 选项。请使用 最新 LangSmith Helm chart 版本 中指定的镜像标签。

    hostBackendImage:
      repository: "docker.io/langchain/hosted-langserve-backend"
      pullPolicy: IfNotPresent
    operatorImage:
      repository: "docker.io/langchain/langgraph-operator"
      pullPolicy: IfNotPresent

（可选）配置基础代理模板

如果需要自定义操作符创建代理 Kubernetes 资源的方式，请覆盖 values.yaml 中的基础代理模板。最常见的用例是添加 imagePullSecrets 来验证私有容器注册表。有关详细信息，请参阅为私有注册表配置身份验证。 </section>

应用更改

运行以下命令来应用更改。在整个指南中，只要要求您应用更改，都会使用此命令。将 <version> 和 <namespace> 替换为您的值：

    helm upgrade -i langsmith langchain/langsmith --values langsmith_config.yaml --version <version> -n <namespace> --wait --debug

验证新 Pod 已运行后再继续：

    kubectl get pods -n <namespace>
您的实例现已准备好创建部署。

启用 Fleet、Insights 和 Chat

信息

Fleet 需要 LangSmith Self-Hosted v0.13 或更高版本。下方描述的独立部署模式需要 v0.15 或更高版本。

每个功能都需要一个 Fernet 加密密钥。您可以在单个 Helm 配置中启用所有三个功能。

组件

启用这些功能将为集群中的每个功能（Fleet、Insights、Chat）配置以下组件：

  • api-server：处理该功能请求的主 API 服务器。
  • queue：后台任务处理队列。
  • postgres：该功能的专用 PostgreSQL 实例。可替换为外部 PostgreSQL 实例。
  • redis：该功能的专用 Redis 实例，用于缓存和发布/订阅。可替换为外部 Redis 实例。

Fleet 额外配置：

  • toolServer：为代理提供 MCP 工具执行。
  • triggerServer：处理 Webhook 和定时触发器。

生成加密密钥

每个功能使用自己的 Fernet 加密密钥来加密功能特定的密钥（如凭证和令牌）。独立的密钥允许独立轮换，并可限制密钥泄露时的暴露范围。使用 Python 为每个功能生成一个密钥：

python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"

我们建议将每个密钥存储在预定义的 Kubernetes Secret 中，而不是直接在配置文件中设置它们。请参阅使用现有 Secret了解相关参数：agent_builder_encryption_key、insights_encryption_key 和 polly_encryption_key。

启用功能

将配置添加到你的 langsmith_config.yaml

使用 Kubernetes Secret（推荐）

通过名称引用你现有的 Secret。图表会自动从中读取 agent_builder_encryption_key、insights_encryption_key 和 polly_encryption_key。

        config:
          existingSecretName: "<your-secret-name>"

        fleet:
          enabled: true

        insights:
          enabled: true

        # Chat (formerly Polly)
        polly:
          enabled: true

        fleetToolServer:
          enabled: true

        fleetTriggerServer:
          enabled: true

Using inline values

直接在配置文件中设置加密密钥。避免将此文件提交到版本控制系统。 </section>

        fleet:
          enabled: true
          encryptionKey: "<fleet-encryption-key>"

        insights:
          enabled: true
          encryptionKey: "<insights-encryption-key>"

        polly:
          enabled: true
          encryptionKey: "<chat-encryption-key>"

        fleetToolServer:
          enabled: true

        fleetTriggerServer:
          enabled: true
警告

如果您正在从旧的 agentBootstrap 部署模型迁移，请禁用 backend.agentBootstrap 以及旧的 config.agentBuilder、config.insights 和 config.polly 标志。这些是 config 部分下的标志，不是上面显示的顶级 fleet、insights 和 polly 标志。

您还必须通过 LangSmith Deployments UI 手动删除现有的 Fleet、Insights 和 Chat 部署。如果您在旧的 agentBootstrap 模型中未使用外部 PostgreSQL 数据库来运行 Fleet，并希望保留现有的 Fleet agents，请在应用此配置之前通过 Support Portal 联系技术支持。 </aside>

提示

Fleet 需要 fleetToolServer 和 fleetTriggerServer。这些从 Helm chart v15 起取代了已弃用的 agentBuilderToolServer 和 agentBuilderTriggerServer 键。 </aside>

默认情况下，每个功能都会部署自己专用的 PostgreSQL 和 Redis 实例。如需改用外部数据库，请在每个功能下配置 postgres.external 和 redis.external 部分。例如：

    fleet:
      enabled: true
      encryptionKey: "<fleet-encryption-key>"
      postgres:
        external:
          enabled: true
          connectionUrl: "<fleet-postgres-connection-url>"
      redis:
        external:
          enabled: true
          connectionUrl: "<fleet-redis-connection-url>"

应用更改

    helm upgrade -i langsmith langchain/langsmith --values langsmith_config.yaml --version <version> -n <namespace> --wait --debug

验证 Fleet、Insights 和 Chat pods 是否正在运行：

    kubectl get pods -n <namespace>

提供商设置指南

Google OAuth 提供商

要为 Fleet 启用 Google OAuth，请在 GCP 中创建一个 OAuth 客户端，并使用所需的 URL 和凭据对其进行配置。

在 GCP 中创建 OAuth 客户端

在 Google Cloud Console 中创建一个新的 OAuth 客户端应用（Web 应用）。 </section>

将 URL 添加到 GCP

将以下 URL 添加到您的 OAuth 客户端，将 <hostname> 替换为您的 LangSmith 主机名，将 <provider-id> 替换为您将使用的提供商 ID（例如 google）：

已授权的 JavaScript 来源：

  • https://<hostname>

已授权的重定向 URI：

- https://<hostname>/api-host/v2/auth/callback/<provider-id> - https://<hostname>/host-oauth-callback/<provider-id> </section>

复制凭据

从 GCP OAuth 应用中复制客户端 ID和客户端密钥。 </section>

在 LangSmith 中配置 OAuth 提供商

在 LangSmith 中，转到设置 > OAuth 提供商并添加一个新提供商：

- 客户端 ID：来自 GCP - 客户端密钥：来自 GCP - 授权 URL：https://accounts.google.com/o/oauth2/auth - 令牌 URL：https://oauth2.googleapis.com/token - 提供商 ID：唯一字符串，例如：google </section>

应用更改

将 LangSmith OAuth 提供商 ID 添加到您的 langsmith_config.yaml 并部署：

    fleet:
      oauth:
        providerOrgId: "<your-org-id>"
        googleOAuthProvider: "<provider-id>"
    
    helm upgrade -i langsmith langchain/langsmith --values langsmith_config.yaml --version <version> -n <namespace> --wait --debug

创建 Azure 应用注册

    https://<hostname>/host-oauth-callback/<provider-id>

创建客户端密钥

在证书和密钥中，创建新的客户端密钥。复制应用程序(客户端)ID和生成的客户端密钥值。 </section>

添加 Microsoft Graph 委托权限

  • Mail.Send
  • Calendars.ReadWrite

  • Channel.Create
  • ChannelMessage.Send
  • ChannelMessage.Read.All

提示

LangSmith 会自动为 Microsoft 提供程序请求 offline_access，以便用户可以接收刷新令牌。 </aside>

授予租户同意

如果您的 Microsoft 365 策略要求这些委托权限获得管理员同意，请为租户授予管理员同意。 </section>

在 LangSmith 中配置 OAuth 提供程序

在 LangSmith 中，转到设置 > OAuth 提供程序并添加新的提供程序：

  • 名称：例如，Microsoft
  • 提供程序 ID：唯一字符串，例如：microsoft-oauth-provider
  • 客户端 ID：来自 Azure 的应用程序（客户端）ID
  • 客户端密钥：来自 Azure 的客户端密钥值
  • 授权 URL：https://login.microsoftonline.com/common/oauth2/v2.0/authorize
  • 令牌 URL：https://login.microsoftonline.com/common/oauth2/v2.0/token
  • 提供程序类型：microsoft
  • 令牌端点身份验证方法：client_secret_post

Salesforce OAuth 提供商

要为 Fleet 启用 Salesforce OAuth，请创建一个 Salesforce 外部客户端应用，配置其 OAuth 设置和策略，获取其凭证，然后在 LangSmith 中配置 Salesforce OAuth 提供商。

创建外部客户端应用

在 Salesforce 设置中，使用快速查找打开外部客户端应用管理器，然后点击新建外部客户端应用。

在基本信息下，设置：

  • 外部客户端应用名称：例如，LangSmith Fleet
  • 联系人电子邮件：管理员电子邮件地址
  • 分发状态：本地
提示

外部客户端应用是 Salesforce 当前用于 OAuth 集成的框架。如果新建外部客户端应用不可用，请在设置 > 外部客户端应用设置下确认是否为您的组织启用了应用创建。 </aside>

启用 OAuth 并配置 OAuth 设置

展开API（启用 OAuth 设置）并选择启用 OAuth。然后配置：

  • 回调 URL，将 <hostname> 替换为您的 LangSmith 主机名，将 <provider-id> 替换为您的提供商 ID：
    https://<hostname>/host-oauth-callback/<provider-id>

打开应用，选择 策略 选项卡，然后点击 编辑：

  • 刷新令牌策略：选择 刷新令牌在撤销前有效。
  • 允许的用户：保留 所有用户均可自行授权。如果选择 管理员批准的用户已预先授权，则必须先将应用分配给权限集或配置文件，否则授权将失败。

点击 保存。

提示

外部客户端应用需要在两个位置进行配置：设置（上一步中的 OAuth 定义）和 策略（此步骤）。两个位置都必须保存。 </aside>

复制凭据

在 设置 选项卡的 OAuth 设置 下，选择 Consumer Key 和 Secret。Consumer Key 是您的 Client ID，Consumer Secret 是您的 Client Secret。

提示

创建应用后，请等待最多 30 分钟以完成传播，然后再进行首次连接尝试。 </aside>

在 LangSmith 中配置 OAuth 提供商

在 LangSmith 中，转到 设置 > OAuth 提供商，点击 OAuth 提供商，然后填写：

  • 提供商 ID：唯一字符串，例如：salesforce-oauth-provider。在下一步中，将此值用于 salesforceOAuthProvider。
  • 显示名称：例如，Salesforce
  • 客户端 ID：来自 Salesforce 的 Consumer Key
  • 客户端密钥：来自 Salesforce 的 Consumer Secret
  • 授权 URL：https://.my.salesforce.com/services/oauth2/authorize
  • 令牌 URL：https://.my.salesforce.com/services/oauth2/token

LangSmith 从令牌 URL 自动识别 Salesforce，因此无需设置 provider-type 或 token-auth-method 字段。保持 启用 PKCE 关闭，以匹配上述配置的 Web 服务器流程。

提示

将 ` 替换为组织的 My Domain，可在 设置 > My Domain 中找到。对于沙箱环境，请使用 https://--.sandbox.my.salesforce.com/services/oauth2/authorize` 和相应的令牌 URL。 </aside>

应用更改

将以下内容添加到您的 langsmith_config.yaml 并部署：

    fleet:
      oauth:
        providerOrgId: "<your-langsmith-org-id>"
        salesforceOAuthProvider: "<provider-id>"
    
    helm upgrade -i langsmith langchain/langsmith --values langsmith_config.yaml --version <version> -n <namespace> --wait --debug
警告

如果登录失败：请确认 Salesforce 中的 Callback URL 与 https://<hostname>/host-oauth-callback/<provider-id> 完全匹配（HTTPS，无尾部斜杠）；如果您选择了 Admin approved users are pre-authorized，请通过权限集分配应用；如果您的组织强制执行登录 IP 范围，请在用户配置文件上允许列 Fleet 服务器的出口 IP，或在应用策略中将 IP Relaxation 设置为 Relax IP restrictions。

（可选）为 Fleet 启用 GitHub App

Fleet 通过专用 GitHub App（而非 OAuth 应用）与 GitHub 集成。GitHub App 为 Fleet 的 GitHub 工具提供仓库访问权限，并支持私有仓库访问所需的用户授权流程。

设置包括创建 GitHub App、收集其凭证、将凭证存储为 Kubernetes Secret，以及从 langsmith_config.yaml 中引用它们。

创建 GitHub App

前往 GitHub Settings > Developer settings > GitHub Apps 并点击 New GitHub App。

提示

可以在个人账户或组织下创建该应用。如果有多人需要管理此集成，建议使用组织拥有的应用。 </aside>

填写基本信息

- GitHub App name：任意唯一名称，例如 acme-langsmith-fleet。请记下 GitHub 生成的 slug（名称的小写、连字符形式），因为这是 FLEET_GITHUB_APP_SLUG 使用的值。 - Homepage URL：您的 LangSmith 主机名，例如 https://langsmith.acme.com。 - 在 Webhook 下取消选择 Active，稍后会在生成 webhook secret 后启用。 </section>

设置回调 URL

在 Identifying and authorizing users 下，添加以下 Callback URL，将 <hostname> 替换为您的 LangSmith 主机名：

    https://<hostname>/v1/platform/fleet/providers/github-app/auth/callback

选择 Redirect on update。

在 Post installation 下，添加以下 Setup URL：

    https://<hostname>/v1/platform/fleet/providers/github-app/callback

选择 Redirect on update。 </section>

设置 webhook URL 并生成 webhook 密钥

生成一个随机的 webhook 密钥：

    python3 -c "import secrets; print(secrets.token_urlsafe(48))"

在 Webhook 下：

  • 选择 Active。
  • 将 Webhook URL 设置为：
      https://<hostname>/v1/platform/fleet/providers/github-app/webhooks
  • 将生成的值粘贴到 Webhook secret 中并保存，因为在后续步骤中创建 Kubernetes secret 时需要使用相同的值。

设置仓库权限

在 Permissions > Repository permissions 下，授予以下权限：

  • Contents：读写
  • Issues：读写
  • Pull requests：读写
  • Metadata：只读（自动选中）

在 Permissions > Account permissions 下，授予 Email addresses: Read-only 权限。

提示

这些是 Fleet 内置 GitHub 工具（问题管理、拉取请求创建、仓库内容访问）所需的最低权限。如需其他工具功能，请进行调整。 </aside>

选择安装可见性

在 Where can this GitHub App be installed? 下，选择符合您分发需求的选项。对于大多数自托管部署，应选择 Only on this account。 </section>

创建应用

点击 Create GitHub App。在应用设置页面中，记下以下值：

值在哪里找到环境变量
App ID数字，位于页面顶部FLEET_GITHUB_APP_ID
Public link例如，https://github.com/apps/acme-langsmith-fleetFLEET_GITHUB_APP_PUBLIC_LINK
App slug公共链接的最后一段路径FLEET_GITHUB_APP_SLUG
Client ID在 About 下FLEET_GITHUB_APP_CLIENT_ID

生成客户端密钥

在 Client secrets 下，点击 Generate a new client secret 并复制该值。这就是 FLEET_GITHUB_APP_CLIENT_SECRET。GitHub 仅显示一次。 </section>

生成私钥

滚动到 Private keys 并点击 Generate a private key。GitHub 会下载一个 .pem 文件。请妥善保管此文件，因为它授予对 GitHub App 的完全访问权限。PEM 内容就是 FLEET_GITHUB_APP_PRIVATE_KEY。 </section>

生成状态 JWT 密钥

LangSmith 使用 HMAC 密钥对短期 OAuth 状态令牌进行签名。生成一个：

    python3 -c "import secrets; print(secrets.token_urlsafe(48))"

This is FLEET_GITHUB_APP_STATE_JWT_SECRET. </section>

创建 Kubernetes 密钥

将敏感值存储在 Kubernetes 密钥中：

    kubectl create secret generic fleet-github-app \
      --namespace <your-langsmith-namespace> \
      --from-literal=client_secret="<client-secret>" \
      --from-literal=webhook_secret="<webhook-secret>" \
      --from-literal=state_jwt_secret="<state-jwt-secret>" \
      --from-file=private_key=/path/to/fleet-app.private-key.pem

添加以下内容，将占位符值替换为上面收集的非敏感值：

    commonEnv:
      - name: FLEET_GITHUB_APP_ID
        value: "<app-id>"
      - name: FLEET_GITHUB_APP_SLUG
        value: "<app-slug>"
      - name: FLEET_GITHUB_APP_PUBLIC_LINK
        value: "https://github.com/apps/<app-slug>"
      - name: FLEET_GITHUB_APP_CLIENT_ID
        value: "<client-id>"
      - name: FLEET_GITHUB_APP_CLIENT_SECRET
        valueFrom:
          secretKeyRef:
            name: fleet-github-app
            key: client_secret
      - name: FLEET_GITHUB_APP_PRIVATE_KEY
        valueFrom:
          secretKeyRef:
            name: fleet-github-app
            key: private_key
      - name: FLEET_GITHUB_APP_WEBHOOK_SECRET
        valueFrom:
          secretKeyRef:
            name: fleet-github-app
            key: webhook_secret
      - name: FLEET_GITHUB_APP_STATE_JWT_SECRET
        valueFrom:
          secretKeyRef:
            name: fleet-github-app
            key: state_jwt_secret

    fleetToolServer:
      deployment:
        extraEnv:
          - name: FLEET_GITHUB_APP_ENABLED
            value: "true"
提示

FLEET_GITHUB_APP_ENABLED 必须在工具服务器上设置，以便注册 GitHub 工具。其余的 FLEET_GITHUB_APP_* 变量由平台后端使用，位于 commonEnv 下。

为私有镜像仓库配置身份验证

如果您的 Agent Server 部署 将使用来自私有容器镜像仓库（例如 AWS ECR、Azure ACR 或 GCP Artifact Registry）的镜像，请配置镜像拉取密钥。此配置会自动应用到所有部署，使它们能够通过私有镜像仓库进行身份验证。

创建 Kubernetes 镜像拉取密钥

    kubectl create secret docker-registry langsmith-registry-secret \
        --docker-server=myregistry.com \
        --docker-username=your-username \
        --docker-password=your-password \
        --docker-email=your-email@example.com \
        -n langsmith

替换以下值为您的镜像仓库凭据：

- myregistry.com：您的镜像仓库 URL - your-username：您的镜像仓库用户名 - your-password：您的镜像仓库密码或访问令牌 - langsmith：LangSmith 安装所在的 Kubernetes 命名空间 </section>

在 langsmith_config.yaml 中配置部署模板

要使代理服务器部署能够使用私有镜像仓库密钥，请在操作符的部署模板中添加 imagePullSecrets：

    operator:
      templates:
        deployment: |
          apiVersion: apps/v1
          kind: Deployment
          metadata:
            name: ${name}
            namespace: ${namespace}
          spec:
            replicas: ${replicas}
            revisionHistoryLimit: 10
            selector:
              matchLabels:
                app: ${name}
            template:
              metadata:
                labels:
                  app: ${name}
              spec:
                enableServiceLinks: false
                imagePullSecrets:
                - name: langsmith-registry-secret
                containers:
                - name: api-server
                  image: ${image}
                  ports:
                  - name: api-server
                    containerPort: 8000
                    protocol: TCP
                  livenessProbe:
                    httpGet:
                      path: /ok
                      port: 8000
                    periodSeconds: 15
                    timeoutSeconds: 5
                    failureThreshold: 6
                  readinessProbe:
                    httpGet:
                      path: /ok
                      port: 8000
                    periodSeconds: 15
                    timeoutSeconds: 5
                    failureThreshold: 6

应用更改

    helm upgrade -i langsmith langchain/langsmith --values langsmith_config.yaml --version <version> -n <namespace> --wait --debug

所有通过 LangSmith UI 创建的用户部署都将继承这些注册表凭证。 </section>

有关注册表特定的认证方法，请参阅 Kubernetes 文档：从私有注册表拉取镜像。
