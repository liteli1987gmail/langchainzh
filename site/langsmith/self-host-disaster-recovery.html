您正在恢复的内容


自托管 LangSmith 由无状态服务组成，由四个状态存储支持。恢复规划几乎完全围绕状态存储展开。您可以随时通过重新应用 Helm chart 来重新创建无状态服务。

| 层级 | 组件 | 状态 | 恢复操作 |
|-------|------------|--------|-----------------|
| LangSmith 服务 | `langsmith-frontend`、`langsmith-backend`、`langsmith-platform-backend`、`langsmith-queue`、`langsmith-ingest-queue`、`langsmith-playground`、`langsmith-ace-backend` | 无状态 | 重新安装 Helm chart |
| PostgreSQL | 运营数据：组织、工作区、用户、API 密钥、数据集、提示词、项目、部署元数据 | **持久** | 从备份或副本恢复 |
| ClickHouse | 追踪和反馈（高容量分析数据） | **持久** | 从备份或副本恢复 |
| Blob 存储（S3/GCS/Azure Blob） | 运行输入、输出、错误、清单、额外数据、事件、附件（启用时） | **持久** | 从版本化存储桶或副本恢复 |
| Redis（或 Valkey） | 临时队列状态、pub/sub、缓存、运行心跳 | 临时 | 重新配置；无需恢复 |
| Kubernetes 对象 | Helm values、`Secret`、TLS 材料、IRSA / 工作负载身份绑定 | 配置 | 从源代码控制重新应用或备份集群状态 |


<div class="callout"><strong>警告</strong>


所有持久数据存储必须一起保护。Postgres、ClickHouse 和 blob 存储是保存持久数据的三个存储；Redis 是临时的，不需要备份。在没有 ClickHouse 和 blob 存储的情况下恢复 Postgres（反之亦然）会产生不一致的安装。Postgres 中对 ClickHouse 中运行和 blob 存储中对象的引用会在分歧点之后断开。请始终采用协调一致的备份，或使用跨存储接近的时间点恢复 (PITR) 目标。

</div>



规划您的 RPO 和 RTO


在设计 DR 架构之前，请定义两个目标：

- **恢复点目标 (RPO)：** 您的组织可以容忍的最大数据丢失量，以时间衡量。使用托管 Postgres PITR，RPO 通常小于 5 分钟。仅使用夜间快照，RPO 可能长达 24 小时。
- **恢复时间目标 (RTO)：** 故障后恢复服务的最大时间。热跨区域副本可以在几分钟内实现 RTO；从快照冷恢复可能需要数小时，特别是对于大型 ClickHouse 数据集。

以下部署模式假设三种目标配置文件之一：

| 配置文件 | 典型 RPO | 典型 RTO | 方法 |
|---------|-------------|-------------|----------|
| 仅快照 | 6 到 24 小时 | 数小时 | 每个存储的每日托管备份。成本最低，恢复时间最长。 |
| 多可用区高可用 | 秒级 | 分钟级（可用区故障） | Postgres 和 ClickHouse 在另一个可用区设置同步备用，多可用区 Redis，区域冗余 blob 存储。标准生产姿态。 |
| 跨区域 DR | 分钟到小时 | 小时 | Postgres、ClickHouse 和 blob 存储的备份复制到第二个区域，按需恢复。可选的 Postgres 跨区域副本以获得更紧密的 Postgres RPO。成本最高，恢复速度比多可用区慢，但可防止区域中断。 |


Postgres


LangSmith 使用 PostgreSQL 作为运营和事务数据的主要存储。**与 Postgres 的所有通信都使用可重试错误的重试机制**，因此故障转移期间的短暂中断通常不会显示为用户可见的错误。长时间中断将使 LangSmith API 不可用。

### 使用托管服务

我们强烈建议在生产环境中使用托管服务运行 Postgres。托管服务提供内置的自动备份、PITR 和高可用性故障转移。有关设置，请参阅[连接外部 Postgres](/langsmith/self-host-external-postgres.html)。


  
#### AWS


    在多可用区模式下运行 [Amazon RDS for PostgreSQL](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/CHAP_GettingStarted.CreatingConnecting.PostgreSQL.html) 或 [Aurora PostgreSQL](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/Aurora.Overview.html)。

    - **备份：** 启用[自动备份](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_WorkingWithAutomatedBackups.html)，保留窗口与您的合规姿态匹配（通常为 7 到 35 天）。
    - **PITR：** 自动备份在保留窗口内包含 PITR。
    - **高可用性：** 多可用区部署在第二个可用区维护同步备用，具有自动故障转移。
    - **跨区域 DR：** 对于 Aurora，配置 [Aurora 全局数据库](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/aurora-global-database.html)。对于 RDS，使用[跨区域只读副本](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_ReadRepl.XRgn.html)或将自动快照复制到辅助区域。
    - **加密：** 使用客户管理的 [KMS](https://aws.amazon.com/kms/) 密钥启用存储加密。
  

  
#### GCP


    使用[高可用性](https://cloud.google.com/sql/docs/postgres/high-availability)运行 [Cloud SQL for PostgreSQL](https://cloud.google.com/sql/docs/postgres)。

    - **备份：** 启用带 PITR 的[自动备份](https://cloud.google.com/sql/docs/postgres/backup-recovery/backups)。
    - **高可用性：** 区域实例同步复制到第二个区域中的备用。
    - **跨区域 DR：** 配置[跨区域只读副本](https://cloud.google.com/sql/docs/postgres/replication/cross-region-replicas)并在区域故障时提升。
    - **加密：** 使用 [Cloud KMS 客户管理加密密钥](https://docs.cloud.google.com/sql/docs/postgres/cmek)。
  

  
#### Azure


    使用区域冗余高可用性运行 [Azure Database for PostgreSQL Flexible Server](https://learn.microsoft.com/en-us/azure/postgresql/flexible-server/overview)。

    - **备份：** 启用带异地冗余备份存储的[自动备份](https://learn.microsoft.com/en-us/azure/postgresql/flexible-server/concepts-backup-restore)。
    - **高可用性：** 区域冗余高可用性在不同可用区维护同步备用。
    - **跨区域 DR：** 在辅助区域使用[只读副本](https://learn.microsoft.com/en-us/azure/postgresql/flexible-server/concepts-read-replicas)，以便在区域故障时提升。
  



### 集群内 Postgres

如果您必须从捆绑的 chart 在集群内运行 Postgres，您负责备份底层 PersistentVolume。定期使用 CSI 驱动程序的快照类对 PVC 进行快照，并将快照复制到对象存储或不同区域。**不建议将此路径用于生产。**


ClickHouse


ClickHouse 保存高容量追踪和反馈数据，通常是 LangSmith 部署中最大的数据存储。备份和复制需要规划成本和恢复时间影响。

### 托管 ClickHouse

实现弹性 ClickHouse 的最快路径是托管选项。请参阅[连接外部 ClickHouse](/langsmith/self-host-external-clickhouse.html)。

- **[LangSmith 托管 ClickHouse](/langsmith/langsmith-managed-clickhouse.html)：** LangChain 操作 ClickHouse 集群，包括备份和复制。VPC 对等连接将其连接到您的 LangSmith 安装。
- **[ClickHouse Cloud](https://clickhouse.cloud/)：** 提供内置备份、复制和高可用性。可在 AWS、GCP 和 Azure 市场上使用。

### 自管理复制集群

如果出于合规性或气隙原因自行管理 ClickHouse，请使用复制集群。单个节点 ClickHouse 实例无法满足有意义的 RPO。

- 使用 [Keeper 或 ZooKeeper](https://clickhouse.com/docs/architecture/replication) 通过复制配置多节点 ClickHouse 集群。
- 在 LangSmith chart 中设置 `cluster` 值，以便迁移从一开始就创建 `Replicated` 表引擎。**集群设置必须针对全新架构配置**，您不能稍后将独立实例转换为集群。
- 将副本分布到可用区。
- 安排[`BACKUP TABLE` 或 `BACKUP DATABASE`](https://clickhouse.com/docs/operations/backup)到对象存储，频率与您的 RPO 匹配。社区 [`clickhouse-backup`](https://github.com/Altinity/clickhouse-backup) 工具也是定时增量备份的流行选项，内置支持 S3、GCS 和 Azure Blob。
- 对于跨区域 DR，将备份存储桶复制到辅助区域。跨区域 ClickHouse 复制在自管理部署中通常不受支持，ClickHouse Cloud 也不提供此功能，因此请计划备份/恢复故障转移模型，而不是热副本。

有关复制的示例配置，请参阅 Helm 仓库中的[复制 ClickHouse 示例](https://github.com/langchain-ai/helm/blob/main/charts/langsmith/examples/replicated-clickhouse/README.md)。


<div class="callout"><strong>警告</strong>


恢复 ClickHouse 可能比恢复相同数据量的 Postgres 花费更长时间，因为追踪表很大。在设置 RTO 时请考虑这一点。在 DR 演练期间使用代表性数据集验证恢复时间。

</div>



Blob 存储


如果您已启用 [blob 存储](/langsmith/self-host-blob-storage.html)（推荐用于生产），您的运行输入、输出、错误、清单、额外数据、事件和附件存储在 S3、GCS 或 Azure Blob Storage 中。云 blob 服务设计上是持久的，但您仍应配置防止意外删除和区域中断的保护。


  
#### AWS


    - 启用 [S3 版本控制](https://docs.aws.amazon.com/AmazonS3/latest/userguide/Versioning.html)以防止意外删除和覆盖。
    - 为高安全存储桶启用 [MFA 删除](https://docs.aws.amazon.com/AmazonS3/latest/userguide/MultiFactorAuthenticationDelete.html)。
    - 对于跨区域 DR，配置[跨区域复制 (CRR)](https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html)到 DR 区域中的存储桶。
    - 使用 [S3 Object Lock](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lock.html)实现一次写入多次读取 (WORM) 保留。
    - 使用 [SSE-KMS](https://docs.aws.amazon.com/AmazonS3/latest/userguide/UsingKMSEncryption.html)加密。LangSmith 支持传递特定的 KMS 密钥 ARN，请参阅 [KMS 加密头支持](/langsmith/self-host-blob-storage.html#kms-encryption-header-support)。
  

  
#### GCP


    - 在存储桶上启用[对象版本控制](https://cloud.google.com/storage/docs/object-versioning)。
    - 使用[双区域或多区域存储桶](https://cloud.google.com/storage/docs/locations)实现地理冗余。
    - 对于跨区域 DR，使用 [Storage Transfer Service](https://cloud.google.com/storage-transfer-service)或[对象生命周期管理](https://cloud.google.com/storage/docs/lifecycle)与复制策略。
    - 使用[客户管理加密密钥 (CMEK)](https://cloud.google.com/storage/docs/encryption/customer-managed-keys)加密。
  

  
#### Azure


    - 选择与您的 DR 目标匹配的[冗余层](https://learn.microsoft.com/en-us/azure/storage/common/storage-redundancy)。使用 **RA-GRS** 或 **RA-GZRS** 以便在主区域中断期间跨区域读取访问。
    - 启用[软删除和 blob 版本控制](https://learn.microsoft.com/en-us/azure/storage/blobs/soft-delete-blob-overview)。
    - 使用 [Key Vault 中的客户管理密钥](https://learn.microsoft.com/en-us/azure/storage/common/customer-managed-keys-overview)加密。
  




<div class="callout"><strong>警告</strong>


**在 DR 存储桶中保留 TTL 生命周期规则。** 如果您将数据复制到 DR 存储桶，还要复制 `ttl_s/`、`ttl_l/` 和任何自定义 `ttl_XXd/` 前缀的生命周期规则。DR 存储桶中缺少规则会导致故障转移后数据无限期保留。请参阅 [TTL 配置](/langsmith/self-host-blob-storage.html#ttl-configuration)。

</div>



Redis


Redis 存储临时元数据、队列状态和跨实例 pub/sub。**Redis 中不存储持久数据，因此您不需要备份它。** 与 Redis 的通信会重试可重试错误。恢复设计是使 Redis 在活动区域内高度可用，并在 DR 区域从头重新配置。

- 使用云托管服务：[Amazon ElastiCache](https://aws.amazon.com/elasticache/redis/)、[Google Cloud Memorystore](https://cloud.google.com/memorystore) 或 [Azure Cache for Redis](https://azure.microsoft.com/en-us/products/cache)。
- 启用多可用区故障转移。
- 对于跨区域 DR，在故障转移期间在 DR 区域配置新的 Redis 实例；**不要**在新区群中重用活动区域的 Redis URI。


<div class="callout"><strong>警告</strong>


每个 LangSmith 安装必须使用自己专用的 Redis 实例。**不要在两个安装之间共享 Redis 实例**，包括可能在任何时候都处于活动状态的主安装和 DR 副本。共享 Redis 会导致部署任务被路由到错误的集群。请参阅[连接外部 Redis](/langsmith/self-host-external-redis.html)。

</div>



Kubernetes 配置和密钥


Helm chart values、Kubernetes `Secret` 和身份绑定与您的数据备份同样重要。完整恢复需要两者。

- **Helm values：** 将 `values.yaml` 存储在源代码控制中。单独跟踪每个环境的覆盖。
- **镜像版本：** 固定 LangSmith chart 版本和镜像标签，以便恢复安装相同版本的软件。请参阅[自托管升级](/langsmith/self-host-upgrades.html)和[依赖版本](/langsmith/self-host-dependency-versions.html)。
- **密钥：** LangSmith 从 Kubernetes `Secret` 读取数据库、blob 和许可凭证。将这些镜像到 DR 集群的密钥管理器（[AWS Secrets Manager](https://aws.amazon.com/secrets-manager/)、[GCP Secret Manager](https://cloud.google.com/secret-manager) 或 [Azure Key Vault](https://azure.microsoft.com/en-us/products/key-vault/)）。请参阅[使用现有密钥](/langsmith/self-host-using-an-existing-secret.html)。
- **TLS 材料：** 如果您在 LangSmith 入口终止 TLS，请备份证书和密钥，或在 DR 区域从您的私有 CA 重新颁发。请参阅[自定义 TLS 证书](/langsmith/self-host-custom-tls-certificates.html)。
- **IRSA / 工作负载身份绑定：** 在 DR 区域重新创建 IAM 角色和服务账户绑定；服务账户 ARN 和注解是区域范围的。
- **许可证密钥：** 将 LangSmith 许可证密钥与其他恢复密钥一起保存。


参考部署模式


### 具有多可用区高可用的单区域（推荐基线）

这是最低生产姿态，可防止可用区故障。它不能防止区域中断。

- Kubernetes 节点池跨越至少两个可用区。
- Postgres 以多可用区高可用模式运行（RDS Multi-AZ、Cloud SQL HA 或 Flexible Server 区域冗余）。
- ClickHouse 作为托管服务，或跨可用区分布的 3 节点复制集群。
- 启用多可用区故障转移的 Redis。
- 启用版本控制的 blob 存储，至少具有区域冗余的冗余层。
- 每个数据存储的每日快照，保留至少 7 天。

### 跨区域主动/被动 DR

这可以防止区域中断。它成本更高，但对于一级部署是正确的模式。

- DR 区域中的第二个 Kubernetes 集群，安装了 LangSmith Helm chart，但缩减到低副本数（热）或零（冷）。
- Postgres 跨区域副本（RDS 或 Aurora 跨区域副本、Cloud SQL 跨区域副本、Azure Flexible Server 跨区域副本）。在故障转移时提升。
- ClickHouse Cloud 或 LangSmith 托管 ClickHouse 以及区域故障转移计划，**或** ClickHouse 备份复制到 DR 区域并在故障转移时恢复到新的自管理集群。跨区域 ClickHouse 复制通常不受支持（ClickHouse Cloud 也不提供此功能），因此请计划备份/恢复而不是热 DR 副本。
- Blob 存储复制到 DR 存储桶，具有版本控制并匹配生命周期规则。
- 在故障转移期间在 DR 区域配置新 Redis。
- DNS 由 [Route 53](https://aws.amazon.com/route53/)、[Cloud DNS](https://cloud.google.com/dns) 或 [Azure DNS](https://azure.microsoft.com/en-us/products/dns/) 管理，具有健康检查和故障转移策略，指向每个区域的 LangSmith 前端入口。


<div class="callout"><strong>提示</strong>


LangSmith 是一个单写平台。跨区域部署应该是**主动/被动**，而不是主动/主动。同时写入两个区域与同一逻辑安装会导致数据不一致。

</div>



恢复程序


### 可用区故障后恢复

在单区域多可用区部署中，可用区故障由云提供商自动处理：

1. 托管 Postgres 故障转移到另一个可用区中的备用。LangSmith Pod 通过集群端点重试后重新连接。
2. 托管 Redis 类似地故障转移。LangSmith 自动重试重新连接。
3. Kubernetes 将 LangSmith Pod 重新调度到剩余可用区中的健康节点。验证节点池和水平 Pod 自动扩缩器限制允许此余地。
4. 通过从 SDK 提交测试追踪来验证摄入，并确认它出现在 UI 中。

### 区域故障后恢复

这是跨区域故障转移手册。根据您的特定基础设施进行调整。


  
### 宣布故障转移


    确认主区域不可用。向利益相关者传达您正在故障转移以及预期的 RTO。
  

  
### 提升数据存储


    将 Postgres 跨区域副本提升为 DR 区域中的主副本。对于 ClickHouse Cloud 或 LangSmith 托管 ClickHouse，启动文档化的区域故障转移。对于自管理 ClickHouse，将最新备份恢复到 DR 集群（这通常是最长的步骤）。
  

  
### 重新指向 blob 存储


    更新 LangSmith Helm `config.blobStorage.bucketName` 和 `apiURL` 指向 DR 存储桶。确认存储桶具有相同的 TTL 生命周期规则。请参阅 [Blob 存储配置](/langsmith/self-host-blob-storage.html#configuration)。
  

  
### 配置 Redis


    在 DR 区域创建新的托管 Redis 实例。更新 LangSmith Helm `redis.external` 值以指向它。**不要从主 Redis 导入转储**；配置空的。
  

  
### 扩展 DR 集群


    如果运行热/冷，请将 LangSmith 部署扩展到其生产副本数。从源代码控制应用任何待处理的 Helm value 更新。
  

  
### 运行冒烟测试


    提交测试追踪，验证它进入 ClickHouse 并且（如果启用了 blob 存储）进入 DR 存储桶。打开 UI 并确认追踪、数据集和项目加载。验证身份验证。请参阅[诊断](/langsmith/diagnostics-self-hosted.html)。
  

  
### 切换 DNS


    更新 DNS 将流量路由到 DR 入口。向利益相关者传达切换。
  

  
### 计划回切


    一旦主区域恢复健康，计划受控回切。这通常安排在维护窗口内，涉及将主区域重建为新的 DR 副本，然后再切换。
  



### 从快照恢复

如果您完全丢失了主数据存储，需要从快照恢复：


  
### 停止摄入


    将 `langsmith-queue` 和 `langsmith-ingest-queue` 缩减到零，以便在恢复时不写入新追踪。
  

  
### 恢复 Postgres


    将 Postgres 备份恢复到新实例，或执行 PITR 到最新的事故前时间戳。更新 LangSmith Helm `postgres.external` 连接详情以指向恢复的实例。
  

  
### 恢复 ClickHouse


    恢复与 Postgres 恢复点时间最接近的最新 ClickHouse 备份。恢复时间随数据大小而增加。
  

  
### 恢复 blob 存储


    如果您丢失了 blob 数据（很少见），从 S3/GCS/Azure 恢复版本化对象，或从复制的 DR 存储桶复制。
  

  
### 恢复摄入


    将 `langsmith-queue` 和 `langsmith-ingest-queue` 扩展回生产副本数。提交冒烟测试追踪并验证它已到达。
  




<div class="callout"><strong>警告</strong>


始终将 Postgres、ClickHouse 和 blob 存储恢复到尽可能接近的协调时间点。将 Postgres 恢复到比 ClickHouse 更新的点会产生悬空项目引用和 UI 中缺失的追踪。

</div>



测试您的 DR 计划


备份的好坏取决于上次成功恢复。请安排以下练习：

- **每季度：** 将 Postgres 和 ClickHouse 快照恢复到非生产环境，运行[诊断工具](/langsmith/diagnostics-self-hosted.html)和冒烟追踪测试。测量实际恢复时间并确认其在 RTO 内。
- **每半年：** 对暂存安装执行完整的跨区域故障转移演练。提升副本、重新指向 blob 存储、扩展 DR 集群、运行冒烟测试，然后回滚。
- **每次 chart 升级时：** 验证升级路径不会使您的 DR 计划失效（例如，仅应用于主数据库的架构迁移需要复制到 DR 副本）。请参阅[自托管升级](/langsmith/self-host-upgrades.html)。
