提示

自托管版本要求：访问告警功能需要 Helm chart 版本 0.10.3 或更高版本。

有效的 LLM 应用可观测性需要主动检测故障、性能下降和回归问题。LangSmith 的告警功能帮助识别以下关键问题：

  • 模型提供商的 API 速率限制违规。
  • 应用的延迟增加。
  • 影响反映最终用户体验的反馈分数的应用变更。
  • LLM 使用导致的意外成本激增。

LangSmith 中的告警是按项目作用域的，每个被监控的项目需要单独配置。

建议

告警可以通过 webhook 路由到 Slack、PagerDuty、Dynatrace 或任何 HTTP 端点。Webhook 选项卡包含自托管部署的 Microsoft Teams、电子邮件和 Slack 的示例配置。

按照以下步骤配置告警。

步骤 1：导航到创建告警

在 UI 中，导航到你想要配置告警的 Tracing 项目。点击页面右上角的告警图标，查看该项目的现有告警并设置新告警。

步骤 2：选择指标类型

LangSmith 提供以下指标的阈值告警：

指标类型说明使用场景
运行次数跟踪时间窗口内 runs 的总数。监控管道是否以预期数量生成运行，并在意外下降时告警。
成本跟踪时间窗口内运行的总成本。监控 LLM 支出，在成本超过预期阈值时告警。需要配置成本跟踪。
错误跟踪具有错误状态的运行。根据错误总数或错误百分比（错误运行占总运行的比例）告警。监控应用中的失败，或在错误率超过可接受阈值时告警。
反馈分数测量平均反馈分数。跟踪来自最终用户的反馈或在线评估结果，以在回归时告警。
延迟测量平均运行执行时间。跟踪应用的延迟以在峰值和性能瓶颈时告警。

此外，对于错误和延迟，你可以使用过滤器构建器对状态、运行类型、标签和错误等字段堆叠条件。例如，你可以将错误告警范围限定为状态为 error、运行类型为 llm、标签为 support_agent 且错误匹配 RateLimitExceeded 的运行。

步骤 3：定义告警条件

告警条件由以下几个组件组成：

  • 聚合方法：平均值、百分比或计数。
  • 比较运算符：>=、<= 或超过阈值。
  • 阈值：触发告警的数值。
  • 聚合窗口：指标计算的时间段（可选择 5 或 15 分钟）。
  • 反馈键（仅限反馈分数告警）：要监控的特定反馈指标。

示例： 截图中显示的配置会在过去 5 分钟内超过 5% 的运行产生错误时生成告警。

你可以预览告警在历史时间窗口中的行为，以了解在选定阈值下有多少数据点以及哪些数据点会触发告警（以红色显示）。例如，为项目设置 60 秒的平均延迟阈值可以让你可视化潜在告警，如以下截图所示。

步骤 4：配置通知渠道

Slack

使用 LangSmith 原生的 Slack 集成直接将告警通知发送到 Slack 频道。无需自定义 webhook 或 Slack 应用配置。

提示

原生 Slack 通知类型仅在 LangSmith Cloud 上可用。对于自托管部署，请改用 Webhook 选项卡中的 webhook Slack 配置。 </aside>

前提条件

  • 一个连接到你的 LangSmith 组织的 Slack 工作区。如果你还没有连接，LangSmith 会在你配置此通知类型时在线提示你连接。

1. 配置 Slack 通知

  1. 在告警设置的通知设置部分，选择 Slack。
  2. 点击频道选择器。如果还没有关联 Slack 工作区，点击连接 Slack 并完成 OAuth 流程以授权 LangSmith。
  3. 将 @LangSmith 应用添加到你要接收通知的频道。该应用必须是频道的成员——在频道中输入 /invite @LangSmith 来添加它。
  4. 从下拉菜单中选择工作区和频道。如果频道没有立即显示，点击刷新图标。
  5. 点击保存以保存通知配置。

2. 测试集成

点击发送测试通知以验证 LangSmith 能够访问该频道。检查频道中是否收到测试消息。

通知格式

当警报触发时，LangSmith 会发布一条结构化的 Slack 消息，包含：

- 标题：警报名称和您的 LangSmith 工作区名称。 - 详情行：指标属性、触发值、比较运算符、配置的阈值、聚合方法和时间窗口 — 例如：Total Cost: $12.50 ≥ $5.00 · avg · 30 min。 - 操作按钮：查看警报（链接到 LangSmith 中的警报预览）和查看运行（链接到触发警报的筛选后的运行）。 </section>

PagerDuty

使用 PagerDuty 的 Events API v2 将 PagerDuty 配置为通知渠道。此集成允许关键的 LLM 应用程序问题触发 PagerDuty 事件，从而通过您已建立的事件管理工作流程实现快速响应。

前置要求

  • 具有管理员访问权限的有效 PagerDuty 账户
  • PagerDuty 中适当的服务级别权限

如果使用 LangSmith 的自定义部署，请确保没有防火墙设置阻止 LangSmith 服务的出口流量。

1. 在 PagerDuty 中创建服务

1. 登录您的 PagerDuty 账户 2. 导航至 Services → Service Directory 3. 点击 + New Service 4. 填写以下字段： - Name：提供一个描述性名称（例如，"LangSmith Monitoring"） - Description：添加有关被监控应用程序的详细信息 - Escalation Policy：选择适当的团队升级策略 - Integration Type：选择"Events API V2" 5. 点击 Add Service 创建服务

2. 获取集成密钥

创建服务后，检索集成密钥：

  1. 从 Service Directory 中，找到并点击您新创建的服务
  2. 选择 Integrations 选项卡
  3. 找到"Events API V2"集成
  4. 复制 Integration Key（一个 32 字符的字母数字字符串）

3. 配置 LangSmith 警报与 PagerDuty 的集成

信息

要在同一警报触发后一小时内再次接收该警报，您必须在 PagerDuty 中解决该警报创建的活动事件。 </aside>

1. 在 LangSmith 警报设置的通知部分，选择 PagerDuty 2. 点击密钥图标将 Integration Key 保存为工作区密钥，或选择现有的工作区密钥。作为最佳实践，我们建议将 Integration Key 保存为工作区密钥，而不是直接添加。这样您可以在工作区的多个警报中重复使用同一个密钥。 3. 配置其他通知选项： - Severity：映射到 PagerDuty 事件优先级 4. 点击 Send Test Alert 发送测试警报 5. 验证 PagerDuty 是否触发了事件，并包含相关的 LangSmith 警报信息

故障排除

如果 PagerDuty 中未创建事件：

  • 验证 Integration Key 在 LangSmith 中输入正确
  • 确保 PagerDuty 服务处于活动状态且未处于维护模式
  • 检查您的 PagerDuty 账户是否启用了 Events API v2
  • 如果 PagerDuty 中缺少警报触发，请检查预期的触发是否发生在同一警报规则上一次触发后一小时内，以及上次警报创建的事件是否仍处于打开状态。
  • 如果您的 LangSmith 实例位于防火墙后面，请检查网络连接

其他资源

- PagerDuty Events API v2 文档 - PagerDuty 集成指南 </section>

Dynatrace

使用 Dynatrace 的 Events API v2 将 Dynatrace 配置为通知渠道。此集成将 LangSmith 警报事件发送到您的 Dynatrace 环境，从而能够与您更广泛的基础设施监控进行关联。

前置要求

  • 一个有效的 Dynatrace 环境（SaaS 或托管版本）。
  • 具有 events.ingest 作用域的 Dynatrace API 访问令牌。

如果您使用的是 LangSmith 的自定义部署，请确保没有防火墙设置阻止 LangSmith 服务的出口流量。

1. 在 Dynatrace 中创建 API 令牌

  1. 登录您的 Dynatrace 环境。
  2. 导航至 Access Tokens。
  3. 点击 Generate new token。
  4. 提供一个描述性名称（例如，"LangSmith Alerts"）。
  5. 在 Scopes 下，搜索并启用 events.ingest（摄取事件）。
  6. 点击 Generate token。
  7. 复制生成的令牌并安全存储。令牌只会显示一次。

2. 获取您的 Dynatrace 环境 URL

您的 Dynatrace 环境 URL 遵循以下格式：

    https://{your-environment-id}.live.dynatrace.com

登录 Dynatrace 后，你可以在浏览器 URL 栏中找到你的环境 ID。

3. 使用 Dynatrace 配置 LangSmith 告警

1. 在 LangSmith 告警设置中的 通知设置 中，选择 Dynatrace。 1. 输入你的 Dynatrace 环境 URL。 1. 点击密钥图标将 API 令牌保存为工作区密钥，或选择现有的工作区密钥。作为最佳实践，建议将 API 令牌保存为工作区密钥，而不是直接添加。这样可以让你在同一个工作区的多个告警中重复使用同一个令牌。 1. 配置其他通知选项： - 事件类型：选择 Dynatrace 事件类型（例如 CUSTOM_ALERT、ERROR_EVENT） 1. 点击 发送测试通知 发送测试告警。 1. 验证事件是否出现在你的 Dynatrace 环境中。

故障排除

如果事件没有出现在 Dynatrace 中：

  • 验证 API 令牌具有 events.ingest 作用域且未过期。
  • 确保环境 URL 正确且包含你的环境 ID。
  • 确认 Authorization 标头格式使用 Api-Token（而不是 Bearer）。
  • 检查你的 Dynatrace 环境是否处于活跃状态且可访问。
  • 如果你的 LangSmith 实例在防火墙后面，请检查网络连接。

其他资源

- Dynatrace Events API v2 文档 - Dynatrace 访问令牌 </section>

Webhook

Webhook 通过在触发告警条件时发送 HTTP POST 请求，实现与自定义服务和第三方平台的集成。使用 webhook 可以将告警数据转发到工单系统、聊天应用程序或自定义监控解决方案。

前提条件

  • 一个能够接收 HTTP POST 请求的端点
  • 接收服务的适当身份验证凭据（如果需要）

1. 准备你的接收端点

在 LangSmith 中配置 webhook 之前，确保你的接收端点：

  • 接受 HTTP POST 请求
  • 能够处理 JSON 负载
  • 可从外部服务访问
  • 具有适当的身份验证机制（如果需要）

如果使用 LangSmith 的自定义部署，请确保没有防火墙设置阻止来自 LangSmith 服务的出口流量。

2. 配置 webhook 参数

在 LangSmith UI 的 Monitoring 部分下的 Alerts 选项卡中，点击 + Alert 创建新告警。

在 Notification Settings 部分，使用以下参数完成 webhook 配置：

必填字段

  • URL：你的接收端点的完整 URL
  • 示例：https://api.example.com/incident-webhook

可选字段

  • Headers：随 webhook 请求发送的 JSON 键值对
  • 常用标头包括：
  • Authorization：用于身份验证令牌
  • Content-Type：通常设置为 application/json（默认值）
  • X-Source：用于标识来源为 LangSmith
  • 如果没有标头，请使用 {}
  • Request Body Template：自定义发送到端点的 JSON 负载
  • 默认值：LangSmith 发送已定义的负载，以及以下附加键值对追加到负载中：
  • project_name：触发的告警所属项目名称
  • alert_rule_id：用于标识 LangSmith 告警的 UUID。这可以用作 webhook 服务中的去重键。
  • alert_rule_name：告警规则的名称。
  • alert_rule_type：告警类型（截至 04/01/2025，所有告警类型均为 threshold）。
  • alert_rule_attribute：与告警规则关联的属性 - error_count、feedback_score、latency 或 cost。
  • triggered_metric_value：触发阈值时该指标的值。
  • triggered_threshold：触发告警的阈值。
  • timestamp：触发告警的时间戳。
信息

LangSmith 不会对请求体执行模板替换。上述自动填充的字段将作为顶级键合并到传出的 JSON 中，与你配置的正文并列。像 {alert_rule_name} 这样的占位符语法会按原样发送到接收服务。只有当接收方本身可以从传入的 JSON 中提取字段时（例如 Power Automate Workflow、AWS Lambda 或自定义 HTTP 处理器），它才会解析为实际值。 </aside>

3. 测试 webhook

点击 Send Test Alert 将 webhook 通知发送出去，以确保通知按预期工作。

故障排除

如果 webhook 通知未被送达：

  • 验证 webhook URL 正确且可访问
  • 确保任何身份验证标头格式正确
  • 检查你的接收端点是否接受 POST 请求
  • 检查你的端点日志以查看收到的但被拒绝的请求
  • 验证你的自定义负载模板是有效的 JSON 格式

安全注意事项

  • 为你的 webhook 端点使用 HTTPS
  • 为你的 webhook 端点实施身份验证
  • 考虑在你的标头中添加共享密钥以验证 webhook 来源
  • 在处理传入的 webhook 请求之前先验证它们

示例配方

通过 webhook 配置 Slack 通知

以下是通过 chat.postMessage API 配置 LangSmith 告警以向 Slack 频道发送通知的示例。

前提条件

  • 访问 Slack 工作区。
  • 用于设置告警的 LangSmith 项目。
  • 创建 Slack 应用程序的权限。

第 1 步：创建 Slack 应用

  1. 访问 Slack API 应用程序页面。
  2. 点击 创建新应用。
  3. 选择 从头开始。
  4. 输入 应用名称（例如 "LangSmith 警报"）。
  5. 选择要安装该应用的工作区。
  6. 点击 创建应用。

步骤 2：配置机器人权限

1. 在 Slack 应用配置的左侧边栏中，点击 OAuth 与权限。 2. 向下滚动到 作用域 下的 机器人令牌作用域，然后点击 添加 OAuth 作用域。 3. 添加以下作用域： - chat:write（以应用身份发送消息）。 - chat:write.public（向应用未加入的频道发送消息）。 - channels:read（查看频道基本信息）。

步骤 3：将应用安装到您的工作区

  1. 向上滚动到 OAuth 与权限 页面顶部。
  2. 点击 安装到工作区。
  3. 审查权限并点击 允许。
  4. 复制显示的 机器人用户 OAuth 令牌（以 xoxb- 开头）。

步骤 4：将机器人添加到 Slack 频道

将机器人添加到您想要接收警报的特定频道。您可以在消息字段中 @机器人来将其添加到 Slack 频道（例如 @机器人名称）。

您还需要频道 ID 来配置 LangSmith 中的 Webhook 警报。您可以通过打开频道详情 > 关于来找到频道 ID。

步骤 5：在 LangSmith 中配置 Webhook 警报

  1. 在 LangSmith 中，导航到您的项目。
  2. 选择 警报 → 创建警报。
  3. 定义您的警报指标和条件。
  4. 在通知部分，选择 Webhook。
  5. 使用以下设置配置 Webhook：

Webhook URL

      https://slack.com/api/chat.postMessage

请求头

提示 将 `xoxb-your-token-here` 替换为你的 Bot 用户 OAuth Token 
      {
        "Content-Type": "application/json",
        "Authorization": "Bearer xoxb-your-token-here"
      }

请求体模板

提示 必须填写第4步中找到的 `{channel_id}` 值。

其余字段：`alert_name`、`project_name` 和 `project_url` 可选择性地为警报消息添加额外的上下文信息。您可以在浏览器的地址栏中找到您的 `project_url`。复制到查询参数之前（不包含查询参数）的部分。 
      {
        "channel": "{channel_id}",
        "text": "{alert_name} triggered for {project_name}",
        "blocks": [
          {
            "type": "section",
            "text": {
              "type": "mrkdwn",
              "text": "🚨{alert_name} has been triggered"
            }
          },
          {
            "type": "section",
            "text": {
              "type": "mrkdwn",
              "text": "Please check the following link for more information:"
            }
          },
          {
            "type": "section",
            "text": {
              "type": "mrkdwn",
              "text": "<{project-url}|View in LangSmith>"
            }
          }
        ]
      }
  1. 点击 Save 以激活 webhook 配置。

步骤 6：测试集成

  1. 在 LangSmith 告警配置中，点击 Test Alert。
  2. 检查您指定的 Slack 频道中的测试通知。
  3. 验证消息包含预期的告警信息。

（可选）步骤 7：链接到请求体中的告警预览

创建告警后，您可以选择在 webhook 的请求体中链接到其预览。

配置方法：

1. 保存您的告警。 2. 在告警表中找到已保存的告警并点击。 3. 复制显示的 URL。 4. 点击 "Edit Alert"。 5. 用复制的告警预览 URL 替换现有的项目 URL。 </section>

通过 webhook 配置 Microsoft Teams 通知

以下是通过 Workflows 应用（Power Automate）配置 LangSmith 告警以向 Microsoft Teams 频道发送通知的示例。推荐使用此方法，因为它在流程中提取传入 JSON 的字段，使自动填充的 LangSmith 告警字段能够在 Teams 消息中正确呈现。

提示

Microsoft 传统的 Office 365 传入 Webhook 连接器即将停用。新集成请使用 Workflows 应用。 </aside>

前提条件

  • 有权添加 Workflows 的 Microsoft Teams 工作区访问权限。
  • 用于设置告警的 LangSmith 项目。

步骤 1：在 Teams 中创建 Workflow

  1. 在 Microsoft Teams 中，导航到您要接收告警的频道。
  2. 点击频道名称旁边的 ...（更多选项）菜单。
  3. 选择 Workflows。
  4. 搜索并选择 Post to a channel when a webhook request is received 模板。
  5. 登录以确认连接，然后点击 Next。
  6. 确认要发布告警的团队和频道，然后点击 Add workflow。
  7. 复制生成的 HTTP POST URL——在 LangSmith 中使用此 URL。

步骤 2：在 Power Automate 中自定义消息（可选）

默认 workflow 将原始 JSON 正文作为卡片发布。要格式化告警详情，请在 Power Automate 中编辑流程：

  1. 打开 Power Automate 门户 并编辑您创建的 workflow。
  2. 点击 Post card in a chat or channel 操作。
  3. 在 Adaptive Card 字段中，使用 triggerOutputs()?['body/alert_rule_name']、triggerOutputs()?['body/project_name']、triggerOutputs()?['body/triggered_metric_value']、triggerOutputs()?['body/triggered_threshold']、triggerOutputs()?['body/timestamp'] 和 triggerOutputs()?['body/alert_rule_url'] 引用传入字段。
  4. 保存流程。

步骤 3：在 LangSmith 中配置 webhook 告警

  1. 在 LangSmith 中，导航到您的项目。
  2. 选择 Alerts → Create Alert。
  3. 定义您的告警指标和条件。
  4. 在通知部分，选择 Webhook。
  5. 使用以下设置配置 webhook：

Webhook URL

粘贴 Teams Workflow 中的 HTTP POST URL：

      https://prod-XX.westus.logic.azure.com:443/workflows/.../triggers/manual/paths/invoke?...

标题

      {
        "Content-Type": "application/json"
      }

请求体模板

LangSmith 自动将自动填充的警报字段（alert_rule_name、project_name、triggered_metric_value、triggered_threshold、timestamp、alert_rule_url 等）合并到请求体中作为顶级 JSON 键。Power Automate 直接从传入的有效载荷中读取这些字段，因此使用空请求体即可：

      {}
  1. 点击 保存 以激活 webhook 配置。

步骤 4：测试集成

  1. 在 LangSmith 告警配置中，点击 发送测试告警。
  2. 检查你指定的 Teams 频道中的测试通知。
  3. 验证卡片包含预期的告警信息。

参考实现

有关将 LangSmith webhook 负载（阈值告警、运行规则和通用事件）转换为格式化 Teams Adaptive Cards 的工作示例，请参阅 langsmith-teams-webhook 示例仓库。该示例作为一个小型的 Python 服务运行在 Teams Workflow URL 前面，这样就无需自定义 Power Automate 流程本身。 </section>

通过 webhook 配置电子邮件通知

以下示例展示了如何使用 SendGrid 的 Mail Send API 配置 LangSmith 告警以发送电子邮件通知。你可以使用任何提供 HTTP API 的事务性电子邮件提供商（例如 Mailgun、Amazon SES、Postmark）。

前提条件

  • 一个具有已验证发件人身份的 SendGrid 账户。
  • 一个具有 Mail Send 权限的 SendGrid API 密钥。
  • 一个用于设置告警的 LangSmith 项目。

步骤 1：创建 SendGrid API 密钥

  1. 登录到你的 SendGrid 仪表板。
  2. 导航到 Settings → API Keys。
  3. 点击 Create API Key。
  4. 选择 Restricted Access 并启用 Mail Send → Full Access。
  5. 点击 Create & View，复制密钥，并安全存储。

步骤 2：验证你的发件人邮箱

  1. 在 SendGrid 中，导航到 Settings → Sender Authentication。
  2. 为你想要发送的地址完成 Domain Authentication（推荐）或 Single Sender Verification。

步骤 3：在 LangSmith 中配置 webhook 告警

  1. 在 LangSmith 中，导航到你的项目。
  2. 选择 Alerts → Create Alert。
  3. 定义你的告警指标和条件。
  4. 在通知部分，选择 Webhook。
  5. 使用以下设置配置 webhook：

Webhook URL

      https://api.sendgrid.com/v3/mail/send

响应头

提示 请将 `SG.your-api-key-here` 替换为您的 SendGrid API 密钥。 
      {
        "Content-Type": "application/json",
        "Authorization": "Bearer SG.your-api-key-here"
      }

请求体模板

提示

请将 alerts@your-company.com 替换为您已验证的发件人地址，将 oncall@your-company.com 替换为收件人地址。SendGrid 不会从任意顶级 JSON 键中提取字段，因此本示例使用固定的主题和正文。若要在邮件中包含特定警报的值，请通过中间件（如 Power Automate flow、AWS Lambda 或 Zapier webhook）路由 LangSmith webhook，让其读取传入的 payload 并渲染 SendGrid 请求。

      {
        "personalizations": [
          {
            "to": [
              {
                "email": "oncall@your-company.com"
              }
            ],
            "subject": "LangSmith alert triggered"
          }
        ],
        "from": {
          "email": "alerts@your-company.com",
          "name": "LangSmith Alerts"
        },
        "content": [
          {
            "type": "text/plain",
            "value": "A LangSmith alert was triggered. Open your LangSmith workspace to view the alert details, including the project, metric value, threshold, and timestamp."
          }
        ]
      }
  1. 点击 保存 以激活 webhook 配置。

步骤 4：测试集成

  1. 在 LangSmith 告警配置中，点击 发送测试告警。
  2. 检查收件人的收件箱中是否有测试通知。
  3. 验证邮件包含预期的告警信息。

使用其他邮件服务商

相同的模式也适用于其他接受静态认证头的交易邮件 API。更改 Webhook URL 和 Headers 以匹配您的服务商：

服务商Webhook URL认证头格式
Mailgunhttps://api.mailgun.net/v3/{your-domain}/messagesAuthorization: Basic <base64(api:<key>)>
Postmarkhttps://api.postmarkapp.com/emailX-Postmark-Server-Token: <token>

调整 请求体模板 以匹配每个服务商预期的 payload 格式。Amazon SES 不直接兼容，因为 SES API 需要每个请求的 AWS SigV4 签名，无法用静态头表示。要使用 SES，请通过中间件路由（例如，带有 HTTP 触发器的 Lambda 函数）。 </section>
