LLM 输出是非确定性的，这使得响应质量难以评估。评估（evals）是一种分解"好"的定义并对其进行测量的方法。LangSmith 评估提供了一个框架，用于在整个应用生命周期中测量质量，从部署前测试到生产监控。

评估内容

在构建评估之前，先确定什么对你的应用最重要。将系统分解为关键组件——LLM 调用、检索步骤、工具调用、输出格式——并为每个组件定义质量标准。

**从手动策划的示例开始。** 为每个关键组件创建 5-10 个"好"的示例。这些示例作为你的基准真相，并指导使用哪些评估方法。例如：

  • - **RAG 系统**：好的检索（相关文档）和好的回答（准确、完整）的示例。
  • - **Agent**：正确的工具选择和正确的参数格式或 agent 执行的轨迹示例。
  • - **Chatbot**：有帮助、符合品牌调性的回复示例，这些回复能解决用户意图。

一旦通过示例定义了"好"，你就可以测量系统产生类似质量输出的频率。

离线评估和在线评估

LangSmith 支持两种类型的评估，在你的开发工作流中服务于不同的目的：

离线评估

使用离线评估进行**部署前测试**：

  • - **基准测试**：比较多个版本以找到最佳性能。
  • - **回归测试**：确保新版本不会降低质量。
  • - **单元测试**：验证单个组件的正确性。
  • - **回溯测试**：用历史数据测试新版本。

离线评估针对_示例_和_数据集_：带有参考输出的策划测试用例，定义了什么是"好"。

在线评估

使用在线评估进行**生产监控**：

  • - **实时监控**：持续跟踪实时流量的质量。
  • - **异常检测**：标记异常模式或边缘情况。
  • - **生产反馈**：识别要添加到离线数据集的问题。

在线评估针对_运行_和_线程_（来自追踪）：真实的生产追踪，没有参考输出。

这种目标的差异决定了你能够评估什么：离线评估可以对照预期答案检查正确性，而在线评估则专注于质量模式、安全性和真实世界行为。

评估生命周期

当你开发和部署应用时，你的评估策略会从部署前测试演变为生产监控。在开发和测试期间，离线评估根据策划的数据集验证功能。部署后，在线评估监控实时流量上的生产行为。随着应用成熟，两种评估类型在迭代反馈循环中协同工作，持续改进质量。

3. 持续改进

将两种评估类型结合在迭代反馈循环中使用。在线评估发现问题，这些问题成为离线测试用例；离线评估验证修复；在线评估确认生产改进。

核心评估目标

根据评估是离线的还是在线的，评估会在不同的目标上运行。

离线评估的目标

离线评估在数据集和示例上运行。参考输出的存在使得预期结果与实际结果之间能够进行比较。

数据集

数据集是用于评估应用程序的_示例集合_。示例是测试输入、参考输出对。

示例

每个示例包括：

  • - **输入**：要传递给应用程序的输入变量字典。
  • - **参考输出**（可选）：参考输出字典。这些不会传递给您的应用程序，仅用于评估器。
  • - **元数据**（可选）：额外信息字典，可用于创建数据集的筛选视图。

了解有关管理数据集的更多信息。

实验

_实验_表示在数据集上评估特定应用程序版本的结果。每个实验捕获数据集中每个示例的输出、评估器分数和执行跟踪。

!实验视图

通常会在给定数据集上运行多个实验，以测试不同的应用程序配置（例如不同的提示词或 LLM）。LangSmith 显示与数据集关联的所有实验，并支持并排比较多个实验。

了解如何分析实验结果。

在线评估的目标

在线评估在生产流量的运行和线程上运行。没有参考输出，评估器专注于实时检测问题、异常和质量下降。

运行

_运行_是来自您已部署应用程序的单个执行跟踪。每个运行包含： - **输入**：应用程序接收到的实际用户输入。 - **输出**：应用程序实际返回的内容。 - **中间步骤**：所有子运行（工具调用、LLM 调用等）。 - **元数据**：标签、用户反馈、延迟指标等。

与数据集中的示例不同，运行不包括参考输出。在线评估器必须不知道"正确"答案应该是什么来评估质量，而是依靠质量启发式、安全检查和无参考评估技术。

了解有关可观测性概念中的运行和跟踪的更多信息。

线程

_线程_是相关运行的集合，代表多轮对话。在线评估器可以在线程级别运行，以评估整个对话而不是单个回合。这使得能够评估对话级别的属性，例如跨轮的一致性、主题保持和整个交互过程中的用户满意度。

评估器

_评估器_是评分应用程序性能的工作区级资源。它们为离线和在线评估提供测量层，根据可用的数据调整其输入。由于评估器的作用域是工作区，您可以将单个评估器附加到多个跟踪项目和数据集，而无需每次重新创建它。

使用以下任何方式运行评估器：

  • - 评估器页面，用于将评估器附加到跟踪项目或数据集
  • - Playground
  • - LangSmith SDK（Python 和 TypeScript）
  • - 规则，用于自动在跟踪项目或数据集上运行评估器

评估器输入

评估器输入根据评估类型而有所不同：

**离线评估器**接收： - 示例：来自您数据集的示例，包含输入、参考输出和元数据。 - 运行：在示例输入上运行应用程序时的实际输出和中间步骤。

**在线评估器**接收： - 运行：包含输入、输出和中间步骤的生产跟踪（没有可用的参考输出）。

评估器输出

评估器返回**反馈**，即评估的分数。反馈是一个字典或字典列表。每个字典包含：

  • - key：指标名称。
  • - score | value：指标值（数值指标使用score，分类指标使用value）。
  • - comment（可选）：分数的附加推理或解释。

评估技术

LangSmith 支持多种评估方法：

  • - 人工
  • - 代码
  • - LLM 即评判者
  • - 配对

人工

_人工评估_涉及对应用程序输出和执行跟踪的手动审查。这种方法通常是评估的有效起点。LangSmith 提供工具来审查应用程序输出和跟踪（所有中间步骤）。

**标注队列**

标注队列简化了对运行的人工反馈的结构化收集。它们通过提供有组织的工作流程、预定义的评分标准、团队协作功能和进度跟踪来补充内联标注。

LangSmith 支持两种队列类型：

  • - **单运行队列**：一次审查一个运行，对抗自定义评分标准项。对于分类问题或从生产跟踪构建数据集很有用。单运行队列还支持断言，即离线评估器可以对未来运行进行评分的自由格式验收标准。
  • - **配对队列**：并排比较两个运行以判断哪个更好。专为实验之间的快速 A/B 比较而设计。

关键功能包括为每个运行配置多个审核者、启用预约以防止冲突，以及将标注的运行直接导出到数据集以供将来评估。

代码

_代码评估器_是确定性的、基于规则的函数。它们非常适合检查，例如验证聊天机器人的响应结构不为空、生成的代码可以编译，或者分类完全匹配。

LLM 即评判者

_LLM 即评判者评估器_使用 LLM 对应用程序输出进行评分。评分规则和标准通常编码在 LLM 提示词中。这些评估器可以是：

  • - **无参考**：检查输出是否包含冒犯性内容或是否符合特定标准。
  • - **基于参考**：将输出与参考进行比较（例如，根据参考检查事实准确性）。

LLM 即评判者评估器需要仔细审查分数和调整提示词。Few-shot 评估器在评分器提示词中包含输入、输出和预期评分的示例，通常可以提高性能。

了解如何定义 LLM 即评判者评估器。

配对

_配对评估器_使用启发式方法（例如哪个响应更长）、LLM（使用配对提示词）或人工审核者来比较两个应用程序版本的输出。

当直接对输出评分困难但比较两个输出很简单时，配对评估效果很好。例如，在摘要任务中，从两个摘要中选择信息更丰富的那个通常比给单个摘要分配绝对分数更容易。

了解如何运行配对评估。

无参考评估器与基于参考的评估器

理解评估器是否需要参考输出对于确定何时可以使用它至关重要。

**无参考评估器**在不与预期输出比较的情况下评估质量。这些适用于离线评估和在线评估： - **安全检查**：毒性检测、PII 检测、内容政策违规 - **格式验证**：JSON 结构、必需字段、模式合规性 - **质量启发式**：响应长度、延迟、特定关键词 - **无参考 LLM 即评判者**：清晰度、连贯性、有用性、语气

**基于参考的评估器**需要参考输出，仅适用于离线评估： - **正确性**：与参考答案的语义相似性 - **事实准确性**：根据事实依据进行事实核查 - **精确匹配**：具有已知标签的分类任务 - **基于参考的 LLM 即评判者**：将输出质量与参考进行比较

在设计评估策略时，无参考评估器在离线测试和在线监控中提供一致性，而基于参考的评估器在开发过程中实现更精确的正确性检查。

评估类型

LangSmith 支持用于不同开发和部署阶段的各种评估方法。了解何时使用每种类型有助于建立全面的评估策略。

离线评估和在线评估服务于不同的目的： - **离线评估类型**在具有参考输出的策划数据集上进行部署前测试 - **在线评估类型**在没有参考输出的实时流量上监控生产行为

了解有关评估类型以及何时使用每种类型的更多信息。

最佳实践

构建数据集

构建数据集有多种策略：

**手动策划的示例**

这是推荐的起点。创建 10-20 个高质量的示例，涵盖常见场景和边缘情况。这些示例定义了您的应用程序的"好"是什么样的。

**历史跟踪**

一旦投入生产，将真实跟踪转换为示例。对于高流量应用程序： - **用户反馈**：添加收到负面反馈的运行以进行测试。 - **启发式**：识别有趣的运行（例如，长延迟、错误）。 - **LLM 反馈**：使用 LLM 检测值得注意的对话。

**合成数据**

从现有示例生成其他示例。当从几个高质量的手工示例开始作为模板时效果最好。

数据集组织

**划分**

划分是数据集的命名子集，用于将示例分段到单独的组中。常见模式包括：

  • - **机器学习风格划分**：将示例分为训练集、验证集和测试集，以避免过拟合，即模型在训练数据上表现良好但在未见过的数据上表现不佳。
  • - **基于类别的划分**：当数据集跨越多个任务类别时，分别评估不同的输入类型。
  • - **分阶段推出**：将探索性示例保持隔离，直到您准备好将它们包含在主评估集中。

划分与元数据不同：使用划分进行评估的高级组织分组，元数据用于每个示例的信息（如标签和来源）。

在机器学习中，最佳实践是每个示例恰好属于一个划分。LangSmith 允许示例属于多个划分，这在示例适合多个评估类别时很有用。

了解如何创建和管理数据集划分。

**版本**

当示例更改时，LangSmith 会自动创建数据集版本。标记版本以标记重要里程碑。在 CI 管道中定位特定版本，以确保数据集更新不会破坏工作流程。

人工反馈收集

人工反馈通常提供最有价值的评估，特别是对于主观质量维度。

**标注队列**

标注队列支持人工反馈的结构化收集。标记特定运行以供审查，在流线型界面中收集标注，并将标注的运行传输到数据集以供将来评估。

标注队列通过提供附加功能来补充内联标注：分组运行、指定标准和配置审核者权限。

评估与测试

测试和评估是相似但不同的概念。

**评估根据指标衡量性能。**指标可以是模糊的或主观的，在相对条件下更有用。它们通常比较系统之间的差异。

**测试断言正确性。**系统只有在通过所有测试后才能部署。

评估指标可以转换为测试。例如，回归测试可以断言新版本必须在相关指标上优于基线版本。当系统运行成本高昂时，一起运行测试和评估以提高效率。

评估可以使用标准测试工具编写，如 pytest 或 Vitest/Jest。
