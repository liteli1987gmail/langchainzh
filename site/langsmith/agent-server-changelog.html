Agent Server 是一个用于创建和管理基于代理的应用程序的 API 平台。它提供内置持久化、任务队列，并支持大规模部署、配置和运行助手（代理工作流）。此更新日志记录了 Agent Server 发布的所有重要更新、功能和修复。

### 概述 - v0.10.0rc1 包含安全性和正确性的破坏性变更。请参阅 安全部分 了解更多详情。 - 包含依赖项和安全维护更新。

### 安全 - **可能破坏兼容性** 现在默认拒绝 loopback webhook 目标以修复身份验证绕过原语（GHSA-2c9q-c2q9-qgqv）。 webhooks.url.disable_loopback 策略现在默认为 true, blocking relative-URL webhooks (which dispatch through the in-process ASGI transport and bypass auth), as well as localhost / 127.x / ::1 / host.docker.internal absolute URLs and any hostname that DNS-resolves into the loopback range (mitigating DNS rebinding). Deployments that legitimately need loopback webhooks (e.g. langgraph dev 使用本地主机 Webhook 接收器的设置，或分派到挂载在同一服务器上的自定义 FastAPI 路由的生产设置，可以通过设置 webhooks.url.disable_loopback: false in langgraph.json （或等效的 LANGGRAPH_WEBHOOKS JSON 环境变量）。仅在您控制循环 Webhook 到达的路由时执行此操作，因为这些路由在分派时不进行身份验证。 - **可能破坏性** POST /runs 和 POST /threads/{thread_id}/runs 现在通过assistants.read 认证事件（匹配 cron 创建和直接 GET）进行授权，而不是之前使用的 assistants.search 带有不完整负载的事件（GHSA-jfj5-wrj9-63x4）。仅注册了 @auth.on.assistants.read （且无 .search handler) were vulnerable to a cross-user authorization bypass; their existing read handler will now be invoked on the run-creation path. As a defense-in-depth follow-up, client-supplied run/cron metadata is no longer forwarded into the assistants.read 来自 Runs.put or Crons.put, and inmem/postgres runtimes now agree on the value shape. Breaking change for deployments with custom auth handlers: (1) any @auth.on.assistants.search 处理程序的认证事件负载在运行创建期间不再被调用——确保您有等效的@auth.on.assistants.read 处理程序返回相同的所有者风格过滤器；(2) value["metadata"] 在 assistants.read event invoked from run/cron creation is no longer populated, so handlers that inspected or mutated it must move that logic into @auth.on.runs.create_run / @auth.on.crons.create 并依赖返回过滤器进行所有权强制执行。 - 部署现在会在服务器启动时看到结构化警告，列出每个未覆盖的分派路径以及要复制的默认拒绝代码片段。对于注册了全局 @auth.on 处理程序或仅使用 @auth.authenticate 而不使用任何资源级处理程序的部署，此警告为静默。

v0.8.0

此小版本将运行队列轮询从 Postgres 移至 Redis，节省了数据库负载并提升了性能。

在底层，Agent Server 使用持久化运行队列来管理运行执行。Worker 轮询队列获取新运行并执行它们。之前，队列轮询逻辑通过 Postgres。这可能导致长时间运行的查询，尤其是在高负载下。通过此更新，队列轮询逻辑现在通过 Redis，然后从 Postgres 获取运行详情。这使得队列轮询的热路径大大加快，并减少了数据库的负载。

这不是破坏性更改，升级时无需代码更改，但有几件事需要注意： - 在升级后的部署中，队列将切换。可能会有一段时间线程被非时间顺序调度。每个线程内的运行执行顺序仍有保证。 - **仅限自托管：** Redis 流量可能略有增加。在内部测试中，增加幅度很小。

此小版本更新了流式API /join-stream 和 /stream 关于 last-event-id 参数的行为以符合SSE规范。以前，传递last-event-id会返回该消息以及后续的所有消息。今后，这些API将仅返回所提供的last-event-id之后的新消息。例如，对于以下流，之前传递last-event-id为 2 会返回id为 2 和 3的消息，但现在只会返回id为 3:

    "id": 1,
    "event": "message",
    "data": {
        "content": "Excluded"
    }
},
{
    "id": 2,
    "event": "message",
    "data": {
        "content": "Passed last-event-id"
    }
},
{
    "id": 3,
    "event": "message",
    "data": {
        "content": "Included"
    }
}
