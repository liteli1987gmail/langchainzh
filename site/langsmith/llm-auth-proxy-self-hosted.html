LLM 身份验证代理允许您的组织对 LangSmith 的所有模型调用强制执行自己的身份验证流程，从而确保提供商凭据永远不会暴露给最终用户，并且每个请求都可以追溯到特定的操作者。

LLM 身份验证代理是一个基于 Envoy 的组件，运行在您的环境中，位于 LangSmith 与上游 LLM 提供商或网关（如 OpenAI、Anthropic 或内部 LLM 网关如 LiteLLM）之间。LangSmith 使用短期 JWT（JSON Web Token）对每个请求进行签名。代理验证 JWT、可选地注入提供商凭据或转换请求和响应正文，然后将请求转发到上游。它适用于 SaaS 和自托管 LangSmith 客户。

信息

LLM 身份验证代理需要 LangSmith Enterprise 计划。更多详情请参阅定价或联系我们的销售团队。

在以下情况下使用 LLM 身份验证代理：

  • - 针对您自己的提供商网关对 Playground 或 LLM 即评判评估 请求进行身份验证。
  • - 注入提供商特定的 API 密钥或身份验证标头，而不将其暴露给最终用户。
  • - 转换请求或响应正文（例如，在 OpenAI 格式和自定义网关格式之间转换）。

对于 OAuth2 client_credentials 特别说明，模型配置上的 OAuth 客户端凭据 是一种按配置的自助服务替代方案，工作区管理员可以在不部署身份验证代理的情况下进行设置。在配置级别上路由是互斥的——启用 OAuth 的配置不会通过身份验证代理。

工作原理

来自 LangSmith 的每个请求都会经过代理中的以下步骤：

1. 验证 JWT（签名、颁发者、受众） 2. 调用您的 ext_authz 服务，该服务接收已验证的 JWT 并返回要作为标头注入的提供商凭据 3. 可选地调用您的 ext_proc 转换器，它可以重写请求和响应正文（例如，在 OpenAI 格式和自定义网关格式之间转换） 4. 使用自定义标头（静态或动态）将请求转发到上游提供商

ext_authz 服务和转换器都是客户部署的组件，与代理一起运行在您的环境中。根据您的用例，可以启用其中一个或两个（何时使用 ext_proc 与 ext_authz）。

前置要求

  • - LangSmith Enterprise 计划（SaaS 或 0.13.33+ 版本的自托管）
  • - 配备 Helm 3 的 Kubernetes 集群
  • - Envoy v1.37 或更高版本（Helm chart 默认使用 envoyproxy/envoy:v1.37-latest）
  • - 您的上游 LLM 提供商或网关的 URL（代理将转发请求的目标）
提示

身份验证代理当前支持 Playground、Evals、Fleet、Chat 和 Insights 功能。 Playground 和 Evals 在 v0.13.33+ 中可用。Chat 和 Insights 在 v0.13.39+ 中可用。

1. 配置 JWT 签名（仅限自托管 LangSmith）

对于 LangSmith SaaS，跳过此步骤。JWT 签名已配置。

**生成 Ed25519 密钥对**，使用 step CLI（或者如果您愿意，也可以使用内部流程）。Ed25519 是 LangSmith 用于签名 JWT 的签名算法。私钥对每个请求进行签名；身份验证代理仅使用公钥验证签名。

TMPDIR_KEYS="$(mktemp -d)"
step crypto keypair "$TMPDIR_KEYS/pub.pem" "$TMPDIR_KEYS/priv.pem" \
  --kty OKP --crv Ed25519 --no-password --insecure
PRIV_JWK=$(step crypto key format --jwk --no-password --insecure < "$TMPDIR_KEYS/priv.pem")
SIGNING_JWKS=$(echo "$PRIV_JWK" | jq -c '{keys: [. + {use: "sig", alg: "EdDSA"}]}')
echo "$SIGNING_JWKS"

**将 JWKS 存储在 Kubernetes Secret 中：**

kubectl create secret generic langsmith-signing-jwks \
  --namespace <namespace> \
  --from-literal=LANGSMITH_SIGNING_JWKS="$SIGNING_JWKS"

JWKS（JSON Web Key Set）是一种用于发布加密密钥的标准 JSON 格式。LANGSMITH_SIGNING_JWKS 包含 Ed25519 私钥，并存储为 Kubernetes secret。它永远不会暴露。LangSmith 自动提取相应的公钥，并在 /.well-known/jwks.json 提供服务。Auth 代理获取此公共端点来验证 JWT 签名，无需使用私钥。

**在您的 LangSmith values.yaml 中引用该 secret：**

platformBackend:
  deployment:
    extraEnv:
      - name: LLM_AUTH_PROXY_ISSUER
        value: "langsmith"        # must match jwtIssuer in the auth proxy chart
      - secretRef:
          name: langsmith-signing-jwks

LLM_AUTH_PROXY_ISSUER 用于设置签名 JWT 中的 iss 声明。使用 langsmith 来匹配 SaaS 默认值，或使用自定义标识符（如 langsmith:self-hosted:<short_identifier>）来区分您的安装。该值必须与步骤 4)中 auth proxy chart 中的 jwtIssuer 匹配。

2. 为您的组织启用 LLM Auth Proxy

Self-hosted

**选项 A：** 为特定组织启用：

在 LangSmith UI 中，导航到 **Settings** 页面，复制左上角 **Organizations** 旁边的组织 ID。

针对您的 LangSmith PostgreSQL 数据库运行以下命令：

UPDATE organizations
SET config = config || '{"can_use_llm_auth_proxy": true}'
WHERE id = '<organization_id>';

**选项 B：**在安装中为所有组织启用：

将以下内容添加到你的 LangSmith values.yaml 的 commonEnv 中：

commonEnv:
  DEFAULT_ORG_FEATURE_CAN_USE_LLM_AUTH_PROXY: "true"
提示

此设置对个人组织无效。

SaaS

请联系 支持门户 的技术支持，为您的组织启用 LLM Auth Proxy。

3. 在 LangSmith 中配置组织设置

在 LangSmith UI 中，导航至 **Settings** > **General**，配置以下内容：

1. **JWT audience:** 代理将验证的 aud 声明值（例如，example-audience）。此值必须与 第 4 步 的 auth proxy chart 中的 jwtAudiences 匹配。 2. **Enable LLM auth proxy:** 为您的组织开启切换开关。 3. **Allowed URLs:** 控制代理允许将 JWT 转发到哪些目标 URL。这可以防止凭据被转发到非预期的主机。选择以下三个选项之一： - **Allow all**（默认）：允许将 JWT 转发到任何上游 URL。等同于无限制。 - **Block all：** 阻止向所有 URL 转发 JWT。 - **Custom：** 指定允许的 URL 模式的明确列表。不接受空字符串和单独的 *。当 LLM auth proxy 切换开关关闭时，此控件将被禁用。

4. 安装 auth proxy Helm chart

添加 LangChain Helm 仓库：

helm repo add langchain https://langchain-ai.github.io/helm/
helm repo update

创建包含上游 URL 和 JWT 验证设置的 values.yaml。JWKS 配置有两个选项：

  • - **jwksUri（推荐）：** 指向 LangSmith 实例的 /.well-known/jwks.json 端点。Envoy 会自动获取并缓存公钥，支持无缝密钥轮换。
  • - **jwksJson（内联）：** 将 JWKS JSON 直接粘贴到 values.yaml 中。适用于测试环境或 auth proxy 无法访问 LangSmith 的隔离环境。需要更新 chart 才能轮换密钥。只包含公钥组件，省略 d 字段（私钥）。

如果同时配置了这两个选项，jwksUri 优先。

authProxy:
  upstream: "https://gateway.example.com"
  jwtIssuer: "langsmith" # must match LLM_AUTH_PROXY_ISSUER in LangSmith values.yaml
  jwtAudiences:
    - "example-audience" # must match the org setting in LangSmith

  # Option A: remote JWKS (recommended for production)
  # Envoy fetches and caches public keys from LangSmith's /.well-known/jwks.json.
  jwksUri: "https://langsmith.example.com/.well-known/jwks.json"       # self-hosted
  # jwksUri: "https://api.smith.langchain.com/.well-known/jwks.json"   # SaaS
  jwksCacheDurationSeconds: 300

  # Option B: inline JWKS (testing or air-gapped environments only)
  # Omit the "d" field (private key); include public key components only.
  # jwksJson: '{"keys": [{"kty": "OKP", "crv": "Ed25519", "x": "<base64url-public-key>", "use": "sig", "alg": "EdDSA"}]}'

安装图表：

helm install langsmith-auth-proxy langchain/langsmith-auth-proxy \
  --namespace <your-namespace> \
  -f values.yaml

编写 ext_authz 服务

当您需要添加、删除或编辑授权头时，请使用 ext_authz。例如，根据 JWT 中的身份注入提供商的 API 密钥。您的服务接收已验证的 JWT，可选择接收请求体，并返回要向上游注入的头部。这使用 Envoy 的 HTTP ext_authz 过滤器（不是 gRPC）。

在 values.yaml 中启用它：

authProxy:
  extAuthz:
    enabled: true
    serviceUrl: "http://my-auth-service:8080"
    timeout: "10s"

工作原理

在转发每个请求之前，Envoy 会使用与原始请求相同的 HTTP 方法调用您的服务，地址为 <serviceUrl>/check<original_path>。您的服务会在 x-langsmith-llm-auth header 中接收经过验证的 JWT。

您的服务返回纯 HTTP 响应：

  • - **2xx：** 允许该请求。所有匹配 allowedUpstreamHeaders 模式（默认：authorization 和 x-*）的 headers 都会被注入到上游请求中。如需在转发前剥离 JWT，请在响应中包含 x-envoy-auth-headers-to-remove: x-langsmith-llm-auth。
  • - **非 2xx：** 拒绝该请求。状态码和所有匹配 allowedClientHeaders 模式（默认：www-authenticate 和 x-*）的 headers 会被返回给客户端。

部署选项

您的 ext_authz 服务可以以下述两种方式运行：

  • - **Sidecar：** 在代理所在的同一个 pod 中运行服务。将容器添加到 authProxy.deployment.sidecars 下，并将所需卷添加到 values.yaml 中的 authProxy.deployment.volumes 下。使用 localhost URL，例如 http://localhost:10002。
  • - **独立部署：** 独立部署服务，并将 extAuthz.serviceUrl 指向它。使用集群内 DNS 名称，例如 http://my-auth-service.my-namespace.svc.cluster.local:8080，如果服务有自己的入口，也可以使用外部 HTTPS URL。

示例部署

下面的示例是一个最小的 Python ext_authz 服务，执行 OAuth2 客户端凭证令牌交换。在每个请求时，它返回一个带有新访问令牌的缓存 Authorization header，并在其过期前从配置的令牌端点刷新它。有关完整示例，请参阅 chart 仓库中的 e2e/oauth/。

ext-authz-oauth.py

"""ext_authz service that performs an OAuth2 client-credentials token exchange.

Runs as a sidecar (or standalone service) alongside the main auth-proxy component.
On each ext_authz check request it returns a cached OAuth access token,
refreshing it from the configured token endpoint when expired.

Environment variables:
  OAUTH_TOKEN_URL    – Token endpoint (e.g. https://login.example.com/oauth/token)
  OAUTH_CLIENT_ID    – Client ID for the credentials grant
  OAUTH_CLIENT_SECRET– Client secret for the credentials grant
  OAUTH_SCOPE        – (optional) Space-separated scopes to request
  LISTEN_PORT        – (optional) Port to listen on, default 10002
"""

from http.server import HTTPServer, BaseHTTPRequestHandler








# ---------------------------------------------------------------------------
# Configuration
# ---------------------------------------------------------------------------
TOKEN_URL = os.environ["OAUTH_TOKEN_URL"]
CLIENT_ID = os.environ["OAUTH_CLIENT_ID"]
CLIENT_SECRET = os.environ["OAUTH_CLIENT_SECRET"]
SCOPE = os.environ.get("OAUTH_SCOPE", "")
LISTEN_PORT = int(os.environ.get("LISTEN_PORT", "10002"))

# Refresh the token this many seconds before it actually expires.
EXPIRY_BUFFER_SECONDS = 30

# ---------------------------------------------------------------------------
# Token cache (thread-safe)
# ---------------------------------------------------------------------------
_lock = threading.Lock()
_cached_token: str | None = None
_token_expiry: float = 0  # epoch seconds


def _fetch_token() -> tuple[str, float]:
    """Perform a client_credentials grant and return (access_token, expiry_epoch)."""
    data = urllib.parse.urlencode({
        "grant_type": "client_credentials",
        "client_id": CLIENT_ID,
        "client_secret": CLIENT_SECRET,
        **({"scope": SCOPE} if SCOPE else {}),
    }).encode()

    req = urllib.request.Request(
        TOKEN_URL,
        data=data,
        headers={"Content-Type": "application/x-www-form-urlencoded"},
        method="POST",
    )
    with urllib.request.urlopen(req, timeout=10) as resp:
        body = json.loads(resp.read())

    access_token = body["access_token"]
    expires_in = int(body.get("expires_in", 3600))
    expiry = time.time() + expires_in - EXPIRY_BUFFER_SECONDS
    return access_token, expiry


def get_token() -> str:
    """Return a valid access token, refreshing if necessary."""
    global _cached_token, _token_expiry
    with _lock:
        if _cached_token and time.time() < _token_expiry:
            return _cached_token
    # Fetch outside the lock so other requests aren't blocked on I/O.
    token, expiry = _fetch_token()
    with _lock:
        _cached_token = token
        _token_expiry = expiry
    print(f"Refreshed OAuth token (expires in {int(expiry - time.time())}s)", flush=True)
    return token


# ---------------------------------------------------------------------------
# ext_authz HTTP handler
# ---------------------------------------------------------------------------
class Handler(BaseHTTPRequestHandler):
    def do_any(self):
        try:
            token = get_token()
        except Exception as exc:
            print(f"OAuth token fetch failed: {exc}", flush=True)
            self.send_response(500)
            self.send_header("Content-Type", "text/plain")
            self.end_headers()
            self.wfile.write(b"OAuth token exchange failed")
            return

        self.send_response(200)
        # Replace the header name as needed - this header will be forwarded to the upstream LLM provider / gateway.
        self.send_header("Authorization", f"Bearer {token}")
        self.end_headers()

    # Handle every method Envoy might send for ext_authz checks.
    do_GET = do_POST = do_PUT = do_DELETE = do_PATCH = do_HEAD = do_OPTIONS = do_any

    def log_message(self, format, *args):
        # Quieter logs — only print errors.
        pass


if __name__ == "__main__":
    server = HTTPServer(("0.0.0.0", LISTEN_PORT), Handler)
    print(f"ext-authz-oauth listening on :{LISTEN_PORT}", flush=True)
    print(f"  token_url={TOKEN_URL} client_id=<redacted>", flush=True)
    server.serve_forever()

有关 extAuthz 参数的完整列表，请参阅 Helm chart README。

编写 ext_proc 转换器

当您需要重写请求或响应正文时，例如在 OpenAI 格式和自定义网关格式之间进行转换，或向请求负载中注入额外字段，请使用 ext_proc。这使用 Envoy 的 ext_proc 过滤器。

与 ext_authz（HTTP）不同，ext_proc 使用双向 gRPC 流。Envoy 在每个处理阶段（请求头、请求正文、响应头、响应正文）向您的转换器服务发送一条消息，您的服务在每个阶段回复相应的修改指令。您的转换器必须实现 envoy.service.ext_proc.v3.ExternalProcessor gRPC 服务。图表仓库中的 e2e/transformer/ 提供了 Go 示例实现。

何时使用 ext_proc 与 ext_authz

| 功能 | ext_authz | ext_proc | |-----------|-----------|----------| | 修改请求头 | 是 | 是 | | 修改响应头 | 否 | 是 | | 修改请求正文 | 否 | 是 | | 修改响应正文 | 否 | 是 | | 协议 | HTTP | gRPC |

如果您只需要注入认证头（例如 API 密钥），请使用 ext_authz。如果您需要重写正文，请使用 ext_proc。两者可以同时启用。

在 values.yaml 中启用 ext_proc：

authProxy:
  transformer:
    enabled: true
    serviceUrl: "grpc://my-transformer:50051"
    timeout: "10s"
    failureModeAllow: false
    processingMode:
      requestHeaderMode: "SEND"
      requestBodyMode: "BUFFERED"
      responseHeaderMode: "SKIP"
      responseBodyMode: "NONE"

设置 failureModeAllow: true 允许在转换器不可用时让请求通过。默认值（false）会拒绝请求。

处理模式

通过 processingMode 控制哪些阶段会发送到您的转换器。只启用您需要的阶段，因为禁用未使用的阶段可以减少延迟。

| 字段 | 选项 | 描述 | |-------|---------|-------------| | requestHeaderMode | SEND, SKIP, DEFAULT | 是否转发请求头。 | | responseHeaderMode | SEND, SKIP, DEFAULT | 是否转发响应头。 | | requestBodyMode | NONE, BUFFERED, STREAMED, BUFFERED_PARTIAL | 如何发送请求体。 | | responseBodyMode | NONE, BUFFERED, STREAMED, BUFFERED_PARTIAL | 如何发送响应体。 | | requestTrailerMode | SEND, SKIP | 是否转发请求拖车。 | | responseTrailerMode | SEND, SKIP | 是否转发响应拖车。 |

  • - 使用 BUFFERED 进行请求体重写：在发送之前缓冲完整主体，最适合 JSON 重写。
  • - 使用 STREAMED 进行流式 LLM 响应体重写：在数据块到达时发送，延迟更低但实现更复杂。
  • - 使用 NONE 完全跳过某个阶段。
警告

在修改主体时，您的 ext_proc 服务还必须通过 HeaderMutation 更新 content-length 头以匹配新的主体大小。Envoy 会拒绝 content-length 与修改后的主体不匹配的响应。

请求流程

在启用 ext_proc 进行头部注入和主体重写的示例：

curl -H "X-LangSmith-LLM-Auth: " -d '{"model":"gpt-4",...}'
  -> Envoy(:10000)
  -> built-in Envoy JWT filter (validate sig, iss, aud)
  -> `ext_proc` filter -> transformer:50051 (gRPC)
    <- phase 1: request_headers -> mutate headers (inject Authorization)
    <- phase 2: request_body   -> mutate body (rewrite JSON) + update content-length
  -> upstream LLM provider or gateway

示例部署

以下示例将一个最小的 Go 转换器部署为 Kubernetes Deployment。它从请求头中读取 JWT，注入 Authorization 头，并将请求体从 OpenAI 格式重写为自定义格式。

transformer-configmap.yaml

apiVersion: v1
kind: ConfigMap
metadata:
  name: transformer-source
data:
  main.go: |
    package main

        "encoding/json"
        "fmt"
        "io"
        "log"
        "net"
        "strings"

        core "github.com/envoyproxy/go-control-plane/envoy/config/core/v3"
        ext_proc "github.com/envoyproxy/go-control-plane/envoy/service/ext_proc/v3"
        "google.golang.org/grpc"
    )

    type server struct {
        ext_proc.UnimplementedExternalProcessorServer
    }

    func (s *server) Process(stream ext_proc.ExternalProcessor_ProcessServer) error {
        for {
            req, err := stream.Recv()
            if err == io.EOF {
                return nil
            }
            if err != nil {
                return err
            }

            var resp *ext_proc.ProcessingResponse
            switch v := req.Request.(type) {
            case *ext_proc.ProcessingRequest_RequestHeaders:
                resp = handleRequestHeaders(v.RequestHeaders)
            case *ext_proc.ProcessingRequest_RequestBody:
                resp = handleRequestBody(v.RequestBody)
            default:
                resp = &ext_proc.ProcessingResponse{}
            }

            if err := stream.Send(resp); err != nil {
                return err
            }
        }
    }

    func handleRequestHeaders(headers *ext_proc.HttpHeaders) *ext_proc.ProcessingResponse {
        var jwtValue string
        for _, h := range headers.Headers.Headers {
            if strings.EqualFold(h.Key, "x-langsmith-llm-auth") {
                if len(h.RawValue) > 0 {
                    jwtValue = string(h.RawValue)
                } else {
                    jwtValue = h.Value
                }
                break
            }
        }

        resp := &ext_proc.ProcessingResponse{
            Response: &ext_proc.ProcessingResponse_RequestHeaders{
                RequestHeaders: &ext_proc.HeadersResponse{},
            },
        }

        if jwtValue != "" {
            // TODO: Replace with your auth logic, e.g. exchange JWT for a
            // provider-specific token, call a secrets manager, etc.
            providerKey := "Bearer your-provider-key"

            headerResp := resp.GetRequestHeaders()
            headerResp.Response = &ext_proc.CommonResponse{
                HeaderMutation: &ext_proc.HeaderMutation{
                    SetHeaders: []*core.HeaderValueOption{
                        {
                            Header: &core.HeaderValue{
                                Key:      "Authorization",
                                RawValue: []byte(providerKey),
                            },
                        },
                    },
                },
            }
        }
        return resp
    }

        var original map[string]interface{}
        if err := json.Unmarshal(body.Body, &original); err != nil {

        // IMPORTANT: update content-length to match the new body size.
        bodyResp := resp.GetRequestBody()
        bodyResp.Response = &ext_proc.CommonResponse{

其他配置

HTTP 代理

Envoy 不遵循 HTTP_PROXY、HTTPS_PROXY 或 NO_PROXY 环境变量。请显式配置 HTTP 代理：

authProxy:
  httpProxy:
    enabled: true
    host: "proxy.example.com"
    port: 3128
    noProxy:
      - "internal.corp"
      - ".internal.corp"

在没有公共入口的情况下部署

当认证代理没有公共入口且仅可通过内部 Kubernetes 网络访问时，必须配置 LangSmith 服务以允许向私有 IP 地址发出出站请求。如果没有这些设置，内置的 SSRF 保护会阻止对私有 IP 的请求。

将以下环境变量添加到您的 LangSmith values.yaml 中：

  • - **SSRF_ALLOW_K8S_INTERNAL** — 所有发出 LLM 调用的服务都需要此变量。对于支持 commonEnv 的服务，请将其添加到 commonEnv；对于不支持 commonEnv 的服务，请将其添加到各服务的 extraEnv。
  • - **SSRF_ALLOW_PRIVATE_IPS_PLAYGROUND** — 仅 playground 服务需要此变量。将其添加到 playground.deployment.extraEnv 中。
# Allow all LLM-calling services to reach the auth proxy on private IPs
commonEnv:
  SSRF_ALLOW_K8S_INTERNAL: "true"

# Allow the playground service to reach the auth proxy on private IPs
playground:
  deployment:
    extraEnv:
      - name: SSRF_ALLOW_K8S_INTERNAL
        value: "true"
      - name: SSRF_ALLOW_PRIVATE_IPS_PLAYGROUND
        value: "true"

如果 commonEnv 不适用于您部署中的所有必需服务，请通过每个进行 LLM 调用服务的 extraEnv 单独设置 SSRF_ALLOW_K8S_INTERNAL。

authProxy:
  upstream: "https://gateway.example.com"   # your LLM gateway or provider
  jwtIssuer: "langsmith"                    # must match LLM_AUTH_PROXY_ISSUER on LangSmith
  jwtAudiences:
    - "example-audience"                    # must match org setting in LangSmith

  # Option A: remote JWKS (recommended for production)
  # Envoy fetches and caches public keys from LangSmith's /.well-known/jwks.json endpoint.
  jwksUri: "https://langsmith.example.com/.well-known/jwks.json"   # self-hosted
  # jwksUri: "https://api.smith.langchain.com/.well-known/jwks.json"  # SaaS
  jwksCacheDurationSeconds: 300             # how long Envoy caches the JWKS (default 5 min)

  # Option B: inline JWKS (testing or air-gapped environments only)
  # jwksJson: '{"keys": [...]}'

  # ext_authz: header-only auth logic (include only if needed)
  # Use this to inject, remove, or modify authorization headers.
  # Your service receives an HTTP request at /check with the validated JWT
  # in the x-langsmith-llm-auth header and responds with headers to inject upstream.
  extAuthz:
    enabled: true
    serviceUrl: "http://localhost:10002"    # sidecar URL
    # serviceUrl: "http://ext-authz.<namespace>.svc.cluster.local:10002"  # separate deployment
    sendBody: false                         # set true to include request body

  # transformer: request/response body transformation (include only if needed)
  # Use this when you need to rewrite request or response bodies (e.g. OpenAI -> custom format).
  # Can be enabled alongside ext_authz.
  transformer:
    enabled: true
    serviceUrl: "grpc://transformer.<namespace>.svc.cluster.local:50051"
    timeout: "10s"
    failureModeAllow: false                 # reject if transformer is unavailable
    processingMode:
      requestHeaderMode: "SEND"             # forward request headers (read JWT, inject auth)
      responseHeaderMode: "SKIP"            # skip response headers
      requestBodyMode: "BUFFERED"           # buffer full body for JSON rewriting
      responseBodyMode: "NONE"              # skip response body
      requestTrailerMode: "SKIP"
      responseTrailerMode: "SKIP"

JWT 声明参考

LangSmith 使用 **Ed25519 (EdDSA)** 对 JWT 进行签名。公钥在 /.well-known/jwks.json 提供，由代理自动获取。认证代理使用这些公钥验证签名。

| 声明 | 描述 | |-------|-------------| | iat, exp, jti, nbf | 标准 JWT 声明（签发时间、过期时间、JWT ID、生效时间） | | iss | 签发者。SaaS 版本为 langsmith；自托管版本通过 LLM_AUTH_PROXY_ISSUER 设置 | | aud | 受众。与 LangSmith 组织设置中的 JWT 受众匹配 | | sub | 主体标识符（用户 ID、评估器 ID、助手 ID 或 API 密钥 ID） | | actor_type | 类型之一：user、evaluator、agent-builder、api_key | | workspace_id | 工作区 ID | | workspace_name | 工作区名称 | | organization_id | 组织 ID | | organization_name | 组织名称 | | request_id | 请求关联 ID | | ls_user_id | LangSmith 用户 ID（仅当 actor_type 为 user 时存在） |

JWT 通过 x-langsmith-llm-auth 请求头传递到您的 ext_authz 或 transformer 服务。

常见问题

Does the auth proxy support corporate proxies?

是的。通过 values.yaml 中的 httpProxy 部分配置 HTTP 代理。详见 HTTP 代理。 </section>

Does the auth proxy support custom certificates?

是的，通过 customCa 支持自定义 CA 证书，通过 mtls 支持双向 TLS。 </section>

Can a single auth proxy route to multiple upstream LLM gateways?

不能。认证代理只有一个 upstream 字段。 </section>

Can the auth proxy serve multiple organizations?

是的。多个组织可以通过 LangSmith 中的模型配置指向同一个认证代理实例。 </section>

Can the LangSmith to auth proxy connection use HTTP instead of HTTPS?

可以，但仅在自托管环境中适用，我们通常建议将认证代理置于专用入口之后，以便通信使用 HTTPS。若要允许 HTTP，请在 LangSmith values.yaml 的 commonEnv 和 playground.deployment.extraEnv 中添加 LLM_AUTH_PROXY_ACCEPT_HTTP。 若要为 Chat and Insights 启用到认证代理的 HTTP 流量，请在相应的 extraEnv 部分设置此环境变量：config.polly.agent.extraEnv（用于 Chat，原名 Polly）和 config.insights.agent.extraEnv。 </section>

Does the auth proxy work without a public ingress?

是的。当认证代理只能通过内部 Kubernetes 网络访问（无公共入口）时，将 SSRF_ALLOW_K8S_INTERNAL 添加到所有发起 LLM 调用的服务中，并将 SSRF_ALLOW_K8S_INTERNAL 和 SSRF_ALLOW_PRIVATE_IPS_PLAYGROUND 添加到 playground 服务。详见 Deploy without a public ingress。 </section>

When should I use the LLM auth proxy versus OAuth client credentials on a model configuration?

当认证需要超出 OAuth2 client_credentials 的自定义逻辑时使用 LLM 认证代理。例如，交换 LangSmith JWT 以获取提供商特定令牌、注入 GCP 或 AWS 身份，或重写请求和响应体。当每个工作区或团队需要对自定义网关的 OAuth2 client_credentials 进行自助服务控制时，使用模型配置上的 OAuth 客户端凭据。两者可以在同一组织内共存；路由按配置进行。 </section>
